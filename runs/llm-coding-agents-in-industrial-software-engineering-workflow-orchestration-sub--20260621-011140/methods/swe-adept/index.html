SWE-Adept: LLM-based Two-Agent Framework for Repository-Level Software Engineering

Repository-level debugging is hard for a reason every developer knows: the bug is rarely in the first place you look, and the fix is rarely a single clean edit. SWE-Adept tackles that by splitting the work into two specialized agents. One agent is responsible for finding the right code locations. The other is responsible for turning those locations into a controlled repair workflow.

That split is the whole paper in miniature. Instead of letting one free-form agent wander through a repository and edit whenever it feels ready, SWE-Adept tries to manage context and state more deliberately. Localization gets a compact view of the codebase. Resolution gets explicit hypotheses, to-do lists, and Git checkpoints.

First, build a map of the repository

SWE-Adept starts by turning the repository into a set of code units. Functions and classes become the main units, and whatever code does not fit those boundaries is covered by fixed-size chunks. Each unit keeps its name, file location, and code text, so the system can refer to a specific piece of source without loading an entire file.

Then comes the more interesting part: the code-structure tree. Each code unit knows which child units it contains or invokes. Instead of building a heavy global graph and looking up adjacency later, SWE-Adept stores a lightweight adjacency list with each unit. That means a search result can return both the local code preview and the next places the agent can go. The repository becomes something the agent can walk, not just search.

That is the right mental model. The paper is not trying to give the model the whole codebase. It is trying to give it just enough structure to choose a promising path.

How the localization agent searches without drowning in context

What to notice in Figure 1 is the two-stage flow. The agent does not jump straight to full source code. It starts with a structured code tree, uses lightweight search results to narrow the field, and only then opens the full code for shortlisted candidates.

The localization agent begins with the issue description. If the issue mentions a file, class, or function name explicitly, that becomes an entry point. If it does not, the agent can still search by partial strings or other content matches. The available tools cover several granularities: file search, definition search, content search, and a direct jump to a child unit.

The key choice is that these tools return compact previews instead of full files. A definition search gives the signature, location metadata, child-unit identifiers, and a short invocation context. A file search gives the file skeleton. A content search points to the containing unit and the relevant lines. That is enough to judge whether a branch is worth following, but not enough to get lost in irrelevant source.

SWE-Adept then uses depth-first traversal. The agent follows one dependency path at a time, choosing the most promising child unit from the preview and recursing downward until it either has enough context or decides the path is not relevant. This is a very different shape from indiscriminate breadth-first expansion. Depth-first search keeps the working context focused on one likely root-cause chain instead of spreading attention across many branches.

After the traversal, the system runs a second filter. First it shortlists candidate locations using the lightweight preview and location metadata. Only then does the runtime load full source code for those shortlisted spots and let the agent re-rank them. That is the main context-management trick in the paper: preview first, full code later. The paper’s ablation shows that this works better than returning full code during search, and it also uses fewer tokens because the agent does not spend context on dead ends.

Turning repair into a checkpointed workflow

Once the likely locations are known, the second agent takes over.

What to notice in Figure 2 is that repair is treated like a version-controlled workflow, not a single patching step. The agent does not just edit and hope. It tracks hypotheses, checkpoints progress, and keeps alternative repair paths separate.

SWE-Adept builds the resolution agent on top of SWE-agent, but adds two tool families. hypothesis_plan manages alternative solutions and the to-do lists attached to them. It also logs insights gathered from execution feedback. hypothesis_git wraps Git operations into higher-level actions with built-in error handling. That matters because raw Git commands are easy to get wrong once the trajectory becomes long and branchy.

Both tool families talk to a shared working memory. The memory stores the relationship between each hypothesis, its to-do steps, and the code-state checkpoint for each completed step. The agent can therefore refer to semantic names like a hypothesis or to-do instead of trying to remember commit hashes in context. The tools keep the bookkeeping; the model keeps the reasoning.

The resolution flow is straightforward once you see it this way. The agent checkpoints the original state, reproduces the issue, and then starts repair planning. For a simple bug, it may only need one hypothesis. For a harder bug, it can branch into several competing ideas. Each hypothesis gets its own branch and its own fine-grained to-do list. If tests reveal a missing edge case or a step that was skipped, the plan can expand while the work is in progress.

After each completed to-do, the agent commits a checkpoint. That commit is stored in working memory together with the semantic step it corresponds to. If the later part of a hypothesis turns out to be wrong, the agent can jump back to the appropriate earlier checkpoint and continue from there on a fresh branch. That is the crucial reliability feature. A failed edit does not have to poison the whole trajectory.

When several hypotheses have been explored, hypothesis_git can compare them by branch, to-dos, commits, diffs, and insights. The agent then selects the best branch and merges it back onto the original code state for submission. In other words, SWE-Adept treats repair as a sequence of recoverable decisions, not a single irreversible edit stream.

Why the extra structure matters

The paper’s main claim is not that two agents are better than one by magic. It is that each agent gets a workflow that matches its job.

The localization agent gets a compact, dependency-aware view of the repository, so it can search deeply without filling the context window with unrelated code. The resolution agent gets explicit hypotheses and version-controlled checkpoints, so it can try alternatives without mixing them together. Those two design choices line up with the two bottlenecks in repository debugging: finding the root cause and keeping the repair state legible.

That is why the results matter. The paper reports better function-level localization and up to 4.3% better end-to-end resolve rate, while often using fewer tokens than graph-heavy baselines. The token savings are not cosmetic. They are evidence that the agent is spending context on the current hypothesis instead of on the whole repository at once.

The ablation on raw Git is also revealing. Simply giving the agent Git commands does not reproduce the same gain. The abstraction matters because long-horizon repair becomes messy when low-level version-control details leak into the model’s working context. SWE-Adept improves reliability by turning Git into a semantic tool, not just a shell interface.

What the paper is really teaching

SWE-Adept is a good example of a broader pattern in software agents: context control beats brute-force context size. First make the repository searchable as a compact structure. Then walk that structure one plausible path at a time. Then turn editing into a checkpointed, hypothesis-driven process.

That is also where the limitations show up. The system depends on proprietary LLMs, so it is not yet a proof that the same orchestration will work unchanged for open models. It also does not learn across tasks, so each issue is still solved in isolation. And because the tool-memory integration is doing real work, the system is heavier than a free-form loop.

Still, the core lesson is strong and easy to remember. Do not ask one agent to read everything and edit everything. Give one agent a narrow, dependency-aware search job. Give the second agent a structured repair job with checkpoints and recoverable branches. That is the shape of SWE-Adept, and it is why the method feels more like an engineering workflow than a clever prompt.

What to remember

SWE-Adept makes repository debugging easier to follow by separating search from repair. Search stays compact until a candidate location is worth opening. Repair stays organized through hypotheses, to-do lists, and Git checkpoints. If you remember that sequence, you remember the method.
