MAGIS: LLM-Based Multi-Agent Framework for GitHub Issue Resolution

MAGIS is easiest to understand as a repair team for repository issues. The paper is reacting to a familiar failure mode: a single LLM prompt is asked to read too much code, pick the wrong files, edit the wrong lines, and then trust its own patch without a serious review loop. On SWE-bench, that direct approach struggles badly; MAGIS raises the resolved rate to 13.94%, but the more interesting point is structural. It works because the task is broken into smaller jobs that match the model’s strengths.

The paper’s empirical study points to three places where direct prompting breaks down: file selection, line localization, and change complexity. MAGIS turns those three pain points into a workflow. A Manager plans the work, a Repository Custodian finds the relevant files, Developers make narrow edits, and QA Engineers review the result before it is merged. That role split is the heart of the method.

What the team looks like

The clearest visual is Figure 2. It makes the system feel less like one giant prompt and more like a small software org.

What to notice in the figure is the handoff pattern. The Manager handles planning, the Repository Custodian shrinks the repository search space, Developers work on file-level tasks, and QA keeps the patch honest. The workflow is not just linear; it also loops back when a change does not yet look right. That is the main idea: control the repair process instead of hoping the model improvises the whole thing in one pass.

Starting with the right files

MAGIS does not begin by editing. It begins by asking, “Which files are even worth looking at?” The Repository Custodian first ranks repository files with BM25 using the issue text as the query. That gives a candidate set, but the paper is careful not to stop there. A bigger candidate list can improve recall and still make the model worse if the extra files add noise. So the custodian filters the shortlist further.

The interesting trick is memory reuse. If a file has already been summarized before and only changed a little, MAGIS does not reread the whole file from scratch. It reuses the earlier summary and updates it with a short description of the diff when that is cheaper. This is a practical way to keep context under control. Instead of feeding the model an ever-growing pile of source code, the system carries forward a compact note about what each file version now means.

That is why the Manager receives a smaller, cleaner set of candidate files. The goal is not maximal recall at any cost. The goal is to preserve enough recall while cutting away irrelevant context that would otherwise distract the repair loop.

Turning one issue into file-level work

Once the relevant files are on the table, the Manager decomposes the issue into tasks. This is where the paper gets closer to how a human team would work. A single GitHub issue often hides several subproblems: one file may need a small fix, another may need a supporting change, and some pieces may be independent enough to run in parallel.

MAGIS uses that structure instead of fighting it. The Manager creates file-level tasks, then defines a developer role for each task. The paper even describes a kick-off meeting, which is a good metaphor because that is basically what it is: a short coordination step where dependencies are sorted before code changes start. If two changes are independent, they can proceed in parallel. If one task depends on another, the schedule has to respect that order.

This is more than organizational theater. The method is trying to reduce the chance that one agent’s patch invalidates another agent’s assumptions. In repository repair, a bad handoff is often the real bug.

Editing at line level, not file level

This is the most technical part of the method, and also the most important. MAGIS does not just say “edit this file.” It asks the Developer to locate the lines that should change before writing the replacement code.

That matters because file-level retrieval is only the first half of the problem. A model can find the right file and still touch the wrong region inside that file. The paper measures this with a coverage ratio, which is just a plain-English way of asking how much of the true edit region the generated patch overlaps. A higher coverage ratio means the model landed on the right lines, not merely in the right file.

MAGIS turns that observation into a workflow. The Developer looks at the task and the file, predicts the line intervals that should be modified, and uses those spans to separate the code that stays from the code that changes. Only then does it generate new snippets for the targeted region. The idea is simple: narrow the edit before you write it. That reduces collateral damage and makes the patch easier to reason about.

The paper’s empirical analysis supports this design choice. Better line localization tends to go with better issue resolution, which is why MAGIS treats line selection as a first-class step rather than an afterthought.

Why QA is not optional

After the Developer produces a change, the QA Engineer reviews it. The reviewer writes a comment and a yes-or-no decision. If the decision is negative, the Developer revises the patch and tries again. The loop continues until the change passes review or the iteration limit is hit.

This is the part of MAGIS that keeps the system from stopping at a plausible patch. Code can look reasonable and still be wrong in the exact way that breaks a repository. QA exists to catch that gap. In the paper’s ablation, removing QA hurts resolution, which is a strong hint that the review stage is actually doing useful work rather than just adding ceremony.

The optional human hints discussed in the paper point in the same direction. When extra issue context is available, it helps. That does not mean the model is helpless without it. It means repair work benefits from every bit of grounded context the workflow can carry forward.

What MAGIS is really buying

The deeper point of MAGIS is not “use four agents.” It is “separate the failure modes.” Each role is attached to one bottleneck.

The Repository Custodian reduces search noise.

The Manager reduces planning ambiguity.

The Developer reduces edit scope and focuses on the right lines.

The QA Engineer reduces the chance that a brittle patch gets treated as finished.

That separation matters because repository repair is a compounded problem. If any step is too broad, the whole workflow becomes fragile. MAGIS works by making each step small enough to fit inside the model’s strengths.

A useful way to remember the method is this: retrieve, plan, localize, edit, review. MAGIS is not trying to make the model magically smarter. It is trying to make the repair process less ambiguous.

Where it is still fragile

MAGIS is still tied to the quality of its retrieval and prompts. It was evaluated mainly on SWE-bench’s Python repositories, so transfer to other languages and repository styles is still open. It also still struggles as edits get larger and more complex. The method narrows the task, but it does not eliminate the hard parts of software maintenance.

That is the right way to read the paper. MAGIS is not a claim that orchestration solves everything. It is a demonstration that workflow design can matter as much as model strength when the task is real repository repair.

What to remember

MAGIS teaches a simple lesson with practical consequences: when an LLM has to fix a repository issue, the problem is often not raw generation. It is finding the right files, touching the right lines, and checking the patch before it spreads. MAGIS builds a small team around those bottlenecks, and that is why it is easier to understand as a repair workflow than as a single model prompt.
