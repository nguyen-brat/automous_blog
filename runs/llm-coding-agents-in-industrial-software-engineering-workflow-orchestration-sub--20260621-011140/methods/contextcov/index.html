ContextCov: Executable Guardrails for LLM Agents in Software Engineering

ContextCov takes a simple but important idea seriously: repository instruction files should behave more like code than like prose. If a team writes AGENTS.md, CLAUDE.md, or copilot-instructions.md, the point is not just to inform a human reader. The point is to constrain what an autonomous agent is allowed to do. ContextCov turns that passive text into executable guardrails that can stop a bad command, flag a style mismatch, or warn about an architectural boundary before the change lands.

You can think of the system as a small compiler for repository policy.

Read the figure left to right. The left side is compilation: extract rules, refine them, and turn them into checks. The right side is enforcement: intercept commands and edits while the agent is working, then send back a concrete violation trace if something breaks the repository policy. The key idea is that the same instruction does not need to be reinterpreted from scratch every time the agent acts. It is compiled once and reused.

That is what makes ContextCov interesting. The paper is not saying “use a bigger prompt” or “add a nicer reflection loop.” It is saying that instruction compliance becomes much more reliable when repository policy is treated as an executable specification.

Why passive instructions fail

The failure mode ContextCov targets is familiar to anyone who has watched a coding agent drift. The repository already contains the rule, but the agent still runs the wrong command, writes code in the wrong style, or places a module in the wrong layer. The problem is not always that the instruction is missing. Often the instruction is there, but it is buried under context pressure, diluted by local code patterns, or too vague to stay salient while the agent is acting.

The paper separates two sources of trouble. One is underspecified instruction text. If the rule is ambiguous, no enforcement system can magically recover the maintainer’s exact intent. ContextCov mostly leaves that problem to the repository owner. The other is agent non-compliance: the rule is clear enough, but the agent still ignores it or overweights the local context. That is the problem ContextCov is built to catch.

The paper organizes those violations into three practical buckets. Process violations are about commands and environment setup. Source violations are about code patterns and style. Architectural violations are about where code may live and how modules may depend on one another. That split matters because each class needs a different kind of checker.

Turning a Markdown file into rules

ContextCov starts with the part humans are already writing: the instruction file itself. But it does not flatten the file into a bag of sentences. It parses the Markdown structure and preserves the heading path above each leaf block. That is a small design choice with a big effect.

If “Use pytest” appears under Backend and Testing, the meaning is not “use pytest everywhere.” It is “use pytest for backend testing.” The header path keeps that scope intact. Without it, a rule can easily leak into places where it does not belong.

Once a raw instruction is tied to its header path, ContextCov rewrites it into a standalone constraint. That rewrite step matters because runtime checks should not need the original document to guess what the rule meant. The refined rule is the one that gets compiled into a check.

The system also gives each constraint a stable identity derived from its path and content. That is what makes incremental updates practical. If a maintainer edits one section of the instruction file, ContextCov can re-extract and re-synthesize only the affected constraints instead of rebuilding everything.

This is the first place where the “compiler” analogy really holds up. The instruction file is the source text. The refined constraint is the intermediate representation. The runtime check is the generated artifact.

The three kinds of guardrails

Once a rule has been refined, ContextCov routes it to the enforcement machinery that best matches the kind of mistake it is supposed to catch. The paper’s main design move is specialization: do not force one generic checker to solve every constraint type.

Process rules need a gatekeeper, not a linter

Process rules are about commands, package managers, and environment setup. They are not really source-code problems at all. If a repository says “use pnpm” or “do not run this build command directly,” the system needs to intercept the shell command before it executes.

ContextCov handles that by inserting small wrapper scripts into PATH. The agent thinks it is calling the normal binary, but the shim sees the command first. The shim loads the relevant checks, evaluates the command against them, and either blocks it with a violation trace or forwards it to the real binary.

That timing is the whole point. If the command is wrong, the agent should learn that before it creates noisy failures downstream. Immediate feedback is much more useful than a post-hoc critique after the damage is already done.

Source rules need syntax-aware inspection

Source constraints are a better fit for static analysis. ContextCov uses Tree-sitter so it can parse code across multiple languages and look for specific AST patterns instead of searching for fragile text fragments.

That gives the checker three advantages at once. It can scan a whole repository or just a diff. It can report the exact file and line where a rule is violated. And it can express rules like “avoid this function pattern” or “prefer this syntax shape” in a way that is precise enough to be actionable.

This is what makes source checks feel like a real review rather than a generic warning. The agent does not just hear “something is wrong.” It gets a location and a pattern that it can actually fix.

Architectural rules need graphs

Some repository rules are about structure rather than syntax. ContextCov treats deterministic architectural constraints as graph problems. It builds a dependency graph from imports and module references, then checks for things like layer crossings, cycles, and boundary violations.

That is the right level of abstraction for rules such as “this directory may only contain certain submodules” or “the UI layer must not import the database layer.” You cannot catch those reliably by looking at one file in isolation. You need the shape of the codebase.

The paper also notes that the graph-building logic is language aware. Python imports are extracted differently from JavaScript or TypeScript imports, and Go needs its own parsing strategy. The point is not the exact parser. The point is that the architectural rule is enforced on the dependency structure the repository actually uses.

Some rules are really about intent

There is one more class of architectural rule that does not fit neatly into syntax or graph analysis. These are the semantic rules, where the issue is not “which file imports what” but “did the agent respect the design principle.”

The paper’s example is a rule like “do not reach into another component’s storage keys; define a proper API instead.” A dependency graph cannot decide that. A file path cannot decide that. The checker has to understand the intent of the change.

ContextCov handles those cases with an LLM-as-judge step. That is a deliberate fallback, not the primary enforcement path. These checks produce warnings rather than hard blocks, which keeps the system from overclaiming certainty when the rule depends on judgment.

That separation is one of the method’s biggest strengths. Deterministic problems get deterministic checks. Judgment-heavy problems get softer review.

What happens when the agent acts

After synthesis, the checks are stored in JSON and deployed into the runtime layer. The enforcement side can then operate without calling an LLM again. That matters because the expensive reasoning work happened up front. Runtime feedback becomes cheap, fast, and reproducible.

At action time, the system looks like this:

  • a shell command is intercepted before execution,
  • source files are parsed and scanned with the relevant Tree-sitter checks,
  • the dependency graph is queried for architectural violations,
  • semantic design rules are reviewed with an LLM only when the rule cannot be reduced to structure.

The flow is active rather than passive. ContextCov is not waiting for a human code review to notice a mistake later. It is acting like a continuous reviewer that sits in the agent’s path and forces a correction loop immediately.

That feedback style matters because it changes how the agent can recover. A deterministic trace is easier to reason about than a vague criticism. If the agent sees “this command violates the repo workflow” or “this file belongs in the wrong layer,” it can revise its next move without guessing what the reviewer meant.

The paper also emphasizes that the checks can be reviewed and edited by developers before deployment. That is important for trust. The generated policy is not hidden inside a model prompt. It is visible, inspectable code.

Why ContextCov is strict

One of the paper’s clearest design choices is fail-closed handling of ambiguity. If the instruction says “use pnpm,” ContextCov does not try to be clever and guess how narrow the rule should be. It blocks the obvious alternatives broadly unless the instruction file makes the exception explicit.

That sounds harsh only if you think of instructions as soft advice. ContextCov treats them as operational policy. In that framing, a visible false positive is often better than a silent false negative. A false positive can be fixed by clarifying the instruction or narrowing the check. A false negative lets the agent drift and quietly accumulate technical debt.

This is also why the paper draws a line between hard blocks and warnings. When the rule is crisp enough to enforce mechanically, the system should stop the action. When the rule depends on semantic judgment, the system should surface the concern without pretending it has complete certainty.

The tradeoff is real. Ambiguous instruction files may be over-enforced. But the paper argues that ambiguity itself is part of the problem. In an autonomous setting, policy needs to be precise enough to compile.

What the evaluation says

The evaluation on SWE-bench Lite is meant to answer two questions at once: does executable compliance actually reduce violations, and does it hurt functional problem solving?

The headline result is that it helps. ContextCov reaches 88.3% constraint compliance, compared with 67.0% for vanilla prompting and 50.3% for LLM reflection. It also resolves 57.3% of the benchmark tasks, which is slightly better than the baselines rather than worse. And because the feedback is deterministic, it is much cheaper to deliver: roughly 3.4 times lower feedback cost than reflection.

The deeper story is even more useful than the raw numbers. Reflection can drift. The paper describes cases where the critic keeps circling the same issue without converging. ContextCov does not rely on a second model to argue about the mistake. It points at a concrete violation trace and lets the agent correct the action directly.

The residue is also informative. The paper reports that the remaining violations are mostly in source-level checks, while the process and deterministic architectural violations are fully eliminated by the runtime feedback. That suggests the hard part is not stopping obviously wrong commands. The hard part is making style and code-pattern rules precise enough to capture the maintainer’s real intent.

What to remember

ContextCov is best thought of as a compiler for repository policy. It preserves the scope of instruction files, rewrites the useful parts into standalone rules, routes each rule to the right kind of checker, and then enforces those checks while the agent is still acting.

That design has a nice practical consequence. The repository does not depend on the agent remembering every rule all the time. The policy is externalized, compiled, and made executable. When the agent violates it, the system responds immediately with feedback the agent can actually use.

The limitations are just as clear. The method is strongest when the instruction file is precise. Ambiguous rules can be over-enforced. Semantic architectural constraints still require model judgment. And the evaluation is limited to 12 repositories from SWE-bench Lite, so the paper is a strong proof of concept rather than a claim that every codebase should use the same exact setup.

Still, the core lesson is durable: if you want autonomous coding agents to respect repository rules, do not leave those rules as passive prose. Compile them into executable guardrails, then let the runtime enforce them before the mistake becomes part of the codebase.
