Unified Software Engineering Agent and Benchmark for Repository-Level Tasks

USEagent starts from a practical mismatch. Repository work is not one stable task. A bug report may need a reproducer first, a partial fix may need more tests, a code-generation request may need real repository context, and a feature task may need code and tests to evolve together. Existing agents often do well inside one fixed workflow, but they become awkward when the task changes shape.

The paper’s answer is to build one controller that can choose among a small set of coarse software actions and keep a shared record of what it has learned. In plain language, USEagent is trying to behave less like a one-shot code generator and more like a project lead coordinating a few reusable work units.

That loop is the heart of the method. The rest of the design exists to make that loop work across repair, testing, code generation, and compound tasks without rewriting the whole agent for every new benchmark.

Why a fixed repair pipeline is too narrow

The closest older pattern is a specialized repair agent such as AutoCodeRover. It is strong because it already knows the rough shape of a debugging job: reproduce the problem, locate relevant code, edit the patch, review the result, then stop when the patch looks good.

What to notice in this figure is the rigidity of the path. The workflow is sensible for program maintenance, but it is still a prewritten route. If the task is actually test generation, feature development, or a partial fix that needs to be rethought, the fixed pipeline becomes a burden. USEagent keeps the useful pieces of that workflow, but turns the order into a decision rather than a script.

That is the key move. The paper does not throw away structure. It makes structure selectable.

The meta-agent that sits above the actions

USEagent’s architecture is easier to read if you picture a controller sitting above a handful of software-engineering actions. The Meta-Agent looks at the task description and the current state, decides which action should happen next, and then reads the updated state after the action finishes.

What to notice here is that the actions are not the point by themselves. The point is the control loop above them. The meta-agent does not need to commit to one fixed sequence at the start. It can begin with retrieval, testing, or reproduction, and it can return to those steps later if the state says more context is needed.

This is why the paper describes the system as ReAct-style orchestration at the action level. The controller reasons about the current situation, chooses one high-level action, observes the result, and decides again. That is a much more stable way to manage a repository task than trying to do everything inside one prompt.

The action menu

The actions are intentionally coarse-grained. They are meant to be reusable building blocks, not raw shell commands.

  • CodeRetrieval finds program code that is likely relevant to the task. It writes those code locations into shared state so later actions do not need to rediscover them.
  • TestRetrieval does the same for tests. This matters because a test-generation task needs the test side of the repository, not only the implementation side.
  • Reproduction turns a bug report into a standalone failing test. That gives the agent something executable instead of leaving it with only natural-language uncertainty.
  • ExecuteTests runs the relevant tests or a subset of the suite. A useful trick here is that the action can look up project-specific test commands from documentation instead of requiring the prompt to know the repo already.
  • EditCode writes the actual patch, whether that means changing production code or test code. The meta-agent specifies the desired outcome and the location, not the low-level edit mechanics.
  • ReviewPatch checks a candidate patch and the reproducer together, then iterates on them if the result still looks weak. This is where the agent can tighten a promising but incomplete solution.
  • A dedicated stop action ends the loop and emits the best diff in the state. If the run reaches its limit first, the controller can still choose the most promising candidate instead of wandering forever.

Compared with a general-purpose agent such as OpenHands CodeActAgent, this is much more structured. OpenHands can issue lower-level commands like shell execution, Python execution, file edits, and finish. USEagent intentionally raises the abstraction level. That makes the trajectory easier to inspect and makes the same improvement to one action useful across multiple task types.

Why the shared state matters

The second half of the design is the task state, which acts like consensus memory among the actions. The paper keeps four kinds of information there: relevant code locations, relevant test locations, execution results, and a diff store of candidate changes.

This sounds small, but it changes the whole feel of the agent. When ExecuteTests finds failing tests, that evidence is not lost in chat history. It stays available to the next action. When EditCode creates a patch, the patch itself becomes something later actions can apply, extend, or compare against. When ReviewPatch sees a partial solution that is close but not quite right, the agent does not have to start from zero.

The diff store is especially important for understanding why USEagent can handle partial fixes. The agent can keep several candidate edits around, test different combinations, and layer one patch on top of another. That is useful when the first patch is promising but incomplete. It is also exactly where the system can get stuck if the patch is in the wrong place and the agent keeps polishing the wrong branch.

In other words, the shared state is not just a notebook. It is the mechanism that lets the controller compare trajectories instead of forgetting them.

How the workflow changes with the task

Once you give the meta-agent a real task type, the loop starts to self-configure. The paper reports a recognizable pattern:

  • repair tasks often begin with Reproduction,
  • regression-testing tasks often begin with TestRetrieval,
  • code-generation tasks often begin with CodeRetrieval,
  • and many tasks then alternate between EditCode and ExecuteTests.

That pattern matters because it shows the workflow is not hard-coded. The agent is not following the same recipe for every dataset. It is choosing the first move based on what kind of software work it thinks it is doing, then adjusting as the evidence changes.

This is also why the dynamic controller is useful later in the run, not just at the start. If a test failure reveals that the current patch is missing context, the meta-agent can go back to retrieval. If a reproduction test exposes a new edge case, the agent can shift from editing to reviewing. The workflow becomes an action graph rather than a single line.

Why USEbench is part of the method

The benchmark is not just evaluation. It is the pressure test that justifies the architecture.

USEbench brings together repair, regression testing, code generation, test generation, and compound tasks such as partial fix and feature development. That mixture matters because it forces the agent to move between different software-engineering modes. A system built only for repair would overfit to one path. A system built only for free-form command use would be too low-level to stay readable across all those task types.

The unified interface behind USEbench matters too. The benchmark tasks sit behind a common containerized environment that lets the agent read files and execute commands in the same basic way across task families. That removes a lot of one-off wiring and makes the comparison about the agent design rather than about special-case setup.

The compound tasks are the clearest example of why the unified design helps. A partial fix is part test-generation, part repair. Feature development asks for both code and tests. These are exactly the kinds of jobs where a rigid single-purpose workflow starts to feel brittle.

What the experiments say

The paper’s main evidence is that this extra structure actually helps. On the full USEbench, USEagent reaches 33.3 percent PASS@1, compared with 26.8 percent for OpenHands CodeActAgent. On SWE-bench-verified, USEagent reaches 45.6 percent, which is close to the specialized AutoCodeRover result of 46.2 percent. That is the important pattern: the more general controller stays competitive on repair while also covering more task types.

The ablations sharpen the point. When dynamic decision-making is disabled, performance drops across tasks by roughly 4 to 33 percent depending on the benchmark. When ExecuteTests is removed, the drop is especially painful on REPOCOD and REPOTEST. That tells you the added action is not decorative. It is one of the pieces that lets the unified agent keep refining its output instead of guessing.

So the evidence supports the paper’s main claim: the gain does not come from making the model more talkative. It comes from giving it a better action menu and a better way to remember what those actions produced.

Where it still breaks

The paper is careful about the limits.

Feature-like tasks with hidden edge cases remain difficult. The agent can produce something that looks almost right but misses a few cases that only show up in the hidden tests. Partial fixes are another trap: once the agent latches onto a promising but misplaced patch, it may keep editing that patch instead of backing up and trying a different route.

There is also the benchmark overfitting problem. Some plausible patches pass the checks without really matching what a human developer would want. That is why the paper treats execution feedback as necessary but not sufficient. Tests help, but they do not guarantee the patch is semantically right.

The open challenge is backtracking. USEagent knows how to keep going, but it is still weak at deciding when to abandon a bad line of attack. That is probably the most important place where a future version of the architecture would need to improve.

What to remember

Think of USEagent as a project lead with a shared notebook. The meta-agent chooses the next useful software action, the actions read and write a common task state, and the loop keeps going until one diff looks good enough to keep.

That simple change is the paper’s main contribution. It replaces a rigid repair recipe with dynamic orchestration over reusable software actions. The result is not a perfect AI engineer, but it is a much more general one: one that can move between retrieval, editing, testing, and review instead of getting trapped inside a single workflow.
