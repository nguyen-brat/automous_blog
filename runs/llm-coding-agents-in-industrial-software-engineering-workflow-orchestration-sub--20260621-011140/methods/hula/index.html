HULA: Human-in-the-loop LLM-based Agents Framework for Software Development

HULA is easiest to understand as a drafting assistant that lives inside Jira. A ticket arrives with a problem description and repository context, but not usually enough detail to safely write code in one pass. HULA answers that by splitting the work into smaller artifacts: first it finds the relevant files, then it writes a plan, then it drafts code, then a human reviews and edits the result before a PR is raised.

That is the real idea behind the paper. It is not trying to replace engineers with a fully autonomous agent. It is trying to reduce the friction of starting, coordinating, and reviewing software work in an enterprise setting, where the issue text may be sparse and the code change may touch several files across a large repository.

The best way to picture the system is as a staged relay. The issue comes in, context gets narrowed, a plan gets written, a patch gets drafted, and the human keeps the authority at each handoff.

What to notice in the figure is the division of labor. The AI Planner does not jump straight to code. It first localizes the task by identifying relevant files, then turns that narrowed context into a coding plan. The AI Coding Agent works from that plan and produces the actual patch. The Human Agent is not a passive reviewer at the end; the human can inspect, edit, and redirect both the plan and the code before anything becomes a pull request.

Why the paper splits the work up

Atlassian’s own software workflow already makes this split feel natural. A Jira issue is the starting point, but a Jira issue is not the same thing as a code change. Between those two points, someone has to decide which files matter, what the change should do, how the change should be written, and whether the result is good enough to review.

HULA turns that messy path into a sequence of smaller decisions. The first decision is “which files are relevant?” That matters because enterprise repositories are too large for a model to treat as one flat prompt. The second decision is “what should change in each file?” That matters because a relevant file list alone does not tell you how to fix the bug or add the feature. The third decision is “what code should be written now?” That is the actual drafting step. The fourth decision is “is the draft acceptable to a human reviewer?” That is what makes the workflow usable in practice.

The paper describes this as decentralized planning and decentralized execution. In plain language, each stage owns its local job instead of asking one central agent to micromanage everything. All agents still see the same issue and repository context, so they are working from a shared source of truth. That keeps communication overhead lower and makes it easier to react when a human adds new detail.

This also explains why HULA is not just a prettier chatbot. It is a process design. The system is trying to preserve human control while still letting the model do the repetitive, context-heavy work that usually burns time.

Finding the right files before writing anything

The planner’s first job is file localization. That sounds modest, but it is one of the most important steps in the whole paper. If the model starts from the wrong files, every later step becomes a cleanup exercise.

The planner receives the Jira issue and the repository, then identifies the files most likely to be involved. Only after that does it write a coding plan. A useful plan is not a vague summary like “fix the bug.” It is more concrete, along the lines of “add tests for the add_subfigure method” or “update this validation path and the related helper.” The human can review that file set, edit it if needed, and even rewrite the plan before the coding stage begins.

This is where HULA fits the enterprise setting better than a one-shot code generator. Real issues often arrive with short descriptions, informal context, or a task statement that assumes the reader already knows the surrounding project discussion. The planner gives the engineer a smaller, inspectable surface instead of forcing them to trust a single generated patch.

The paper’s results make this step feel tangible. On the SWE-bench Verified benchmark, HULA’s file localization recall is strong, which means it often finds the right files to touch. On Atlassian’s internal issues, localization drops much more sharply because the tickets are shorter, the repositories are more diverse, and the changes span more languages and file types. That contrast is a good reminder that “find the right files” is a real technical problem, not a trivial prelude to coding.

Turning a plan into a draft patch

Once the plan is approved, the AI Coding Agent takes over. Its job is narrower than the planner’s job: it receives the plan and generates code changes for the files that were already selected. That narrowing is the whole point. The model no longer has to rediscover the task from scratch.

The coder is also allowed to refine its own draft. HULA explicitly feeds validation feedback from tools such as compilers and linters back into the generation loop. If the patch has a syntax issue, a formatting issue, or some other mechanical problem, the model can revise it before the human sees the final version.

That self-refinement step matters because not every failure is conceptual. Sometimes the model has the right idea but the wrong import, the wrong signature, or a small inconsistency that makes the patch unusable. Tool feedback gives the coder a way to tighten the draft before it reaches review.

The human still stays in the loop here. If the generated code does not look right, the human can add instructions and ask for another pass. In other words, HULA is not trying to make the model perfect on the first shot. It is trying to make the first shot useful enough that a human can correct it quickly.

Why the human remains the final authority

HULA’s most important design choice is that the human does not leave the process after the ticket is opened. The human can review the file list, edit the plan, inspect the proposed code, and decide whether to turn the result into a pull request or keep iterating.

That matters because enterprise coding is not just about producing code that looks plausible. It is about code that fits the team’s conventions, passes the right checks, and is complete enough to survive review. The paper is explicit that code quality and task completeness still require human intervention. HULA helps start the work and structure the work, but it does not magically remove the need for judgment.

In practice, that makes HULA feel more like a drafting desk than an autopilot. The model prepares reviewable artifacts; the engineer decides whether those artifacts are ready to move forward. That is also why the framework fits Jira well. The issue tracker is already the place where work is discussed, edited, and approved, so the agent can join the same workflow instead of creating a separate one.

Reading the evaluation as a workflow story

The evaluation is staged for the same reason the system is staged: benchmark accuracy, real usage, and practitioner perception are different questions. The paper does not treat them as interchangeable.

What to notice in the figure is the progression from offline to online to survey. The offline stage asks whether the workflow can do the job in a controlled setting without human feedback. The online stage asks whether real practitioners will actually accept the generated plan and code inside Jira. The survey asks a third question: even if the workflow works, does it feel helpful to the people who use it?

The offline results show the shape of the method. On SWE-bench Verified, HULA completes the generation workflow for most issues, finds the relevant files with strong recall, and produces code that is often similar to the human-written patch. But the tests also expose the gap between getting the right idea and getting a fully correct patch. Passing every unit test is much harder than locating the files.

The online results are more interesting for deployment. In real use, HULA successfully generated plans for most issues that reached the planning stage, and practitioners approved most of those plans. That is the paper’s strongest practical signal: the early output was often good enough for humans to trust and move forward with. The workflow also produced pull requests that were actually merged, which is the point that matters in an enterprise setting.

The survey adds the final layer. Many participants said the generated code was easy to understand and modify, which is exactly what you want from a drafting assistant. Fewer participants felt that the plan perfectly matched the issue description, and even fewer thought the first generated code was fully complete without human help. That split tells you where HULA is strongest: it helps engineers get moving and gives them a readable starting point, but it does not eliminate the need for review.

What the paper says still breaks

HULA is useful, but the paper does not pretend it is solved.

The first limitation is sparse issue descriptions. If the Jira ticket is too brief, the planner has less to work with, and file localization becomes harder. The paper’s own lessons learned make this point bluntly: detailed input matters a lot.

The second limitation is code quality. HULA can produce understandable drafts, but understandable is not the same as correct or complete. Complex multi-file changes, missing context, and non-functional requirements still trip it up.

The third limitation is evaluation itself. The paper argues that unit tests are not enough to judge enterprise code quality. Passing tests does not guarantee that the patch is readable, maintainable, or aligned with the real intent of the ticket. That is why the human review step stays central, and why the survey results matter as much as the benchmark numbers.

The fourth limitation is scope. The deployment is tightly tied to Atlassian’s Jira-based workflow, so the exact shape of the system may not transfer directly to every company or every repository culture. The broader lesson is transferable, but the implementation is not automatically universal.

What to remember

HULA’s core lesson is simple: for enterprise software work, the useful unit is not a single generated patch. It is a sequence of reviewable artifacts.

The system first narrows the task to relevant files, then turns that context into a change plan, then drafts the code, then lets the human steer the result into a pull request. That staged design is what makes the method practical. It reduces workflow friction without pretending that a model can safely own the whole job on its own.

If you remember one sentence, remember this: HULA is a human-centered coding pipeline, not an autonomous coding oracle.
