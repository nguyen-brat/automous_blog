AgentConductor: RL-Optimized Multi-Agent Systems with Difficulty-Aware Dynamic Interaction Topologies for Code Generation

AgentConductor is easiest to understand as a conductor for an agent team. It does not try to solve the code problem directly. Instead, it learns how to compose the team: which specialist roles should appear, how they should depend on one another, how much parallelism the task deserves, and when the topology should be rewritten after the sandbox says the code failed.

That is the paper’s main shift in perspective. Most multi-agent coding systems spend their effort on better prompts or better role descriptions while leaving the workflow itself fixed. AgentConductor treats the workflow as part of the learned policy. The orchestrator does not just answer a coding question; it writes the interaction graph that decides how the team will work.

The topology is the product

The first thing to notice is that the paper does not hide the workflow inside opaque text. It makes the workflow explicit in YAML, then decodes that YAML into a layered directed acyclic graph. In that graph, each layer is a step, agents inside the same layer can run in parallel, and references point back to earlier layers when a role needs output from a previous agent.

What matters in the figure is not the formatting trick. It is the control surface. The YAML block is both human-readable and machine-checkable, so the orchestrator can emit a compact plan and the runtime can deterministically turn it into an executable graph. That gives the system a clean middle ground between a rigid chain and an unruly fully connected mesh.

The graph choice is the heart of the method. A chain is too serial. A tree branches locally, but it is awkward when a later step needs to reconnect to earlier reasoning. A fully connected mesh can express almost anything, but it wastes tokens and becomes hard to learn. AgentConductor’s layered graph keeps the useful part of each idea: parallel work within a layer, structured dependencies across layers, and enough sparsity to stay readable and efficient.

How the conductor chooses a team

The orchestrator works from a fixed pool of roles instead of inventing a new cast every time. The paper’s role set includes retrieval, planning, algorithmic, coding, debugging, and testing agents. For a given problem, the orchestrator first estimates difficulty, then selects the roles that are likely to matter, then emits a YAML topology that says who should run, in what order, and who should listen to whom.

That topology is only the beginning. Once the agents execute, the environment returns a richer signal than a simple pass/fail bit. The system sees the role outputs, the sandbox execution result, and the trace of the current topology. If the code passes, the loop stops. If not, the orchestrator uses the failure information to regenerate a better topology for the next turn.

This is what makes the method multi-turn instead of one-shot. The orchestrator can reuse a role from memory, rerun a role with a different dependency set, or activate a new role if the previous structure was too sparse. The point is not to keep speaking louder. The point is to change the communication pattern until it matches the problem.

Teaching the model the grammar of workflows

The paper does not ask reinforcement learning to discover valid topology syntax from scratch. It first builds a supervised corpus of synthetic topologies so the model learns the grammar of the YAML and the typical shapes of successful interaction graphs. Those examples are spread across difficulty levels and are validated for format, schema, uniqueness, and basic structural legality before they are used for training.

What to notice here is the progression. Stage 1 teaches the model what a valid topology looks like. Stage 2 uses execution feedback to tune the policy toward topologies that actually solve tasks. Stage 3 freezes the orchestrator and lets it generate and revise topologies on new problems. That order matters because reinforcement learning is much better at refining a structural prior than it is at inventing one from nothing.

The reinforcement-learning step uses GRPO, which is a practical way to compare multiple sampled trajectories from the same problem and reward the ones that do better than their peers. That relative scoring is useful because absolute rewards can be noisy: one candidate may fail with a parse error, another may compile but answer incorrectly, and a third may pass. Ranking them within a group gives the optimizer a clearer signal about which topology was actually the better choice.

The paper also keeps a regularizer on the topology tokens so the policy does not drift too far from the supervised prior. That regularizer only applies to the tokens that describe the topology itself. The agent outputs and sandbox traces are treated as environment results, not as text the policy should directly imitate. This keeps the learning target focused on orchestration rather than on the accidental wording of execution logs.

The reward has two jobs

AgentConductor is not only learning to be correct. It is also learning to be frugal. The reward therefore has two parts: one part checks whether the topology led to a valid and correct solution, and the other part checks whether the interaction graph stayed appropriately small and shallow for the task difficulty.

The correctness side is intentionally strict. If the model fails to emit usable YAML, the reward is strongly negative because the workflow cannot even be executed. If the YAML parses but violates the topology schema or the graph logic rules, that is still a structural failure, just a less severe one. Once the YAML is valid, the code execution result takes over. Wrong answers, timeouts, memory failures, runtime crashes, and compilation errors all produce different signals, because each failure type suggests a different kind of orchestration fix.

That separation is important. A broken topology is not the same thing as a wrong program, and the orchestrator should not receive the same lesson from both.

Why density depends on difficulty

The second part of the reward is the more novel one. The paper treats topology complexity as a cost problem. More agent invocations mean more baseline overhead. More dependency edges mean more communication. More layers mean more latency, because work has to move through more sequential stages before the final answer is ready.

So instead of rewarding the model for making the graph “as sparse as possible,” the paper rewards it for making the graph sparse enough for the task. Easy problems get tight density ceilings. Hard problems get looser ones. The paper sets those ceilings from statistics over successful synthetic examples, so the bounds are not arbitrary hand-tuning.

In plain language, the orchestrator is allowed to spend more communication budget when the task deserves it, but it is penalized for wasting that budget on easy cases. That is the key to the paper’s difficulty-aware behavior. The model is not trying to minimize the number of agents at all costs. It is trying to match the workflow size to the problem size.

The paper’s density score is easier to remember if you think of it as a sparseness score. Higher score means the graph is cheaper and cleaner; lower score means the graph is bulkier or deeper. If the graph goes beyond the difficulty-specific ceiling, the reward drops sharply so the model does not “buy” accuracy with unnecessary coordination.

What the system is really optimizing

The most useful way to think about AgentConductor is that it learns a negotiation between coordination and cost. If the graph is too sparse, the team loses information and important roles never speak. If the graph is too dense, the team burns tokens and still may not solve the problem. The learned policy tries to land in the narrow middle where the topology is just rich enough to support the task.

That explains why the paper’s output is not just a better answer. It is a better workflow. The orchestrator learns when to add a planning branch, when to let retrieval run in parallel, when to send the code back through debugging, and when to stop because the sandbox already passed. In other words, the model is learning how to spend communication the way a careful engineer spends time: only where it moves the solution forward.

The reported results support that idea. The paper shows that the learned topology can improve code quality while also reducing communication overhead relative to stronger static or frozen-workflow baselines. But the deeper point is broader than the benchmark numbers. Workflow structure itself is an optimization target.

Where the method is strong, and where it is still narrow

AgentConductor is strongest when the task has a crisp execution signal, a clear role pool, and a sandbox that can tell the system what went wrong. That makes competition-style code generation a good fit. The method is also conceptually appealing because it keeps the workflow legible: the topology is still readable YAML, not an opaque internal policy.

The tradeoff is that the system becomes more complex to train than a fixed workflow baseline. It needs supervised topology data, a schema validator, an execution sandbox, and reward shaping that balances correctness with sparsity. The paper also remains mainly centered on coding tasks, so the most ambitious transfer questions are still open: whether the same density-control idea scales cleanly to larger industrial teams, new tool ecosystems, or non-code collaborative work.

What to remember

If you want one sentence to carry away, it is this: AgentConductor learns the wiring diagram, not just the answer.

The orchestrator writes a YAML topology, the runtime turns that topology into a layered graph, the agents execute, the sandbox reports what happened, and the next topology is revised from that feedback. SFT gives the model a prior for valid structure. GRPO sharpens that prior with execution-based feedback. The difficulty-aware density reward keeps the graph small when it can be small and larger when the task really needs more coordination.

That is why the paper matters. It shows that in agentic software workflows, communication structure is not fixed plumbing. It is something you can learn, optimize, and adapt per problem.
