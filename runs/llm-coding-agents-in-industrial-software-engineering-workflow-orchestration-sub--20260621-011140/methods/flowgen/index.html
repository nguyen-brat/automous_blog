FlowGen

FlowGen, from the paper FlowGen: Emulating Software Process Models with Large Language Model Agents for Code Generation, starts from a simple but important idea: code generation goes wrong less because a model cannot write code and more because it skips the workflow humans use to keep requirements, design, implementation, review, and testing aligned. So instead of asking one model to do everything in one shot, the paper turns the task into a small software team.

Think of it as a relay race. The baton is not just code. It is the requirement document, the design sketch, the test plan, and finally the failure report that tells the developer what to fix. FlowGen tries to make each handoff more structured than a raw prompt, so the next role gets a cleaner and more checkable artifact than the last one.

The paper’s architecture figure makes that idea easy to see.

What to notice here is not just that there are three variants. It is that the same roles appear in all three, but the communication rules change. Waterfall, TDD, and Scrum are not three unrelated systems; they are three ways of wiring the same team.

A small software team in the loop

FlowGen assigns LLM agents to familiar development roles: requirement engineer, architect, developer, tester, and, in the Scrum variant, scrum master. That role split matters because it turns a vague instruction like “write the code” into a chain of narrower jobs.

The requirement engineer first rewrites the programming problem into a requirement document. The architect then turns that into a high-level design. The developer writes the code from the requirement and design artifacts, and later rewrites the code when a failure report comes back. The tester does more than run a final check: the tester first designs tests, then writes and executes them, and finally summarizes failures in a report. The scrum master appears only in the Scrum version and turns the team discussion into a task list.

The paper makes one subtle but important choice here: it reuses the same prompt template across process models. Only the process vocabulary changes, such as saying “user story” in Scrum instead of “requirement document” in Waterfall. That keeps the comparison focused on workflow structure instead of prompt phrasing.

There is also a second design choice that is easy to miss: all roles can feed back on other roles’ artifacts. So the system is not a one-way pipeline. It is a pipeline with review pressure at every stage.

The tester role is especially important. The paper says the tester is involved in every activity so the system behaves more like DevTestOps than like a final-stage QA gate. The tester does not just verify the code at the end. It also helps shape the requirement, the design, and the implementation. On average, FlowGen generates about four tests before the review meeting and about six after it, which shows that the testing signal gets richer as the workflow iterates.

One more detail matters for understanding the method: the paper keeps the oracles out of the loop. The tester is not handed hidden answers. It has to infer expected behavior from the problem statement, the design, and the evolving artifacts. That keeps the method honest. The improvement comes from workflow discipline, not from cheating with ground truth.

Waterfall, TDD, and Scrum are three control policies

The easiest way to understand FlowGen is to treat the three variants as different control policies on the same team.

Waterfall

FlowGen Waterfall is the most straightforward version. The problem flows through requirement analysis, then design, then implementation, then testing. The order is fixed, which makes the handoff structure easy to follow.

The important detail is that test results are not the end of the story. In the paper’s implementation, the generated test result is redirected back to the developer so the code can be fixed and improved. That turns Waterfall from a one-pass sequence into a sequence with a repair loop at the end.

This version is useful when you want the cleanest separation between phases. The tradeoff is that the workflow can feel rigid. If a later stage reveals that an assumption was wrong, the earlier stages may need to be revisited in a way the linear flow does not naturally encourage.

TDD

FlowGen TDD moves the test thinking earlier. Here, testing comes after design and before implementation. The developer sees the test design first, then writes the code with those tests in mind, and only afterward are the tests executed.

That shift changes the shape of the code. Instead of writing code first and letting tests catch mistakes, the developer is nudged to think about test behavior before the implementation is complete. In practice, that often makes the generated code more explicit about edge cases and failure handling.

When a test fails, the developer is sent back to inspect the code and repair it. So TDD in FlowGen is not just “write tests earlier.” It is a way of making the tests part of the implementation pressure from the beginning.

Scrum

Scrum is the most social version of the three, and in this paper it is also the strongest one.

Compared with Waterfall and TDD, Scrum adds a scrum master and sprint meetings. The paper also changes the terminology to match agile practice, using phrases like user story instead of requirement document. More importantly, it changes the communication pattern. During sprint planning, every role can contribute comments into a shared buffer, and everyone can see the discussion history. Then the scrum master summarizes the discussion and turns it into a task list.

That shared-buffer meeting is a different kind of control mechanism from the linear handoff used in Waterfall and TDD. It gives each role a chance to react before the sprint begins, which helps keep the team aligned on the same goal. After that, the four development roles still execute in sequence during the sprint. At the end of the sprint, they meet again to decide whether to release the code or go back and fix failures.

In plain terms, Scrum lets the agents talk before they lock in the work. That extra coordination is one reason the paper’s results favor it.

Self-refinement is where the method really tightens

FlowGen is not just role assignment. It is role assignment plus revision.

At every stage, other agents review the artifact that was just produced and suggest improvements. The paper explicitly includes the tester in those reviews, so quality feedback is always present even when the current artifact is a requirement document or a design note. The downstream role and the tester both help shape the next version of the artifact.

This matters because the method is trying to simulate how human software teams refine work. A requirement is rarely perfect on the first pass. A design often needs clarification. Code needs review comments and failure reports before it becomes reliable. FlowGen bakes that reality into the pipeline.

The paper repeats the refinement loop three times. If the code still does not pass the tests, the whole software process starts again. That sounds expensive, and it is. But it is also the whole point: the method spends more effort earlier so the final output has a better chance of being correct, readable, and testable.

What the results actually say

The headline result is not just that FlowGen works, but that the process model matters.

On the paper’s zero-shot evaluation setup, FlowGen Scrum is the strongest of the three variants. It reaches 75.2 Pass@1 on HumanEval, 65.5 on HumanEval-ET, 82.5 on MBPP, and 56.7 on MBPP-ET. The paper reports that this is about a 15 percent average improvement over RawGPT, and Scrum also has the lowest run-to-run variation. So the workflow is not only better on average; it is also more stable.

The other important result is that testing is the biggest lever. When the testing activity is removed, Pass@1 drops sharply, by as much as 17.0 percent to 56.1 percent depending on the model and benchmark. That is a strong signal that the generated tests are doing real work. They are not decorative. They are one of the main mechanisms that improve functional correctness.

The ablation study also shows that design and code review improve code quality in a different way. Removing those activities increases code smells and reduces handled exceptions. In other words, FlowGen is not only trying to get the right answer through the test suite. It is also nudging the generated code toward better structure and more defensive error handling.

That distinction matters. A model can pass a benchmark while still producing clumsy code. FlowGen pushes on both fronts: correctness and code quality.

The paper also checks stability across model versions and temperature settings. The overall story is that FlowGen stays relatively steady while RawGPT can swing much more, especially on older GPT-3.5 versions. So the workflow acts like a stabilizer around the base model.

Compared with other techniques, FlowGen Scrum is competitive with CodeT and clearly stronger than Reflexion on these benchmarks. The paper also shows that a CodeT-style repeated-implementation stage can be layered on top of FlowGen Scrum for even better results. That is a useful clue: the process model is not a replacement for every other technique. It is a scaffold that can absorb other improvement ideas.

Where the paper is careful

The paper does not claim that FlowGen is a universal fix for software generation.

First, the evaluation is mostly on algorithmic Python benchmarks, so the results may not transfer cleanly to large repositories or messy product code. Second, the benchmark test suites are not always consistent. The paper gives examples where the generated code is arguably more careful than the test expects, such as a case where the code validates a dictionary input but the test passes a set, or cases where MBPP-ET changes the expected return value or naming behavior in ways that make the test less representative. In those situations, a sensible implementation can still fail the benchmark.

That is why the paper treats the benchmark scores as only part of the story. Some of the observed failures are about the evaluation setup, not just about the model’s reasoning.

There is also a dependency on GPT-3.5 and the exact prompting setup. The workflow structure seems to help, but the paper does not prove that every stronger base model or every different tool setup will behave the same way. The authors themselves raise the open question of how well process-based orchestration transfers to larger software tasks and to stronger, tool-using models.

What to remember

FlowGen is best remembered as a workflow policy, not as a new code generator. It takes a base model and makes it behave like a small software team that keeps handing off requirements, design notes, code, and failure reports until the output is good enough.

The core lesson is simple: process structure matters. Testing is the biggest correctness lever. Design and code review improve code quality and reliability. Scrum-style coordination, with its shared discussion and repeated refinement, is the strongest variant in this paper. And the whole system becomes more stable because it has a place for feedback at every stage instead of only at the end.

If you want to zoom out, this is the FlowGen example inside the broader Process-Structured Software Factories family.
