ChatDev: Communicative Agents for Software Development

ChatDev starts from a simple observation: software development is not one big answer, it is a chain of conversations. A requirement has to become a design, the design has to become code, and the code has to survive review and testing. Within the broader family of process-structured software factories, ChatDev is the clearest example of using language itself as the coordination layer. Instead of asking one model to guess the whole program in a single pass, it asks specialized agents to refine the task step by step.

Suppose the request is “build a Gomoku game.” A one-shot model has to invent the rules, structure the code, anticipate bugs, and decide how to test everything at once. ChatDev breaks that uncertainty apart. It treats the project as a relay: each stage gets a narrower question, and only the resolved answer moves forward.

What to notice here is the control structure. This is not a loose group chat. The paper deliberately chains the work into phases and then into subtasks, so the conversation stays focused. Each subtask is handled by two specialized roles that push and answer in a bounded loop until they reach a usable decision. That makes the handoff cleaner: the next phase receives a settled artifact instead of a noisy transcript.

Why the chain matters

Here is what is happening under the hood. ChatDev organizes the project into design, coding, and testing. The coding phase is further split into writing and completion, and the testing phase is split into review and runtime checking. That matters because the kind of help the system needs changes as the artifact becomes more concrete. Early on, the team needs to settle architecture and intent. Later, it needs to pin down syntax, imports, method names, and failure cases.

The key idea is that ChatDev does not let every agent talk about everything all the time. For each subtask, one agent drives the discussion like an instructor, and the other responds like an assistant. The names are less important than the discipline: a subtask has a narrow goal, a bounded dialogue, and a concrete output that can be handed off.

This is why the framework feels more like a software process than a chatbot. The conversation is there to produce an artifact, not to keep the chat alive.

Memory that keeps the right things

ChatDev also treats memory as a control problem, not just a storage problem. Inside one phase, the agents keep short-term memory so the current conversation stays coherent. Across phases, they carry forward only the solved output as long-term memory. The full back-and-forth does not have to be dragged into every later step.

That split is useful for a simple reason: most of the conversation noise is not worth preserving. The next phase needs the design decisions that were actually settled, not every half-formed idea that came up along the way. If the design phase decides on a GUI, a local file layout, or a specific interaction pattern, that decision becomes the bridge into coding. If a speculative suggestion never became part of the final agreement, it can be forgotten.

This is one of the paper’s quiet strengths. It makes continuity possible without letting the context balloon.

When the system should ask before it answers

The second major trick is communicative dehallucination. Instead of forcing the assistant to answer immediately, ChatDev lets it switch into question-asking mode when the request is too vague. In effect, the assistant can ask for missing specifics before giving a final response.

That is a small role reversal: the agent that normally answers is allowed to ask first.

This sounds small, but it attacks the most damaging failure mode in software generation. If a model guesses the wrong dependency, class, import, or method name, it can produce code that looks plausible but does not run. ChatDev’s answer is to turn uncertainty into a visible part of the conversation. The assistant asks for more detail, the instructor supplies it, and only then does the code get revised.

That is why the mechanism is called dehallucination. It does not magically make the model omniscient. It simply makes it harder for the model to hide uncertainty behind fluent nonsense. The paper’s review and testing analysis shows exactly the kind of issues this helps with: missing methods, missing imports, placeholders, and other small errors that can break the whole program. By handling one concrete issue at a time, the system has a better chance of converging on runnable code.

Why the language changes with the phase

The paper’s communication analysis is helpful because it shows that ChatDev is not trying to speak one fixed dialect throughout. Natural language dominates the design stage, because that is where the team is still negotiating intent, users, interfaces, and structure. Later phases mix in more code-centric communication, because debugging and repair need exact detail.

What to notice is the shift in emphasis. Early design discussions are broad and descriptive. Later discussions become narrower and more technical, especially around review and testing. That is not a bug. It is the point. Natural language is better when the system is still deciding what to build. Programming-language-like communication is better when the system is fixing a specific error or checking a concrete code path.

In the paper’s own terms, the point is to use natural language for system design and programming language for debugging. ChatDev is really a controlled switch between those two modes.

What the results say about the design

The reported numbers line up with the mechanism. ChatDev outperforms GPT-Engineer and MetaGPT on completeness, executability, consistency, and quality, with reported values of 0.5600, 0.8800, 0.8021, and 0.3953 respectively. The most striking number is executability: the generated software actually runs much more often than the baselines.

The ablations make the deeper point. Removing communicative dehallucination hurts quality and executability. Removing roles hurts even more, which is a strong reminder that the role prompts are not cosmetic. Without them, the agents drift toward weaker, less specific behavior. The paper notes that programmers become more likely to produce generic command-line-only solutions, and reviewers give feedback that is too high level to drive useful repair.

Those are not small shifts. They show that the choreography itself matters. ChatDev is not better simply because it has more agents. It is better because the agents are constrained to talk about the right thing at the right time.

Where the method is strong, and where it is not

ChatDev is best suited to clearly specified tasks and prototype-style software. If the requirement is vague, the chain can still drift because the system only works as well as the questions it is able to ask. The paper is also honest about the cost: multiple agents, more turns, and more refinement mean more tokens and more time than simpler methods.

There is also a broader evaluation problem. General-purpose software is hard to judge automatically. Completeness, executability, and consistency are useful signals, but they do not capture everything teams care about, such as robustness, safety, maintainability, or user experience. So ChatDev is best read as a strong orchestration pattern, not a final answer to industrial software engineering.

What to remember

The simplest way to remember ChatDev is this: it turns software generation into a disciplined relay. The chat chain tells the agents what to talk about, selective memory tells them what to keep, and communicative dehallucination tells them when to stop guessing and ask for precision. That combination is what makes the system more coherent than a one-shot code generator.

ChatDev’s bigger lesson is not just “use more agents.” It is “make the workflow itself harder to misunderstand.” When the handoffs are explicit, the roles are narrow, and the missing details are forced into the open, language becomes a much better bridge between a human request and a runnable codebase.
