CLIMP: Contrastive Language-Image Mamba Pretraining

CLIMP is easiest to understand as CLIP after a backbone swap on both sides. The paper keeps the contrastive recipe that made CLIP useful: take a matched image and caption, map them into one shared embedding space, and train the matched pair to be close while other pairs in the batch act as negatives. What changes is how the two encoders read the data. Instead of a Vision Transformer for images and a transformer language model for text, CLIMP uses VMamba for vision and a Mamba language model for text.

That matters because the bottlenecks are no longer the same. Attention-heavy encoders compare tokens against all other tokens, which makes high-resolution images expensive and makes long sequences awkward. Mamba keeps a compact running state and updates it token by token. So CLIMP is not trying to invent a new objective. It is trying to keep CLIP’s alignment contract while making both towers cheaper to scale and less brittle under resolution changes and distribution shift.

The method as a pipeline

The whole system has a very simple shape in your head.

An image is split into patches and passed through VMamba blocks. A caption is tokenized and passed through a Mamba language model. Each tower produces one final embedding, and those two embeddings are pulled into the same space with contrastive training.

That is the whole contract. The model still behaves like CLIP at the surface, because the output is still a pair of aligned multimodal vectors. The difference is that the image side is no longer a transformer that mixes every patch with every other patch, and the text side is no longer a bidirectional encoder that has to fake long-range memory with attention. Both towers now process structure sequentially, with a compact state carrying the context forward.

If you want one mental image, think of CLIMP as two readers with notebooks. The image reader scans the scene and keeps updating a spatial notebook. The text reader scans the caption left to right and keeps updating a linguistic notebook. The final embeddings are the summaries written at the end.

Why the vision tower is the real departure

The image side is where CLIMP feels most different from ordinary CLIP. VMamba does not flatten the picture into a bag of unrelated tokens and then let attention decide everything from scratch. It uses Visual State-Space blocks with a 2D selective scan, so the encoder reads the image along multiple spatial paths and accumulates context as it goes. That gives the model a built-in spatial bias: local structure, order, and neighborhood relationships matter.

This is a useful bias for two reasons. First, it helps the model behave more like an image model and less like a generic token mixer. Second, it makes variable resolution feel natural. Since the backbone is already scanning structured image patches rather than relying on fixed positional interpolation tricks, CLIMP can accept larger images without needing a special resolution-specific recipe.

What that bias looks like in practice is more localized alignment. The paper’s porch example is a good illustration.

The thing to notice is not the exact score. It is where the warm regions land. CLIMP concentrates on the porch and fence, which is what you want from a useful contrastive representation. The transformer baselines are more diffuse, which usually means the model has learned a weaker or noisier image-text match.

This is the hidden payoff of the vision backbone choice. CLIMP is not only cheaper to run at high resolution. It is also learning representations that keep more of the image’s layout structure instead of collapsing the scene into a loose global summary too early.

Why the text tower uses the last token

The text side is just as important, even if the change feels more straightforward. CLIMP uses pretrained Mamba language models, either Mamba-1 or Mamba-2, as the caption encoder. Because these models are autoregressive, each token only has access to the context that came before it. That makes the last non-padding token the natural place to read out a single caption embedding.

So CLIMP pools the hidden state at the final real token. That is not a random choice. It is the position that has seen the entire caption. If you pooled earlier, you would be throwing away words that had not been read yet. Last-token pooling respects the causal structure of the model.

This also explains one of the method’s more practical benefits: CLIMP can go beyond CLIP’s hard 77-token ceiling in dense captioning retrieval. A longer caption is not awkward special case baggage here. It is just a longer sequence that the autoregressive text tower can process in order, then compress at the end.

Mamba-2 makes this even more attractive because it tightens the recurrent computation into a more structured form. You do not need to remember the algebra to understand the point: the text tower is still sequential, but the sequential update is engineered to be faster and more efficient than standard attention.

Why the alignment space looks better

Once both towers produce their embeddings, CLIMP trains exactly where CLIP trains: in the shared space. The interesting part is that the state-space bias seems to change what that space looks like. The paper argues that the model learns tighter cross-modal alignment and less hubness, which is a fancy way of saying that matching pairs become easier to find and a few overpopular embeddings stop dominating the nearest-neighbor search.

That matters because retrieval quality is not only about being close to the right match. It is also about not being distracted by generic, overused embeddings that look good for everything. A representation space with less hubness and better alignment is easier to search, easier to transfer, and often more robust when the input shifts a little.

The retrieval numbers reflect that geometry. On the benchmark set the paper uses, CLIMP reaches strong recall, with the Mamba-1 variant leading the retrieval metrics. The exact table is less important than the pattern: the fully Mamba-based design is not sacrificing alignment quality just to get efficiency.

Why the efficiency story matters

The state-space choice becomes most valuable when the image gets large. Attention grows expensive as the token count rises, because every patch wants to compare itself with every other patch. Mamba grows much more gently because it updates a compact state instead of building all-pairs interactions.

The paper’s efficiency figure shows that clearly.

What to notice here is the slope. At small sizes, the difference is already useful. At larger sizes, the difference becomes the whole story. That is why CLIMP can train at 224x224 and still evaluate at much higher resolutions without feeling like it is fighting its own backbone.

The paper reports that at 896x896, CLIMP keeps strong retrieval while using much less memory and fewer FLOPs than transformer baselines. In plain language, it buys you more room to scale images up without paying the usual attention tax.

High resolution is not an afterthought

This part is easy to miss if you only think about benchmark scores. CLIMP is not simply a CLIP variant that happens to survive larger images. High resolution is part of the point. A model that can read a 896x896 image without a special positional encoding hack has a different operational profile from one that needs interpolation tricks or a dedicated resolution recipe.

That is why the paper’s scaling plot matters.

The curve keeps rising as the training set gets larger, which tells you the backbone swap is not a dead end. CLIMP is still benefiting from more data. That matters because a method can look clever on a small benchmark and still fail to scale. Here the shape of the curve says the opposite: the architecture remains useful as you feed it more supervision.

The high-resolution retrieval results tell the same story. The method keeps its advantage when the image size grows, and the gap to transformer baselines widens at the larger settings instead of shrinking. So the backbone choice is not just a neat architectural footnote. It is the mechanism behind the stronger high-resolution behavior.

What the paper is really claiming

The important claim is not that Mamba magically replaces transformers everywhere. The claim is narrower and more interesting: for a CLIP-style dual encoder, a sequential state-space backbone can preserve the contrastive alignment contract while improving resolution flexibility, robustness, and efficiency.

That is why the paper spends time on out-of-distribution tests and long-caption retrieval. CLIMP does not only work on the in-distribution retrieval benchmark. It also holds up on ImageNet-O, where the Mamba-2 variant reaches 49.8 top-1 accuracy, and it handles dense captioning retrieval beyond CLIP’s usual text-length limit. The strongest retrieval anchor is 65.5 IR@5 and 77.0 TR@5 for the Mamba-1 variant. Those numbers are evidence, but the deeper point is structural: the backbone choice changed the geometry of the model in a useful way.

There is also a nice intuition for why robustness improves. VMamba’s scan order encourages the model to respect spatial layout instead of treating the image as a fully exchangeable set of patches. That can reduce the chance that the model latches onto accidental global shortcuts. The paper’s analysis on shuffled images supports that idea: when the spatial order is destroyed, the advantage changes, which is exactly what you would expect from a model that depends on real spatial inductive bias.

What to keep in mind

CLIMP is a good example of a method that is simple to state but meaningful in practice.

  • It keeps the CLIP contract intact.
  • It replaces both encoders with Mamba-family backbones.
  • It uses spatial scanning for images and last-token pooling for text.
  • It gets better high-resolution behavior without needing transformer-specific resolution tricks.
  • It stays useful as the data scale rises, which is the real test for a backbone idea.

The main limitation is scale. The paper trains on CC12M and uses base-sized models, so the results are a strong architecture proof, not the final answer on LAION-2B or very large ViT-H/L comparisons. The other assumption is that the spatial order is meaningful. CLIMP’s image bias is a feature, not a free lunch. If the structure you are reading does not have coherent order, the advantage can fade.

So the shortest way to remember CLIMP is this: it is CLIP with Mamba on both sides, and that change is enough to make the model more natural at high resolution, more comfortable with long captions, and more robust when the input shifts away from the training set.
