RWKV-CLIP: Diverse Description Generation and RWKV-Driven Vision-Language Representation Learning

RWKV-CLIP is easiest to read as a two-part upgrade to CLIP. First, it repairs the text supervision before training by merging raw web text, synthetic captions, and detection tags into a richer description. Then it swaps the usual Transformer towers for RWKV towers so the model can keep the familiar CLIP dual-encoder setup while aiming for cheaper sequence modeling.

That split matters. If the captions are noisy, contrastive learning gets weaker signal than it should. If the backbone is expensive, scaling the same recipe gets harder than it needs to be. RWKV-CLIP tries to improve both sides without changing the basic CLIP contract.

Here is the whole pipeline in one glance:

The text side: repair the caption before training

The paper’s Figure 2 is the diverse description generation framework. It is easiest to think of it as a three-source text repair step. Raw web text has real-world detail, but it is often noisy, awkward, or only loosely aligned with the image. A synthetic caption from OFA-base is usually more image-grounded, but it can stay coarse and miss finer semantics. Open-set detection tags from RAM++ add concrete anchors from the image itself, which helps constrain what the language model is allowed to say.

The authors first tried to do this rewriting directly with ChatGPT. That showed the idea worked, but it was too expensive to use at dataset scale. So they turned the ChatGPT behavior into an instruction set, fine-tuned LLaMA3-8B, and then used that open-source model for large-scale rewriting. In other words, the LLM is not the end of the method. It is the offline compiler that turns three weak text views into one better training description.

The comparison image below makes the motivation easy to see.

What to notice is not just that the description gets longer. The important part is that the tags keep the language model closer to the image. They give it a checklist of facts to preserve, so the rewrite can add detail without drifting into invented scene content.

That generated description is not the only text the model ever sees. During training, RWKV-CLIP randomly samples from the raw text, the synthetic caption, and the generated description. This is a small but important trick. It keeps the text encoder robust to different phrasings, and it stops the model from overfitting to a single polished caption style.

You can think of it as text-side augmentation. The image already gets to be seen through the lens of one photograph. The text should also be seen through more than one wording.

The backbone side: CLIP shape, RWKV inside

The paper’s Figure 3 is the RWKV-CLIP architecture. The outside still looks like CLIP: one image encoder, one text encoder, and one vector from each side at the end. The change is inside the towers. Instead of stacking Transformer blocks, the model stacks RWKV blocks.

The easiest way to remember RWKV here is that it splits the work into two stages. One stage mixes information across positions, so tokens and patches can influence each other without paying the full attention cost. The other stage mixes information within each channel, so the representation can be remixed before it moves on. A gate decides how much of the mixed signal should pass through, and a learned decay controls how much older context should still matter.

That sounds low level, but the intuition is simple:

  • the spatial mixing stage is the long-range communication path,
  • the channel mixing stage is the feature remixing path,
  • the gate keeps the model from over-trusting every mixed signal,
  • the decay keeps memory flexible instead of fixed.

The image and text towers are adapted slightly differently. The image side uses local shifts that let neighboring patches exchange information in multiple directions. The text side uses a simpler two-way shift so words can talk to what comes before and after them. The goal is the same in both cases: let the model carry context forward without falling back to full quadratic attention.

This is why the architecture ablation is interesting. A hybrid setup, where only one tower is RWKV and the other stays a Transformer, is not the clean win. The paper finds that the best result comes when both towers switch to RWKV. That tells you the gain is not just “replace one block and hope.” The whole dual-encoder needs to speak the same sequence-model language.

The loss still feels like CLIP on purpose

Once the two towers produce one image embedding and one text embedding each, training returns to a standard CLIP-style contrastive loss. The paper uses the usual symmetric setup: matched image-text pairs should score higher than mismatched pairs in both retrieval directions.

That choice is deliberate. RWKV-CLIP is not trying to invent a new retrieval objective. It is trying to make the training signal better before the loss ever sees it, and then to make the encoder cheaper and more scalable once the loss starts doing its job. The familiar objective is what keeps the method easy to slot into the CLIP ecosystem.

In plain language, the method says: keep the contrastive game, but give it better text and a better backbone.

Why the results line up with the design

The reported numbers follow the story the method tells. On YFCC15M, RWKV-CLIP-B/32 reaches 74.1 average linear-probe accuracy on 10 datasets, compared with 72.2 for ALIP-ViT-B/32. It also reaches 76.0 Flickr30k text retrieval R@1 and 57.6 image retrieval R@1, plus 44.4 average zero-shot accuracy across 11 datasets.

The most useful ablation is the one on text views. Raw text alone, synthetic captions alone, and generated descriptions alone each help in different ways, but the best setting is to keep all three. That tells you the value is not in replacing one source with another. The value is in letting the model see a mixed text diet: natural web phrasing, image-grounded captioning, and the LLM-refined version.

The paper’s comparison with CapsFusion says the same thing from the data side. Detection tags make the fused description more reliable, because they constrain the language model and reduce hallucinations. That is why the proposed generator beats CapsFusion on average linear probe and zero-shot scores.

Where it can break down

RWKV-CLIP is strong, but it is not free.

  • The fused description is only as good as the caption generator and the detector.
  • The rewriting step adds preprocessing cost before training even starts.
  • The paper only tests tens-of-millions scale, not billion-pair scale.
  • It is still an open question how much of this transfer survives in multilingual or video settings.

Those limits are not surprising. The method relies on two upstream helpers, OFA-base and RAM++, and on a rewriting model that tries to clean up weak supervision instead of replacing it completely. If either helper is weak, the final text can still inherit missing details or hallucinations.

What to remember

RWKV-CLIP is a good example of a CLIP successor that does two different jobs at once:

  • it turns noisy web captions into a richer description before training,
  • it keeps the CLIP dual-encoder contract but swaps the Transformer towers for RWKV towers,
  • it leaves the contrastive objective familiar so the improvement comes from the data and backbone, not from a new loss.

The durable lesson is simple. If the training captions are noisy, you can often get more from the same data by synthesizing a better text view first. And if the Transformer backbone is the part that is getting in the way, you can replace it without giving up the CLIP-style retrieval setup.
