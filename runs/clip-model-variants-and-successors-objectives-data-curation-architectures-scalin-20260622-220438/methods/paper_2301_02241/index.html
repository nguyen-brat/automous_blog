Curation in Training: Efficient Pre-Training from Noisy Image-Text Data

CiT is CLIP with a feedback loop around the data. Instead of treating the dataset as something that gets cleaned once and then frozen, the method keeps asking a simpler question over and over: given the current model, which image-text pairs look useful for the task concepts we care about?

That changes the whole pretraining workflow. The model does not just learn from the data. It also helps decide what counts as good training data next. In practice, that means CiT can start from a noisy web crawl, keep only the pairs that look semantically relevant, train on that subset, and then repeat the selection with a better text encoder. The source is treated like a stream, not like a fixed dataset with one pass over it.

The best way to remember CiT is this: it is not a better offline filter. It is a data-selection loop that lives inside training.

The two loops

Figure 1 is the cleanest picture of the method, because it shows the idea that makes CiT different from ordinary CLIP-style pretraining.

What to notice is the split between the outer loop and the inner loop. Vanilla CLIP assumes the data distribution is fixed before training starts. CiT breaks that assumption. It uses the current text encoder to curate a better slice of the source data, then uses that curated slice to improve the model, then uses the improved model to curate again.

The image encoder is frozen, following the LiT style, so the vision backbone does not get dragged around by noisy captions. The text encoder and the projection layers stay trainable, which is important because the text side is what drives the curation decisions. That text tower is also much cheaper to run than the image tower, so using it as the curation engine keeps the extra cost manageable.

So the picture is not just a neat diagram. It shows the core design choice: move data selection from preprocessing into the learning loop.

How CiT decides what is worth keeping

The curation rule is simple enough to explain without equations.

First, CiT embeds the downstream metadata, usually class names or a taxonomy of task concepts. Then it takes a raw image-text pair, encodes the text side, and measures how close that caption embedding is to the metadata embeddings. The closest metadata match becomes the relevance score for that sample.

If the score is high enough, the pair enters the curated set.

If too few pairs pass the threshold, CiT does not stall and wait forever for better data. It falls back to keeping the best fraction of the batch by score. That fallback is the safety valve that makes the loop work on noisy or weakly aligned sources.

The threshold is the quality knob. A stricter threshold gives cleaner data, but it can leave you with too little of it. The minimum keep ratio is the quantity knob. It prevents the curation step from starving the training loop when the source data is messy or the metadata is sparse.

One subtle implementation detail matters here: the paper finds that curation works better when it uses the text representation before the projection head, not after it. That makes sense. The projection head is still moving around during training, while the earlier text embedding is already a better semantic similarity space. For the same reason, a SimCSE-tuned BERT text encoder helps more than a plain text encoder.

A good mental model is that CiT does not ask, “Is this caption perfect?” It asks, “Does this caption look close enough to the task vocabulary that it is worth spending training budget on?”

What happens inside the training loop

Once CiT has a curated subset, the inner loop looks familiar.

The model still learns a dual-encoder representation, and the contrastive idea is the same one CLIP made popular: matched image-text pairs should come together, and mismatched pairs should be pushed apart. The difference is that CiT uses the curated subset instead of the full noisy source, so the contrastive objective gets cleaner examples to work with.

CiT also simplifies the loss a bit. Instead of using the full bidirectional CLIP loss, the paper keeps only the image-to-text direction. That keeps the method simpler and, in the paper’s ablations, it is slightly better in this setting. The gain is not huge, but it fits the overall design: keep the inner loop straightforward and let the outer loop do the heavier data work.

The training loop is not run blindly over the whole source. CiT alternates curation and training periodically. The paper’s default behavior is roughly every 100 training steps, which is a practical middle ground. Curate too often and you spend all your time scoring data. Curate too rarely and the model keeps training on an outdated slice of the source.

This periodic alternation is what makes the system feel like a sieve that tightens as it learns. Early in training, the text space is still rough, so CiT keeps a broader slice of the data. Later, as the text encoder becomes better aligned with the target concepts, the curated ratio drops and the selected pairs become more precise.

Why this works on noisy web data

The paper’s main claim is not that it can make the web clean. It is that it can make the web usable.

That matters because static filtering pipelines are expensive, fragile, and often tied to one release of one dataset. CiT moves the filtering burden into the model itself. Because the text encoder is cheap, the curation signal can be refreshed often. Because the image encoder is frozen, the visual features stay stable. Because the fallback keeps a minimum number of pairs, the loop does not die when the metadata is imperfect.

Those design choices are exactly what let CiT handle raw sources. The paper shows that vanilla CLIP and LiT can become unstable on an unfiltered web crawl, while CiT still trains successfully. That is the strongest sign that the method is doing something real: it is not just benefiting from cleaner input, it is actively reshaping the input distribution while learning.

The same logic also explains why CiT can be fast. Once the model learns to select data better, it does not need to spend as much time on weak or irrelevant samples. Training budget goes into pairs that are more likely to move the downstream task needle.

What the experiments say, in plain language

The results are best read as proof that the data loop matters as much as the model loop.

On LAION400M, CiT reaches OpenCLIP-level accuracy with 37.7x less training time. On a raw web image-text crawl, vanilla CLIP and LiT are unstable, while CiT reaches 77.9% ImageNet zero-shot accuracy. On YFCC15M, CiT improves over LiT and is more than 5x faster in the cited comparison.

Those numbers matter because they point in the same direction: better data distribution can be as important as bigger scale. CiT is not a universal replacement for every pretraining recipe, but it shows that online data curation can remove a lot of the cost and brittleness that usually come with web-scale multimodal training.

Where the method is strong, and where it is not

CiT is deliberately more targeted than fully task-agnostic pretraining. It depends on metadata such as class names or a taxonomy, so it works best when you can describe the target concepts ahead of time. If the metadata is sparse, ambiguous, or poorly aligned with the source data, the curation score becomes weaker.

Threshold tuning also matters. Too strict, and the curated set shrinks too much. Too loose, and low-quality examples leak in. The fallback keeps the method from stalling, but it does not magically solve the quality-quantity tradeoff.

That is the real tradeoff in CiT. You gain speed, robustness, and control over the data distribution, but you give up some of the open-endedness of fully general pretraining. The method is strongest when you know the kind of downstream vocabulary you care about.

What to remember

CiT turns data selection into part of training.

Instead of cleaning a dataset once and hoping it stays useful, the model keeps re-evaluating the raw source with its current text encoder, keeps the pairs that look relevant, trains on them, and then repeats. The threshold decides how picky the filter is, the fallback keeps the loop from starving, and the frozen vision backbone keeps the visual side stable while the text side learns to judge relevance better.

If you want one sentence to hold onto, make it this: CiT uses the model’s own text space to decide what to learn next.
