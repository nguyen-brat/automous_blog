Multi-Modal Alignment Prompt Learning for CLIP in Multi-Task Learning

MmAP is easiest to understand as a two-part answer to the same adaptation problem. First, it refuses to let CLIP’s image side and text side drift apart, so both prompts are born from one shared source. Then it refuses to treat every task as unrelated, so related tasks share one prompt group while still keeping their own task-specific prompts.

That combination is the whole method in one sentence: keep the modalities aligned, then share that aligned structure across compatible tasks.

Why one-sided prompting is not enough

CLIP already does classification by comparing image and text embeddings in a shared space. That means adaptation is not just about adding more trainable parameters. It is about deciding where to make the smallest useful change without breaking the alignment CLIP learned during pretraining.

Figure 2 makes the design choice very clear.

What to notice here is the difference between independent prompting and coordinated prompting. Text-only tuning changes the language branch while leaving the visual branch alone. Visual-only tuning does the opposite. Even a multimodal prompt setup can still become awkward if the two branches are generated by a heavier, less structured module.

MmAP takes a stricter position. It starts from one compact source prompt and expands that same seed into both modalities. The point is not simply to have two prompts. The point is to make the two prompts share a common origin so the model keeps a coordinated text-image geometry while still allowing each branch to specialize.

Formula Card: one source prompt becomes two aligned prompts

Purpose: Generate text and visual prompts from the same latent seed while still letting each modality specialize.

Equation:

Pl=Ml⊗Ps,  Pv=Mv⊗PsP_l = M_l \otimes P_s,\; P_v = M_v \otimes P_sPl​=Ml​⊗Ps​,Pv​=Mv​⊗Ps​

Intuition: The source prompt is the shared core. The modality-specific matrices stretch that core into a text prompt or a visual prompt without making the two branches unrelated.

Symbols:

  • P_l is the text prompt.
  • P_v is the visual prompt.
  • M_l is the text scaling matrix.
  • M_v is the vision scaling matrix.
  • P_s is the shared source prompt.
  • \otimes is the Kronecker product, which expands the shared prompt in a structured way.

Read-aloud meaning: Expand one prompt seed into two branch-specific prompts using different structured scalings.

Why it matters: This keeps multimodal adaptation aligned and keeps the trainable prompt budget small.

This is also where MmAP differs from a simpler prompt generator. A standard MLP can map one prompt into another, but the paper argues that Kronecker-structured expansion preserves more of the source prompt’s information while remaining much lighter. In other words, the source prompt is not being hidden behind a large generator. It stays visible in both branches.

That detail matters because the paper is trying to solve a specific failure mode: single-modality prompt tuning can move one side of CLIP so far that the image-text match becomes less faithful. MmAP is designed to keep the correction compact and synchronized instead of letting each branch wander independently.

From a single prompt to a multi-task system

MmAP becomes more interesting once the paper stops treating the downstream setting as one task at a time. The real setting here is multi-task learning: several related domains, each with its own data, all competing for a very small adaptation budget.

The first question is whether all tasks should be treated the same. The paper says no. Related tasks should share more, while unrelated tasks should not be forced into the same prompt. To make that decision, it measures how tasks move a shared prompt during training.

Figure 3 shows the full framework.

What to notice here is the split between compatibility and specialization. The group-shared prompt captures what related tasks agree on, while the task-specific prompt protects each task from being flattened into the group average. That is why the framework has two prompt levels instead of one.

Formula Card: gradient similarity for task grouping

Purpose: Estimate whether two tasks should be grouped because they tend to help each other when they share a prompt.

Equation:

sim(Ti,Tj)=∇PglbLTi(Pglb)⋅∇PglbLTj(Pglb)sim(\mathcal{T}_i,\mathcal{T}_j) = \nabla_{P_{glb}} L_{\mathcal{T}_i}(P_{glb}) \cdot \nabla_{P_{glb}} L_{\mathcal{T}_j}(P_{glb})sim(Ti​,Tj​)=∇Pglb​​LTi​​(Pglb​)⋅∇Pglb​​LTj​​(Pglb​)

Intuition: If two tasks push the same shared prompt in similar directions, they are probably compatible. If their gradients point in opposite directions, they are more likely to fight over the same shared capacity.

Symbols:

  • \mathcal{T}_i and \mathcal{T}_j are two downstream tasks.
  • P_{glb} is a temporary global shared MmAP used to probe task compatibility.
  • L_{\mathcal{T}_i} and L_{\mathcal{T}_j} are the losses for the two tasks.
  • \nabla_{P_{glb}} means “take the gradient with respect to the shared prompt.”
  • \cdot is the dot product used to compare the two gradient directions.

Read-aloud meaning: Tasks with similar gradients on the shared prompt should probably be trained together.

Why it matters: This is the mechanism that turns multi-task sharing into a data-driven choice instead of a guess. It is the paper’s guardrail against negative transfer.

The paper does not trust a single gradient snapshot. It averages multiple similarity measurements during training so the grouping decision is less noisy. That is a sensible choice because task similarity is not static early in training, and one batch can easily be misleading.

The practical training loop is easy to remember:

Inside each task, the current group prompt and the task prompt are concatenated into the frozen encoders along with the normal tokens. The backbone never moves. Only the prompts do.

That separation is the key to the method’s balance. The group-shared MmAP collects common cues from related tasks, while the task-specific MmAP keeps each task from being over-smoothed by the group. If you only use shared prompts, you risk washing out useful differences. If you only use task-specific prompts, you lose the benefits of related-task sharing. MmAP uses both so neither failure mode dominates.

Why the Kronecker prompt helps the whole system

The prompt-generation trick and the task-sharing trick are not separate ideas. They fit together.

The Kronecker-structured source prompt gives every task a compact aligned starting point. The task grouping then decides which tasks are allowed to share that aligned prompt more aggressively. After that, the task-specific prompt restores the details that the group would otherwise blur.

That is why the paper’s design choices line up so neatly:

  • tune both modalities instead of only one,
  • use one source prompt so the two branches stay aligned,
  • compress the prompt generator with Kronecker structure,
  • share prompts across similar tasks, not across all tasks blindly.

The paper’s ablations reinforce the same story. The Kronecker version improves over a heavier MLP-based multimodal prompt baseline by about 0.54 points while using far fewer parameters. Task grouping also matters: grouping by gradient similarity beats random grouping, and it also beats lumping all tasks into one group. That is a strong sign that the sharing rule is doing real work, not just adding more shared parameters.

What the results actually tell you

The headline result is not just that MmAP works. It is that it works while staying tiny.

On Office-Home, the method reaches 86.5% average accuracy at the 10% split with only 0.13M trainable parameters. At the 20% split, it matches multi-task full fine-tuning at 87.8% average while using far fewer parameters. On MiniDomainNet, it reaches 84.9% at 1% data and 86.1% at 2% data, which is the best average performance reported in the paper’s comparison.

The more important pattern is not the exact number. It is the shape of the result. Full fine-tuning can still be competitive, but MmAP gets very close to it without touching the backbone. That tells you the method is not winning by brute force. It is winning by placing a small, aligned correction in the right place.

The ablations also show why the multi-task part matters. Similar tasks benefit from sharing. Random groups are worse. Putting all tasks in one bucket is also worse. So the method is not just “share more prompts.” It is “share the right prompts for the right tasks.”

Where to be careful

MmAP is strong, but its assumptions are visible.

It is validated mainly on cross-domain image classification. The paper does not establish the same behavior for retrieval, dense prediction, or broader vision-language backbones. It also relies on gradient-based task grouping, so if the grouping signal is poor, the shared prompt can create negative transfer instead of helping.

That means the method is best seen as a careful adaptation recipe for CLIP-style dual encoders, not as a universal multimodal adapter. Its strength comes from a very specific combination: frozen backbone, aligned prompt generation, and task-aware sharing.

What to remember

MmAP is easiest to remember as “one prompt seed, two aligned branches, then two levels of sharing.”

First, one source prompt is expanded into text and vision prompts with Kronecker-structured scaling, so the modalities stay coordinated. Then tasks are grouped by how similarly they move that shared prompt, and each group gets a shared MmAP plus a task-specific MmAP.

If you remember only one sentence, make it this: MmAP adapts CLIP by preserving alignment first and sharing only where the tasks actually agree.
