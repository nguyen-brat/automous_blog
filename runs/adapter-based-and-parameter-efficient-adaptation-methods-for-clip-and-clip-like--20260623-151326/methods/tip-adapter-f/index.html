Tip-Adapter: Training-free Adaption of CLIP for Few-shot Classification

Tip-Adapter starts from a very practical observation: CLIP is already strong enough that the few-shot problem is not always “learn a better model.” Sometimes it is simply “remember the few labeled examples you already have, then let them correct CLIP at test time.” The paper turns that idea into a tiny cache. The cache is built from the few-shot support set, queried by similarity, and then blended back into CLIP’s own zero-shot score.

That is why the method feels more like a memory system than a classical adapter. The base version of Tip-Adapter does not train a new neural module. It turns the support set into key-value pairs and uses retrieval as the adaptation step. Tip-Adapter-F keeps exactly that memory layout, then fine-tunes only the cache keys so the support examples land in a slightly better geometry.

What to notice in the pipeline is the direction of the data flow. The few-shot images are encoded once, stored as cache keys, and never turned into a deep trainable branch. At inference time, the test image does not go through a new classifier head first. It looks up nearby support examples, collects their label votes, and only then mixes that evidence with CLIP’s original judgment.

Turning a few-shot set into memory

The cache is built from the support set in the most literal way possible. Freeze CLIP’s visual encoder, run the few-shot images through it, and store the resulting normalized image features as keys. Then convert the labels into one-hot vectors and store those as values.

Purpose. Turn the labeled support set into a small key-value memory that can be queried later without training a new adapter from scratch.

Equation.

Ftrain=VisualEncoder(IK),Ltrain=OneHot(LN)\mathbf{F}_{\mathrm{train}} = \mathrm{VisualEncoder}(I_K), \qquad \mathbf{L}_{\mathrm{train}} = \mathrm{OneHot}(L_N)Ftrain​=VisualEncoder(IK​),Ltrain​=OneHot(LN​)

Intuition. Each support image becomes a point in CLIP’s embedding space, and each label becomes a vote vector. The “adapter” is not a learned tower of parameters. It is the support set itself, translated into a form that CLIP can search quickly.

Symbols.

  • I_K is the labeled K-shot support set, with K examples per class.
  • L_N is the set of class labels for the N classes in the downstream task.
  • \mathbf{F}_{\mathrm{train}} is the matrix of cached support-image features, shaped NK x C.
  • \mathbf{L}_{\mathrm{train}} is the matrix of one-hot label vectors, shaped NK x N.
  • C is the CLIP feature dimension.

Read-aloud meaning. Encode the support images with frozen CLIP, and store their labels as one-hot values.

Why it matters. This is the training-free part of Tip-Adapter. The method does not need SGD to build its first adaptation layer, because the layer is already present in the support set.

The key idea is that nothing fancy is happening yet. The cache is just a memory bank, but it is a memory bank in CLIP’s own feature space, which makes the later retrieval step cheap and meaningful.

Let the support examples vote

At test time, a query image is encoded by the same frozen CLIP visual encoder. The resulting feature vector, f_{\mathrm{test}}, is compared to every cached support feature. Because both sides are L2-normalized, their dot product is cosine similarity. Tip-Adapter then passes that similarity through an exponential sharpening function.

Purpose. Convert similarity in embedding space into retrieval weights that decide which support examples should matter most.

Equation.

A=exp⁡(−β(1−ftestFtrainT))A = \exp\left(-\beta\left(1 - f_{\mathrm{test}} \mathbf{F}_{\mathrm{train}}^{T}\right)\right)A=exp(−β(1−ftest​FtrainT​))

Intuition. This is soft nearest-neighbor retrieval. If one support example looks much closer to the test image than the rest, it should get a larger vote. If several examples are nearby, they should all contribute. The \beta parameter controls how focused that vote becomes.

Symbols.

  • f_{\mathrm{test}} is the normalized CLIP feature of the test image.
  • \mathbf{F}_{\mathrm{train}} is the matrix of cached support features.
  • A is the vector or matrix of retrieval affinities over the support set.
  • \beta is the sharpness ratio.
  • f_{\mathrm{test}} \mathbf{F}_{\mathrm{train}}^{T} is the cosine similarity between the test image and all cached support features.

Read-aloud meaning. Turn test-support similarity into a positive retrieval weight, and make the weights sharper or softer with beta.

Why it matters. Tip-Adapter is not doing a hard kNN lookup. It is using a soft vote, so the method can combine multiple nearby examples instead of being forced into a single neighbor.

Once the affinities are computed, the support labels are aggregated into cache logits and then added to CLIP’s own classification score.

Purpose. Mix support-set evidence with CLIP’s original zero-shot prior.

Equation.

logits=αALtrain+ftestWcT=αφ(ftestFtrainT)Ltrain+ftestWcT\mathrm{logits} = \alpha A\mathbf{L}_{\mathrm{train}} + f_{\mathrm{test}}W_c^T = \alpha\varphi\left(f_{\mathrm{test}}\mathbf{F}_{\mathrm{train}}^T\right)\mathbf{L}_{\mathrm{train}} + f_{\mathrm{test}}W_c^Tlogits=αALtrain​+ftest​WcT​=αφ(ftest​FtrainT​)Ltrain​+ftest​WcT​

Intuition. First ask the cache what the support set thinks. Then add CLIP’s own answer. The \alpha term decides how much the support memory is allowed to override the pretrained prior.

Symbols.

  • \alpha is the residual ratio between cache evidence and CLIP evidence.
  • \mathbf{L}_{\mathrm{train}} is the matrix of one-hot support labels.
  • W_c is CLIP’s text-derived classifier weight matrix.
  • f_{\mathrm{test}}W_c^T is CLIP’s zero-shot class score for the test image.
  • \varphi(x) = \exp(-\beta(1-x)) is the same sharpening function used in the affinity computation.

Read-aloud meaning. Final class scores are cache votes plus CLIP’s own scores, with alpha controlling the balance.

Why it matters. This residual fusion is the whole method in one line. Tip-Adapter does not throw away CLIP’s prior knowledge; it corrects that prior with retrieved few-shot evidence.

The balance between the two terms is important. If the downstream domain is far from CLIP’s pretraining distribution, the cache should matter more. If the task is close to CLIP’s world, CLIP’s original score should keep more of the vote. In that sense, \alpha is not just a tuning knob. It is the method’s way of saying how much the support set should be trusted.

Figure 3 is useful because it reveals a second way to think about the same formula. The support set is the visual cache, but CLIP’s text classifier is also a kind of memory: it encodes the class names that CLIP already knows. Tip-Adapter’s final prediction is therefore not just “support retrieval plus zero-shot CLIP.” It is a blend of two memories, one built from images and one built from text.

Why the fine-tuned version only moves the keys

Tip-Adapter-F keeps the same cache, but makes the cached keys learnable. That is the important detail. The values stay frozen as one-hot labels, and the CLIP encoders stay frozen too. The model is only allowed to move the support features themselves.

That choice makes sense if you think about what the keys do. The keys control retrieval geometry. If the keys are slightly misplaced, the test image may vote for the wrong neighbors. A short fine-tuning run can tighten that geometry without changing what the labels mean. The values, by contrast, are the class identity itself. Updating them would not make the memory better; it would make the memory less faithful.

The t-SNE view makes the effect easy to see. Before tuning, the cache already has a decent class structure because CLIP features are strong. During fine-tuning, same-class keys pull closer together and different classes separate a bit more cleanly. That is exactly the kind of change you want from a cache-based adapter: not a wholesale rewrite, just a cleaner retrieval space.

The paper’s appendix supports that design choice. Fine-tuning the cached values hurts performance, and trying to tune everything is even worse. The one-hot labels are not a trainable nuisance to be optimized away. They are the thing the cache is trying to remember.

On ImageNet, this small amount of tuning is enough to push the method further. The training-free version already improves zero-shot CLIP on 16-shot ImageNet, and Tip-Adapter-F reaches 65.51% after 20 epochs, which is notably more efficient than the older prompt- or adapter-heavy alternatives the paper compares against.

How this differs from CLIP-Adapter

Tip-Adapter and CLIP-Adapter solve a similar problem, but they live on different sides of the parametric / non-parametric boundary. CLIP-Adapter learns a tiny MLP residual branch on top of frozen CLIP features. Tip-Adapter builds the adapter directly from the support set.

The comparison in Figure 2 is the quickest way to remember the difference. CLIP-Adapter says, “learn a small correction module.” Tip-Adapter says, “use the support set itself as the correction module.” The paper even shows that the two are related mathematically if you reinterpret the learned adapter through a cache lens, but the practical distinction is cleaner than the algebra: one is a learned residual branch, the other is retrieval over real support features and labels.

That difference also explains the efficiency story. Building the cache is a single pass over the support set. Retrieval is just matrix multiplication. Tip-Adapter-F gives up the pure training-free property, but even then it only fine-tunes the cached keys for a short schedule instead of training a whole new module through the backbone.

Where it works, and where it stops

The method is strongest when the support set is representative and CLIP’s embedding space is already useful. In that regime, a cache can capture task-specific evidence cheaply and often surprisingly well. The paper also reports that the training-free version is robust under distribution shift, which fits the intuition: a cache-based correction is less likely to overfit than a larger trainable adapter.

The weak point is equally intuitive. If the support examples are sparse, noisy, or badly matched to the downstream classes, the cache can amplify the wrong neighbors. The gains can also plateau when the cache size is constrained and no tuning is allowed. Tip-Adapter-F helps by learning better keys, but that comes at the cost of giving up the pure zero-training story.

So the method is not magic. It is a carefully chosen compromise: keep CLIP’s prior, store the few-shot evidence explicitly, and only move the smallest part of the system that actually controls retrieval geometry.

What to remember

Tip-Adapter is a memory-based view of CLIP adaptation. The support images become keys, the labels become values, the test image retrieves neighbors by similarity, and the final class score is a residual blend of cache votes and CLIP’s own zero-shot judgment.

Tip-Adapter-F keeps the same idea but lets the keys move. That is why it gets more accuracy without becoming a full adapter stack. If you want one sentence to keep in your head, make it this: do not retrain CLIP when the support set can already act as the adapter.
