FedMVP: Federated Multimodal Visual Prompt Tuning for Vision-Language Models

FedMVP is easiest to understand as a refusal to trust static prompts in a federated CLIP system. When clients only see disjoint classes and each client adapts on a tiny local slice of the world, a prompt learned from seen classes can become too local to transfer. FedMVP changes the question from “what prompt should this class always use?” to “what prompt should this image ask for right now?” The answer is a tiny PromptFormer that looks at the image and at class attributes, synthesizes visual prompt tokens, and leaves the CLIP backbone frozen.

What to notice in that comparison is the direction of the adaptation. Textual prompt tuning keeps the prompt idea on the language side and lets the class name carry most of the signal. FedMVP moves the adaptation to the vision side and feeds it richer context: not just the label, but also image evidence and descriptive attributes. That is the paper’s main bet. If the local data are heterogeneous, a prompt that can change with the current image should transfer better than one that only remembers a class name.

At a high level the flow looks like this:

Why static prompts break in federated CLIP

The starting problem is not CLIP itself. CLIP already gives a strong image-text prior. The problem is what happens when you drop that prior into federated learning. Each client sees only a narrow local slice of the data, and those slices are often disjoint in class and domain. If a prompt is trained to fit that local slice too closely, it learns a static context that is good for the seen classes but weak for the classes and domains that never appeared locally.

That failure mode shows up in plain prompt tuning and even in some multimodal prompting setups. The prompt is still learned as if the world were stable, but the world is not stable in federated learning. Different users, devices, and domains all push the same tiny prompt budget in different directions. Once the prompt becomes a memorized summary of the seen clients, the aggregated model can stop behaving like a transferable CLIP adapter and start behaving like a local classifier with a frozen backbone attached.

FedMVP responds by making the prompt dynamic. Instead of learning one fixed prompt template, it generates a visual prompt for the current input by combining image evidence with semantic side information. That extra conditioning matters because not every useful cue is visible in the same way. Some classes share shape or part structure, some share texture or abstract properties, and some share both. A prompt generator that can consult both the current image and class meaning has a better chance of finding a transferable cue.

Where the attribute side comes from

The second ingredient is a set of class attributes generated by an LLM. For each class name, the paper asks for a short descriptive list of traits: things like shape, parts, texture, or other recognizable properties. A giraffe becomes a bundle of clues such as a long neck and a patterned coat, while a hen might be described by legs, feathers, and similar details. The attributes are not meant to be poetic. They are meant to be discriminative.

The paper also wraps those attributes in normal text prompts with dataset-specific prefixes, such as “a photo of a [class]” or “a sketch of a [class].” That gives the CLIP text encoder a class description that is still natural language, but richer than a bare label. The same attribute list then plays two different roles:

  • PromptFormer consumes the raw attribute embeddings to decide how to generate visual prompts.
  • The CLIP text branch consumes the composite text prompt to produce class-side features for training and inference.

That split is important. FedMVP is not just “use more text.” It is “use text in two places for two different jobs.” One part helps the prompt generator decide what to synthesize, and the other part gives the classifier a better semantic anchor.

What to notice in the examples is that the paper does not rely on a single hand-crafted phrase and hope it fits every dataset. It keeps the class name, adds attributes, and then adapts the surrounding prefix to the dataset style. That matters because a sketch, a photo, and a clipart image do not need exactly the same textual framing. The attributes carry class meaning; the prefix carries domain context.

PromptFormer: image patches ask the attributes what matters

Figure 2 shows the full pipeline. The image is encoded into patch embeddings, the class attributes are encoded into text embeddings, and PromptFormer cross-attends between the two to create visual prompt tokens. Those tokens are then appended to the frozen CLIP vision encoder input.

The easiest way to think about PromptFormer is as a tiny editor. The image patches pose the question: which semantic fragments from the class attributes are useful for this image? The attributes answer, and the feed-forward layers turn that attended signal into prompt tokens that the vision encoder can use.

Formula Card: image patches query the attribute memory

Purpose: Generate visual prompt tokens that depend on both the current image and the class semantics.

Equation:

P(A′,E)=FFN(CrossAttention(QE,KA′,VA′))\mathbf{P}(\mathbf{A}', \mathbf{E}) = \mathrm{FFN}(\mathrm{CrossAttention}(\mathbf{Q}_{\mathbf{E}}, \mathbf{K}_{\mathbf{A}'}, \mathbf{V}_{\mathbf{A}'}))P(A′,E)=FFN(CrossAttention(QE​,KA′​,VA′​))

Intuition: The image patches act like questions, the projected attributes act like a memory bank, and the prompt generator turns the attended result into prompt tokens for CLIP’s vision encoder.

Symbols:

  • \mathbf{P} is the set of generated visual prompt tokens.
  • \mathbf{A}' is the projected attribute embedding matrix.
  • \mathbf{E} is the matrix of image patch embeddings.
  • \mathbf{Q}_{\mathbf{E}} is the query projection made from image patches.
  • \mathbf{K}_{\mathbf{A}'} and \mathbf{V}_{\mathbf{A}'} are the key and value projections made from attribute features.
  • W_Q, W_K, and W_V are the learned linear maps inside the attention block.
  • \mathrm{FFN} is the small feed-forward network that turns the attended features into prompt tokens.

Read-aloud meaning: Let the image decide which attribute fragments matter, then convert that attended signal into visual prompt tokens.

Why it matters: This is the core mechanism that makes FedMVP multimodal without putting the prompt on the text side. The prompt is still lightweight, but it is no longer static.

The architecture is intentionally small. The paper uses two multi-head cross-attention blocks, two bottleneck FFNs, a linear projection from the text feature space to the patch space, and a learnable query prompt of length 4. That is enough capacity to synthesize a useful prompt, but not so much that the prompt generator becomes a second backbone.

There is also a useful intuition hidden in the cross-attention direction. If a patch looks like an animal leg, it can pull on the “four-legged” part of the attribute bank. If it looks like a neck and a coat pattern, it can pull on the corresponding semantic descriptors. Cross-attention lets the image decide which attribute fragments survive into the final prompt.

Training the prompts without moving CLIP

Once PromptFormer has produced the visual prompts, the prompted input to the vision encoder becomes the [CLS] token, the patch embeddings, and the new prompt tokens concatenated together. Everything else in CLIP stays frozen. The only trainable parameters are the PromptFormer weights. That is the main efficiency idea: keep CLIP’s pretrained knowledge intact and train only the small module that generates the prompts.

The training signal is still standard CLIP-style classification. The model compares the prompted image feature with text features made from the composite prompts, and the class with the highest similarity should win. The difference from a normal prompt-tuning setup is that the text side is not the only place where context enters. The attribute-conditioned visual prompt has already pushed the vision encoder toward the right semantics before the similarity score is computed.

The paper then adds a second term that keeps the representation stable under augmentation.

Formula Card: the consistency penalty

Purpose: Keep the prompted image representation stable when the same image is seen through a second augmentation.

Equation:

Lcon=1−cos⁡(Ev(I),Ev(x′))\mathcal{L}_{\text{con}} = 1 - \cos(\mathcal{E}_{v}(\mathbf{I}), \mathcal{E}_{v}(\mathbf{x}'))Lcon​=1−cos(Ev​(I),Ev​(x′))

Intuition: If two views of the same image are semantically the same, their vision features should stay close even after prompting.

Symbols:

  • \mathcal{E}_{v} is the frozen CLIP vision encoder.
  • \mathbf{I} is the prompted vision input, meaning the [CLS] token, the patch embeddings, and the generated visual prompts.
  • \mathbf{x}' is a second augmented view of the same image.
  • \cos(\cdot, \cdot) is cosine similarity.

Read-aloud meaning: Make the prompted view and the augmented view land near each other in feature space.

Why it matters: This keeps the prompt generator from overfitting to one exact crop or augmentation. In a federated setting, that extra stability is valuable because local data are already scarce and uneven.

Formula Card: the total objective

Purpose: Balance class discrimination with view stability.

Equation:

Ltotal=Lce+α⋅Lcon\mathcal{L}_{\text{total}} = \mathcal{L}_{\text{ce}} + \alpha \cdot \mathcal{L}_{\text{con}}Ltotal​=Lce​+α⋅Lcon​

Intuition: The classification loss teaches the model the right label, while the consistency term keeps the representation from becoming brittle.

Symbols:

  • \mathcal{L}_{\text{ce}} is the CLIP cross-entropy classification loss.
  • \mathcal{L}_{\text{con}} is the consistency penalty from the previous card.
  • \alpha is the balance between the two terms.

Read-aloud meaning: Train for the correct class and for stable features at the same time, with alpha deciding how much weight the stability term gets.

Why it matters: This is the actual training objective for PromptFormer. The paper uses alpha = 10 in its experiments, and the sensitivity analysis shows that pushing alpha too high can weaken the classification signal.

Formula Card: the federated aggregation step

Purpose: Combine the PromptFormer updates coming back from the selected clients in each communication round.

Equation:

ρr+1=1∣Sr∣∑i∈Srρir+1\rho^{r+1} = \frac{1}{|S_r|}\sum_{i \in S_r} \rho_i^{r+1}ρr+1=∣Sr​∣1​i∈Sr​∑​ρir+1​

Intuition: Each client nudges the prompt generator toward its local data, and the server averages those nudges into one global prompt generator.

Symbols:

  • \rho^r is the PromptFormer parameter set at round r.
  • S_r is the set of clients selected in round r.
  • \rho_i^{r+1} is the updated parameter set returned by client i.
  • |S_r| is the number of participating clients in that round.

Read-aloud meaning: Average the prompt-generator weights returned by the participating clients.

Why it matters: FedMVP is still a federated method. The novelty is not that it skips aggregation. The novelty is that it federates only the lightweight prompt generator instead of the whole backbone.

There is one more practical detail that matters in heterogeneous clients. Some clients have so little data that repeatedly fine-tuning the full PromptFormer can overfit badly. FedMVP handles that by switching to LoRA on the query, key, and value matrices once the local training loss drops below 0.5. In that case, the client freezes the main PromptFormer parameters and trains only the low-rank adapters, then shares those LoRA weights instead. The paper reports that this cuts the trainable parameter count by about 267x for those clients.

That fallback is a good sign. It shows the paper is not treating all clients as equally easy. Some clients can safely train the full prompt generator; smaller clients get a narrower, safer update path.

What the results say

The main pattern in the experiments is that richer contextual prompts help generalization, especially when the evaluation distribution changes. On the base-to-new setting, FedMVP reaches a harmonic mean of 78.27, which is 1.57 points better than FedTPG. On DomainBed multi-source single-target, it reaches 73.02 on average, about 1.99 points better than the next best method. On ImageNet domain generalization, it reaches 60.91 average and gains 3.28 points on ImageNet-Sketch. On the cross-dataset setting, it reaches 67.57 average and beats the next best baseline by 2.26 points.

Those numbers matter less than the shape of the pattern. FedMVP is not just better on the clients it saw. It is better where static prompts usually fail: unseen classes, unseen domains, and unseen datasets. That is exactly what you would hope if the prompt generator is really using image-specific cues together with semantic side information instead of memorizing a fixed local template.

The ablation study makes the story even cleaner. PromptFormer alone already helps, adding the consistency loss helps a little more, and the full method with the LoRA fallback is best. The paper also compares different prompting placements and finds that visual prompting is stronger than textual or multimodal prompting inside this federated setup. That is an important clue: in this problem, the vision side seems to be the best place to spend the prompt budget.

The limitations are just as informative. FedMVP still depends on the quality of the LLM-generated attributes. If the attributes are noisy, sparse, or too similar across fine-grained classes, the cross-attention can become confused. The paper notes that performance can fall below zero-shot CLIP on some fine-grained datasets such as OxfordPets and EuroSAT, which is a reminder that better prompts do not automatically solve overlapping semantics. There is also a communication tradeoff: FedMVP sends more parameters per round than FedTPG, even though it converges much faster.

What to remember

FedMVP is a prompt-tuning method that refuses to let the prompt be static. It turns class attributes into semantic side information, lets the current image query those attributes through a tiny PromptFormer, and injects the result as visual prompts into a frozen CLIP vision encoder. The backbone stays fixed, the prompt generator gets federated, and the model gets a better chance of generalizing beyond the local clients that trained it.

If you keep one sentence from the paper, keep this one: when static prompts overfit in federated CLIP, the fix is not a bigger prompt template, but a prompt that can change with the image and with the class semantics.
