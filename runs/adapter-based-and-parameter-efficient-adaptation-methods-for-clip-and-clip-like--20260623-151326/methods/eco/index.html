ECO: Ensembling Context Optimization for CLIP Few-Shot Classification

ECO is easiest to remember as prompt-budget diversification. CoOp-style prompt learning usually spends the whole trainable budget on one learned prompt. ECO keeps that budget fixed, but splits it across several shorter prompts, lets each prompt produce its own text embedding for every class, and then averages those embeddings before CLIP does the final image-text match. The backbone stays frozen, and the averaged class embeddings can be precomputed, so the method improves few-shot classification without changing inference cost.

What to notice in the figure is the budgeting choice. CoOp puts all learned context into one block, while ECO spreads the same number of learned tokens across multiple blocks. The paper’s claim is not that longer prompts are bad in general. It is that, under a fixed parameter budget, a small committee of prompts can be more useful than one larger prompt.

Why a single prompt can be too narrow

CLIP already knows how to compare images and text, but the way you phrase the class name still matters a lot. A tiny wording change can move accuracy, which is why hand-crafted prompt engineering became such a big deal in the first place. Prompt ensembling helped because different wordings produce slightly different text features, and the average is often more robust than any single phrasing.

ECO takes that old idea and makes the prompts learnable. Instead of manually writing several versions of “a photo of a [CLASS]”, the model learns several context blocks from data and lets them play the role of the ensemble. That is the key conceptual shift: the ensemble is no longer a hand-built list of prompts, but a set of learned views over the same class.

Intuition: CLIP does not predict a class directly. It scores how well the image embedding matches each class’s text embedding, then turns those similarities into probabilities.

Symbols: I is the input image. K is the number of classes. \psi_I = f_{\theta}(I) is the image feature from the frozen visual encoder. \psi_T^k is the text feature for class k. \tau is the temperature. \cos(\cdot, \cdot) is cosine similarity.

Read-aloud meaning: Compare the image to every class text feature, sharpen the similarities with temperature, and normalize them into class probabilities.

Why it matters: ECO does not change this classifier. It changes only how \psi_T^k is built, which is why the method fits cleanly into CLIP.

How ECO builds one class feature from many prompts

The core of the method is simple once you see the data flow. ECO freezes CLIP’s image encoder and text encoder, then learns multiple sets of context tokens. For class k, each learned prompt is paired with the class token and sent through the frozen text encoder. That gives one text feature per prompt. ECO then averages those text features into a single class prototype.

The paper also keeps the prompt parameters shared across classes. In other words, the learned context blocks are not class-specific parameters. The same learned prompts are reused with every class token, which keeps the model compact and forces the prompts to capture reusable structure rather than memorizing one label at a time.

Intuition: Each prompt bank gives CLIP a slightly different way to phrase the same class. ECO keeps those views separate long enough for the text encoder to process them, then averages the resulting embeddings into a single class prototype.

Symbols: D is the number of prompts. N is the number of context tokens per prompt. v_{i1}, \ldots, v_{iN} are the learned context tokens for prompt i. c_k is the class-token embedding for class k. g_{\phi} is the frozen CLIP text encoder. \psi_T^k is the final text feature used for class k.

Read-aloud meaning: Encode each learned prompt with the class name attached, then average the D text embeddings into one vector.

Why it matters: The ensemble happens in feature space, so the result is still just one class vector. That keeps inference cheap and makes the ensemble behave like a consensus representation instead of a runtime voting system.

This also explains why the paper averages text features rather than logits. The mean happens before the image is scored, so the output is already a single reusable embedding. If the ensemble were only combined after class scoring, you would keep the cost of evaluating every prompt branch. ECO avoids that by collapsing the branch outputs into one prototype per class.

Training is ordinary, but only the prompts move

Training is straightforward. The paper uses cross-entropy on the few-shot labels, backpropagates through the text encoder’s computation graph, and updates only the learned context tokens. The CLIP backbone stays frozen throughout. That choice keeps the method in the parameter-efficient regime and prevents the few-shot data from overwriting the pretrained representation.

At inference time, nothing exotic happens. The averaged text features do not depend on the input image, so they can be computed once and cached. The classifier then behaves like standard CLIP classification: take the image feature, compare it to the stored class text features, and pick the most similar class. That is why ECO improves accuracy without adding inference overhead.

Why the split between prompt count and prompt length matters

The interesting part of ECO is not just “more prompts.” It is the way the same fixed prompt budget gets divided. One long prompt has more room for detailed context, but it is still only one view of the class. Several shorter prompts give the model more diversity, but each prompt becomes thinner if you split the budget too aggressively.

That is the tradeoff the paper studies. The number of learnable tokens stays fixed, but changing how many prompts you use and how long each prompt is changes the answer. That means the split itself is part of the design, not a bookkeeping detail.

The paper’s best average setting is D = 4 prompts with N = 4 tokens each. That is a useful middle ground. It gives enough diversity for an ensemble effect, but each prompt still has enough context to say something meaningful. If you make the prompts too tiny, they stop carrying useful class information. If you use only one prompt, you lose the diversity that ensembling gives you.

The result is a very clean intuition: under a fixed budget, distribution matters. ECO is not asking for more parameters. It is asking you to spend the same parameters differently.

What the experiments are really showing

The headline result is not just that ECO wins somewhere on a leaderboard. The important pattern is that the method helps most when data is scarce. The paper reports an average gain of +3.47 points over CoOp at 1 shot, and the gains stay positive from 1 to 16 shots. That is exactly the regime where prompt diversity should matter most, because one tiny support set does not give you much room to overfit a single phrasing.

The paper also reports that the best setting is not the longest prompt or the largest ensemble. It is the balanced split. That is a good sanity check: the method is not “more is always better.” It is “a better partition of the same budget can matter more than concentrating everything in one prompt.”

One more useful takeaway is that ECO still works within the standard CLIP pipeline. It does not need a new backbone, a new image branch, or a new inference-time voting scheme. That makes it a good example of a design that improves data efficiency without changing the basic CLIP machinery.

What to keep in mind

ECO is validated mainly in the CoOp-style prompt-learning setting and on few-shot classification. The paper suggests that the idea should extend to other single-prompt learners, including methods like CoCoOp and MaPLe, but that transfer is left open. The optimal split between prompt count and prompt length is also empirical, not derived from a closed-form rule.

So the method is best understood as a design principle rather than a one-off trick: if the prompt budget is fixed, you do not have to spend it all on one long learned context. You can distribute it across several shorter prompts, let them vote in feature space, and keep the frozen CLIP backbone exactly where it is.

If you want one sentence to remember, make it this: ECO turns prompt tuning into prompt ensembling, and under the same parameter budget, that simple change is often enough to make CLIP a better few-shot learner.
