CapS-Adapter: Caption-based Multimodal Adapter in Zero-Shot Classification

CapS-Adapter is trying to fix the support memory, not the CLIP backbone. The paper starts from a simple complaint: training-free CLIP adapters often treat a tiny support set as if it were a faithful slice of the target domain, but the stored images can be repetitive, narrow, or just too far from what the test set actually looks like. CapS-Adapter changes that memory before inference even begins. It captions a few target samples, turns those captions into prompts, generates more target-like support images, and then caches both the image features and the caption features. At inference time, a test image consults that multimodal memory instead of an image-only cache.

That is why the method matters. Across 19 benchmark datasets, the paper reports 64.94% average accuracy, which is 2.19 points above the previous leading training-free method.

Figure 2 is the whole pipeline

Figure 2 is the cleanest way to see the method as a whole.

What to notice is the split. The left half is support-set construction. The right half is inference. CapS happens before any prediction, so the cache already carries more of the target domain when the test image arrives. M-Adapter then reads that cache with both image and text features instead of forcing everything through one image-only memory.

Start by making the target samples speak

The first move is captioning a small set of target-training images per class with a multimodal large language model, ShareCaptioner. The class prompt is kept deliberately plain: “A photo of .” for most datasets, and “In .” for Country211. The point is not prompt engineering. The point is to attach an instance-level clue to the class name.

The paper concatenates the class prompt and the caption into what it calls a caption-based prompt, or CBP. That prompt carries both pieces of information at once:

  • the class identity from the class prompt
  • the instance-level detail from the caption

This matters because the class name alone is too generic. A caption can say something about pose, background, style, or texture that the label does not mention. That extra detail is what gives the text-to-image model a better starting point for generating support images that look like the target domain instead of a generic internet version of the class.

The generation step is also practical. The paper uses Stable Diffusion v1-4, and when the same caption-based prompt has to be reused many times, it changes the random seed so repeated prompts do not collapse into copies.

Figure 3 shows why the synthetic support set is less repetitive

Figure 3 makes the main visual argument immediately obvious.

The old support sets look too safe. For apple pie, they keep gravitating toward one round shape. For arctic tern, they keep producing the same kind of static image. CapS is more varied: different pie shapes, more image styles, and both static and dynamic bird shots. That is the whole point of using instance-level captions. The captions give the generator enough detail to diversify the support set instead of cloning one narrow template.

CapS is better in feature space too

The paper does not stop at visual examples. Figure 4 shows the image part of CapS in CLIP feature space, and the same story appears there as well.

What to notice is that the CapS features are closer to the target distribution and more spread out, while the SuS-X style support features are tighter and farther away. The paper also measures this directly with average CLIP similarity, and CapS comes out higher than the SuS-SD baselines. So this is not just a prettier set of generated images. It is a cache that better matches the data the model will actually see.

The cache keeps both views of the same support item

Once the support set exists, CapS does not throw half of it away. The generated images are encoded with CLIP’s visual encoder, and the caption-based prompts are encoded with CLIP’s text encoder. So every support item carries two cached vectors: one visual and one textual.

That is the key difference from image-only caches. A test image can now match the support memory by appearance, by caption semantics, or by both. In practice, that means the cache has a visual side for texture and style and a text side for the instance-level meaning that the generator used to make the image in the first place.

Purpose: combine text-image similarity and image-image similarity when matching a test image to the CapS memory.

Equation:

AM=exp⁡(−β(1−δftestt⋅Fcap−(1−δ)ftestt⋅Fimg))\mathbf{A}_{M}=\exp(-\beta(1-\delta f^{t}_{\text{test}}\cdot\mathbf{F}_{cap}-(1-\delta)f^{t}_{\text{test}}\cdot\mathbf{F}_{img}))AM​=exp(−β(1−δftestt​⋅Fcap​−(1−δ)ftestt​⋅Fimg​))

Intuition: the query image should score highly when it matches the caption-derived support features, the image-derived support features, or both. The two similarity paths are blended instead of forcing the cache to behave like an image-only nearest-neighbor store.

Symbols:

  • AM\mathbf{A}_{M}AM​: multimodal association scores between the test images and cached support items
  • β\betaβ: sharpness factor that makes the scores more or less selective
  • δ\deltaδ: balance between text-feature similarity and image-feature similarity
  • ftesttf^{t}_{\text{test}}ftestt​: CLIP visual features for the test images
  • Fcap\mathbf{F}_{cap}Fcap​: cached text features from the caption-based prompts
  • Fimg\mathbf{F}_{img}Fimg​: cached image features from the generated support images

Read-aloud meaning: each support item gets a higher score when the test image agrees with its caption side, its image side, or both, with δ\deltaδ deciding which side matters more.

Why it matters: this is the core change from image-only training-free caches. The cache is no longer just a list of support images. It is a multimodal memory that can be read through both text-aligned and image-aligned similarity.

The paper treats δ\deltaδ as a real tuning knob, not a cosmetic constant. Different datasets prefer different balances between the two views, which is exactly what you would expect if some domains are easier to capture through captions while others are easier to capture through appearance.

M-Adapter keeps zero-shot CLIP in the loop

M-Adapter is built on the same residual fusion idea as TIP-X, but it swaps in CapS’s multimodal cache. The zero-shot CLIP classifier still provides the base logits from the class prompts. The cache then adds a correction term on top of that base score instead of replacing it. That is important: the method is trying to nudge CLIP, not overwrite it.

Formula card: Equation 12, final multimodal logits

Purpose: combine zero-shot CLIP logits, the multimodal cache vote, and the TIP-X-style intimacy correction.

Equation:

logits=ftestt⋅WT+αAML+γφ(−ML)\text{logits}=f^{t}_{\text{test}}\cdot\mathbf{W}^{T}+\alpha\mathbf{A}_{M} \mathbf{L}+\gamma\varphi(-\mathbf{ML})logits=ftestt​⋅WT+αAM​L+γφ(−ML)

Intuition: start with CLIP’s own class scores, add the multimodal cache vote from CapS, and keep the intimacy-based correction from TIP-X in the mix.

Symbols:

  • ftesttf^{t}_{\text{test}}ftestt​: CLIP visual features for the test images
  • W\mathbf{W}W: the zero-shot CLIP text classifier built from the class prompts
  • AM\mathbf{A}_{M}AM​: the multimodal association matrix from Equation 11
  • L\mathbf{L}L: the one-hot label matrix for the support set
  • M\mathbf{M}M: the TIP-X intimacy matrix
  • φ\varphiφ: the scaling function that matches the range of M\mathbf{M}M to the other terms
  • α\alphaα: residual weight for the cache vote
  • γ\gammaγ: residual weight for the intimacy correction

Read-aloud meaning: final class scores equal CLIP’s original prediction plus a multimodal support-set vote plus a scaled intimacy term.

Why it matters: CapS-Adapter stays training-free at inference time because the heavy lifting already happened when the support set was built. The adapter is in the memory and in the readout rule, not in a new trainable backbone.

What Table 2 isolates

Table 2 is the cleanest ablation in the paper because it separates the two ideas.

Replacing SuS-X style support images with CapS lifts average accuracy from 62.75% to 63.72% under TIP-X. Adding M-Adapter on top lifts it again to 64.94%. So the gain comes from two different places:

  • better support construction through caption-guided synthesis
  • better inference through multimodal cache association

That is a useful pattern to remember. CapS is not just a better generator, and M-Adapter is not just a better scorer. The method works because both pieces line up: the support set is closer to the target domain, and the inference rule knows how to read both the image and the caption side of that support set.

What the method costs

The tradeoff is straightforward. CapS-Adapter depends on a captioner and a text-to-image model, so it adds an offline support-construction step before inference. If captions are generic or hallucinated, or if the generator collapses into repetitive images, the cache drifts and larger support sets stop helping as much. The paper also tunes δ\deltaδ per dataset, so the balance between text and image similarity is still a hand-set knob rather than an automatic rule.

That is the right way to think about the limitation. CapS-Adapter is strong because it makes a better synthetic memory bank, but it is still only as good as the captioning and generation pipeline that builds that memory bank.

What to remember

Build a small synthetic memory bank that looks and sounds like the target domain, then let test images retrieve from both the visual and textual views. That is CapS-Adapter in one sentence.

The deeper lesson is simple: when a support set is the adapter, its quality matters as much as the fusion rule. CapS improves the support set itself, and M-Adapter gives CLIP a multimodal way to read that support set back.
