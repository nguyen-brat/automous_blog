DGL: Dynamic Global-Local Prompt Tuning for Text-Video Retrieval

DGL is a very specific answer to a very specific problem: CLIP already knows how to align images and text, but video retrieval asks it to do two things at once. It has to keep the word-level alignment that CLIP is good at, and it has to stop thinking of a video as just a bag of separate frames. DGL keeps CLIP frozen, then learns a small prompt system that steers both branches together. One part of that system couples text and frame prompts through a shared latent space. The other part makes the visual encoder read the clip at two scales, one local and one global.

That combination is the whole method. DGL is not a new video backbone. It is a way to make a frozen CLIP backbone behave more like a video model without paying the cost of full fine-tuning.

This overview figure is the right place to start because it shows the method as a control system. The frozen text encoder is on the left, the frozen visual encoder is in the center, and the trainable prompt logic sits between them. The fire icons mark the only parts that learn. What to notice is that DGL does not let text prompts and frame prompts evolve independently. They are generated together, so the text side and the video side stay coupled while the model adapts.

Why plain prompt tuning is not enough for video

The paper begins from a simple complaint about existing prompt-tuning methods for text-video retrieval. Some methods only add learnable text prompts. Others add separate prompts to the text and visual branches, but still train them as if they were unrelated. That works better than doing nothing, but it leaves two gaps.

The first gap is cross-modal alignment. If the text prompts and frame prompts are optimized separately, they can drift apart. The model may still retrieve by rough semantic overlap, but it is not really learning a shared adaptation strategy.

The second gap is video understanding. CLIP was pretrained on image-text pairs, so its visual encoder is naturally biased toward single-frame evidence. In video retrieval, that is often too weak. A caption like “a golf player hits the ball into the pit” is not just about one frame. It is about a short chain of events. If the model only watches one frame at a time, it may notice the golfer, or the ball, or the pit, but miss the action that connects them.

DGL is built to close both gaps at once. It learns prompts in a way that forces the two modalities to share structure, and it gives the visual encoder a way to summarize both local frame details and the whole clip.

A shared prompt space, not two separate prompt islands

DGL starts with deep prompts. That is an important detail. Instead of adding a single prompt only at the input, it places prompts through the encoder stack. The text branch gets prefix prompts and postfix prompts at each layer. The visual branch gets frame prompts at each layer, plus a global prompt that is only introduced once and then carried forward.

The point of using deep prompts is not just extra capacity. For video retrieval, the model needs to adjust how it reads the clip at several depths, because local object cues and temporal cues emerge at different layers. A shallow prompt would be too blunt for that.

The interesting part is how DGL generates those prompts. The paper gives two variants.

The first is DGL-Transformer. It merges the text prompts and frame prompts into one small unified prompt transformer, lets them interact, then splits them back into text-side and visual-side prompts. This is the expressive version. It is still small, but it gives the two modalities a real place to talk to each other.

The second is DGL-Linear. This is the minimalist version, and the one that makes the cross-modal design easiest to see. It uses the frame prompts as the source, then maps them into the text prompts with two shared linear projections, one for the prefix and one for the postfix. The paper deliberately chooses the visual-to-text direction because video prompts are richer and more complex than text prompts. In other words, the video side has more structure to donate, so the text side borrows from it.

Purpose: Reuse one prompt basis for both modalities so the adaptation stays coupled instead of splitting into two unrelated branches.

Read-aloud meaning: Use shared linear layers to turn a frame prompt into the text prompts that wrap the query at the same layer.

Why it matters: This is the cheapest version of DGL’s cross-modal coupling. It keeps the text branch and the visual branch from drifting apart, but it does so with almost no trainable state.

You can think of this shared prompt space as a small control room. The prompts are not the model’s knowledge by themselves. They are the knobs that decide how the frozen backbone should read the current task. DGL’s contribution is that the same control logic shapes both modalities.

How the video branch reads a clip twice

The other half of the method is inside the visual encoder. This is where DGL stops treating a video as a sequence of independent frames and starts treating it as both a set of details and a story.

This figure is the key to the video side. The top path is local frame attention. The bottom path is global video attention. The visual shorthand is useful: the local path is for inspecting one frame closely, while the global path is for collecting evidence across the whole clip. What matters is that the two paths are not separate heads with unrelated weights. They share the same attention parameters, so the model learns one reusable video-reading mechanism and applies it at two scales.

Local frame attention is the microscope. Each frame prompt is used to inspect the local frame tokens, while the global prompt is also copied into the local attention input so the frame does not lose sight of the broader clip. That is a nice design choice because local reasoning is not supposed to be blind. Even when the model zooms in on one frame, it still carries a weak notion of the whole video.

Global video attention is the storyline. The global prompt is the query, and it reads from all frame tokens and prompts. That lets it accumulate a summary of the clip instead of just a snapshot. In practice, this is what helps DGL capture actions that unfold over time. The global prompt is not a static CLS token copied from image classification. It is a token that is allowed to collect motion evidence from the whole sequence.

The shared-parameter setup matters here. If local attention and global attention had separate parameter sets, the visual branch would become heavier and less coherent. DGL instead reuses the CLIP visual attention weights and changes the query-key layout to change the behavior. The result is a very efficient kind of video adaptation. The model learns two views of the clip, but it does not double the visual encoder.

Why the first global prompt becomes the video representation

After the video passes through the global-local attention stack, DGL does not average everything and call it done. It uses the first global prompt as the final video embedding.

That choice looks small, but it is revealing. The first global prompt is the token that has been explicitly designed to gather information from the whole clip. Because it has already been exposed to both local frame detail and global context, it is a natural place to read out the final video representation. Averaging the global prompts or averaging the local prompts dilutes that role. The paper’s ablation shows the first global prompt works best, which fits the intuition: the representation should come from the token that was trained to summarize the video, not from a post-hoc average of everything.

Once the text embedding and the global video embedding are produced, the similarity step is refreshingly plain. DGL does not add a heavy temporal fusion head or a query-specific re-ranking module. It uses the frozen CLIP geometry and a simple inner-product similarity. That keeps retrieval fast and query-agnostic, which matters when you want to score many text-video pairs without running a second pass.

What the training actually changes

Training is deliberately narrow. The CLIP text encoder and visual encoder stay frozen. Only the prompt parameters and the local prompt generator are updated. The loss is the standard symmetric cross-entropy used in CLIP-style retrieval training, so the model still learns to align matched text and video pairs, but it does so through a tiny set of trainable vectors and projections.

That is why the efficiency story is so strong. The paper reports that DGL-Transformer reaches 45.8 R@1 on MSR-VTT with 9.57M trainable parameters, while DGL-Linear uses only 0.83 MB of trainable parameters and still stays competitive with much heavier baselines. The exact number is less important than the shape of the tradeoff: DGL is trying to get video-aware retrieval without abandoning the CLIP prior or spending a fine-tuning budget on the whole model.

What to remember

DGL is best remembered as a shared-prompt, two-scale attention method.

The shared-prompt part says that text prompts and frame prompts should be generated together, so the two modalities adapt in the same direction instead of growing apart. The two-scale attention part says that a video needs both local frame detail and a global clip summary, and that the same visual backbone can support both if the query layout is changed carefully.

The method is strong because it is disciplined. It preserves CLIP’s pretrained geometry, adds a small amount of cross-modal control, and reads the video through both a microscope and a storyline token. That is enough to make frozen CLIP substantially better at retrieval.

The limits are just as important to keep in mind. DGL is still a retrieval method, not a generation method. It still depends on sampled frames and on CLIP’s image-centric pretraining. And if the task needs much richer motion reasoning or longer temporal structure, prompt-only adaptation may eventually hit the ceiling of the frozen backbone.

So the mental model is simple: when you need to adapt a frozen multimodal model for video, do not let the text side and the video side live in separate prompt worlds. Tie them together with a shared prompt generator, then let the video branch read the clip once at the frame level and once at the story level.
