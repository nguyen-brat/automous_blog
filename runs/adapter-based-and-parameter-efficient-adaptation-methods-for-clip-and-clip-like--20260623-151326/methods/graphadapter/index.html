GraphAdapter: Tuning Vision-Language Models With Dual Knowledge Graph

GraphAdapter is easiest to understand as CLIP adapter tuning with a neighborhood map. Instead of adjusting each class embedding in isolation, the paper first builds two class graphs, one in text space and one in visual space, then lets the current class prompt borrow structure from both graphs before classification. The goal is not to replace CLIP. The goal is to keep CLIP as the anchor and add a structure-aware correction where few-shot data is too sparse to discover class relationships on its own.

That first picture is the key shift. Earlier adapter-style methods usually correct one side of CLIP at a time. GraphAdapter says the missing ingredient is not just another small module, but a better way to represent how classes relate to each other. If a few-shot task never shows you enough evidence that “wolf” and “dog” live near each other, or that some aircraft classes are visually close but semantically distinct, then the adapter has to invent that structure somewhere. GraphAdapter chooses to make it explicit.

Why a single-modality adapter runs out of room

The paper starts from a practical complaint about low-data adaptation. Prompt tuning changes the classifier through trainable prompts, but it still treats each class embedding as a separate item. Adapter-style methods are cheaper than full fine-tuning, but most of them still adapt either the text side or the image side without modeling class-to-class relations directly.

That leaves a blind spot. In few-shot learning, the model does not just need a better class vector. It needs a better view of the class neighborhood. The important question is not only “what does this class mean?” but also “which other classes should it be allowed to borrow evidence from?” GraphAdapter is built around that question.

Build the class map once

The method builds a dual knowledge graph, and the graphs are class-level rather than instance-level. That matters because the model wants to learn structure that is stable across the whole task, not a fresh graph for every image.

The pipeline figure is worth reading slowly. The left branch is the textual knowledge graph, the right branch is the visual knowledge graph, and both feed the same adapted classifier. The graph is not rebuilt at every step. The class nodes are created once, then reused throughout training, which keeps the method light enough to fit the efficient-transfer setting.

The two graphs play different roles:

  • The textual graph captures how class names and prompts relate in CLIP’s language space.
  • The visual graph captures how classes relate in feature space when you average the available training images.

That split is not cosmetic. Text tends to give you cleaner semantics. Vision tends to give you more fine-grained appearance cues. GraphAdapter keeps both because the downstream task can benefit from either kind of structure.

Why it matters: GraphAdapter wants its graph structure to match CLIP’s own geometry. If CLIP already believes two classes are close, the graph should let them exchange information more easily.

The text graph is built from prompt embeddings. For a class, the model takes the prompt feature, and the class node becomes the mean feature for that class. The visual graph does the same thing, but with image features instead of text features. In practice that means the visual graph is built from class-wise visual prototypes, which are often more sensitive to appearance-level relationships than the text graph.

Let the prompt ask both graphs for help

Once the two class graphs exist, the current prompt embedding is treated like a node that can temporarily join each graph. This is the part that makes GraphAdapter feel more like graph reasoning than a plain residual MLP.

Think of the prompt feature as a visitor entering two neighborhoods. In the text neighborhood, it can ask semantically close classes for help. In the visual neighborhood, it can ask visually similar classes for help. After that message passing step, the prompt is no longer a lonely class vector. It has been nudged by the local class structure around it.

The paper uses a graph convolution network for that refinement. The important idea is not the exact algebra. The important idea is that the prompt feature is inserted into the graph, message passing happens over similarity-weighted edges, and the output at the prompt node becomes a structure-aware version of the original text feature.

The paper names the two refined prompt features z_tt and z_vt:

  • z_tt is the prompt after reading from the textual graph.
  • z_vt is the prompt after reading from the visual graph.

That naming is easy to remember if you read it literally. The first t is the textual prompt feature. The second symbol says which graph refined it.

This is also where the method becomes “dual knowledge” rather than just “two graphs.” The prompt does not only read from text neighbors and only read from visual neighbors separately. It is allowed to collect both inner-modality structure and cross-modality structure. Inner-modality means text-to-text or image-to-image relations inside one graph. Cross-modality means the text prompt is being corrected by the visual graph as well.

The final classifier stays close to CLIP

After graph propagation, GraphAdapter does not throw away the original CLIP feature. It blends the graph-refined features back into the original text embedding in two stages: first text-vs-vision, then adapted-vs-original.

  • β\betaβ controls how the two graph corrections are mixed.

Read-aloud meaning: the final classifier embedding is part original CLIP and part graph-refined correction.

Why it matters: this is the safety mechanism of the method. The graph can adjust the classifier, but it cannot completely overwrite CLIP unless you ask it to.

The two weights have distinct jobs. beta decides whether the text graph or the visual graph should dominate the correction. alpha decides how much the correction should move the original CLIP prompt feature at all. The paper finds that fixed coefficients work better than learning them directly in the few-shot setting, which makes sense: with very little data, even a small balance parameter can be hard to optimize reliably.

One useful detail from the experiments is that the cross-domain variant leans more heavily on the original text feature. For generalization, the paper increases alpha to keep more of the CLIP prior and reduce overfitting. That is a good way to remember what alpha really does. A bigger alpha means “trust CLIP more.”

Why the text branch is the main story

GraphAdapter is designed to support both textual and visual adaptation, but the paper’s ablations make the ranking pretty clear. The text-branch version, GraphAdapter-T, is the main winner. The visual-branch-only version is noticeably weaker, and using both branches together gives only a tiny improvement over the text branch alone.

That result is important because it tells you what kind of structure is most useful here. The graph is not valuable just because it is a graph. It is valuable because it gives the text classifier a better class neighborhood. The visual branch still helps, but mostly as a supporting signal.

This also explains why the method stays focused on textual adaptation in the main paper. If the text side already contains most of the transferable class semantics, then the best use of the graph is to refine that side rather than split attention across two separate adapters.

The headline numbers line up with that story. On 16-shot learning across 11 datasets, GraphAdapter reaches 76.22 average accuracy and beats TaskRes and Tip-Adapter-F. That is a strong result, but the deeper point is that the gain comes from modeling relations between classes, not from simply adding more trainable capacity.

What the training loop actually changes

The training recipe is deliberately narrow. CLIP’s encoders stay frozen. The class graphs are built once. The trainable parameters are the graph convolution layers that refine the prompt node inside each graph. Training then uses the usual cross-entropy loss against the downstream labels.

That is why GraphAdapter still counts as efficient transfer learning. It does not require full fine-tuning. It does not keep rerunning the text encoder every iteration. And it does not add a large amount of task-specific state. The extra work is mostly in the graph construction and the small GCN update.

The paper also checks the method on different CLIP backbones and on cross-domain ImageNet variants. The point of those tests is not just to show that the method works on one benchmark. It is to show that the structural correction survives when the backbone changes and when the target distribution shifts. The generalization variant, called Ours_g in the paper, is a reminder that the right amount of residual correction depends on the domain shift. When the target is farther from the source, you usually want to lean a little more on the frozen CLIP prior.

Where it can break

GraphAdapter is strong, but it is not free of assumptions. The most obvious limitation is that the textual graph depends on simple hand-crafted prompts. If the prompt set is weak, the textual structure is weak too. The paper also notes that graph quality can degrade when class prototypes are noisy, which is exactly the kind of problem you expect in a scarce-label regime.

There is also a practical asymmetry in the method. The visual graph is useful, but the textual branch does most of the heavy lifting. So the method is not really “balanced dual-branch adaptation” in the way a symmetric two-tower system would be. It is more like a text classifier that consults vision structure when it needs a better correction.

That tradeoff is probably the right one for this paper. The class graph is there to stabilize the classifier, not to become a second giant model.

What to remember

GraphAdapter is a class-structure adapter for CLIP. It builds one graph from text prototypes and one from visual prototypes, lets the current prompt node read from both, and then folds the result back into the original text feature with a residual mix.

If you want a one-line memory hook, use this:

the model does not adapt classes one by one; it adapts the class neighborhood and keeps CLIP as the anchor.
