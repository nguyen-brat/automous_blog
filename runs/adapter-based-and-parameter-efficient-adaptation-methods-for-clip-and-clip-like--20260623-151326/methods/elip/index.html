ELIP: Enhanced Visual-Language Foundation Models for Image Retrieval

ELIP is easiest to understand if you start from the part of retrieval that CLIP-like models are already good at. They are usually strong at pulling a broad shortlist of relevant images. The weakness shows up after that first pass, when the system has to separate near-misses: a parasailer versus a wakeboarder, an occluded bicycle versus a different object with a similar silhouette, or a target class that is visually present but easy to overlook. ELIP is built for that second problem. It is a reranker that keeps the backbone frozen and uses the query text to steer how the image encoder reads each shortlisted candidate.

That design choice is the whole paper in miniature. Instead of learning a bigger joint model, ELIP turns the text query into a small set of visual prompt tokens, inserts those prompts into the frozen image encoder, and recomputes the image feature before scoring. So the query does not just ask for a similarity score. It temporarily changes how the image is encoded.

What to notice in this figure is the direction of control. The text branch does not become a separate late scorer. It produces prompt vectors that enter the vision transformer early, so the image representation itself becomes query-aware before reranking happens.

One shortlist, then a second look

The method only makes sense in a two-stage retrieval pipeline. The first stage is the usual fast retriever, which scans the gallery and produces a top-k shortlist. ELIP does not try to replace that stage. It assumes the shortlist is already reasonably good and asks a narrower question: among these few candidates, which ones are actually the best match once the query is allowed to guide the image encoder?

That is why the method is lightweight. It does not need to rerun the whole gallery through an expensive cross-modal model. It only recomputes features for the shortlisted candidates, and it does so with a tiny text-to-prompt module rather than a full fine-tuned backbone.

The main intuition is simple: if the problem is hard-negative discrimination, the best place to spend extra compute is not on a bigger global model. It is on a sharper second look at the candidates that are already close.

How the text becomes visual prompts

The central trick is a small mapping network. The text encoder first turns the query into a compact summary, and then a trainable MLP converts that summary into a handful of prompt vectors in the visual embedding space. Those prompt vectors are appended to the image tokens before the frozen ViT runs again.

Formula Card: Text-to-Prompt Mapping

Purpose. Turn the query text into visual prompt tokens that can steer the frozen image encoder during reranking.

Equation.

[tp1,…,tpm,tCLS]=Φt(T),v=Φv([xp1,…,xpn,xCLS;ψmap(tCLS)])[t_p^1, \dots, t_p^m, t_{\texttt{CLS}}] = \Phi_t(T), \quad v = \Phi_v([x_p^1, \dots, x_p^n, x_{\texttt{CLS}}; \psi_{\mathrm{map}}(t_{\texttt{CLS}})])[tp1​,…,tpm​,tCLS​]=Φt​(T),v=Φv​([xp1​,…,xpn​,xCLS​;ψmap​(tCLS​)])

Intuition. The text encoder gives you a compact summary of the query. ELIP then uses that summary to generate a small set of visual prompts. Those prompts are placed next to the image tokens, so the image encoder rereads the candidate through the lens of the query.

Symbols.

  • T is the input text query.
  • \Phi_t is the frozen text encoder.
  • t_{\texttt{CLS}} is the global text summary token from the text encoder.
  • \psi_{\mathrm{map}} is the trainable MLP that maps text information into visual prompt vectors.
  • t_p^1, \dots, t_p^m are the generated visual prompt tokens.
  • x_p^1, \dots, x_p^n are the image patch tokens.
  • x_{\texttt{CLS}} is the visual CLS token.
  • \Phi_v is the frozen visual encoder.
  • v is the recomputed image feature used for reranking.

Read-aloud meaning. Encode the query text, turn its summary into visual prompt tokens, append those tokens to the image tokens, and run the frozen image encoder again to get a query-aware image embedding.

Why it matters. This is the mechanism that keeps ELIP parameter-efficient. The backbone does not need to be rewritten. The text query only needs to learn how to steer the image encoder.

The notation in the paper is compact, but the practical flow is straightforward: text summary in, prompt tokens out, prompted image encoding back in. That is the whole adapter.

Two details in the design are worth pausing on. First, the prompts are inserted at the first ViT layer, not later. That matters because early prompting can influence the entire attention cascade, while late fusion only tweaks a representation that is already mostly formed. Second, the paper finds that more than one prompt token is helpful. A single token is too cramped for the amount of information a query can carry. Ten prompt tokens performed best in the reported ablations because they give the mapper enough room to spread different parts of the query across several visual slots.

You can think of those prompt tokens as a tiny steering wheel rather than as a new sentence. They are not meant to replace the text. They are meant to bias which visual evidence the frozen encoder pays attention to.

Why the reranker is stronger than a late score tweak

ELIP is not doing post-hoc score smoothing. It is changing the image representation before scoring. That difference is easy to miss, but it is the reason the method works on hard negatives.

If you only adjust the final similarity score, the image encoder has already committed to its representation of the candidate. ELIP instead asks the image encoder to reinterpret the candidate with the query present as a visual prompt. That early conditioning is a better fit for retrieval, where the important question is often not “is this image globally relevant?” but “which tiny cue makes this near-match the right one?”

This also explains why the method is especially useful for reranking. The first-stage retriever does the broad search. ELIP spends its compute on the narrower discrimination problem that remains after broad recall has already done most of the work.

What changes for BLIP-2

The same text-to-prompt idea also works with BLIP-2, but the tail of the model is different. BLIP-2 does not use the same pure dual-encoder scoring path as CLIP or SigLIP. In ELIP-B, the prompted image features are passed through BLIP-2’s Q-Former and then into an image-text matching head.

What to notice here is that the front end is still the same idea: text becomes visual prompts, and the image encoder is recomputed with those prompts. The difference is what happens next. Instead of using a simple dot product between text and image embeddings, ELIP-B lets the Q-Former and the ITM head produce the final match score. The paper also fine-tunes the lightweight ITM head, because the features reaching it have changed.

At inference time, the BLIP-2 variant first produces an initial ranking from the original model, then reranks the shortlist using the prompted image features and the ITM score. So the core story remains unchanged. Only the scoring tail changes.

How the model learns to care about hard negatives

The reranking trick only helps if the model sees enough difficult comparisons during training. ELIP therefore spends real effort on batch construction.

The paper’s first move is global hard sample mining. Instead of drawing random batches, it builds batches around a reference text and collects examples whose images are highly similar under CLIP features. That creates batches filled with lookalikes, which is exactly what a reranker needs. Random batches are usually too easy; they do not force the model to learn the fine distinctions that matter at the top of the ranking.

What to notice in this figure is that the batches are intentionally crowded with semantically close examples. The first item is not just a seed. It is a way of pulling in hard negatives so the model has to learn a sharper boundary.

ELIP then uses a different training loss depending on the backbone:

  • ELIP-C trains with InfoNCE on the text CLS token and the recomputed image features.
  • ELIP-S and ELIP-S-2 use a pairwise Sigmoid loss.
  • ELIP-B uses binary cross entropy through the ITM head.

The exact loss changes, but the teaching signal is the same. The trainable prompt mapper has to produce visual prompts that make the prompted image feature line up better with the query than the hard negatives do.

For ELIP-B, the paper adds one more selection step inspired by JEST. After grouping hard samples, it keeps the batches with the highest learnability, meaning the ones where the learner still has more room to improve than the reference model does. That is a practical way to spend limited compute on the batches that are most worth learning from.

Why the method is useful beyond one backbone

The interesting thing about ELIP is not that it wins on one model. It is that the same mechanism works across several frozen backbones. The paper reports gains for CLIP, SigLIP, SigLIP-2, and BLIP-2, which is a strong sign that the idea is architectural rather than backbone-specific.

The reported numbers are best read as evidence for the reranking idea, not as the main point of the paper. ELIP-C raises CLIP on COCO from Recall@1 40.16 to 45.61. ELIP-S raises SigLIP on Flickr average Recall@1 from 82.96 to 87.62. ELIP-S-2 fine-tuned reaches 85.05 average mAP on Occluded COCO and ImageNet-R. ELIP-B also slightly surpasses earlier BLIP-2 reranking baselines while keeping the trainable part small.

The more durable lesson is that query-conditioned visual prompting is a good fit when the base retriever already has decent recall but still fails on hard negatives. In that regime, the safest way to adapt is to keep the backbone semantics stable and only let the query steer the image encoder.

Where it stops

ELIP is not a magic fix for retrieval. It has a few clear limits.

The first limit is inherited from the pipeline itself: if the first-stage retriever misses a relevant image entirely, ELIP cannot recover it, because it only reranks the shortlist it is given. The second limit is cost. Reranking top-k candidates means extra image encoding work, so ELIP is more expensive than a one-pass similarity lookup. The third limit is scope. The strongest evidence in the paper is for retrieval and OOD reranking, not for arbitrary multimodal tasks.

Those limits are not flaws in the idea. They are the natural price of doing a more careful second pass. ELIP is intentionally narrow: it spends a little extra compute where the ranking is hardest.

What to remember

ELIP is a query-conditioned reranker. The text query is encoded once, turned into visual prompt tokens, inserted into the frozen image encoder, and used to recompute the image feature before reranking. That simple move is what lets the method sharpen hard negatives without full fine-tuning.

If you want one mental image to keep, make it this: the query does not replace the image encoder. It leaves a small note for the encoder, then asks it to look again.

For the family-level pattern this method belongs to, see Cross-Modal Prompt Coupling for Adapting Frozen CLIP Models.
