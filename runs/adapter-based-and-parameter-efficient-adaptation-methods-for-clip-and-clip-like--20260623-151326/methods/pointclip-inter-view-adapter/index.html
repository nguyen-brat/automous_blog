PointCLIP: Point Cloud Understanding by CLIP

PointCLIP is a simple idea that solves a hard mismatch. CLIP already knows how to match images and text, but a point cloud is neither a natural image nor a dense 3D tensor. Instead of retraining a 3D backbone from scratch, PointCLIP first turns the point cloud into several depth-map views, then lets frozen CLIP score those views, and finally adds a tiny inter-view adapter when a few labeled 3D examples are available.

That order matters. The paper does not try to teach CLIP an entirely new modality all at once. It first makes the data look like something CLIP already understands, and only then learns the smallest extra module needed to close the gap between 2D pretraining and 3D recognition.

What to notice in this figure is the split between two regimes. The solid path is the zero-shot route: project the point cloud, run CLIP, vote across views. The dotted path is the few-shot route: keep that same pipeline, but slip in a small adapter that lets the views share context before the final prediction.

First make the point cloud look like an image

A point cloud is sparse and unordered, so it is awkward input for a 2D vision-language model. PointCLIP sidesteps that by using online perspective projection. Raw points are projected onto a fixed set of image planes, and the pixel values store depth rather than color.

This is a deliberately cheap bridge. There is no mesh rendering, no texture synthesis, and no learned image preprocessor that tries to invent a more realistic picture. The point is not visual realism. The point is compatibility. Once the 3D object has been converted into a handful of depth maps, CLIP can read it using the same machinery it already uses for ordinary images.

The paper also keeps the projection lightweight by writing the depth value directly into all three channels. That means the point cloud is represented as a scatter-like depth image rather than a fully rendered RGB scene. The result still preserves the rough 3D shape from multiple angles, which is enough for CLIP to start reasoning about the object.

Zero-shot is just CLIP, repeated across views

Once the point cloud has been projected, the zero-shot part is almost embarrassingly simple. Each depth map is encoded by CLIP’s visual encoder. The class names are placed into a prompt, passed through CLIP’s text encoder, and turned into class weights. Then each view is scored against those weights.

A single view follows CLIP’s usual scoring rule

Purpose: Use CLIP’s normal image-text alignment rule to turn one projected depth map into class scores.

Equation:

logits=fvWtT,pi=softmaxi(logits)\mathrm{logits}=f_v W_t^T,\qquad p_i=\mathrm{softmax}_i(\mathrm{logits})logits=fv​WtT​,pi​=softmaxi​(logits)

Intuition: The visual encoder gives one feature for the depth map. The text encoder gives one feature per class name. Classification is just the similarity between the two.

Symbols: f_v is the visual feature for one view, W_t is the matrix of text features for all classes, \mathrm{logits} is the vector of class scores, and p_i is the predicted probability for class i.

Read-aloud meaning: Multiply the view feature by the class-name features, then turn those scores into probabilities.

Why it matters: PointCLIP reuses this exact CLIP scoring rule on each projected view, which is what makes the method a transfer method instead of a new 3D classifier from scratch.

The prompt matters because the text side is part of the classifier, not decoration. The paper finds that the class prompt should describe the projected representation as a depth map, so the text encoder is aligned with what the visual encoder is actually seeing. If the prompt sounds like an ordinary photograph, the text side drifts away from the 3D bridge the model is trying to build.

Multiple views vote together

Purpose: Fuse the per-view predictions into one point-cloud prediction.

Equation:

logitsi=fiWtT,logitsp=∑i=1Mαilogitsi\mathrm{logits}_i=f_i W_t^T,\qquad \mathrm{logits}_p=\sum_{i=1}^{M}\alpha_i \mathrm{logits}_ilogitsi​=fi​WtT​,logitsp​=i=1∑M​αi​logitsi​

Intuition: Each view sees a different side of the object, so each view gets a vote. But the votes are not equally useful, so the paper weights them.

Symbols: f_i is the CLIP visual feature from view i, M is the number of projected views, \alpha_i is the weight of view i, and \mathrm{logits}_p is the final point-cloud logit vector.

Read-aloud meaning: Score every view separately, then add the scores together with view-specific weights.

Why it matters: A point cloud is not a single image. The weighted sum is what turns a set of partial depth maps into one prediction for the whole object.

This zero-shot path is already useful, but it still has a limitation: the views never really talk to each other. Each depth map gets its own score, and the final answer is only a weighted aggregate. That is fine when the pretrained CLIP prior is already close to the target domain, but it leaves performance on the table when a few labeled examples are available.

The inter-view adapter is where few-shot learning happens

The few-shot version keeps the same projection bridge and the same frozen CLIP encoders, but adds a small inter-view adapter on top of the multi-view features. This is the paper’s main adaptation trick.

The motivation is easy to understand. A single projected view only sees part of the object, and different views can miss different cues. If the model can build one shared summary of all the views, it can send that summary back to each view and correct the parts that were too local or too ambiguous. That is exactly what the adapter does.

What matters in this figure is the direction of information flow. The views are not treated as independent mini-models. They are first compressed into one global representation, and then that global representation is used to improve every view through a residual update.

First compress all views into one global feature

Purpose: Gather the evidence from every projected view into a compact object-level summary.

Equation:

fglobal=ReLU⁡(Concat⁡(f1,…,fM)W1T)W2Tf_{\mathrm{global}}=\operatorname{ReLU}(\operatorname{Concat}(f_1,\dots,f_M)W_1^T)W_2^Tfglobal​=ReLU(Concat(f1​,…,fM​)W1T​)W2T​

Intuition: Concatenate the view features, squeeze them through a bottleneck, and expand them back into CLIP width. The adapter is learning “what the object looks like overall,” not just “what one view looks like.”

Symbols: f_1,\dots,f_M are the view features, \operatorname{Concat} joins them along the channel dimension, W_1 and W_2 are the first two linear layers of the adapter, \operatorname{ReLU} is the nonlinearity, and f_{\mathrm{global}} is the shared object feature.

Read-aloud meaning: Stack the view features together, compress them into a global summary, then lift that summary back to the CLIP feature size.

Why it matters: This is the “inter-view” part of the adapter. The model learns from all views at once instead of trying to patch each view in isolation.

Then send that global summary back to each view

Purpose: Use the global object summary as a residual correction for each individual view.

Equation:

fia=fi+ReLU⁡(fglobalW3iT)f_i^a=f_i+\operatorname{ReLU}(f_{\mathrm{global}}W_{3i}^T)fia​=fi​+ReLU(fglobal​W3iT​)

Intuition: Keep the original CLIP feature, but add a learned correction built from the whole object. The adapter does not replace CLIP’s representation; it nudges it.

Symbols: f_i is the original feature of view i, f_i^a is the adapted feature for that view, W_{3i} is the part of the third adapter layer assigned to view i, and f_{\mathrm{global}} is the shared summary computed above.

Read-aloud meaning: Take each view feature, add a correction derived from the global object feature, and use the result as the adapted representation.

Why it matters: This residual path is what lets PointCLIP keep CLIP’s pretrained knowledge intact while injecting a small amount of 3D-specific context learned from the few-shot data.

Training stays intentionally narrow. CLIP’s visual and textual encoders remain frozen, and only the lightweight adapter is trained with the few labeled point clouds. The paper also lets the view weights become learnable in the few-shot setting, so the model can adapt not just the view features but also how strongly each view should count in the final vote.

Why this small adapter helps so much

The striking thing about PointCLIP is how much improvement comes from a very small amount of training. The zero-shot version on ModelNet40 reaches 20.18%, but the inter-view adapter pushes 16-shot PointCLIP to 87.20% while keeping CLIP frozen. That is not a tiny gain from a tiny module. It is a sign that the main missing ingredient was not model size, but a better interface between CLIP’s 2D prior and the 3D target data.

The adapter helps because it learns what single views cannot. Some angles carry more shape evidence than others. Some views are redundant. Some views become useful only when they are interpreted together. The global bottleneck gives the model a place to reconcile those differences before it makes a final decision.

The paper also shows that PointCLIP is useful as a companion to classical 3D networks, not just as a standalone classifier. Because PointCLIP brings in CLIP’s contrastive 2D knowledge, its errors are not identical to those of a fully supervised 3D model. Simple logit addition can therefore improve stronger 3D baselines such as PointNet++ and CurveNet instead of just averaging the same kind of evidence twice.

Where the method is fragile

PointCLIP is powerful, but it is not magic.

  • It is specialized to 3D classification, not a general 3D foundation model.
  • Its zero-shot accuracy is still modest, especially on noisy real-world scans.
  • Projection choices, prompt wording, and view weighting matter more than they do in ordinary image classification.
  • If the depth-map bridge loses important geometry, the adapter cannot fully recover what never reached CLIP in the first place.

That is the real tradeoff in the paper. PointCLIP saves you from retraining a big 3D network, but it also asks you to be careful about how you convert 3D into 2D.

What to remember

The easiest way to remember PointCLIP is as a five-step loop:

  1. Project the point cloud into multiple depth-map views.
  2. Let frozen CLIP score each view against text prompts.
  3. Weight and aggregate the view scores.
  4. In few-shot mode, add a tiny inter-view adapter that builds one global feature and feeds it back to every view.
  5. Keep the backbone frozen and train only the small bridge.

If you want a one-line summary, it is this: PointCLIP turns a modality mismatch into a view-fusion problem, then solves the remaining gap with a small residual adapter.

For the wider design pattern this method belongs to, see Domain- and Modality-Transfer PEFT.
