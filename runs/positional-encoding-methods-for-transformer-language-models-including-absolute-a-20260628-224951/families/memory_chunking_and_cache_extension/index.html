How Transformers Stretch Context with Memory, Chunking, and Cache Reuse

The hard part of long-context language modeling is not just “making the window bigger.” It is deciding what the model should keep active, what it can compress, what it can fetch later, and what it can safely ignore. Once prompts become book-length or decode sessions become very long, full attention turns into the wrong abstraction: too much memory, too much noise from distant tokens, and too much repeated work when the same context must be reused across turns.

That is why this area is better understood as an interface problem rather than a single algorithm. The common theme is not one shared mechanism. It is a shared bottleneck: distant context becomes distracting, replaying old context becomes expensive, positional limits make naive extension brittle, and the KV cache grows until throughput collapses. Different methods attack different points in that interface.

The Recurring Long-Context Pipeline

Many of these systems fit the same high-level flow:

The details change, but the pattern stays recognizable. Some methods keep a local sliding window and move older KV states into a memory archive. Some split context into chunks and build a side memory that the decoder can consult later. Some treat the KV cache itself as a recurrent state and simply carry it forward across chunks. Others keep the cache bounded by evicting low-value history or by compressing old segments into learned latent vectors.

What matters is the contract at the interface:

  • Local tokens must stay faithful, because nearby syntax and discourse still dominate next-token prediction.
  • Long-range dependencies must remain recoverable, even when old tokens are no longer fully active.
  • Per-step compute must stay bounded enough for streaming decoding.
  • Causal order cannot be broken, even if storage format changes.

Why Long Context Fails in Practice

The first failure mode is distant-context distraction. A model may technically “see” old tokens but still fail to use them because too much irrelevant history competes for attention. This is the main lesson behind InfLLM and Focused Transformer: long context is useful only if the model can separate signal from archive noise.

The second failure mode is context reuse cost. If a user wants to keep returning to a long document, replaying the entire token stream every time is wasteful. Methods like Cartridges and Recurrent Context Compression treat long context as something that should be distilled into a reusable state rather than reread verbatim.

The third failure mode is context length scaling. Pretrained decoders often break when asked to operate far beyond the positional regime they saw during training. CEPE and Parallel Context Windows show two different escapes: do not stretch one global positional sequence if you can instead encode chunks separately or reuse positions across isolated windows.

The fourth failure mode is attention-memory scaling during generation. Even if retrieval quality stays high, the KV cache can grow linearly with every new token. KV-Fold accepts that growth to preserve exact retrieval, while MorphKV forces a fixed budget by keeping recent tokens plus a selected subset of older ones.

The Main Design Axes

The most useful way to organize this space is by design choices, not by paper names.

1. What is the memory unit?

A system can remember old context as contiguous KV blocks, individual retained tokens, compressed latent vectors, or chunk-level encoder states.

  • InfLLM stores evicted history as contiguous KV blocks, then summarizes each block with a few representative tokens.
  • MorphKV keeps actual tokens, but only a curated set of distant ones plus a recent local window.
  • RCC turns segments into learned vectors that condition the decoder later.
  • CEPE turns chunks into encoder states and uses those states as a side memory bank.

This choice determines both retrieval quality and storage cost. Blocks preserve exact detail inside retrieved regions. Learned vectors save more memory but risk hiding task-critical facts.

2. Is the local path strong or weak?

Most successful systems keep a strong local path. Nearby tokens are cheap to use and often carry the tightest syntactic and discourse constraints. The distant path should therefore be selective, not symmetric.

InfLLM uses local sliding-window attention and adds a relevance-gated distant memory path. MorphKV explicitly reserves part of the cache for recent tokens. Even when RCC compresses long history, it still treats the decoder as the place where current token generation happens. The pattern is durable: keep local detail hot, make distant access conditional.

3. How is old context retrieved?

Some methods retrieve by learned or emergent relevance, others by architectural routing.

  • InfLLM performs block-level top-k lookup from representative tokens.
  • Focused Transformer trains memory attention to retrieve relevant context while resisting distractors.
  • CEPE avoids token-by-token retrieval during reading by pre-encoding chunks and exposing them through cross-attention.
  • Parallel Context Windows avoids retrieval altogether during context reading by isolating windows and letting the final task tokens attend globally.

This axis matters because the bottleneck is often not raw capacity but selection quality. If the model cannot find the right old evidence, a bigger archive does not help.

4. How much compression is acceptable?

Compression can be hand-designed or learned, coarse or fine, reusable or one-shot.

  • Cartridges aim for a reusable learned state that can stand in for prior raw context.
  • RCC performs segment-wise learned compression and then transfers the compressed representation into decoder layers.
  • MorphKV does not learn a new latent memory; it compresses by selection under a fixed token budget.

The tradeoff is clear: stronger compression cuts replay cost and cache growth, but every compression step creates a new failure mode where later reasoning depends on details that were discarded.

5. Where do positions live?

Long-context systems often work by refusing to represent one gigantic global position range.

  • CEPE uses local positions inside each chunk and shifts the burden to encoder-decoder alignment.
  • Parallel Context Windows reuses the same positional range in multiple windows and blocks cross-window context attention.
  • InfLLM keeps far-context access compatible with its memory lookup instead of relying on the model to extrapolate cleanly across massive unseen positions.

This is a deep point: many context-extension wins come from rerouting information flow, not from teaching the model a longer number line.

How Selective Memory Works

InfLLM is a clean example of the archive-and-reload idea. Instead of asking the model to attend over the entire past, it keeps recent context active, evicts older KV states into memory blocks, summarizes each block, and reloads only the few blocks that match the current query.

When Reuse Matters More Than Raw Length

Some workloads do not just need longer prompts. They need persistent state that can be carried into later interactions. That shifts the goal from “fit more tokens once” to “reuse what was learned from earlier context.”

Cartridges represents this idea in a pure form. It tries to distill long context into a lightweight reusable state, learned through self-study, so downstream processing can consume the cartridge instead of replaying the full sequence. The appeal is obvious: long context becomes an object you can keep and reuse, not just a prompt you must resend.

RCC pushes the same intuition into an encoder-decoder compression pipeline. Instead of preserving every token in the live cache, it compresses each segment into learned vectors and passes those vectors to the decoder.

This family of ideas is powerful, but it changes the problem. Once context is compressed, instruction handling becomes part of memory design. RCC explicitly separates instruction reconstruction from answer generation because a compressed memory that keeps facts but loses task intent is not enough.

When the Real Limit Is Positional Plumbing

Another branch of the field starts from a blunt observation: sometimes the model does not need a perfect long sequential workspace. It needs a better route from distant evidence into the decoder.

CEPE does this by encoding extra context in parallel chunks with a small bidirectional encoder, concatenating the resulting states, and letting each decoder layer cross-attend to that side memory.

Parallel Context Windows reaches a related destination with even less machinery. It reuses positions inside multiple separate windows, blocks context-to-context interaction across those windows, and lets the final task tokens attend to all of them. The insight is that independent evidence can be processed in parallel channels. What you gain is breadth; what you lose is rich interaction among distant windows while reading.

Cache Growth Is a Systems Problem, Not Just a Modeling Problem

Generation creates a separate long-context problem: even if retrieval stays accurate, the KV cache can dominate memory and bandwidth.

KV-Fold takes the simplest possible stance. Keep the model frozen, split the sequence into chunks, and carry the accumulated KV cache forward unchanged. In effect, the cache becomes a recurrent state. This is attractive when exact long-range retrieval matters more than bounded memory.

MorphKV makes the opposite trade. Keep the last recent tokens for coherence, let those recent tokens vote on which older tokens still matter, and retain only a fixed-budget set of distant survivors.

This difference is important because it exposes a real design fork:

  • If your priority is exact recovery of old details, append-only cache growth may be acceptable.
  • If your priority is stable long-form generation under a strict memory budget, eviction and retention policy become first-class design choices.

The same fork appears in other forms across the family: preserve exact history in a larger structure, or preserve only the parts of history that still earn their place.

What the Evidence Actually Supports

The compact input gives the clearest direct evidence for selective retrieval methods, especially InfLLM. Three anchors are enough to ground the overall lesson.

  • InfLLM reportedly maintains 100% passkey retrieval accuracy up to 1,024K tokens, which supports the claim that selective archive reload can preserve very long-distance access without retraining.
  • On infinity-Bench, Mistral-based InfLLM reaches 57.7 average score versus about 21.5 for sliding-window streaming baselines, which suggests that retrieval quality, not just window size, is driving the gain.
  • CEPE reaches 9.90x throughput at 128K total tokens in the provided summary, which supports the architectural claim that side encoding can reduce long-context cost substantially.

The input also includes useful qualitative ablation evidence. For InfLLM, removing lookup nearly destroys distant retrieval, and using lookup only during decoding underperforms using it during both encoding and decoding. That matters because it shows memory retrieval is not just a late-stage answer trick; it helps while the model is reading as well.

For the rest of the family, the evidence in this compact packet is thinner and more uneven. That is enough for tutorial use, but it is not enough to make every compression or cache strategy look equally mature.

What to Remember

Memory, chunking, and cache extension methods are all trying to answer one practical question: what should remain live when a transformer cannot afford to keep all past tokens equally active?

The strongest recurring pattern is not “use a bigger context window.” It is:

  • keep local processing reliable,
  • move distant context into a different storage form,
  • make retrieval or reuse selective,
  • and choose explicitly between exact history, compressed history, and bounded-memory history.

That is why these methods matter. They turn long context from a monolithic prompt-length problem into a set of storage and routing decisions that can be engineered.
