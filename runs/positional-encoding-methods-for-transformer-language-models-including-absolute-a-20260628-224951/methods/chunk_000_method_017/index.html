Dialogue Without Limits: Constant-Sized KV Caches for Extended Responses in LLMs

MorphKV is a cache-management method for long decoding. The problem it attacks is simple but expensive: every new generated token adds more key-value pairs to memory, so the KV cache grows linearly with response length. On short answers that is manageable. On long responses it becomes the bottleneck, because the model starts spending bandwidth and memory on a huge pile of past tokens, many of which no longer matter.

The paper’s basic move is to stop treating the cache as a passive log of everything the model has seen. Instead, MorphKV keeps a small recent window for local continuity and uses those newest tokens as witnesses for what should survive from the distant past. In plain terms: the latest tokens vote on which old tokens are still useful.

That is the whole method in one loop. Everything else in the paper is about making that vote trustworthy enough to preserve quality while keeping memory bounded.

Before opening MorphKV, it helps to see what it is pushing back against.

What to notice here is the difference in selection philosophy. Scissorhands is almost purely recent. StreamingLLM keeps a few early anchors, but it still mostly leans on recency. H2O is more conservative and keeps many old tokens, which protects context but leaves less room for compression. Keyformer and SnapKV are smarter about importance, but they still decide based on earlier attention patterns in ways that do not fully adapt as generation continues. MorphKV’s twist is that it keeps only the older tokens that the newest tokens still correlate with.

That is why the paper frames the cache as two compartments instead of one undifferentiated list.

What matters in the figure is not just that the cache is smaller. It is that the cache has a job split:

  • The recent window carries local coherence, so the model does not lose the immediate conversational flow.
  • The distant bucket holds only the older tokens that still look useful according to the recent tokens’ attention patterns.

That split is the method’s main design choice. MorphKV is not trying to compress everything evenly. It is trying to preserve the few older tokens that are still structurally alive in the current generation.

The ideal thing we would like to do

The paper first states the problem as an optimization question. If we could afford to search every possible subset of the cache, we would keep the subset that changes the attention output as little as possible.

So the real question becomes: how do we cheaply estimate which old tokens matter most right now?

MorphKV’s answer is beautifully simple. It assumes the newest tokens already carry compressed evidence about the past, because those tokens were generated after attending to earlier context. If several recent tokens keep leaning on the same older token, that older token probably still matters. If they ignore it, the token is probably safe to evict.

Why it matters. This is the core ranking mechanism behind MorphKV. It does not ask which old tokens were globally popular long ago. It asks which old tokens are still being used by the newest context.

The fusion rule is the only real knob inside this step. In the paper, sum fusion favors old tokens that keep getting repeated attention across the recent window, while max fusion favors old tokens that receive one especially strong signal from any recent token. That difference matters because long-response generation and long-context retrieval do not reward the same kind of evidence.

With the scores in hand, the cache update is straightforward.

Why it matters. This is how MorphKV stays constant-sized. The cache does not accumulate forever; it continuously morphs as generation advances.

A small walk-through makes the idea click

Figure 5 turns the scoring rule into something you can watch happen step by step.

The easiest way to read the example is as a tiny voting problem. At one step, the recent tokens pay much more attention to one older token than another, so the higher-scoring old token survives and the weaker one is evicted. A few steps later, the recent window changes, so the score changes too. If an old token is still useful, it can survive for many steps. If it stops mattering, it falls out of the cache.

That dynamic behavior is the key difference from recency-only retention. MorphKV is not saying older tokens are bad. It is saying old tokens should stay only if the current neighborhood still needs them.

This also explains why the method can preserve long-range structure in open-ended generation. If a story keeps returning to the same entities or facts, recent tokens keep re-attending to the relevant older tokens, and those tokens remain in memory. If a detail was useful only briefly, it gets replaced by newer evidence.

What changes with multiple heads

MorphKV is designed to work with both multi-head attention and grouped-query attention.

In ordinary multi-head attention, each head has its own KV cache. MorphKV can prune each head independently, which is conceptually simple but memory-heavy. In grouped-query attention, several query heads share key-value heads. MorphKV first aggregates the attention signals from the grouped query heads, then scores the shared KV entries. That fits modern models better because they already use GQA to reduce memory pressure.

The practical consequence is that MorphKV is not tied to one specific attention layout. It can run with MHA, and it fits GQA well enough that the paper uses GQA by default for memory efficiency.

Why the method works better than a crude heuristic

The main lesson is that relevance beats age. A naive recent-only rule throws away too much long-range context. A prompt-only rule keeps too much stale context. A popularity rule based only on old tokens can become biased toward early, noisy, or overly generic tokens. MorphKV avoids those traps by letting the newest tokens decide what still matters.

That is also why the method can sometimes do better than full attention in practice. Full attention keeps everything, but not everything is useful. Extra tokens can create redundancy and noise, and that can show up as repetition or drift in long generation. The paper reports the lowest repetition rate among the compared settings for this reason: pruning is not only about shrinking memory, it can also make the remaining context cleaner.

On the reported long-response tasks, MorphKV is not just smaller. It is better on the accuracy-memory tradeoff. The paper reports 52.9% memory savings and 18.2% higher average accuracy over H2O on long-response tasks, and the system trade-off table shows 4.68x throughput relative to SnapKV with slightly better reported accuracy. The useful point is not the exact number by itself. It is that the method manages to cut the cache without turning the generation process into a brittle compression game.

Where the method is fragile

MorphKV is still a pruning method, so it has the usual cost of pruning. Per-step eviction adds runtime work, because the selector has to read attention profiles and update the cache continuously. The paper reduces that overhead with implementation tricks, but the extra work does not disappear.

The design also depends on two choices that really matter: window size and fusion function. If the recent window is too small, it may miss details that become important much later, especially in structured long-response tasks where the model needs to remember specific parts of a plan or outline. The fusion choice matters too. Sum fusion tends to be better when the task benefits from broad evidence across the recent window. Max fusion can be better when a single strong signal is enough, especially in retrieval-heavy long-context settings.

So MorphKV is best thought of as a tunable memory policy, not a universal compression theorem. It works because it uses the model’s own recent attention as a relevance signal, but it still needs reasonable settings.

It is also worth saying plainly that the paper is about cache management rather than positional encoding itself. The method lives in the inference path: it changes what gets kept, not how positions are embedded inside the transformer.

What to remember

MorphKV is easiest to remember as a loop:

  • keep the last RRR tokens for local coherence,
  • let those tokens vote on which older tokens still matter,
  • keep only the top CCC distant tokens,
  • repeat at every decoding step.

That is why the paper calls the cache constant-sized but still accurate. The cache is not frozen; it is continuously reshaped by the most recent context. If you remember one sentence, make it this: MorphKV keeps the tokens that the newest tokens still need, not the tokens that merely arrived first.
