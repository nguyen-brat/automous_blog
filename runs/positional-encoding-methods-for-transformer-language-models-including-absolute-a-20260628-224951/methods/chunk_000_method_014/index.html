Context Expansion with Parallel Encoding (CEPE)

CEPE is a retrofit for decoder-only LLMs. Instead of forcing the decoder to carry every extra token in one giant causal stream, it moves the long prefix into a small bidirectional encoder, processes that prefix chunk by chunk, stitches the chunk states into one side memory, and lets every decoder layer read that memory through cross-attention. The decoder stays frozen. The long context becomes a notebook the decoder can consult.

You can keep the whole method in one mental picture:

That is the real move. CEPE does not try to stretch one decoder context until it breaks. It gives the model a second path for long context.

What to notice here is that the decoder never has to process the entire long prefix as its own full self-attention problem. The extra context is encoded separately, then exposed to the decoder as a side memory. The cross-attention modules sit between self-attention and feed-forward layers, so every decoder block gets another chance to read the long context after it has refined its own local state.

The core trick: turn long context into side memory

The paper splits the input into two parts. The last part is the main decoder stream X. The earlier part is the extra context C, which may be a long document or a batch of retrieved passages. Instead of reading C as one giant sequence, CEPE breaks it into short chunks C_i and encodes each chunk separately.

That matters for two reasons.

First, each chunk gets its own local positional encoding, so the encoder does not need to extrapolate one global position axis across the whole long input. Second, the encoder is bidirectional, so it can build richer representations inside each chunk than a causal decoder could. The encoder is not trying to predict the next token. It is building a memory bank that is easy for the decoder to read later.

Formula card: how CEPE builds the memory bank

Purpose. Turn many short chunks into one concatenated memory that the decoder can query.

Equation.

ϕi=Menc(Ci),Φ=concat({ϕi}i=1k)\phi_i=\mathcal{M}_{\mathrm{enc}}(C_i),\quad \Phi=\mathrm{concat}(\{\phi_i\}_{i=1}^k)ϕi​=Menc​(Ci​),Φ=concat({ϕi​}i=1k​)

Intuition. Encode each chunk on its own, then stitch all the chunk states together end to end. The decoder sees one memory bank, but that bank was built in parallel from many smaller pieces.

Symbols.

  • \phi_i: the token-wise hidden states produced by the encoder for chunk i
  • \mathcal{M}_{\mathrm{enc}}: the small bidirectional encoder
  • C_i: the i-th chunk of extra context
  • \Phi: the concatenated encoder memory
  • k: the number of chunks

Read-aloud meaning. Run the encoder on each chunk, then concatenate all encoded chunks into one memory for the decoder.

Why it matters. This is the core reason CEPE can handle long inputs without stretching one global decoder timeline. Long context is moved into a side notebook instead of forcing the decoder to own it directly.

How the decoder reads that notebook

Once \Phi is built, the decoder uses it as the keys and values for cross-attention. The decoder’s own hidden states from X act as the queries. In plain terms, the decoder keeps reasoning over the tokens it is currently generating, but it can pause and ask the side memory for help whenever a long-range fact matters.

That cross-attention block is inserted into every decoder layer, between self-attention and the feed-forward network. This placement is important. Self-attention still handles the decoder’s local causal stream. Cross-attention then pulls in the long-context evidence. The feed-forward network finally fuses the two sources into the next layer’s representation.

The paper also uses the cross-attention projection matrices as an adapter from the smaller encoder width to the larger decoder width. That is a quiet but useful detail: the encoder can stay compact, while the decoder still receives representations in the shape it expects.

Why the method is efficient

The efficiency win comes from what CEPE does not ask the decoder to do. A normal decoder-only model has to cache every token in every layer. CEPE does not. It caches the encoder memory once and only keeps the decoder-side cache for X.

That gives the method a much better scaling story for very long prompts. The paper summarizes the contrast as:

Formula card: memory cost with and without CEPE

Purpose. Compare the memory footprint of a plain decoder-only model with CEPE.

Equation.

O((m+n)Lddec)vs.O(mdenc+nLddec)\mathcal{O}((m+n)Ld_{\mathrm{dec}})\quad \text{vs.}\quad \mathcal{O}(md_{\mathrm{enc}}+nLd_{\mathrm{dec}})O((m+n)Lddec​)vs.O(mdenc​+nLddec​)

Intuition. A plain decoder stores the whole long prompt in every decoder layer. CEPE stores the extra context once in the encoder and only keeps the decoder-side tokens across decoder layers.

Symbols.

  • m: number of extra-context tokens
  • n: number of decoder-stream tokens
  • L: number of decoder layers
  • d_{\mathrm{dec}}: decoder hidden size
  • d_{\mathrm{enc}}: encoder hidden size

Read-aloud meaning. Decoder-only memory grows with the whole prompt at every layer, while CEPE pays once for the encoder side and only layers the decoder cache over the shorter stream.

Why it matters. This is the basic reason CEPE can scale to much longer inputs with much lower memory pressure. The long prefix is moved out of the expensive part of the model.

How training teaches the side memory to work

CEPE does not train the whole decoder from scratch. The decoder is frozen. The paper trains the added cross-attention modules and fine-tunes the encoder so the new side path learns to speak the decoder’s language without disturbing the base model’s generation ability.

That choice is not just about cost. It also keeps the original LM behavior stable. You are teaching the model a new way to access context, not replacing the model itself.

The encoder is pretrained first as a bidirectional masked-language model on RedPajama. That gives it a compact, general-purpose representation space before it ever has to cooperate with the decoder.

The main long-context stage is shaped like the target use case. The paper trains on 8,192-token sequences, keeps the last 4,096 tokens in the decoder, and chunks the first 4,096 tokens into 16 encoder chunks of 256 tokens each. That means the model learns the exact division of labor CEPE needs later: a short causal stream on one side and a chunked side memory on the other.

The data mix matters too. The paper blends filtered long ArXiv and Books documents with concatenated RedPajama sequences in a 2:1 ratio. The filtered side gives the model naturally long documents to learn from, while the concatenated side adds broader domain coverage. That balance helps the retrofit transfer from long-document modeling to retrieval and instruction-following settings.

Then the paper uses a warmup stage that is easy to understand as a rehearsal. The encoder and decoder see the same short sequence, and the model is trained to copy through cross-attention. In other words, CEPE first learns that the side memory is real and useful. Only after that does it move on to the harder long-context setting.

The paper also makes the cross-attention less disruptive at initialization by copying the decoder’s self-attention projection weights into the new cross-attention module and zeroing the output projection. That way the new branch begins from a familiar shape instead of random noise.

During the main stage, the paper also masks encoder chunks or suffixes at random. The goal is to make the model less brittle to variable passage lengths and variable numbers of chunks at inference time, which is exactly what retrieval-heavy prompts tend to create.

Why retrieval and many passages fit so naturally

Once you think of extra context as chunked side memory, retrieval-augmented prompts stop looking special. A retrieved passage is just another chunk. A larger set of retrieved passages is just a larger set of chunks. CEPE can encode those passages in parallel and let the decoder query the resulting memory when needed.

That is why the method transfers well beyond plain long-document language modeling. It is a good fit whenever the prompt is really “main question plus a lot of supporting evidence.” The decoder can keep its own causal stream short while the evidence lives in the encoder bank.

The trade-off is that chunks do not fully interact with one another inside one shared decoder context. CEPE can read many chunks, but it is not the same as letting every token attend to every other token directly. That shows up in tasks where far-apart demonstrations need to influence each other strongly.

CEPED: the same idea for instruction-tuned models

The paper extends CEPE to instruction-tuned models with CEPED. The problem here is practical: long instruction-following data are scarce and expensive, but we still want an instruction-tuned decoder to gain a much longer context window.

The solution is to reuse the original instruction-tuned decoder as a teacher. The CEPE-style model becomes the student. The teacher runs on a short context that still fits inside the original window, and the student learns to match the teacher’s output while reading the longer context through the new encoder path.

Because storing full teacher distributions for every token would be expensive, the paper keeps only the top-50 token probabilities and their indices for each target position. That is enough to define the distillation signal without storing an enormous amount of soft targets.

Formula card: the KL distillation used by CEPED

Purpose. Keep the new long-context student close to the original instruction-tuned model while it learns the side-memory path.

Equation.

LKL=DKL(Mdec(S)∥MCEPE(C,X))\mathcal{L}_{\mathrm{KL}}=D_{\mathrm{KL}}(\mathcal{M}_{\mathrm{dec}}(S)\Vert \mathcal{M}_{\mathrm{CEPE}}(\mathcal{C},X))LKL​=DKL​(Mdec​(S)∥MCEPE​(C,X))

Intuition. Ask the CEPE student to match the teacher’s next-token preferences on the same short context, then let the student learn how to use extra context through the encoder.

Symbols.

  • \mathcal{L}_{\mathrm{KL}}: KL-divergence distillation loss
  • D_{\mathrm{KL}}: KL divergence between two probability distributions
  • \mathcal{M}_{\mathrm{dec}}(S): teacher distribution from the original decoder on the full short sequence S
  • \mathcal{M}_{\mathrm{CEPE}}(\mathcal{C},X): student distribution from the CEPE model
  • S: \mathrm{concat}(\mathcal{C},X), the teacher’s short input sequence
  • \mathcal{C}: encoder-side extra context
  • X: decoder-side main stream

Read-aloud meaning. Make the CEPE student imitate the original model’s token probabilities while reading long context through the new architecture.

Why it matters. This is how the paper extends instruction-tuned models without needing a large long-context instruction dataset. The teacher preserves instruction-following behavior while the student learns the longer memory path.

In the final training objective, this KL term is combined with the normal cross-entropy loss, so the student still learns to predict the next token while staying close to the teacher’s behavior.

Where the trade-off shows up

CEPE’s biggest strength is also its limit. Chunking makes long-context access much cheaper, but it weakens direct interaction among far-apart pieces of the context. If a task really needs one demonstration to reshape another demonstration inside one shared causal stream, CEPE is less natural than a full-context decoder.

That is why the paper can do well on long-document modeling, retrieval, and many instruction-following tasks, yet still leave some room between CEPE and a full decoder on very interaction-heavy in-context learning settings. The model can consult many chunks, but the chunks are still separate before they reach the decoder.

The method is also most clearly validated on LLaMA-2-7B and LLaMA-2-Chat-7B. The paper’s own limitations section is open about that scope: it would be useful to know how the same idea behaves on other base models, other retrievers, and much more aggressive length extrapolation.

What to remember

CEPE is not “a longer decoder.” It is “a decoder with a side notebook.” The notebook is built by a small bidirectional encoder that reads long context in chunks, the decoder reads that notebook through cross-attention in every layer, and the base decoder stays frozen so the retrofit is cheap and stable.

If you want the shortest memory of the method, keep these three pieces:

  • split extra context into chunks,
  • encode chunks in parallel and concatenate their states,
  • let every decoder layer cross-attend to that memory.

CEPED applies the same idea to instruction-tuned models by adding KL distillation, so the model keeps its instruction-following behavior while gaining a longer context path.
