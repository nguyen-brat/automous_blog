LongEmbed: Extending Embedding Models for Long Context Retrieval

LongEmbed is not trying to invent a brand-new embedding architecture. It is trying to answer a more practical question: if you already have a short-context embedding model, how far can you stretch it before long documents stop behaving like one coherent input?

That question matters because retrieval is where embeddings actually earn their keep in search and RAG. If the encoder only works well on the first 512 tokens, then a long document can hide the answer in plain sight.

Here is the basic flow:

The benchmark is the measurement tool. The extension tricks are the knobs the paper turns after the measurement says there is still a lot of headroom.

Why the benchmark had to be rebuilt

Existing retrieval benchmarks were not enough for two reasons. Some, like BEIR, mostly use short documents, so they never really force a model to deal with thousands of tokens. Others, like LoCo, do contain long documents, but the target information is not always evenly spread out. A model can look good by learning where the easy clues usually sit instead of actually handling arbitrary long-range evidence.

LongEmbed fixes both problems by combining synthetic tasks, where the paper controls length and target placement, with real tasks, where the evidence is naturally dispersed in long source texts.

What to notice in this image is the difference between the two synthetic settings. The passkey test buries a unique key inside mostly repetitive filler. The needle test hides a factual sentence inside a more natural essay. Both force the model to retrieve information from far away, but the needle version is less artificial and less friendly to shortcut learning.

The synthetic tasks sweep across a wide range of lengths, from a few hundred tokens up to about 32k, so the same model can be tested at several scales instead of only one. The real tasks push on the same bottleneck from another angle: NarrativeQA, QMSum, 2WikiMultihopQA, and SummScreenFD all require the model to gather evidence that is scattered across long source documents.

For QA datasets, the query is the question. For summarization datasets, the summary becomes the query and the set of source documents becomes the candidate pool. That turns summarization into a retrieval problem, which is exactly what embeddings are supposed to support.

The position scheme is the whole game

A long document is not hard because it has many words. It is hard because the model must keep positional structure intact while the sequence stretches. This is where the paper divides the world into two camps.

Absolute position embeddings, or APE, attach a learned vector to each absolute slot. That works well inside the trained window, but once you step beyond it, you either have to invent new slots or reuse old ones. RoPE is different. It does not attach a stand-alone vector to each position. Instead, it rotates queries and keys so the attention score depends on relative offset.

Formula Card: RoPE keeps attention tied to relative distance

Purpose. Show why RoPE extrapolates more cleanly than absolute position embeddings.

Equation.

a(q,k):=g(q,k,(m−n)θ)a(\mathbf{q},\mathbf{k}) := g(\mathbf{q},\mathbf{k},(m-n)\theta)a(q,k):=g(q,k,(m−n)θ)

Intuition. RoPE does not only ask “what are these two tokens?” It also asks “how far apart are they?” The score depends on the relative offset between the query position m and the key position n.

Symbols.

  • \mathbf{q}: the query vector
  • \mathbf{k}: the key vector
  • m-n: the relative position offset between two tokens
  • \theta: the vector of RoPE rotation frequencies across dimensions
  • g: the compact mapping from content vectors and relative offset to attention score

Read-aloud meaning. Attention depends on the query, the key, and how far apart the tokens are.

Why it matters. This is the structural reason RoPE usually stretches better than APE. The model is already organized around relative distance, so extension can preserve more of the original geometry instead of rebuilding position identities from scratch.

That one difference explains almost everything that follows. APE extension is mostly about how to reuse or smooth a position table. RoPE extension is mostly about how to stretch a relative-distance geometry without making it collapse.

Stretching APE models without destroying the old ones

The APE side of the paper is the more familiar “make the lookup table longer” problem. LongEmbed compares four cheap strategies, then asks whether a little fine-tuning can recover more performance without breaking short-context behavior.

The left side of the figure shows the basic idea: you can group positions, reuse them recurrently, or interpolate them onto a longer grid. The right side shows the tuning trick. The original position vectors stay frozen while the new long-context positions are trainable. That is the important tradeoff: preserve the old behavior, but let the new slots learn what long context needs.

The four plug-and-play variants are easy to remember:

  • PCW splits the document into short chunks, runs each chunk separately, and then combines the chunk embeddings. It is the simplest fallback, but it weakens interactions across chunk boundaries.
  • GP groups positions so several tokens share one coarser slot. That is cheap, but distant tokens can collide.
  • RP reuses the original position ids cyclically with modulo. It keeps the model running on longer inputs, but it also aliases positions that should be distinct.
  • PI stretches the position ids linearly and fills the gaps by interpolation. This is smoother than hard reuse, but it can blur high-frequency positional detail.

The paper’s point is not that one of these tricks is magical. It is that they are all different ways of deciding how much of the original 512-slot structure you are willing to sacrifice.

The best APE results come from a small extra step: keep the original position vectors frozen, and fine-tune only the newly added long-context positions. That preserves short-context behavior much more cleanly than full retraining. The authors also simulate long training samples by shifting position ids during fine-tuning, so the model gets exposed to long spans even though the underlying training data is still short.

In practice, that extra tuning matters. E5Base jumps from 41.0 average to 56.6 after tuning at 4k context. The gain is not just from more parameters. It comes from giving the new positions a chance to learn long-range behavior without disturbing the old ones.

Why RoPE extensions are cleaner

RoPE models are the more interesting case because their positional geometry already lives inside attention. That makes them better suited to long-context extrapolation, but only if the extension method respects the rotation structure instead of flattening it.

The paper compares three RoPE-friendly moves:

  • PCW, as a baseline chunk-and-aggregate fallback.
  • SelfExtend, which keeps true relative positions in a local neighborhood and only coarsens the farther range.
  • NTK, which remaps the RoPE frequencies so long-context extrapolation is less aggressive.

SelfExtend is the most intuitive to picture. Near neighbors still get their ordinary relative offsets, so the model can keep local ordering sharp. Farther tokens are grouped more coarsely, which saves the relative geometry where it matters most. That is why it tends to work so well on retrieval tasks with both local clues and distant evidence.

NTK is a different kind of fix. Instead of scaling every frequency band in the same blunt way, it changes the RoPE base so the extrapolated schedule is less distorted. In the paper’s language, the pressure is spread across dimensions instead of forcing every band to stretch by the same amount.

Formula Card: NTK-aware interpolation slows the rotary schedule

Purpose. Show the frequency remapping used by NTK-aware RoPE extension.

Equation.

θj′=(10000λ)−2j/d\theta'_j = (10000\lambda)^{-2j/d}θj′​=(10000λ)−2j/d

Intuition. Increasing the RoPE base slows the rotation. That makes long positions less unfamiliar, while still keeping the different frequency bands ordered.

Symbols.

  • \theta'_j: the new angular frequency for frequency band j
  • j: the index of the rotary frequency band
  • d: the hidden size of the attention head
  • \lambda: the scaling factor used by NTK-aware interpolation
  • 10000: the original RoPE base

Read-aloud meaning. Replace the original RoPE frequencies with a slower schedule that is easier to extrapolate.

Why it matters. This is why NTK beats naive uniform interpolation. It does not crush the whole frequency spectrum in one move, so the model keeps more of the long-range structure it already learned.

A useful detail from the paper is that the RoPE comparison is made fairly. The authors train E5-RoPEBase with the same data and recipe as E5Base, which makes the APE versus RoPE comparison much cleaner than if the models came from unrelated pretraining pipelines.

What the results actually tell you

The headline result is not just that extension helps. It is that the best extension method depends on the positional scheme.

On the APE side, plug-and-play methods are all in the same rough range, but tuning the new positions gives a clear boost. On the RoPE side, the specialized methods win much more decisively. E5-RoPEBase with SelfExtend reaches 60.8 average on LongEmbed, up from 40.5 without extension. E5Base with tuning reaches 56.6, up from 41.0. And E5-Mistral with NTK reaches 75.3 at 32k context, up from 64.4.

Two things are worth noticing there.

First, the gains are large enough to matter. This is not a tiny benchmark shuffle. The model is finding a lot more of the right documents once the context window is extended.

Second, the best unextended long-context model still only scores 64.4 average. So even the strong long-context baseline leaves a lot of room for improvement. LongEmbed is not polishing an already-solved problem. It is exposing a real bottleneck.

Limits to keep in mind

The paper is careful about what it does not solve.

  • Most of the extension methods it studies are training-free, so the comparison is practical but not exhaustive.
  • RoPE fine-tuning is explored less deeply than APE fine-tuning.
  • The benchmark is retrieval-centric, so it tells you a lot about long-context search and document selection, but not everything about embedding quality in every downstream use case.

That said, the paper still makes a strong and durable point: if you want longer-context embeddings, the positional mechanism matters more than a lot of people assume. The easier path is not always “train a bigger model from scratch.” Often it is “stretch the existing geometry without breaking the short-context behavior.”

What to remember

LongEmbed is best remembered as two things at once: a benchmark that makes long-context retrieval hard in the right way, and a comparison of extension strategies that shows how much the positional scheme matters.

If you remember one sentence, make it this: APE can be extended by reusing or interpolating positions and then tuning the new slots, but RoPE usually stretches more naturally because attention already depends on relative distance.

That is also why the paper’s practical advice is simple. For APE models, gentle fine-tuning helps. For RoPE models, methods like SelfExtend and NTK are stronger bets. And if the retrieval task is long enough, the gap between those choices becomes impossible to ignore.
