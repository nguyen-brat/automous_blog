When Precision Meets Position: BFloat16 Breaks Down RoPE in Long-Context Training

AnchorAttention is not really a new RoPE formula. It is a way of changing the attention graph so RoPE can keep behaving like a relative positional system even when training runs in BFloat16 and the context window gets very long. The paper’s diagnosis is sharper than “long context is hard.” It says the relative-position property itself starts to drift numerically, and the drift concentrates around the first token.

That matters because RoPE is supposed to be shift-invariant. If you slide every token in a sequence by the same amount, the attention score should not change. In ideal math, the model should care about relative distance, not the absolute offset of the whole window. The paper shows that this ideal stops holding cleanly in practice once long sequences and BFloat16 meet.

Formula card: RoPE’s ideal shift invariance

Purpose: state the property RoPE is supposed to preserve.

Equation:

A(i+Δ)(j+Δ)=qi⊤Rm,θkj=AijA_{(i+\Delta)(j+\Delta)} = q_i^{\top} R_{m,\theta} k_j = A_{ij}A(i+Δ)(j+Δ)​=qi⊤​Rm,θ​kj​=Aij​

Intuition: if every token moves by the same offset, the attention score should stay the same because RoPE should depend only on relative distance.

Symbols:

  • A_{(i+\Delta)(j+\Delta)}: attention logit after shifting both token positions by \Delta
  • A_{ij}: original attention logit between tokens i and j
  • q_i: query vector at token i
  • k_j: key vector at token j
  • R_{m,\theta}: RoPE rotation matrix for relative distance m
  • m: relative distance between i and j, with m = j - i
  • \Delta: uniform shift added to every position index

Read-aloud meaning: sliding the whole sequence should not change the attention score.

Why it matters: the paper’s main claim is that BFloat16 makes this identity drift in long contexts, and the first token is where the error shows up most clearly.

The paper’s first figure makes that failure concrete.

What to notice is the shape of the discrepancy, not just the final number. The BFloat16 curve diverges from the Float32 curve, which means the same shifted sequence no longer produces the same attention behavior. The middle plot shows something even more telling: the first token carries most of the difference. The right plot then shows that the discrepancy grows as the sequence gets longer, so the problem is not a small fixed offset. It gets worse as the window expands.

That is the key diagnostic. The issue is not simply that long contexts are expensive. It is that the positional reference itself becomes unstable under low precision, and the instability compounds with length.

The obvious fix helps, but only partway

Once you see the first-token pattern, a natural question comes up: can we just reset the position IDs?

The paper tests that idea with intra-document attention. The sequence is split into documents, cross-document token-to-token attention is masked, and then the authors compare two ways of assigning position IDs. One version keeps positions continuous across the whole long window. The other resets the first token of each document back to the starting position.

The reset version helps. That is an important clue, because it says the problem is not only in the content of the documents. The starting position itself is part of the failure. If the same model gets a more consistent first-token reference, performance improves.

But resetting positions introduces a new bottleneck. If every document restarts from the same point, the model only sees the full rotary span when the training examples are long enough to fill the window. That is awkward for continual long-context training, because truly long documents are scarce and expensive to assemble.

So the paper is looking for a middle ground:

  • keep the first-token reference stable,
  • keep the rotary span continuous,
  • and avoid wasting compute on unnecessary cross-document links.

That is where AnchorAttention comes in.

One shared anchor instead of many first tokens

AnchorAttention turns the first token into a shared landmark. In the paper’s setup, the beginning-of-sequence token acts as a common anchor that every document can see. Tokens inside a document still attend causally to earlier tokens in that same document, but tokens from different documents are masked from one another. The only cross-document visibility is the shared anchor.

You can think of it as giving every document the same doorway before it enters the long hallway of positions. The doorway is the anchor. Once the doorway is fixed, the model does not have to reinterpret a different “start of document” every time it sees a new segment.

The important detail is that the anchor is semantically cheap. The paper uses the beginning-of-sequence token, which usually carries little direct meaning, so any precision noise concentrated there is less damaging than noise spread across content tokens. The method is therefore trying to trap the numerical fragility in a harmless place.

The architecture figure makes that design easier to see.

What to notice is the difference between “many starts” and “one start.” Standard intra-document attention has separate document starts, so the model still has to reconcile multiple first-token positions. Resetting positions makes those starts consistent, but it still leaves each document isolated. AnchorAttention goes one step further: the documents stay isolated from one another, but they all share the same stable anchor. That lets the model preserve document-local coherence without creating multiple inconsistent reference points.

This is the real trick in the paper. It does not try to make every token interact with every other token. It instead changes which token is allowed to act as the global reference, and then it cuts away the redundant document-to-document attention that would otherwise pile up cost and noise.

Why the anchor helps

The method works because it fixes three things at once.

First, it removes the inconsistency exposed by the reset-position experiment. Every document now begins from the same landmark, so the model does not have to juggle several competing first-token positions.

Second, it still lets the model walk through the full rotary span during training. Continuous position IDs are preserved across the long window, so the model does not need a single document to be enormous in order to practice long-range angles.

Third, it makes attention cheaper. If tokens from different documents cannot attend to each other directly, the model avoids a lot of redundant cross-document computation. That is one reason the paper reports a training-time reduction of more than 50% compared with standard full attention.

This combination is why AnchorAttention feels like a structural fix rather than a numerical patch. It does not just smooth the error. It changes where the error can live.

What the paper finds in practice

The broad empirical picture matches the design story.

AnchorAttention improves long-context RULER performance over full attention across the 8K to 128K range. The cleanest headline number is on LLaMA-3-8B at 128K, where the score rises from 34.02 with full attention to 51.49 with AnchorAttention. That is the kind of jump you expect when the model stops fighting its own positional bookkeeping.

At the same time, the method does not collapse the model’s shorter-range skills. The paper reports competitive results on HellaSwag, MMLU, and LongBench, which matters because a long-context retrofit is only useful if it does not destroy the model’s everyday abilities.

There is also a useful negative result. Interleaved chunks consistently hurt performance when combined with cross-document masking. That tells you the method needs coherent documents and a clean shared-anchor structure. If you try to make the training data more synthetic by shuffling chunks across documents, the masked attention layout no longer gets the coherence it needs.

So the method is not “any way of making the window longer will work.” It is much more specific than that. The shared anchor matters, and document coherence matters with it.

Why this is different from just changing RoPE

The deeper point of the paper is that the failure is not only in the RoPE equations. It is in the interaction between RoPE, BFloat16, and the attention graph used during continual long-context training.

That is why the paper’s remedy is structural:

  • keep RoPE,
  • keep BFloat16,
  • but change the attention mask and the position-ID layout.

In other words, the method does not ask the model to learn a new positional encoding from scratch. It asks the existing system to stop amplifying a numerical weakness across many document starts. The result is a more stable reference structure and a smaller attention graph at the same time.

What the method gives up

The tradeoff is straightforward. AnchorAttention drops direct token-to-token interactions across documents, so the documents are not fully merged into one giant attention space. The model now relies on the shared anchor as the bridge across the long window.

That is a good deal when the problem is long-context continual pretraining under BFloat16, but it is still a constraint. The paper’s evidence is also not full pretraining evidence. It is strongest in continual long-context training, mostly at roughly 10B scale or smaller and with a 2B-token budget. So this is a strong proof-of-concept, not the final word on scaling.

There is another open question the paper leaves behind: the “first token behaves like an absolute reference” story is very plausible, but it is still more of a supported hypothesis than a fully proven mechanism. The experiments point in that direction, but the paper does not claim a complete theory.

What to remember

AnchorAttention is best remembered as a way of concentrating positional fragility where it hurts least. RoPE should be relative, but BFloat16 makes the first token behave like a noisy reference point in long contexts. The method responds by giving every document the same shared anchor, masking away redundant cross-document links, and keeping position IDs continuous so the model still sees the full rotary range.

If you want the shortest possible summary, it is this:

when positional precision gets brittle, do not spread the error across many starts. Give the model one stable landmark and make the rest of the window organize around it.
