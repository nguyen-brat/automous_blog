Long-Context Multilingual Text Representation and Reranking Models for Retrieval

The main idea of this paper is straightforward: if a retrieval system needs to read multilingual documents up to 8192 tokens, do not patch long-context support onto a short encoder as an afterthought. Build the encoder for long context first, then reuse it for retrieval. That is why the method is really a pipeline. The encoder learns long-context multilingual language modeling, the retriever learns dense and sparse matching, and the reranker cleans up the top candidates.

What to notice in the pipeline is the ordering. The paper does not jump straight to retrieval. It first teaches the model to read long multilingual text natively, and only then does it turn that backbone into a retriever and reranker. The positional design is the base layer. The retrieval stack sits on top of it.

Start with a longer-view encoder

The encoder is still a BERT-style base model, but several details change the game. Absolute positional embeddings are replaced with RoPE, the feedforward block is upgraded to a gated linear unit, attention-score dropout is removed for compatibility with FlashAttention-style kernels, and the token embedding matrix is padded to a multiple of 64 for better throughput. None of these changes is flashy on its own. Together, they turn a standard encoder into something that can actually survive 8k-token multilingual training.

The paper also uses unpadding. That matters more than it sounds like, because long multilingual batches contain a lot of padding. Instead of spending attention compute on tokens that carry no information, the model strips them out and runs variable-length attention with xFormers. The MLM labels are unpadded too, so the training loss does not waste work on the padded positions either.

What to notice here is that the architecture is still familiar. The novelty is not a brand-new transformer shape. The novelty is the set of practical changes around it: RoPE instead of absolute tables, unpadding instead of padded attention, and a few efficiency tweaks that make long multilingual pretraining feasible.

Formula card: multilingual sampling distribution

Purpose: keep large languages from dominating multilingual pretraining.

Equation:

qi=piα∑j=1Npjα,pi=ni∑j=1Nnjq_i = \frac{p_i^{\alpha}}{\sum_{j=1}^{N} p_j^{\alpha}}, \quad p_i = \frac{n_i}{\sum_{j=1}^{N} n_j}qi​=∑j=1N​pjα​piα​​,pi​=∑j=1N​nj​ni​​

Intuition: start from each language’s raw corpus share, soften the imbalance with an exponent, and renormalize so low-resource languages still appear often enough.

Symbols:

  • q_i: sampling probability for language i
  • p_i: raw proportion of language i in the corpus
  • \alpha: flattening exponent, set to 0.5 in the paper
  • n_i: number of texts in language i
  • N: total number of languages

Read-aloud meaning: sample languages according to a softened version of their data share, not according to raw corpus size alone.

Why it matters: long-context multilingual training fails if the biggest languages monopolize every batch. This rule keeps the pretraining mixture broad enough that the encoder learns multilingual behavior instead of mostly English behavior.

The pretraining corpus is large: about 1,028B tokens across 75 languages, drawn from sources such as C4, Skypile, mC4, CulturaX, Wikipedia, and books. The exact source list is less important than the training lesson. Long-context multilingual encoders need enough breadth that they see many scripts, tokenization patterns, and document lengths during masked language modeling.

The paper trains the encoder in two stages. First comes MLM-2048, where inputs are chunked to 2048 tokens and the RoPE base is set to 10000. Then comes MLM-8192, where the model is stretched to 8192 tokens and the RoPE base is raised to 160000. That is the core curriculum idea: let the model learn stable multilingual MLM behavior at a manageable length first, then extend the same backbone to the native long context.

This is not just a longer training run. The second stage changes the rotary scale as well, so the phase progression still makes sense at 8k. In other words, the model is not asked to guess long distance behavior from a short-position clock that was never meant to stretch that far.

From encoder to retriever

Once the encoder can handle long multilingual sequences, the paper reuses it for retrieval. The retriever is trained with contrastive learning, which means the model is not predicting masked tokens anymore. It is learning a similarity space where a query should sit close to its relevant document and far from negatives.

What to notice in this diagram is the split between roles. The dense branch gives semantic recall, the sparse branch keeps lexical matching explicit, and the reranker reads the query and document together only after the first-stage retriever has done the expensive filtering.

Formula card: contrastive InfoNCE loss

Purpose: teach the retriever to score a positive document above the negatives for each query.

Equation:

L=−log⁡exp⁡(s(q,d+)/τ)∑i=1Nexp⁡(s(q,di)/τ)\mathcal{L} = -\log \frac{\exp(s(q,d^{+})/\tau)}{\sum_{i=1}^{N}\exp(s(q,d^{i})/\tau)}L=−log∑i=1N​exp(s(q,di)/τ)exp(s(q,d+)/τ)​

Intuition: make the similarity of the query and its relevant document stand out from the similarities of the query and all other candidates in the batch.

Symbols:

  • \mathcal{L}: contrastive loss
  • q: query text
  • d^{+}: positive, relevant document
  • d^{i}: candidate document i, including negatives
  • N: number of candidate documents in the denominator
  • s(q,d): similarity score between query and document representations
  • \tau: temperature that controls how sharp the softmax is

Read-aloud meaning: push the positive pair above the negatives, and control how hard the model separates them with a temperature.

Why it matters: this is the training signal that turns the long-context encoder into a retrieval model instead of just a language model. In the contrastive pretraining stage, the paper uses the [CLS] embedding and cosine similarity, plus in-batch negatives, so every batch teaches the model to discriminate among plausible candidates.

The paper also adds a subtle but important trick here: reverse NTK scaling, or revNTK. During contrastive pretraining, the RoPE base is scaled down from 160000 to 20000, and the retrieval stage is trained with shorter maximum lengths, such as 512 for queries and 1024 for documents. That sounds backwards at first, but the goal is practical. The contrastive stage does not need to spend all its compute on full 8k inputs. It needs to learn a long-context geometry that still transfers when the retriever is trained on shorter query-document pairs.

The appendix makes this even more explicit: the authors describe revNTK as training the 8k encoder with a 1k max length after reducing the RoPE base to one-eighth of the original value. The point is not to fake long context. The point is to keep the long-context rotary behavior stable while the retrieval objective is learned on shorter pair inputs.

What to notice in this plot is the tradeoff. revNTK can be a little less flattering at the shorter evaluation length, but it makes 8k retrieval performance more stable across training steps. That is a good deal for a long-context system, because the model will eventually be judged on the long documents it was built to read.

Dense vectors are only half the story

The retriever is not only dense. The paper combines two retrieval views: Matryoshka dense embeddings and sparse token weights. That is the main reason the model is called a hybrid TRM. Think of Matryoshka as nested dolls: the first few dimensions are already useful on their own, and later dimensions add extra resolution when you can afford them.

Formula card: hybrid TRM objective

Purpose: train dense prefixes and sparse token matching together in one retriever.

Equation:

LTRM=λLsparse+∑d∈Dwd L:d\mathcal{L}_{\text{TRM}} = \lambda \mathcal{L}_{\text{sparse}} + \sum_{d \in D} w_d \, \mathcal{L}_{:d}LTRM​=λLsparse​+d∈D∑​wd​L:d​

Intuition: combine a sparse lexical loss with several dense losses computed from different embedding prefixes, so one model can support both fast and accurate retrieval modes.

Symbols:

  • \mathcal{L}_{\text{TRM}}: total retriever loss
  • \lambda: weight on the sparse branch
  • \mathcal{L}_{\text{sparse}}: sparse retrieval loss
  • D: set of Matryoshka prefix dimensions, here the 32-step prefixes up to the full hidden size
  • w_d: weight for the loss at prefix dimension d
  • \mathcal{L}_{:d}: contrastive loss computed on the first d dimensions of the embedding

Read-aloud meaning: train the retriever to support both sparse token matching and multiple dense embedding sizes at the same time.

Why it matters: a retriever that only knows one embedding size is harder to serve flexibly. Here, the same model can support smaller or larger dense prefixes while still keeping a sparse branch for exact term overlap.

The sparse branch is easy to understand once you see the token weight formula. The encoder predicts a weight for each token, and that weight is then used to score the overlap between query and document.

Formula card: sparse token weight

Purpose: turn each token into a learned lexical importance score.

Equation:

wt=ReLU(Wht)w_t = \mathrm{ReLU}(\mathbf{W}\mathbf{h}_t)wt​=ReLU(Wht​)

Intuition: feed each token hidden state through a tiny linear head, keep only positive evidence, and interpret the result as token importance.

Symbols:

  • w_t: sparse weight for token t
  • \mathbf{h}_t: encoder hidden state for token t
  • \mathbf{W}: learned projection from hidden size to a single score
  • \mathrm{ReLU}: nonlinearity that removes negative values

Read-aloud meaning: the model assigns a nonnegative importance weight to each token.

Why it matters: sparse retrieval is useful when exact terms matter, especially for names, rare words, and multilingual text where lexical overlap still carries signal.

If a token appears more than once, the paper keeps the maximum weight. Then the query and document are scored only on shared tokens. That is a very deliberate design choice. It makes the sparse branch focus on lexical agreement instead of trying to summarize the whole document again.

Formula card: sparse relevance score

Purpose: score a query-document pair by the importance of the terms they share.

Equation:

ssparse(q,d)=∑t∈q∩d(wtq⋅wtd)s_{\text{sparse}}(q,d) = \sum_{t \in q \cap d}(w_t^{q} \cdot w_t^{d})ssparse​(q,d)=t∈q∩d∑​(wtq​⋅wtd​)

Intuition: only tokens that appear in both the query and the document can contribute, and each shared token contributes according to how important it is on both sides.

Symbols:

  • s_{\text{sparse}}(q,d): sparse relevance score
  • q \cap d: tokens shared by the query and the document
  • w_t^q: sparse weight of token t in the query
  • w_t^d: sparse weight of token t in the document

Read-aloud meaning: add up the importance of the overlapping tokens in the query and document.

Why it matters: this is what lets the retriever behave like a hybrid system instead of a pure dense embedder. When exact lexical overlap is important, the sparse branch can carry that signal directly.

The fine-tuning stage then blends the two ideas. Each query gets one positive passage and several hard negatives, and the model is trained on the hybrid loss above. The paper uses dynamic batching by length, so short examples can be packed densely while long 8k examples get smaller batches and sub-batches with gradient checkpointing. That is the kind of implementation detail that looks boring until you realize it is what makes the 8k retrieval stage fit on real hardware.

The reranker reads the pair directly

The reranker is a cross-encoder. It does not compare separate embeddings and it does not need sparse overlap features. It takes the query and document together, in the familiar [CLS] q [SEP] d format, and scores the pair from the final [CLS] state.

Formula card: reranker score

Purpose: produce a direct relevance score after the retriever has already narrowed the candidate set.

Equation:

srerank=Wh[CLS]s_{\text{rerank}} = \mathbf{W}\mathbf{h}_{[\mathrm{CLS}]}srerank​=Wh[CLS]​

Intuition: let the model look at the full query-document pair and turn the final pooled representation into a single relevance number.

Symbols:

  • s_{\text{rerank}}: reranking score
  • \mathbf{h}_{[\mathrm{CLS}]}: final hidden state at the [CLS] token
  • \mathbf{W}: learned linear projection to a scalar

Read-aloud meaning: the reranker maps the pair representation to one relevance score.

Why it matters: this final pass can resolve ties that a retriever cannot. Once the first-stage system has produced a candidate set, the cross-encoder can spend more compute on the top documents and judge them more carefully.

The reranker is fine-tuned with InfoNCE in one step. The paper says that a separate contrastive pretraining stage for the reranker did not help, so the final recipe stays simple: reuse the long-context encoder, keep the same training data family as the TRM fine-tuning stage, and adjust the hard negatives for the cross-encoder setting.

Why the stack works

The reason this method is strong is not any single component. It is the way the pieces line up. RoPE and the two-stage MLM curriculum give the encoder native 8k behavior. revNTK keeps the contrastive retrieval stage stable when the pair lengths are shortened. Matryoshka embeddings make the dense branch flexible. Sparse token weights preserve lexical matching. The reranker then uses a fuller cross-encoder view to clean up the final ranking.

That stack is reflected in the results. The encoder beats XLM-R-base on XTREME-R by 3.22 points for the 2048 model and 2.42 points for the 8192 model. The hybrid retriever reaches 71.3 nDCG@10 on MLDR, which is well above the dense-plus-sparse BGE-M3 baseline at 64.8. The reranker reaches 67.4 average and beats bge-reranker-v2-m3 while using fewer parameters. On efficiency, the unpadding plus memory-efficient attention path cuts encoding time to 52 seconds versus 744 seconds for BGE-M3 in the paper’s setup.

The limitations are worth keeping in mind. This is not uniformly best on every language and benchmark. It depends on large-scale staged pretraining and fine-tuning infrastructure. And some of the strongest gains come from the hybrid retrieval stack rather than dense embeddings alone. The paper is honest about that tradeoff.

What to remember

If you want one sentence to carry around, it is this: teach the encoder long context first, then shape it into a hybrid retriever, then finish with a cross-encoder reranker.

The bigger lesson is that long-context multilingual retrieval is not solved by a single positional trick. It usually needs three things at once: a native long-context backbone, a training curriculum that makes the long context learnable, and a serving-time retrieval design that combines dense semantics with sparse lexical matches. That is what makes mGTE feel like a practical retrieval stack instead of just a larger encoder.
