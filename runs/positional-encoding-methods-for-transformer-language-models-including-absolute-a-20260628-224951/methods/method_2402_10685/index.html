LongHeads: Head-Tail Retrieval for Efficient Long-Context Processing

LongHeads starts from a simple but important observation: a long prompt does not need every attention head to look everywhere. It is enough if each head can cheaply find a few useful chunks, keep a stable anchor at the beginning, keep the recent tail for local coherence, and then read only that short reconstructed window carefully.

That makes the method feel more like retrieval than like brute-force attention. The model is still using its own Q/K/V machinery, but it uses it in two stages. First it summarizes chunks. Then it uses the current query to retrieve the chunks that matter. The selected chunks are remapped into a short, familiar position range, so the model can keep working inside the length it was trained on.

You can keep the whole flow in mind as:

The title is literal enough to remember: the head of the sequence and the tail of the sequence are always preserved, and the middle is searched per head. That is the paper’s main move.

The first figure makes the tradeoff very clear.

What to notice:

  • Dense attention sees everything, but once the prompt goes beyond pretraining length it can become unreliable and expensive.
  • Fixed local windows are safe and cheap, but they throw away important middle evidence.
  • LongHeads sits between those extremes. It does not keep everything, and it does not keep only the tail. It keeps a short, query-dependent shortlist of chunks.

That is why LongHeads is different from a fixed sliding window method like LM-Infinite, and also different from Landmark Attention, which relies on learned landmark tokens and extra training. LongHeads stays training-free and uses the model’s own attention geometry as the router.

The method in one glance

The overview figure shows the method as it is actually used during inference.

What to notice here:

  • The model does not build one giant attention matrix over the whole document.
  • It first converts the document into chunk summaries.
  • Each head can then pick its own chunk shortlist for the current token.
  • The selected chunks are packed back into a short window before the normal causal attention step runs.

That last point matters a lot. LongHeads does not ask the model to extrapolate its positional behavior across the whole original document. It rewrites the selected pieces into a short, in-distribution span first, and only then performs attention.

Turning a chunk into something a head can score

LongHeads needs a compact representation for each chunk before it can decide which chunks are worth keeping. A plain average would be too crude, because not every token inside a chunk carries the same amount of semantic weight. Content words matter more than function words, and some token states are simply better representatives of the whole block than others.

The paper handles that with a two-step, attention-based summarization. First it lets the tokens inside a chunk interact and create a semantic query. Then it uses that query to pull out the most representative chunk state.

Formula card: build a semantic chunk query

Purpose: turn the tokens inside one chunk into a query vector that summarizes what that chunk is about.

Equation:

Oi=flash-attention⁡(Qi,Ki,Vi),qic=mean⁡(Oi)\mathbf{O}_i = \operatorname{flash-attention}(\mathbf{Q}_i, \mathbf{K}_i, \mathbf{V}_i), \quad \mathbf{q}_i^c = \operatorname{mean}(\mathbf{O}_i)Oi​=flash-attention(Qi​,Ki​,Vi​),qic​=mean(Oi​)

Intuition: let the tokens inside the chunk talk to each other first, then average the contextualized token states into one chunk-level summary query.

Symbols:

  • \mathbf{Q}_i, \mathbf{K}_i, \mathbf{V}_i: the token-level query, key, and value states inside chunk C_i.
  • \mathbf{O}_i: the contextualized token outputs after attention inside that chunk.
  • \mathbf{q}_i^c: the chunk-level query vector used to summarize the chunk’s semantics.
  • \operatorname{flash-attention}: the efficient attention kernel used for the chunk-local aggregation.

Read-aloud meaning: let the tokens in a chunk interact, then compress their combined meaning into one query vector.

Why it matters: LongHeads does not need a separate learned retriever. It reuses the model’s own attention states to build a query that is rich enough to represent the chunk.

The next step uses that chunk query to decide which token states best stand for the chunk itself.

Formula card: turn the chunk query into a chunk representation

Purpose: produce a representative vector for each chunk that can be compared with the current token’s query.

Equation:

ci=flash-attention⁡(qic,Ki,Ki)\mathbf{c}_i = \operatorname{flash-attention}(\mathbf{q}_i^c, \mathbf{K}_i, \mathbf{K}_i)ci​=flash-attention(qic​,Ki​,Ki​)

Intuition: use the chunk’s own semantic summary to ask which internal token states are the best representatives of the whole block.

Symbols:

  • \mathbf{c}_i: the final chunk representation for chunk C_i.
  • \mathbf{q}_i^c: the semantic query built from the chunk itself.
  • \mathbf{K}_i: the key states of the tokens inside chunk C_i.

Read-aloud meaning: ask the chunk to point to the token states that best describe itself.

Why it matters: this is the representation LongHeads later compares against the current head query. A coarse average would ignore the fact that some tokens inside a chunk are much more informative than others.

So the chunk summary is not just a shortcut. It is the thing that makes the retrieval step meaningful.

How a head decides which chunks survive

Once every chunk has a representation, LongHeads can score chunks against the current token. The score is the familiar dot product, which is important because it keeps the retrieval step aligned with the model’s native attention behavior.

The paper makes two chunks mandatory for every head.

  • The first chunk is always kept because the beginning of the sequence acts like a stability anchor. If the early tokens disappear, the model can lose its expressive behavior.
  • The last chunk is always kept because generation needs the recent local context to stay fluent.

Only after those two anchors are reserved does the model spend the remaining budget on the most relevant middle chunks.

Formula card: choose the chunks for one head

Purpose: select a short set of chunks for the current token and current head.

Equation:

P={C1}∪{Ci∣rank⁡(qj⋅ci)≤k−2}∪{C−1}P=\{C_1\}\cup\{C_i \mid \operatorname{rank}(\mathbf{q}_j \cdot \mathbf{c}_i)\le k-2\}\cup\{C_{-1}\}P={C1​}∪{Ci​∣rank(qj​⋅ci​)≤k−2}∪{C−1​}

Intuition: always keep the beginning and the end, then fill the remaining slots with the middle chunks that best match the current query.

Symbols:

  • P: the final set of selected chunks for one head.
  • C_1: the first chunk in the sequence.
  • C_{-1}: the last chunk in the sequence.
  • C_i: a candidate chunk.
  • \mathbf{q}_j: the current token query at decoding step j.
  • \mathbf{c}_i: the representation of chunk C_i.
  • k: the total chunk budget for the head.
  • \operatorname{rank}(\cdot): the ordering of a chunk’s similarity score among the candidate chunks.

Read-aloud meaning: keep the first chunk, keep the last chunk, and use the current query to choose the best middle chunks.

Why it matters: this is the actual retrieval policy that replaces full-sequence dense attention. The method only pays for the chunks that matter to the current token.

Because each attention head has its own query state, different heads do not have to agree on the same middle chunks. That is the useful part. One head may lock onto the answer-bearing chunk, another may keep a broader narrative chunk, and another may preserve a local transition chunk. The result is a distributed retrieval pattern rather than a single fixed mask.

The paper’s head visualizations make that behavior concrete.

What to notice here:

  • The selected chunks are not random. Heads repeatedly converge on the chunk that contains the important evidence.
  • Lower layers tend to spread their selections more broadly, while upper layers become more focused.
  • When the key sentence crosses a chunk boundary, the selection can fail even though the model is searching in the right region. That is one of the paper’s main limitations.

This is also the cleanest reminder that LongHeads is still a heuristic retriever. It is strong because it uses the model’s own attention signal, but it can still miss information when the chunk split is unfortunate or the query-chunk similarity is weak.

Why the positions get rewritten

Chunk selection alone is not enough. The chosen chunks may come from far apart in the original sequence, so their original positions can still look out of distribution to the pretrained model. LongHeads fixes that by rebinding the selected chunks into one contiguous short window.

The paper’s example is easy to visualize: if the selected chunks are (1, 2, 7, 8) out of eight chunks, they are packed into a single short interval instead of being left at their scattered original positions.

What to notice:

  • The causal order inside each selected chunk is preserved.
  • The original spacing between selected chunks is discarded.
  • The remapped tokens now live inside the pretrained length range, so the model does not need to extrapolate its positional behavior to a faraway coordinate system.

That is the second half of the method’s name in spirit. Retrieval finds the useful pieces, and position remapping makes them look normal to the model again.

This is why LongHeads fits naturally with many LLMs that already use relative positional encoding. The method does not invent a new positional mechanism. It rearranges the context so the existing one stays in-distribution.

What inference looks like when the model is actually running

The paper separates inference into encoding and generation, but the idea is the same in both cases: summarize the long history first, then only load the small selected window into the expensive attention step.

During encoding, the model computes chunk summaries in parallel. The paper describes this as two short passes of flash-attention over chunk-sized windows, which is cheap compared with attending over the full document. After the summaries are ready, each head selects its k chunks, loads their key/value states, and performs length-restricted causal attention only inside that selected window.

During generation, the model repeats the same idea step by step.

  • It selects the relevant chunks for the current token.
  • It loads only the key/value states for those chunks.
  • It keeps the most recent tokens on GPU so the output remains locally coherent.
  • If the context is extremely large, the older KV cache can be offloaded to CPU, and only the chosen chunks need to be moved back for the current step.
  • When the recent tokens fill one chunk, they are summarized and appended to the chunk cache.

The important consequence is not that attention becomes free. It does not. The important consequence is that the expensive attention step now runs over a short selected window instead of the whole history. That is why LongHeads can stay usable at much longer lengths than ordinary dense attention.

What the experiments are really checking

The results line up with the mechanism.

On passkey retrieval, LongHeads stays near perfect from 4k to 32k context, and it reaches 100% at 128k when the KV cache is offloaded to CPU. That is a strong sign that the chunk shortlist is actually finding distant evidence rather than just relying on local clues.

On LongBench, LongHeads beats the other restricted-attention baselines and remains competitive with full-attention extensions, especially when the context gets longer. The paper also shows that LongHeads keeps perplexity moderate at 32k where the vanilla LLaMA-2 baseline and position interpolation break down.

So the experiments are not just a score table. They are evidence that the retrieval-and-remap loop really preserves useful context. The model does not need to read every token to recover the important one.

Where the method can still break

LongHeads is strong, but its failure modes are very clear.

  • Chunk boundaries can split the needed evidence, so a key sentence may be divided across two chunks.
  • The chunk budget is finite, so the total accessible length is bounded by the number of chunks, heads, and layers that can participate.
  • The selector is non-parametric, so more complex reasoning tasks can expose weaknesses in the chunk scoring itself.

That is the main tradeoff. LongHeads gives you a cheaper and longer usable window, but you still depend on chunk quality. If the answer is split across a boundary or the query-chunk similarity is not sharp enough, the method can miss evidence even though the model is still “looking” in the right direction.

What to remember

LongHeads is easiest to remember as a retrieval loop built around attention heads:

  • turn each chunk into a semantic summary with the model’s own attention states,
  • keep the first chunk and last chunk as anchors,
  • let each head retrieve a few relevant middle chunks,
  • remap the selected pieces into a short, in-distribution window,
  • run normal causal attention only on that window.

That is why the method matters. It does not stretch attention over the whole document. It teaches each head to fetch a small, relevant context subset and then read that subset as if it had always fit inside the pretrained window.
