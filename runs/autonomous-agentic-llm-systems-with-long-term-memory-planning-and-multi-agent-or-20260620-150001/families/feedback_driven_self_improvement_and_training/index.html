How LLM Agents Learn From Feedback, Memory, and Retries

Many LLM agents can explain their mistakes, retry a task, or coordinate with other agents, but that does not mean they actually improve. A frozen agent may produce the same shallow reflection every time. A robot team may need real-world interaction before it learns a plan is bad. A coding agent may sample five trajectories that all fail in almost the same way. And even when an agent does discover a useful strategy, it may forget it by the next task.

That is the real topic here: not a single algorithm, but a shared interface problem between rollout, feedback, memory, and update. These methods ask a practical question: how do you turn failed or partial attempts into something that makes the next attempt better?

The Basic Improvement Loop

The recurring setup is a closed loop in which an agent acts, receives some signal about what went wrong, rewrites its internal guidance, and tries again:

Some methods keep the actor frozen and improve the reflection writer. Some insert a critic before execution so the system can reject weak plans earlier. Some optimize a multi-agent workflow directly. Others treat entire trajectories as editable search objects or distill many past trajectories into reusable memory.

The important idea is that “feedback-driven” does not always mean the same thing. Feedback can be a reward delta across retries, a learned advantage score, a centralized team critic, a selection score over competing trajectories, or a memory-quality signal accumulated over time.

Why Improvement Is Hard

The first bottleneck is frozen self-reflection. A model can describe a failure without producing a reflection that is actually useful for the next retry. That is the lesson behind Retroformer: if the actor is fixed, the component that writes reflections and prompt updates may be the real learning target.

The second bottleneck is expensive feedback loops. In embodied or multi-agent settings, waiting for full environment execution just to learn that a plan was weak is slow and wasteful. ReAd is an example of moving correction earlier by scoring candidate actions before they are executed.

The third bottleneck is coordination and credit assignment. In a multi-agent chain, one agent’s output becomes another agent’s input. When the team succeeds or fails, it is not obvious which step deserves credit. MARFT focuses on this sequential dependency problem rather than on single-agent retry behavior.

The fourth bottleneck is homogeneous rollouts. If every retry explores the same narrow region of the search space, “more attempts” just means “more copies of the same mistake.” SE-Agent attacks that by revising and recombining whole trajectories instead of sampling independent retries.

There is also a longer-horizon problem: forgetting. An agent may solve one task, learn a useful tactic, and then fail to retain it in a reusable form. EvolveR addresses that by distilling trajectories into compact principles that can be retrieved later.

Design Axis 1: What Gets Trained?

A clean way to read this area is to ask which component is allowed to change.

One option is to keep the actor frozen and train the retrospective module. Retroformer does exactly that: the acting LLM stays fixed, while a smaller model reads past states, actions, rewards, and goals, then writes a better reflection and plan for the next retry. This is useful when you trust the actor’s raw competence but not the quality of the self-critique wrapped around it.

Another option is to keep the planner mostly intact and train a scoring component around it. ReAd learns a critic from offline trajectories, then uses that critic to decide whether a joint or sequential action should be kept or rewritten before environment interaction.

A third option is to optimize the whole multi-agent workflow. MARFT treats agent teams as a sequential, role-dependent process and updates policies with trust-region style reinforcement learning so the team improves without losing language ability too abruptly.

The final pattern is to update behavior indirectly through memory. EvolveR compresses trajectories into principles and triples, curates them, retrieves them during later tasks, and then reinforces the policy on those experience-guided rollouts.

Design Axis 2: Where Does the Learning Signal Come From?

These systems do not all learn from the same signal.

Retroformer uses the return difference between retries. If a rewritten prompt produces a better next attempt, the reflection writer should get credit. That is a useful pattern whenever the quality of the text itself is hard to judge, but downstream task outcome is measurable.

ReAd learns from offline trajectory data, then turns critic estimates into actionable feedback during inference. The idea is not “wait and see” but “score and refine first.”

SE-Agent uses selection pressure over a pool of trajectories. Its feedback is not only final success but also reasoning quality and efficiency, which helps it escape local optima created by near-duplicate rollouts.

EvolveR adds a different kind of signal: whether a stored principle was actually useful when retrieved later. That turns memory from a passive log into something closer to a maintained experience base.

When Reflection Should Be Trained, Not Trusted

A common mistake in agent design is to assume that if a model can produce a reflection, that reflection is automatically useful. In practice, frozen reflectors often restate the trace, complain vaguely, or repeat the old plan in nicer words.

Retroformer teaches a sharper lesson: reward the next retry, not the reflection text. The reflector reads the failed trajectory and produces a concise failure summary plus an action plan. Those reflection-response pairs are stored, and the retrospective model is optimized with PPO so that later reflections become more action-guiding.

This matters because the best self-improvement target is often the interface layer between memory and action. On HotPotQA, the reinforced retrospective model reaches 54% success after four retries, compared with 52% for Reflexion and 40% for ReAct under the same GPT-4 actor setup. On AlfWorld, the same idea reaches 100% success after four retries, above Reflexion’s 85.07%. The smaller gain on WebShop is just as informative: better reflection helps, but it does not remove an exploration bottleneck.

When Feedback Arrives Too Late

If a plan is expensive to test in the world, waiting for full execution is the wrong place to learn. ReAd moves the correction point forward. Instead of treating the environment as the only judge, it trains a critic that scores candidate joint or per-agent actions, then asks the planner to keep or rewrite them based on that score.

That is easier to understand with one simple formula card.

Purpose: define whether a proposed team action is better than doing nothing.

Intuition: compare the candidate move with a joint WAIT baseline. If it does not beat that baseline by enough, rewrite it before execution.

The evidence in the compact input is selective, but it is enough to support the concept: ReAd’s joint and sequential variants outperform strong coordination baselines on DV-RoCoBench and adapted Overcooked-style settings, and the sequential version stays ahead under sudden environment resets as disturbances grow.

When the Team, Not the Prompt, Is the Problem

Single-agent retry logic does not transfer cleanly to multi-agent systems. In a team, agents may act asynchronously, hold different roles, and depend on predecessor outputs. That is why MARFT reframes the problem as a flexible Markov game instead of assuming synchronous, homogeneous workers.

The key design choice here is centralized credit assignment with agent-aware updates. Rather than optimizing each agent independently, MARFT uses a shared critic and PPO-style updates so the workflow stays aligned with the final team objective. It also pays attention to output granularity: long language actions can distort training unless token-level backups and normalization are handled carefully.

This is the broader takeaway: once agent outputs become inputs for other agents, the training target is the workflow. A chain of dependent role decisions needs a learning rule that matches the chain.

When Retries Are Too Similar to Help

Not every failure is caused by a weak reward model or a bad critic. Sometimes the agent simply keeps generating the same shape of trajectory. SE-Agent tackles that failure mode by treating whole trajectories as objects that can be revised, recombined, and refined.

That changes the search unit. Instead of asking for another fresh rollout, the system keeps a pool of pilot trajectories, reflects on where each one is weak, splices together useful segments across different attempts, and retains the best evolved trace. The result is a more population-based view of self-improvement: diversity is not noise to filter out, but raw material for better trajectories.

The representative result in the compact input is strong enough to anchor the point: on SWE-bench Verified, SE-Agent consistently beats SWE-Agent and SWE-Search across all five evaluated LLMs, with up to 55% relative improvement. The exact benchmark is less important than the lesson: when retries are too similar, better selection and recombination can matter more than just taking more samples.

When Memory Must Be Curated, Not Just Stored

Long-term improvement requires a memory that stays useful. Raw logs are too verbose, too redundant, and too noisy to retrieve directly. EvolveR responds by distilling trajectories into compact principles, tracking how often those principles are used and how often they help, then pruning the experience base over time.

Its scoring rule is simple enough to teach directly.

Purpose: rank a stored principle by how reliably it helps after retrieval.

Intuition: a good rule should succeed often relative to how often it is used, not just appear frequently in memory.

Formula:

s(p)=csucc(p)+1cuse(p)+2s(p)=\frac{c_{\text{succ}}(p)+1}{c_{\text{use}}(p)+2}s(p)=cuse​(p)+2csucc​(p)+1​

Symbols: ppp is one distilled principle. csucc(p)c_{\text{succ}}(p)csucc​(p) is the number of successful uses of that principle. cuse(p)c_{\text{use}}(p)cuse​(p) is the total number of times it was used or retrieved. s(p)s(p)s(p) is the resulting quality score.

Read-aloud meaning: a principle gets a high score when successful uses keep pace with or exceed total use.

Why it matters: this is what keeps the memory base compact and biased toward advice that actually improves future rollouts.

The main conceptual move is easy to miss: EvolveR does not just remember more. It stores less raw history and more reusable guidance, then trains the agent to act with that guidance in the loop.

What to Remember

The common theme is not a single algorithm. It is a shared interface problem. Agents improve only when feedback is converted into a useful object at the right place in the loop: a better reflection, an earlier action score, a team-level credit signal, a more diverse trajectory pool, or a curated long-term principle.

That framing helps separate surface behavior from real learning. An agent that retries is not necessarily improving. An agent that reflects is not necessarily learning from reflection. And an agent that stores history is not necessarily building memory. The real question is always the same: what structure carries information from one attempt into a better next attempt?
