Captain Agent: Adaptive Multi-agent Team Building

Captain Agent starts from a simple observation: for a complicated task, the right team is usually not a fixed team. A search-heavy question needs different help from a math derivation. A coding step needs different help from a verification step. A data analysis task may need one specialist early and another later. If the system keeps the same roster alive for every step, it drags irrelevant context through the whole conversation and makes the active task harder to solve.

Captain Agent is the paper’s answer to that problem. It behaves like a manager that repeatedly asks, “What does this subproblem need right now?” It then builds a small team for that subproblem, lets the team work in a nested group chat, and uses a reflector to decide whether the task is actually converging or whether the team should be reconfigured before the next round.

When you look at Figure 1, the main thing to notice is that the system lives outside the nested chat and keeps steering the process from the outside. It does not hand the whole task to one permanent committee. It creates a committee for the current step, listens to the result, and then decides what kind of committee should exist next.

What should stand out in the figure is the separation between the main conversation and the nested conversation. That separation is the core design move. It keeps the outer task state clean while allowing the inner team to debate, use tools, and produce a subtask-level outcome.

Why a fixed roster is the wrong starting point

Static multi-agent systems look convenient at first. You define a few roles, give them all the same task, and let them talk. The weakness shows up as soon as the task changes shape. The expert who is useful at the start may be noise later. The group may keep carrying around roles that are no longer relevant. The result is longer context, more duplicated work, and a higher chance that the team will keep reasoning from stale assumptions.

Captain Agent attacks that failure mode directly. The main conversation does not assume the full team is already known. Instead, it begins by decomposing the current task into subtasks and role requests. Each role request is just a short natural-language description of the kind of help needed for this round. That seems modest, but it matters because the system is trying to match capability to the current subproblem rather than to a fixed label.

The next step is semantic retrieval over two libraries: an agent library and a tool library. Think of this as a structured search over a catalog of available specialists and utilities. The role description is compared against the library descriptions, and the closest matches are surfaced as candidates. The paper does not need an exact library hit for every role. That would be unrealistic. So it adds an LLM filter after retrieval.

That filter is an important design choice. Retrieval is broad and cheap; filtering is narrow and deliberate. Captain Agent first gathers a candidate set, then asks an LLM to choose the best agent from those candidates or abstain entirely. The abstention option is not a corner case. It is what keeps the system from forcing an irrelevant specialist into the team just because something in the library looked vaguely similar.

If the filter says none of the candidates is a good match, Captain Agent does not get stuck. It generates a new agent step by step. The paper describes this as first drafting a basic profile, then combining that profile with general task-solving and environment instructions, and finally binding suitable tools from the tool library. Once the new agent is created, it gets written back into the agent library for future reuse.

That gives the system a nice property: it can cache expertise it had to invent on the fly.

The cache is lightweight on purpose. Captain Agent stores only basic team details, not the full conversation history. That means it can reuse the same team later with a new instruction without paying the full team-building cost again. You can think of it as keeping the contact list and skill summary, not the transcript of every meeting.

This is also why the agent and tool libraries matter so much. The agent library gives the system reusable personas and task styles. The tool library gives those personas something concrete to do. The ablations in the paper show that removing either library hurts performance, and that the best results come when both are present. In other words, this is not just a prompting trick. It is a search-and-assemble system.

What happens inside the nested conversation

Once the team is built, Captain Agent opens a nested group chat for the current subtask. This is separate from the main conversation with the user. The nested team shares its own conversation history, but it does not get to see the outer conversation history directly. That boundary is useful because it keeps the specialist discussion focused on the subproblem instead of letting every earlier message leak back in.

The nested chat also has a manager. It adaptively chooses the next speaker from the team members based on the conversation history. So the system is not just “everyone talks in turn.” It is closer to a moderator deciding which specialist should respond next.

Another detail is easy to miss but important: each agent can speak in natural language and can also emit Python code blocks. The paper uses raw Python function format for tools, which means agents can do more than pick a single tool from a menu. They can stitch together the functions they were given and write a one-off helper in code. That turns the tool library from a static list into something more like a programmable kit.

This is the part of the method where the system starts to feel like a small runtime instead of a chatroom. The agent team is not only debating. It is also composing and executing little pieces of action logic.

But group discussion has a risk: it can get stuck. If one agent disputes another and neither side has enough evidence to settle the issue, the chat can keep circling until the round budget is exhausted. The paper explicitly calls out this failure mode.

That is why reflection exists.

The reflector is a separate LLM that reads the nested conversation after it reaches a terminal condition. It summarizes the discussion, checks for disputes, and sends a reflection report back to Captain Agent. Captain Agent then uses that report to decide whether the current team is still appropriate, whether the subtask instruction needs to change, or whether it should stop and return a final answer.

The reflector is not trying to solve the task directly. It is acting like a control signal. It tells the outer loop whether the inner loop produced a healthy result or a conflict that is likely to poison the next step.

Figure 3 makes that role easier to see. The main point is not the exact score in the ablation. The point is the mechanism: when the reflector is present, the system gets an explicit checkpoint that can catch bad conversations before they spread into the next round.

What to notice in the figure is that reflection is not a cosmetic add-on. It is a separate control stage that helps prevent error propagation. The reflector cannot magically repair a poorly framed subtask, but it can tell Captain Agent that the current path looks unstable and should not be blindly continued.

Why adaptivity beats a frozen roster

The strongest intuition behind Captain Agent is also the simplest: different steps need different teams.

Figure 2 shows the paper’s static-versus-adaptive comparison. The adaptive version is allowed to change team members and instructions across rounds. The static version builds a team once and keeps using it. That difference sounds small, but in practice it changes how much irrelevant context survives from one step to the next.

What should stand out is that the adaptive line generally stays ahead. The reason is not mysterious. When the problem changes, the roster should change too. A team that is perfect for subtask A may be bloated or misaligned for subtask B. Captain Agent can throw away duplicate roles, drop stale context, and recruit a different specialist when the task demands it.

That also explains why the method is a good fit for heterogeneous workloads. In the paper’s evaluation, the same framework is used across mathematics, programming, data analysis, chemistry, physics, and information retrieval. Those scenarios do not require the same kind of help. A fixed team would either have to be overstuffed with every possible specialist or risk missing the one that matters most at the current step.

Adaptive team building avoids that tradeoff. It only brings in the experts needed now.

What the results say, without turning this into a benchmark dump

The evaluation mainly supports one message: adaptive orchestration is doing real work here.

On the main comparison set, Captain Agent reaches 84.25 average accuracy across the five task families reported in the paper, above AutoGen Assistant + Executor at 79.89. On GAIA validation, it also edges out AutoGen: GAIA_Orchestrator, with 40.60 average accuracy versus 39.39. Those numbers are not the interesting part by themselves. The interesting part is that the gains come from changing the organization of the work, not from asking for a different prompt template.

The paper’s ablations sharpen that point. Adaptive team building beats a static team in four of five scenarios, and the reflector improves performance across all scenarios. The agent library and tool library also matter independently, but the best behavior shows up when both are used together. That makes sense if you think about what each one contributes. The agent library supplies the right kind of specialist. The tool library supplies the right kind of action surface. The full method needs both the right actor and the right instrument.

There is also a useful cost lesson. The paper shows that a smaller backbone can still be competitive inside this orchestration pattern, especially when the backbone follows instructions reliably. So Captain Agent is not only a story about scaling up model size. It is also a story about getting more out of a less expensive model by controlling how and when it works.

What the paper assumes, and where it is still expensive

Captain Agent is flexible, but it is not free.

The most obvious cost is orchestration overhead. Every step can involve retrieval, filtering, possible agent generation, a nested chat, and reflection. That is more work than a single-agent baseline. The paper is explicit about that tradeoff.

There is also a dependency on library quality. If the agent library is thin or poorly named, retrieval will not find the right specialist. If the tool library does not expose the right operations, the nested team will have to improvise more than it should. The method is therefore only as strong as the libraries it can search and assemble.

Instruction following matters too. Captain Agent works because the outer controller, the nested team, and the reflector all respond to the prompt structure in a predictable way. If a backbone is weak at following those instructions, the whole orchestration becomes less reliable.

Finally, the implementation is built on a customized AutoGen-style communication stack. That is fine inside the paper’s setup, but it also limits portability. One of the open questions the authors raise is whether the same team-building idea can move cleanly across runtimes.

The reflector has its own limitation as well. It can spot disputes and summarize them, but it cannot deeply repair a bad subtask decomposition on its own. If the original instruction is too ambiguous, reflection can warn you that something went wrong, but the system still has to choose a better next move.

What to remember

Captain Agent is easiest to remember as a manager that refuses to keep the wrong team alive.

It breaks a task into subtasks, searches for or creates the right specialists, gives them the tools they need, lets them debate inside a bounded nested conversation, and then uses a reflector to decide whether the current team should continue or be rebuilt. The method works because it treats team composition as a live control problem instead of a one-time setup step.

If you keep one sentence from the paper, make it this: for complex tasks, do not freeze the expert roster up front, assemble the right specialists for the current subproblem, and use reflection to decide what the next team should look like.
