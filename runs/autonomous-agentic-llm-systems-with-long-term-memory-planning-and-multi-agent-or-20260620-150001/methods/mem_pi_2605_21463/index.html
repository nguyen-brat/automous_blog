Mem-π

Mem-π turns memory from a retrieval step into a separate policy. Given the current task context, a dedicated memory model either writes a short hint or stays silent. That sounds like a small change, but it attacks two common failures at once: retrieval often returns a static snippet that is close in wording but wrong for the current situation, while always-on generation can inject a confident clue that the agent should have ignored.

The useful mental model is not “bigger memory”. It is “smarter memory behavior”. Mem-π learns when memory is worth speaking and what form that speech should take.

This figure is the whole pipeline in one view. First the paper distills an offline experience bank into a parametric memory model. Then it adapts that memory model with downstream task reward so it learns the generate-versus-abstain choice. At inference time, the memory policy sits in front of the agent: it either prepends a short hint or leaves the agent alone.

What matters in that flow is that memory is now a learned module with its own behavior. It is not just a database.

Stage 1: teach memory what a useful hint looks like

The first stage is supervised experience distillation. The paper starts with paired examples: a task context and a short guidance hint that proved useful before. The memory model learns to predict the hint token by token from the context.

The goal is not to memorize the reference hints verbatim. It is to internalize the reusable pattern behind them. If the current task is similar to a past one but not identical, the model should still be able to produce a fresh hint that fits the new context.

This warm start matters because it gives the memory policy a useful prior before any RL begins. If you tried to learn both “how to write a hint” and “when to write one” from scratch, the second stage would have to discover the format and the policy at the same time. That is slower and much less stable.

A helpful way to think about Stage 1 is: the model is learning the vocabulary of good advice.

Stage 2: teach it when to speak

The second stage is where Mem-π becomes adaptive. The memory model now emits either [GENERATE] plus a hint, or [ABSTAIN] and nothing else. The downstream agent is the thing that actually tries to solve the task; the memory model is judged by whether its help improved that downstream outcome.

That is a better objective than imitation alone. A hint can look plausible and still be harmful. If the agent already knows what to do, extra memory may only add noise. If the task is brittle, one short and well-targeted hint can flip failure into success. The policy has to learn both cases.

Mem-π uses group-relative policy optimization (GRPO) for this adaptation stage, because it can compare a group of sampled outputs without training a separate value model. In practice, the paper samples a structured group and measures which branch leads to better downstream success. The important part is that the reward comes from the agent’s outcome, not from how close the generated text looks to a reference.

The paper also adds a small length penalty to generated hints. That keeps the memory policy honest: it should produce concise guidance, not a second answer wrapped in memory form.

Why the decision and the content need different credit

The hardest part is credit assignment. The first few tokens decide whether memory exists at all. The rest of the tokens decide what that memory says. If you dump one flat policy-gradient signal onto the whole output, the long hint can dominate the short routing decision.

That is the reason for the decision-content decoupled objective. Mem-π does not just ask “Was this output good?” It asks two separate questions:

  • Was generating memory better than abstaining?
  • If generation was better, which generated hint was best?

That split is the core novelty.

Purpose: give the routing tokens and the hint tokens different learning signals.

Intuition: the model should learn when to stay silent using a direct generate-vs-abstain comparison, but the content of the generated hint should only be trained when generation actually beats abstention.

Read-aloud meaning: use the routing signal for the decision tokens, and only let the hint tokens learn when generating a hint is actually the better choice.

Why it matters: this stops long generated hints from overpowering the short routing decision. It is the part that makes abstention learnable instead of being drowned out by the hint text.

Read that formula in plain language: the early tokens learn whether memory should appear at all. The later tokens learn what the memory should say, but only when speaking was actually the right call.

Why the structured rollout matters

This is the subtle part that makes the whole system work. If you sample outputs in the usual way, the model may keep generating hints and rarely try abstention, especially after supervised warm start. Then it never gets a clean comparison between “no memory” and “memory” on the same context.

Mem-π fixes that with a structured counterfactual rollout. Every training group contains one forced abstain branch and several generate branches. That guarantees that the policy sees both choices on the exact same task. It also makes the comparison fair: the model is not learning from different tasks or different observations, just from the downstream effect of adding or removing memory.

A toy way to picture it: if the task is easy, the abstain branch may already solve it, so generation gets no reward and the policy learns to stay quiet. If the task is brittle, maybe a count, identifier, or format is easy to get wrong, and a short hint changes failure into success. Then the generate branch wins, and only then does the content part of the policy get reinforced.

That is why the paper insists on the generate-versus-abstain comparison. Without it, the model would mostly learn to be a verbose generator.

What the case studies show

The ablations tell you the design is necessary, but the case analysis shows why it feels different from retrieval. The interesting cases are not just “memory helps”. They are cases where generated memory can adapt the stored advice to the current task, or refuse to inject a misleading one.

What to notice here is the contrast with retrieval. In one example, the retrieved memory is close but numerically off, so a copied snippet would tell the agent the wrong count. Mem-π rewrites the hint so it matches the current query instead of copying the stale number. In another example, the retrieved hint is too narrow and would bias the agent toward the wrong subtask. Mem-π abstains, which is often better than forcing a bad clue into the prompt.

That is the real advantage over retrieval-based memory. Retrieval can surface a relevant precedent, but it cannot decide to paraphrase it or to ignore it entirely. Mem-π can do both.

What the results say about the method

The headline result is not just that Mem-π does well. It is that the method improves both success and efficiency. The paper reports 55.4% average success rate versus 45.3% for the base agent, and Mem-π uses about 138 memory tokens per task on average. That is the right kind of gain: better success with less prompt clutter.

The ablations are just as informative. Skipping the supervised warm start hurts. Removing structured rollouts hurts. Removing the \Delta gate hurts. Collapsing everything into one RL stage hurts the most. In other words, the method depends on all three ideas working together:

  • a supervised prior for what a useful hint looks like,
  • a forced comparison between generation and abstention,
  • a separate learning path for routing versus content.

That combination is what turns memory into a selective policy instead of a noisy extra prompt.

The same policy also transfers to a stronger unseen agent, which suggests the memory hints are useful beyond the executor they were trained with. That is a good sign for agent memory in general: the memory model is learning task guidance, not just overfitting to one downstream policy.

Limits and tradeoffs

Mem-π is cleaner than retrieval-only memory, but it is not free.

  • It needs a separate memory model plus extra RL training.
  • It depends on reward signals and a curated experience bank.
  • The generated hints are useful, but they are not inherently attributable to a source.

There is also an open question about online updates. The paper shows a good offline-to-adaptation pipeline, but it does not fully solve continuous self-updating memory. That is the next obvious step if you want a memory system that keeps learning from new experience.

What to remember

Mem-π is easiest to remember as a small advisor with two jobs: speak when a hint helps, and stay silent when it does not. Stage 1 teaches the advisor how to write short, useful hints. Stage 2 teaches it when those hints are worth paying for. The decoupled objective is what keeps the two jobs from interfering with each other.

If you want the one-line version, it is this: selective generative memory beats always-on memory because the best hint is sometimes no hint at all.
