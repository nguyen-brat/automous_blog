OneFlow: Single-Agent Execution of Multi-Agent Workflows

OneFlow asks a blunt question: if every role in a workflow uses the same base model, why run several separate copies of that model? The paper keeps the workflow but collapses the executor. One conversation plays the roles in sequence, carries the history forward, and reuses the same KV cache instead of paying the prefill cost again at every turn.

You can think of the whole method as a small control loop:

That is the core trick. The paper is not saying multi-agent workflows are useless. It is saying that, when the agents are homogeneous, the expensive part is often the duplication, not the decomposition.

What to notice in the figure is that a workflow is a scripted interaction pattern, not a pile of independent brains. The code on the right is just a way to express the order of specialized turns on the middle panel. Once you see the workflow that way, the natural question becomes: if the roles are already just prompts, tools, and routing logic on the same base model, why not let one model play all of them one after another?

Why a single conversation can replay the workflow

The paper’s formal claim is simple in spirit. If the workflow only depends on what is visible in the conversation, and the outside world behaves deterministically, then replaying the roles inside one conversation can match the original multi-agent transcript behavior.

The conditions matter:

  • Tool outputs should be deterministic given the same inputs.
  • Routing should depend only on the visible history and tool results.
  • Decoding should be fixed, or the same randomness should be shared across the replay.

Under those assumptions, the model is not inventing a new behavior. It is just wearing different role prompts in sequence while seeing the same history the original workflow would have seen.

A simple toy example makes this easier to remember. Imagine a planner drafts a solution, a critic checks it, and a formatter rewrites it. If all three roles are the same base model, the only thing that changes between turns is the instruction and the visible conversation state. One model can play planner, then critic, then formatter, without changing the workflow logic at all.

This is also where the boundary shows up. If different roles need different base models, the replay trick stops being exact, because KV cache cannot be shared across model boundaries. OneFlow is strongest when the workflow is homogeneous all the way down.

What OneFlow is optimizing

The search stage is not trying to find the most elaborate workflow. It is trying to find the workflow that gives the best balance between quality and cost.

Formula Card: OneFlow’s search objective

Purpose: choose the workflow that solves the task well without spending more than necessary.

Intuition: the best workflow is not the biggest team. It is the smallest workflow that still gives you the right answer at an acceptable cost.

Formula:

W∗=arg⁡max⁡W∈W[α⋅P(W,T)−β⋅C(W,T)]W^* = \arg\max_{W \in \mathcal{W}} [\alpha \cdot P(W,T) - \beta \cdot C(W,T)]W∗=argW∈Wmax​[α⋅P(W,T)−β⋅C(W,T)]

Symbols:

  • W^* is the best workflow for task T.
  • \mathcal{W} is the space of candidate workflows.
  • P(W,T) is the performance of workflow W on task T.
  • C(W,T) is the cost of running workflow W on task T.
  • \alpha and \beta are weights that trade off quality against cost.
  • T is the task being solved.

Read-aloud meaning: pick the workflow that gives the best mix of performance and cost.

Why it matters: this objective is why OneFlow can prefer a compact workflow with richer prompts over a wider workflow with more agent copies. The method is explicitly rewarding efficiency, not just accuracy.

The paper uses two phases, and it helps to keep them separate. First, OneFlow searches for the workflow. Then OneFlow, executed as a single agent, replays that workflow in one conversation. The first phase is optimization. The second phase is runtime.

What to notice in the figure is the loop shape. The search starts from the simplest input-output workflow. Selection chooses a promising candidate, but it does not commit blindly. Expansion uses two meta-models: one pushes creative workflow edits, and the other checks whether those edits are actually worth the cost. Evaluation runs the candidate on held-out data, and backpropagation stores the result so the tree remembers which edits helped.

That is why the paper uses dual meta-LLMs. The designer is there to explore useful new workflow shapes. The critic is there to stop the search from drifting into a baroque design that looks clever but is too expensive to run. Together they search for a workflow that still solves the task but is easier to replay in one model.

From search to replay

Once the compact homogeneous workflow has been chosen, execution changes in a very practical way. The same base model now plays each role in order, but it stays inside one conversation.

That matters because a transformer does not have to reread the entire prefix from scratch on every turn. It can keep the attention state for the shared history in its KV cache and only process the new tokens that were just added. In plain language, the model keeps the notebook open instead of photocopying the whole notebook for every role.

That is the source of the cost win. Separate agent instances repeatedly pay for the same shared context. A single conversation pays for that context once, then reuses it as the roles change.

The paper also points out a practical ceiling: long workflows still grow the context window. If the sequence gets too long, the system may need compaction or summarization to keep the useful history while trimming noise. The important constraint is that the compaction must preserve routing decisions and the information that later turns actually depend on.

This is why OneFlow is most attractive when the workflow differs mainly by prompts, tools, and routing, not by model identity. In that regime, the method is basically saying: keep the decomposition, throw away the duplicate prefill.

Where the idea stops working

The limitation is not subtle. If a workflow truly needs multiple base models, then the single-agent replay trick cannot share KV cache across those model boundaries. That is the hard edge of the method.

The paper’s heterogeneous pilot is useful here because it keeps expectations honest. A heterogeneous workflow can be worthwhile when specialization really matters, but the pilot does not clearly show that model diversity automatically beats the best homogeneous design. That suggests a lot of the gain in many agentic systems comes from decomposition and prompt structure, not from multiplying model identities.

The open-weight experiment tells a similar story. Even when the executor is not a closed API model, the replay mode stays competitive because the cache reuse still helps. Tool-heavy planning tasks show the same pattern: the workflow can stay effective while the runtime gets cheaper.

So the right mental model is not “single-agent is always better.” It is “test single-agent replay first when the workflow is homogeneous, and only keep multiple models when the extra specialization is worth the cache loss.”

What to remember

OneFlow is easiest to remember as a two-step move.

First, search for the smallest workflow that still solves the task well. Second, let one model replay that workflow with a shared cache instead of splitting the same model into several separate instances.

That is the larger lesson of the paper. A lot of agentic performance comes from task decomposition, routing, and clear role prompts. If the roles are really just different instructions on the same base model, then the expensive part is often the duplication. OneFlow turns that insight into a practical baseline: keep the script, collapse the cast, and see how far one conversation can go.
