Plan-and-Act

Long-horizon web tasks break when one model has to do two jobs at once: decide the strategy and execute the next action. Plan-and-Act separates those jobs. A planner writes the route at a high level, an executor grounds that route in the current page, and after every action the planner gets another look at the state so it can revise the route if the page changed underneath it.

That is the core idea. The paper is not trying to make one giant prompt smarter. It is trying to move the right kind of thinking to the right layer.

What to notice here is the division of labor. The planner does not need to know how to click a button, and the executor does not need to invent the whole strategy from scratch. The environment feeds back into both modules, so the system can keep its plan live instead of pretending the page is static.

Why the split matters

If one model has to both plan and act, it gets overloaded. It must keep the goal in mind, figure out what the page means, decide which step comes next, and also translate that step into the exact action syntax the environment expects. On a simple task that can work. On a long task with changing page content, it tends to drift.

Plan-and-Act avoids that by making the planner responsible for strategy and the executor responsible for grounding. The planner can say “go to contributors, identify the top contributor, then follow them.” The executor then converts the current step into a click or type action that actually fits the current HTML. After the action, the executor trims away useless context so it does not carry around a growing pile of redundant page text.

That last detail is small but important. The executor is not meant to hold the whole world in memory. It only needs enough current state to ground the next move.

Static plans are where things go stale

The paper’s main warning is easy to understand: once the page changes, a static plan can become wrong. Search results appear, labels shift, a query returns nothing, or the content you need is only visible after a click. A one-shot plan assumes the page will keep behaving as expected, and web pages rarely cooperate.

That is why the paper adds dynamic replanning. After each executor step, the planner gets the updated HTML, the previous plan, and the action history, then writes a new plan for the next step.

What to notice in this figure is that the plan is not a frozen outline. It is rewritten after the environment changes. The new observation can be folded into the remaining steps, so the planner carries forward what it just learned instead of forcing the old route to survive unchanged.

That is also how the method handles memory without a separate memory module. The evolving plan itself becomes the place where the useful state is preserved.

How the planner gets enough supervision

This is where the paper does something practical. It does not try to train the planner from scratch using hand-written plans, because that is expensive and brittle. Instead, it builds training data from successful trajectories and then expands those trajectories into more plan examples.

The synthetic data pipeline has three broad stages:

  1. collect successful action trajectories,
  2. reverse-engineer grounded plans from those trajectories,
  3. expand the plan set with synthetic variants and failure-targeted examples.

What to notice here is the direction of the pipeline. The paper starts from actions that really worked, then works backward to the plan that would have produced them. That is much safer than asking a model to invent plans in the abstract for websites it has not actually navigated.

Start with trajectories that really work

For the executor, the simplest useful data is a trajectory: a user instruction, the current page state, an action, the next page state, and so on until the task succeeds. The paper generates those trajectories in an Alpaca-style loop. It samples seed user queries, uses a teacher model to produce similar instructions, filters out impossible ones, has a demonstrator agent try the task, and then keeps only the successful trajectories with an outcome reward model.

That gives the executor exactly what it needs: HTML-action pairs grounded in real page behavior.

It also explains why prompting alone is not enough. The executor is not just choosing actions in the abstract. It has to learn how those actions look against the actual HTML structure of a live page.

Reverse-engineer the plan from the action trail

Planner data is trickier. A naive approach would be to feed a user query to a teacher model and ask for a plan. The paper argues that this fails because the teacher does not see the live website and does not know which steps are actually executable there.

So the paper reverses the problem. It gives the teacher a successful trajectory and asks it to infer the high-level plan behind it. The result is a grounded plan: each plan step is tied back to the low-level actions that made it real.

That grounding is the key trick. The plan is not just a nice-looking outline. It is supposed to match how the task was actually carried out on the page. In other words, the model is not learning to write generic advice. It is learning to write plans that can survive contact with the browser.

Why plan expansion is necessary

There is a data imbalance hidden inside the pipeline. One successful trajectory creates many executor examples, because each action step becomes a training pair. But that same trajectory only produces one planner example.

Plan-and-Act fixes that by expanding the planner set. It samples existing query-plan pairs and asks the teacher model to generate similar but new pairs. This is the part that scales the planner from a small seed set into something broad enough to generalize.

The paper reports generating 10,000 additional query-plan pairs this way, which also helps keep the planner from overfitting to the small original set. The important point is not the exact count. It is the mechanism: use existing grounded plans as seeds, then generate more plans that stay close enough to the real structure of the task.

The paper goes one step further with targeted augmentation. After running the model on a held-out validation set, it classifies training examples that look relevant to the observed failure patterns. Those examples become seed data for another round of plan generation. That is a simple form of curriculum design: when the model struggles in a particular way, feed it more examples that look like the failure it needs to overcome.

A small reasoning trace helps both sides

Plan-and-Act can also add chain-of-thought traces before the plan or before the action. The paper uses this as an optional extra layer, not as the core method.

The idea is straightforward. Some models do better when they first write down a short reasoning trace and then commit to a plan or action. That extra step can help the planner stay organized and can help the executor choose a grounded action from a crowded page. It is not magic, but it often gives the rest of the system a cleaner intermediate state to work with.

What the results are really saying

The results fit the story the method tells.

  • A base executor with no planner is very weak on WebArena-Lite.
  • Adding a planner helps, but only when that planner is trained on grounded synthetic data instead of being left zero-shot.
  • Expanding the planner data matters a lot, because plan quality turns out to be the bottleneck.
  • Dynamic replanning is the biggest jump, because it keeps the strategy aligned with the live page state.
  • Adding CoT gives a final bump, pushing WebArena-Lite to 57.58%.

The most useful numbers are the ones that show the progression. Without a planner, the base executor reaches only 9.85% on WebArena-Lite. With a stronger planner and synthetic plan expansion, the system rises to 29.63%. Dynamic replanning lifts it to 53.94%, and CoT pushes it to 57.58%. On the wider benchmarks, the same approach reaches 48.15% on WebArena and 81.36% on WebVoyager with stronger backbones.

The pattern is more important than the leaderboard position. The paper is showing that long-horizon web agents improve when strategy is made explicit, grounded in real trajectories, and refreshed after each action.

What still hurts

The method is not free of assumptions.

  • It still needs some successful trajectories to bootstrap the synthetic data pipeline.
  • Replanning after every action can slow execution.
  • WebVoyager-style settings without training data still need a model to collect the first trajectories.

So Plan-and-Act is not a universal answer to web automation. It is a strong training and control recipe for cases where you can get a foothold in the environment and then use that foothold to teach the planner better behavior.

There is also an open design question in the paper itself: should the executor learn when replanning is actually necessary? If the environment is stable, replanning every step may be more work than needed. If the environment is volatile, frequent replanning is exactly what keeps the trajectory on track. The paper chooses the simple version first and leaves that efficiency tradeoff for later work.

What to remember

Plan-and-Act is easiest to remember as plan above, action below, refresh after each step. The planner carries strategy, the executor carries grounding, and the plan is rewritten whenever the page state changes enough to matter.

The deeper lesson is that long-horizon agents do not fail only because they lack intelligence. They fail because strategy and execution get tangled together. Plan-and-Act untangles them, trains the planner on grounded traces, and then keeps the plan alive as the web changes under the agent.
