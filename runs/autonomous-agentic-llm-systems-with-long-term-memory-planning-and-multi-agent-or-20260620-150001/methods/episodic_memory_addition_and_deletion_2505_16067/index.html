How Memory Management Impacts LLM Agents: An Empirical Study of Experience-Following Behavior

LLM agents often treat memory like a notebook of past wins. This paper argues that the notebook is not passive storage. It is a replay buffer that keeps shaping the next answer, because the agent does not just look up old experiences - it tends to imitate them.

The authors study one simple question: what happens when you keep adding to that notebook, and what happens when you delete from it? They test that question on one synthetic agent and three real ones spanning EHR query answering, autonomous driving, and IoT traffic classification. Then they follow the same memory bank over long runs to see whether it helps, stalls, or slowly poisons future behavior.

The memory loop is the real object of study

The key move is to treat memory management as a closed loop, not a one-time setting. On each task, the agent retrieves past query-execution pairs, uses them as demonstrations, produces a new execution, and then memory management decides whether that new experience should stay or be removed later.

This diagram is the backbone of the paper. What matters is not just “retrieve memory” but “retrieve, execute, judge, add, and sometimes delete”. Every finished task can become future evidence, which means memory quality can improve or decay over time depending on what you keep.

A nice way to think about it is this: the memory bank is not a static archive. It is a living prompt bank. If you add a good example, future tasks near it may benefit. If you add a noisy or misleading example, the next similar task may inherit that mistake.

Why add-all is such a dangerous baseline

The paper compares four addition policies:

  • fixed memory, where the bank never grows
  • add-all, where every experience is stored
  • coarse selective addition, where an automatic judge filters records
  • strict selective addition, where a strong oracle-like judge keeps only high-quality records

This setup reveals a useful surprise: more memory is not automatically better. Across all four agents, the strict selector is the strongest addition policy, while add-all usually makes the bank much larger and long-term performance worse. That is the first sign that memory management is an optimization problem, not a storage problem.

The reason is simple. A memory bank is only useful if the retrieved examples are both relevant and reliable. If you keep everything, you also keep failures, half-correct trajectories, and examples that look plausible but are bad templates for the next task.

The fixed-memory baseline shows the opposite tradeoff. It cannot grow polluted, so it can stay surprisingly strong. But it also cannot adapt much, because the bank is frozen. That gives the paper a clean lesson: the best policy is not “always add” and not “never change”. It is “add only what will still help later”.

Experience-following: memory acts like replay

The most important empirical pattern in the paper is what the authors call experience-following. When a new query looks similar to a retrieved memory input, the agent’s output tends to look more like the stored output too. In other words, retrieval is not just providing context. It is steering behavior.

This scatterplot is the paper’s signature result. The important thing to notice is the upward trend: as input similarity rises, output similarity also rises. The agent is not merely consulting examples; it is copying them more strongly when the current task feels close to the retrieved one.

The synthetic RegAgent makes this easiest to see because the task is controlled and the answer depends heavily on demonstrations. In that environment, the paper can separate pure replay effects from unrelated noise. The same pattern still appears in the real agents, which makes the claim much stronger than a toy-only result.

The practical meaning is straightforward. A good memory example behaves like a good replay sample in reinforcement learning: it can reinforce the right move. But the same mechanism also means a bad example can be replayed and amplified.

Why bad memories spread

Once you accept experience-following, the next failure mode becomes obvious. If a retrieved record contains an error, the agent may copy that error into the current execution. If that new execution is then written back into memory, the mistake is no longer isolated. It has become part of the future training set.

That is error propagation.

The paper makes this point with an oracle-style comparison: it keeps the same retrieved memories, but replaces the agent’s own execution with error-free output. When that closes the performance gap, you learn that the damage was not just bad retrieval. The damage came from the agent’s own imperfect execution being fed back into memory and then reused again.

This is the part that changes how you should think about episodic memory. A stored episode is not a neutral log. It is a source of imitation. So even one sloppy execution can contaminate later tasks if it keeps getting retrieved.

Deletion should look at utility, not just age

The second half of the paper turns to deletion. That matters because real memory cannot grow forever, and because some entries should be removed even before capacity becomes a problem.

The authors compare three deletion policies:

  • periodic deletion, which removes records based on age or retrieval frequency
  • history-based deletion, which deletes records that repeatedly prove unhelpful downstream
  • combined deletion, which uses both ideas together

The most interesting idea here is history-based deletion. Instead of asking only “Has this record been used a lot?” it asks “When this record was used, did it actually help future execution?” That is a much better question for agent memory, because a record can be correct in isolation and still be a poor demonstration for similar future tasks.

In the paper’s terms, this captures misaligned experience replay: a record may look fine, but once it is reused in a slightly different context, it gives the wrong kind of guidance. The point is not that the original episode was worthless. The point is that its downstream utility was low.

That is why future task outcomes are so valuable here. They act like free labels for memory quality. You do not need a separate annotation pass for every stored trajectory. The agent’s own later performance tells you whether a record deserves to stay.

What deletion actually buys you

This is where the paper gets practical. Periodic deletion does what a simple cleanup rule should do: it keeps the bank bounded and often removes some stale clutter with little damage. History-based deletion can do better when the evaluator is reliable, because it removes records that look acceptable on paper but keep leading the agent astray.

The combined strategy is useful when you want both control and compactness. It prunes old records and also removes low-utility ones, so the bank stays smaller without collapsing performance.

The best intuition is that deletion is not only about saving space. It is also about protecting the prompt from bad exemplars. If the bank is a living training set, deletion is the part that prevents bad replay from taking over.

Why the paper tests shift and memory limits

The final experiments matter because they make the whole story feel real. In a shifted task distribution, yesterday’s useful example can become today’s stale example. Under those conditions, even a simple periodic rule can help stabilize performance by removing memories that belong to the old regime.

When the memory budget is fixed, the same logic still applies. The bank should keep the most useful records, not the largest pile of records. The paper shows that a capped, selective memory can still perform well, which is a nice reminder that naive growth is not a requirement for long-term improvement.

What to remember

The paper is really making one compact claim: memory policy is part of the model’s behavior, not just a storage detail.

If you remember only three things, keep these:

  • retrieved memories are often imitated, not merely consulted
  • wrong or misaligned memories can compound their own damage over time
  • the best memory bank is selected by downstream utility, not by simple accumulation

The limits

This is an empirical study, not a theory paper. It focuses on episodic memory addition and deletion, so it does not cover structural edits, merging, summarization, or reflection. It also relies on evaluator quality, which means the same policy may transfer unevenly across agent families.

Even with that caveat, the paper gives a useful mental model: treat memory as a replay buffer with quality control. If you manage it well, the agent can keep getting better. If you manage it naively, the bank can become a machine for repeating its own mistakes.
