Agentic Memory: Learning Unified Long-Term and Short-Term Memory Management for Large Language Model Agents

AgeMem starts from a complaint that long-horizon agents run into almost immediately: they do not merely need more memory, they need better control over memory. If the system keeps everything, the prompt fills with noise. If it keeps too little, the agent loses the few facts that matter later. If it retrieves blindly, it drags old context back at the wrong moment. AgeMem treats that as a policy problem, not a storage problem.

The key move is to make memory actions part of the agent itself. Instead of hiding long-term memory and short-term context behind separate heuristics, AgeMem lets one policy decide when to Add, Update, Delete, Retrieve, Summary, and Filter.

What matters in that figure is the architectural shift. The older designs leave short-term context mostly static and bolt long-term memory onto a separate controller. AgeMem brings both sides into the same decision loop. That means the model is not just answering with memory nearby. It is actively deciding how memory should change.

Two memory jobs, one policy

AgeMem still respects the difference between the archive and the live window. It just stops treating them as two separate systems with two separate rule sets.

Long-term memory is the durable store. It is where the agent writes facts that should survive across turns or sessions, revises them when later information changes the picture, and removes entries that have gone stale.

Short-term memory is the active context. It is the material the model is currently thinking over, so it needs to stay compact and relevant enough to fit inside the context window.

The six tools split cleanly across those two jobs:

  • Add, Update, Delete keep long-term memory coherent.
  • Retrieve, Summary, Filter keep the live context usable.

The short-term tools are about keeping the working window healthy. Retrieve brings useful long-term memories into the current context. Summary compresses a span of conversation so the prompt does not keep growing forever. Filter removes irrelevant or distracting turns before they crowd out the important parts.

The long-term tools are about making the archive trustworthy. Add stores new information. Update rewrites a memory when a later observation makes the old version incomplete. Delete clears out obsolete content so the agent does not keep consulting a bad record. The paper also exposes these tools as structured actions, which matters because the policy is then learning memory management directly instead of hiding it inside free-form text.

This is the main conceptual boundary in the method: STM tools shape what the model sees right now, while LTM tools shape what the model will know later.

Teaching the policy in three passes

AgeMem does not try to teach all of this in one shot. That would be too brittle, because a policy that has never learned to store durable facts should not be asked to manage distractors and solve the final task at the same time.

Instead, the paper uses a three-stage curriculum.

Stage 1 is about long-term memory construction. The agent sees conversational information and practices deciding what deserves to be stored. The important skill here is selectivity. If it stores everything, the archive becomes useless. If it stores too little, later retrieval has nothing to work with.

One subtle detail is that the agent can look back at its own long-term store during this phase. That is not because the final query has arrived yet. It is because the policy needs to stay aware of what it already remembers, so it can notice duplicates, conflicts, or stale entries before they snowball.

Stage 2 resets the short-term context but keeps long-term memory intact. Then the agent is given distractor messages that look conversational but are deliberately unrelated to the target task. This is where the short-term tools matter. The policy has to learn that not every incoming turn deserves equal status. It should summarize early when the window is getting crowded and filter out noise before that noise takes over.

Stage 3 brings the two sides together. The agent finally gets the actual query and has to combine retrieval, context cleanup, and reasoning to produce the answer. At that point, memory is no longer a side task. It is part of the answer path itself.

Think of it like training a study assistant. First it learns which notes are worth writing down. Then it learns which side conversations can be compressed or ignored. Only then does it get tested on a real question that depends on both of those earlier choices.

Why the final reward has to travel backwards

The hardest part of the method is not calling tools. It is teaching the policy which early memory decisions mattered once the final answer is known. The reward arrives late, but the memory actions happened early.

Formula Card: Send the final outcome back to every earlier step

Purpose: make the eventual task result supervise the memory actions that happened earlier in the episode.

Intuition: a good storage, pruning, or retrieval choice may only prove its value much later. AgeMem scores a rollout by its final outcome, normalizes that score within the rollout group, and then uses the same signal for every earlier step in that trajectory.

Why it matters: this is the paper’s answer to fragmented credit assignment. Without this step-wise broadcast, the model would only know whether the final answer was good. It would not know whether the Add, Filter, or Retrieve choices that shaped that answer were good.

The update itself stays close to GRPO-style training, with a KL penalty against a reference policy so the model learns without drifting too far while it explores better memory behavior.

What the reward is really asking for

The reward is not just “solve the question.” It is trying to balance three things at once.

  • Task success asks whether the final answer is right.
  • Context management asks whether the agent kept the working window compact and preserved the important query facts.
  • Memory management asks whether the archive contains reusable facts, meaningful updates or deletions, and relevant retrievals.

That balance is important. If the policy only optimizes answer quality, it can win by stuffing the prompt with too much history. If it only optimizes compactness, it can throw away the evidence needed later. If it only optimizes memory reuse, it can keep consulting stale or irrelevant entries. AgeMem trains all three concerns together so the memory policy supports the answer rather than fighting it.

The paper also penalizes obvious failure modes such as overflowing the context budget or exceeding the allowed interaction limit. So the model cannot win by being verbose and careless. It has to solve the task while keeping the memory loop disciplined.

What this looks like in one episode

A toy example makes the flow easier to remember.

Suppose a user says they prefer 120-minute study blocks and visual learning materials. In Stage 1, AgeMem should store those preferences because they are likely to matter later. In Stage 2, the conversation gets cluttered with unrelated chatter about hobbies or side interests. The policy should summarize or filter that noise instead of letting it crowd out the useful preferences. In Stage 3, when the user asks for a personalized study plan, the agent retrieves the stored preferences and answers with those constraints still intact.

The important point is that no single tool solves the whole story.

  • Add alone would hoard memory.
  • Retrieve alone would keep dragging the agent back to old text.
  • Summary alone would shrink the prompt but could erase useful detail.
  • Filter alone would become too aggressive if used blindly.

AgeMem is trying to learn the coordination policy that chooses among those tools at the right time.

What changed compared with older memory systems

Older memory systems often follow one of two patterns. They either leave short-term context mostly static and add a separate long-term memory manager, or they rely on retrieval-augmented prompting and hope the prompt layout will do the rest. That gives you a memory pipeline, but not a memory policy.

AgeMem changes that by unifying both memory scopes under one learned controller.

That matters for two reasons. First, it reduces the gap between writing and using memory. The same policy that decides what to store is later responsible for deciding what to fetch and what to keep out of the prompt. Second, it removes a lot of hand-built heuristics. Instead of separate rule sets for long-term and short-term memory, the model learns a single control problem over several kinds of memory action.

The tradeoff is real, though. The fixed tool vocabulary keeps the system interpretable and trainable, but it also limits how fine-grained the control can become.

What the paper shows in practice

The experimental story is straightforward: learned memory control beats a static or heuristic memory pipeline.

Across five long-horizon benchmarks, AgeMem gets the best average performance on both backbones. The strongest headline number is the Qwen3-4B run, where the average reaches 54.31. The version without RL is clearly weaker, which is the clearest sign that the staged curriculum is doing real work rather than just adding extra machinery.

Two other patterns line up with the design. The learned policy stores better memories, and the STM side uses less prompt budget than a RAG-style replacement. Tool usage also shifts in the expected direction: more writing and maintenance, less reactive retrieval. That is the behavior you would want if the archive is getting better before the agent consults it.

The limits to keep in mind

AgeMem is still built around a fixed set of six tools, so it cannot express every possible memory action.

The training setup is also controlled. It relies on curated trajectories and synthetic distractors, so the evidence is stronger for benchmarked long-horizon tasks than for open-ended real-user dialogue.

So the right way to read the paper is not “memory is solved.” It is “if later reasoning depends on earlier storage, compression, and retrieval, those choices should be learned as actions instead of being hidden behind heuristics.”

What to remember

AgeMem’s core idea is easy to keep in your head: memory is not a database; it is a policy.

The model sees live context, persistent memory, and a task goal together. It learns a small but expressive set of memory tools. It is trained in stages so the policy can first learn to store, then to prune, then to coordinate both. And step-wise GRPO makes the final task outcome reach back to the earlier memory decisions that actually shaped it.

If you remember only one sentence, make it this: when the answer depends on what the agent wrote down earlier, expose memory as an action space and train it with delayed reward.
