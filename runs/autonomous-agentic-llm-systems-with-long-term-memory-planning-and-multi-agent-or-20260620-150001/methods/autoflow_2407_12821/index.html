AutoFlow: Automatic Workflow Generation for LLM-based AI Agents

AutoFlow starts from a simple but annoying truth: agent workflows are often the real bottleneck. The model may already be smart enough to answer parts of the task, but the overall procedure still has to be written by hand. If the workflow is wrong, incomplete, or hard to execute, the whole agent fails even when the language model itself is strong.

The paper’s answer is to stop treating workflow design as a one-shot prompt-writing task. Instead, it turns workflow creation into a closed loop. A generator LLM writes a workflow in a CoRE-style natural-language program, a frozen interpreter LLM executes that workflow on validation data, the resulting score becomes feedback, and the generator improves the next draft. In other words: write, run, score, refine.

What to notice in this figure is the control loop. One model proposes the workflow, another model runs it, and the validation score flows back to the writer. The important idea is not the exact box layout. It is the fact that workflow quality becomes something the system can optimize, not just something a human has to guess.

You can think of the method like teaching a recipe writer by actually cooking the recipe. The writer does not just invent steps in the abstract. The recipe gets executed, the result is judged, and the score tells the writer what kind of structure worked.

Why the workflow itself matters

The paper uses a fake-news-style example early on because it makes the bottleneck easy to see. A useful agent workflow might need to check the URL, inspect the language, run commonsense evaluation, inspect the standpoint, summarize the evidence, and only then classify the result. That is not just a list of actions. It is an ordered procedure with branching, stopping conditions, and tool use.

That is where free-form prompting starts to break down. A plan can sound good and still fail when the interpreter tries to execute it. It may skip a necessary step, branch incorrectly, or point to the wrong kind of input for a tool. AutoFlow is trying to learn workflows that are not only readable, but actually executable.

The key design choice is to represent the workflow as a natural-language program rather than code. That keeps the structure legible to humans while still giving the interpreter enough organization to follow step by step. The paper builds on CoRE for exactly that reason: it is a workflow language, but one that stays close to plain language.

What a CoRE workflow stores

CoRE turns a workflow into a small set of fields, and that small structure is doing a lot of work:

  • Step name: a unique label for the step.
  • Step type: the role of the step, usually Process, Decision, or Terminal.
  • Step instruction: the natural-language action to carry out.
  • Step connection: the next step to jump to after execution.

The paper separates those four parts with a ::: delimiter. That sounds minor, but it is what makes the workflow parseable. A human can still read the step as prose, yet the interpreter can still tell where the instruction ends and where the next step begins.

The step types matter because they encode control flow. Process moves forward in a straight line. Decision creates branching, like an if-else statement. Terminal ends the program. Without those distinctions, the workflow would collapse into a flat checklist, which is not enough for a multi-step agent.

So the workflow is not just “what should the model say next.” It is closer to a compact executable plan: what step exists, what kind of step it is, what it says, and where control should go next.

How the interpreter actually runs the workflow

The interpreter LLM is the other half of the system, and it is the part that makes the natural-language program behave like a program.

Here is the execution loop the paper describes:

  1. The interpreter looks into memory and retrieves the pieces relevant to the current step.
  2. It combines that memory with the current instruction into a structured prompt.
  3. If the step needs external help, it decides whether to call a tool, chooses the tool and arguments, executes the tool, and stores the result back in memory.
  4. It uses the output of the step to decide which step comes next.

This is the real reason AutoFlow cares about workflow syntax. The workflow is not the final output. It is the control script for another model.

The memory step matters because agents do not need to carry the entire history into every action. They need the relevant history. The tool step matters because many tasks are only partly language problems. The next-step decision matters because the workflow may branch rather than proceed linearly. Put together, those four moves turn a natural-language workflow into an executable control loop.

One loop, two ways to improve the generator

AutoFlow has the same core loop in both settings, but it updates the generator differently depending on whether the generator is open-source or closed-source.

The common starting point is always the same: the generator sees a task description and a few-shot example workflow. The example is important because it teaches the CoRE grammar by demonstration, not by asking the model to infer the syntax from scratch. The generator then writes a candidate workflow, the interpreter runs it, and the validation score becomes the signal for improvement.

What to notice here is that the open-source branch turns validation feedback into actual parameter updates. The workflow is not only judged; it is used to train the writer.

When the generator is open-source

For open-source models such as Mixtral, AutoFlow uses LoRA plus reinforcement learning. The LoRA adapter keeps the update efficient, so the model can specialize in workflow writing without retraining the entire backbone.

The training loop works like this:

  • The generator writes a CoRE-style workflow from the task description and example.
  • If the open-source output is malformed, a parser cleanup step repairs it into valid CoRE syntax.
  • The frozen interpreter executes the workflow on validation data.
  • The average validation score becomes the reward.
  • REINFORCE updates the LoRA parameters.
  • The process repeats until the score stops improving enough to matter.

That parser cleanup step is a practical workaround, not a conceptual centerpiece. It exists because a workflow only helps if the interpreter can read it. If the generator is grammatically off, the system would otherwise waste reward on syntax failures instead of learning which workflow structure is better.

The frozen interpreter is also important. Because the evaluator stays fixed, a better score really means a better workflow, not a moving target in the executor itself. That makes the reward comparable across iterations.

When the generator is closed-source

For closed-source models such as GPT-4, AutoFlow skips parameter fine-tuning and uses in-context learning instead. The loop still executes the workflow and scores it, but the score is fed back into the next prompt as a natural instruction for revision.

So the next prompt might say, in effect, “the previous workflow scored 0.6415, now write a better one.” The model is not being updated by gradients. It is being steered by reward-conditioned prompting.

This is a clever compromise. Closed models cannot be fine-tuned here, but they can still use performance feedback as a search signal. The model sees the previous score, infers that it should improve the structure, and proposes a revised workflow. In practice, that gives AutoFlow a way to run the same optimization idea even when the generator weights are off-limits.

Why the two branches are doing the same job in different ways

The open-source branch and the closed-source branch look different on the surface, but the underlying logic is the same. Both branches are searching for workflows that execute well. Both branches use validation performance as the score. Both branches iterate until the improvement plateaus.

The difference is just where the update lives:

  • Open-source: update the generator weights.
  • Closed-source: update the prompt with reward feedback.

That split is one of the most useful design choices in the paper. It makes the framework usable in both settings instead of locking it to one model family.

There is also a deeper point here. Workflow writing and workflow execution are not the same skill. A model that is good at proposing a workflow is not automatically the best model for running it. The paper’s experiments end up showing that the best generator and the best interpreter are sometimes different models, which is a nice reminder that agent systems can benefit from complementary roles rather than symmetry.

What the results are really saying

The benchmark story is straightforward: AutoFlow beats the manually designed CoRE baseline on OpenAGI under both interpreter choices. With Mixtral as the interpreter, the average score rises from 0.2483 for CoRE to 0.3597 for AutoFlow. With GPT-4 as the interpreter, it rises from 0.6104 to 0.6501.

The more interesting part is not just the averages. AutoFlow reaches the top score in each OpenAGI task type under both interpreter settings. That suggests the system is not just gaming one metric. It is learning workflow structure that transfers across the different task flavors in the benchmark.

There is also a small but telling synergy effect. The best average score with Mixtral as interpreter comes from pairing it with GPT-4 as the workflow generator, while the best average score with GPT-4 as interpreter comes from pairing it with Mixtral as the generator. That cross-model pattern hints that the two roles are not identical: one model may be better at drafting workflows, while another is better at executing them.

What still hurts

AutoFlow is not free automation. Every improvement cycle adds compute and latency because the system has to generate, execute, score, and refine again. That cost is part of the design.

Open-source generators also do not always emit valid grammar on their own, which is why the parser repair step exists. That is a practical fix, but it is also a reminder that the workflow language is only useful if the model can stay inside it.

The paper’s evidence is also benchmark-bounded. The results are shown on OpenAGI-style tasks, so broader generalization is still an open question. The method looks strong as a workflow synthesis loop, but the paper does not claim that the same recipe will solve every agent setting.

What to remember

AutoFlow is easiest to remember as a write-run-score-refine loop for agent workflows.

The workflow is written in CoRE-style natural language so people can read it and the interpreter can execute it. The interpreter stays frozen so the reward stays stable. The generator then improves either by weight updates, if it is open-source, or by reward-conditioned prompting, if it is closed-source.

The larger lesson is simple: when plans must be executable, do not optimize for a beautiful one-shot prompt. Optimize the workflow against execution feedback. That is what turns workflow authoring from manual craft into a learning problem.
