GoAgent: Group-of-Agents

GoAgent changes one small-looking choice that turns out to matter a lot: it stops treating individual agents as the basic unit of topology design. Instead of wiring one agent to another edge by edge, it asks which subteams should exist, then how those subteams should communicate. That is a better fit for tasks where a decomposer, solver, and verifier belong together, and it is also a cleaner way to stop irrelevant messages from spreading through the whole graph.

Figure 1: node-centric vs group-centric graph generation shows that shift directly.

What to notice is the unit of decision. A node-centric generator asks whether one agent should connect to another. GoAgent asks whether a whole collaborative group should be present at all. That sounds like a small change, but it changes the search space from “which individual edges look plausible?” to “which subteams actually belong in the workflow?”

That is the full story in miniature. GoAgent first builds a library of candidate groups, then grows a graph one subteam at a time, and finally compresses the graph state so old noise does not dominate later decisions.

Why the candidate groups matter

The first move is to turn the task query into a global task representation. The paper uses a sentence encoder followed by a small feed-forward network so the task becomes a vector that can guide every later decision. That vector is not the answer by itself. It is the condition that keeps the rest of the generator pointed at the right problem.

At the same time, GoAgent asks an LLM to enumerate a fixed pool of task-relevant collaborative groups. Each group is more than a name. It comes with a role description, the kinds of expertise it contains, and the internal wiring pattern inside the group. In the paper, that internal wiring is treated as a template, such as a fully connected group for brainstorming or a sequential group for refinement.

That design does two things at once. It gives the generator a cleaner search space, because it no longer has to invent every possible role from scratch. And it gives each group a stable internal structure, so once the model chooses a subteam, the subteam behaves like a reusable macro rather than a random bundle of agents.

The tradeoff is also clear. The pool is generated offline, so the system is only as flexible as the group library it was given. If a task requires a brand-new role or an unusual collaboration pattern, GoAgent cannot freely invent it during inference. That is the price of making the topology space smaller and easier to control.

Figure 2: GoAgent overview puts the whole pipeline on one page.

What to notice in Figure 2 is the flow from task encoding to candidate discovery to autoregressive graph growth. The paper is not choosing agents one by one in isolation. It is comparing the task against a library of candidate subteams, then wiring those subteams together as the graph expands.

How the graph grows one subteam at a time

Once the candidate pool exists, GoAgent generates the final communication graph autoregressively. At time step t, it needs two kinds of context. First, it needs to remember what groups have already been selected. Second, it needs to stay aligned with the task query rather than drifting into whatever looks locally similar.

The historical side is handled by a GRU, which acts like a compact running summary of the groups chosen so far. The task side is handled by a gate that compares that history summary with the task vector. If the current history already matches the task well, the model can lean on history. If the graph has started to drift, the task vector pulls the state back toward the query.

That gate is a simple but important idea. It stops the generator from treating the past as equally useful at every step. Early choices matter, but they should not become a blind commitment. The model should keep checking the task brief against the current partial graph.

After that, the generator produces a feature for the next group and scores that feature against every candidate group in the pool. In other words, it asks: “Which collaborative group should come next?”

Once a group is chosen, the model switches to edge prediction. For each previously built group, it builds a feature from the source group state, the new group embedding, and the task vector, then decides whether an incoming edge should exist. So the generation loop is really two decisions repeated together:

  • pick the next subteam,
  • decide which earlier subteams should send information into it.

That is the main structural change relative to node-centric methods. GoAgent is not just predicting local edges. It is predicting a new unit of coordination and then attaching that unit to the earlier graph.

Why the bottleneck exists

The graph can still get noisy as it grows. Even if the model starts with a sensible subteam pool, later history can carry irrelevant co-occurrences, spurious connectivity, and leftover signals that no longer matter for the current step. That is where the conditional information bottleneck comes in.

Formula Card: Conditional information bottleneck objective

Purpose: keep the graph state compact while preserving only the information needed to predict the next group or edge for the current task.

Intuition: the model should not pass every historical detail forward. It should compress the current graph state into a latent summary that still predicts the next choice, but only in a way that makes sense for the current query.

Symbols:

  • c is the compressed latent feature after the bottleneck.
  • x is the raw group or edge feature before compression.
  • y is the next thing to predict, either a group choice or an edge label.
  • z_Q is the encoded task query, the global task representation.
  • beta controls how strongly the bottleneck compresses the representation.

Read-aloud meaning: learn a compact summary of the current graph state that still predicts the next group or edge for this task.

Why it matters: this is the filter that keeps noisy history from becoming the default fuel for later decisions. Without it, the generator is much more likely to amplify redundant communication instead of pruning it.

GoAgent uses the same bottleneck idea for both group prediction and edge prediction. That is a nice design choice because the graph only needs one compression rule, not two competing ones. The paper also makes the prior task-conditioned instead of using one generic prior for every task. That matters because the latent space for math reasoning, code generation, and general reasoning should not be forced into the same baseline shape.

In practice, the encoder and prior are parameterized as Gaussians, and the model samples with the reparameterization trick during training. The point is not the exact sampling formula. The point is that the bottleneck stays trainable end to end while still acting like a learned filter.

How it learns a good topology

The hard part of training is that nobody hand-labels the perfect graph for each task. GoAgent solves that by building a supervised dataset from successful search. For a set of training queries, it samples diverse candidate topologies from simple structural families such as chain, star, and fully connected graphs. It then runs those graphs on the task and keeps the ones that actually solve it.

But it does not keep every successful graph. It filters them down to minimal viable topologies. That is an important detail. If a task succeeds with three groups and a handful of edges, training on a much denser graph would teach the model that extra communication is acceptable. By keeping the smallest successful graphs, the paper teaches the generator to preserve only the coordination that really mattered.

The model is then trained with teacher forcing on those curated trajectories. That keeps the optimization stable and avoids the high variance of reinforcement learning. The CIB penalty is added on top, and the KL strength is warmed up over early epochs so the bottleneck does not squeeze the representation before the generator has learned any useful structure.

At inference time, the process becomes simpler. The model generates groups autoregressively, uses the deterministic mean from the bottleneck instead of sampling noise, and then samples incoming edges with a Bernoulli decision. If a new group would otherwise end up disconnected, the implementation adds the highest-probability edge to guarantee connectivity. That fallback is practical and important: a good subteam that cannot receive information is not actually usable.

What the results mean

Table 1: Overall performance comparison is the scoreboard, but the more interesting story is what it says about the design.

GoAgent reaches 93.84% average accuracy across six benchmarks, and it does that while cutting token consumption by about 17% relative to the strongest baseline. That combination is the real point. The method is not just more accurate. It is also cheaper to run because it avoids unnecessary agent-level chatter.

The ablation study tells the same story from the other direction. Removing group-level generation hurts. Removing the bottleneck hurts. Removing both hurts the most. That means the method’s two central ideas are complementary, not redundant. The group-centric structure gives the model the right unit of coordination, and the bottleneck keeps that coordination from turning into noisy over-communication.

The robustness analysis points the same way. When a bad prompt is injected into one agent, the group-centric design is less brittle than node-centric baselines because the bottleneck and the group boundaries make it harder for one local error to spread everywhere.

The main limitation is the same one that follows from the design. GoAgent depends on an offline pool of collaborative groups, so it is not free to invent completely new roles at inference time. The evaluation also stays in static reasoning and coding settings, so the method’s behavior in dynamic interactive environments is still an open question.

What to remember

GoAgent is easiest to remember as a three-step habit:

  1. choose the subteams first,
  2. connect the subteams second,
  3. compress the communication so noisy history does not take over.

That is why the method feels cleaner than a node-by-node generator. It matches the way many real tasks are solved: not by one agent talking to one other agent in a long chain, but by a small set of roles that work as a coherent unit and exchange only the information that is still worth keeping.

If you want the one-line takeaway, it is this: when coordination repeats across roles, make the subteam the atomic decision, then use a task-conditioned bottleneck to keep only the signals that help the next decision.
