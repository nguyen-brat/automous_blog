Retroformer: Reinforcing Retrospective Language Agents with Policy Gradient

Retroformer is built around a simple but sharp idea: if the actor LLM is frozen, do not try to train the actor. Train the smaller piece that writes the next prompt.

That smaller piece is the retrospective model. It reads the past trajectory, the rewards, and the user goal, then compresses them into two things the next attempt can actually use: a concise failure summary and a short action plan. The paper’s claim is that this reflection writer is not just a prompt trick. It is a trainable policy component, and once you train it with reward, the agent can improve without touching the large actor model.

You can think of the whole system like a coach-and-player setup. The player executes the move. The coach watches the replay, figures out what went wrong, and writes the next play card. Retroformer is about training the coach, not the player.

That loop is the whole method in one line. The rest of the paper is about making each step do useful work instead of just repeating the same mistake in nicer language.

The failure mode is easy to miss

The paper opens with a problem that shows up in a lot of reflective agents: the model can explain what happened without explaining what should change.

What to notice here is the gap between description and diagnosis. A frozen reflector often paraphrases the failed trace, then presents that paraphrase as if it were guidance. That sounds helpful, but it is not. If the reflection only restates the old steps, the actor is likely to repeat the old steps.

This is why Retroformer matters. The paper is not saying self-reflection is useless. It is saying that self-reflection itself should be optimized. A reflection should earn its keep by helping the next retry do better, not by sounding polished.

The core system is a loop, not a one-shot prompt

The architecture is easiest to understand as a loop with two language models and one memory system.

What to notice in the figure is the division of labor:

  • The actor model takes the current prompt and produces actions.
  • The retrospective model reads the trajectory and writes the reflection.
  • The memory module keeps both short-term trace and longer-term lessons around for the next attempt.

That separation is the key design choice. Retroformer does not ask one model to be everything at once. It keeps the actor fixed and learns only the layer that rewrites the prompt.

The frozen actor stays out of the training loop

The actor is the model that actually talks to the environment. In the paper it is a cloud-hosted LLM with inaccessible parameters. The actor sees the current observation plus the recent history of states, actions, and rewards, and it emits the next action or thought in a ReAct-style format.

This matters because the actor is treated as part of the environment from the learning perspective. Retroformer does not need gradients through the actor, and it does not need control over the actor weights. That makes the method practical for closed models or very large models that are expensive to fine-tune directly.

The tradeoff is simple: if you cannot change the actor, you have to make the prompt better. Retroformer is the prompt-improvement layer.

The retrospective model learns to summarize failure, not replay it

The retrospective model is the part that reads the past episode and writes the next prompt. Its input includes the trajectory, the rewards, and the user goal. Its output is a refined prompt containing a short explanation of what went wrong and what the next attempt should do differently.

The important nuance is that it does not need to preserve every token from the trace. It needs to preserve the parts that matter for the next decision:

  • the task objective,
  • the failed trajectory,
  • the reward signal,
  • the most recent observations,
  • and the actionable change to make next.

Everything else can be compressed away. That is why the model is called retrospective rather than replay: it is supposed to extract the reason for the failure, not mirror the transcript.

The paper frames this as a credit assignment problem. With sparse reward, especially binary success or failure, the model has to infer which earlier decision led the agent into trouble. A good retrospective model can look at the same trajectory that a frozen reflector saw, but produce a much more useful summary of cause and next action.

Memory is split into short-term trace and long-term reflection

Retroformer uses two kinds of memory in the prompt:

  • Short-term memory is the current episode trajectory. It keeps the local context of what just happened.
  • Long-term memory is the history of prior reflection responses. It stores the lessons from previous failed attempts so the next prompt can reuse them.

The paper also keeps a replay buffer of reflection training data. In that buffer, each record is a triplet of the reflection prompt, the reflection response, and the episode return. That is the part that makes the reflection writer trainable over time. Good reflections become reusable examples; bad reflections become negative evidence.

This is a small but important point. The memory is not just a scratchpad for the actor. It is also the dataset that teaches the retrospective model what useful feedback looks like.

The reward has to say something useful

Retroformer needs a measurable signal between retries, so the paper uses rewards that match each environment instead of forcing every task into the same scoring shape.

On HotPotQA, the reward is based on F1-style overlap with the ground-truth answer, which is more forgiving than exact match when the answer is partially correct. On AlfWorld, the reward is terminal success or failure, because the task is a crisp completion problem. On WebShop, the reward measures how well the chosen product matches the target attributes, options, and price, while also penalizing obviously wrong product types.

The principle is more important than the exact scoring rule: if the environment can tell you whether the next retry was better, then the reflector can be trained from that change.

The score for a reflection comes from the next retry

Retroformer turns reward into a training target by comparing consecutive attempts. If a reflection helps the next attempt do better, that reflection should get credit. If it does not help, it should not.

Formula Card: score a reflection by the improvement it causes

Purpose: measure whether a reflection made the next retry better.

Intuition: if the next attempt earns a higher return than the current one, the reflection probably pointed the actor in a better direction.

Formula:

r(xk,i,yk,i)≜Gk,i+1−Gk,ir(x_{k,i}, y_{k,i}) \triangleq G_{k,i+1} - G_{k,i}r(xk,i​,yk,i​)≜Gk,i+1​−Gk,i​

Symbols:

  • x_{k,i} is the reflection prompt for task k at retry i.
  • y_{k,i} is the reflection response written by the retrospective model.
  • G_{k,i} is the episode return for retry i.
  • G_{k,i+1} is the episode return for the next retry.
  • r(x_{k,i}, y_{k,i}) is the rating assigned to that reflection-response pair.

Read-aloud meaning: the quality of a reflection is the gain in return it creates on the next attempt.

Why it matters: this is the credit assignment trick that makes the retrospective model learn from actual downstream improvement instead of from text quality alone.

That little difference-of-returns idea is the heart of the paper. It is what turns “this sounded like a good reflection” into “this reflection led to a better retry.”

PPO is used to reinforce the better reflections

Once the reflection pairs have scores, the paper trains the retrospective model with a standard reinforcement-learning pipeline. The setup is deliberately familiar: collect offline data, train a reward model from the ratings, then fine-tune the retrospective model with PPO.

The practical reason for PPO here is stability. The model should learn to generate better reflections, but it should not drift so far from the reference model that its outputs become bizarre or unusable. The paper therefore keeps a KL regularizer in the training objective so the updated model stays close to the frozen reference model.

That regularization matters more than it might look at first glance. The retrospective model is only useful if the actor can still consume its output naturally. A reflection that is technically high-reward but stylistically off the rails would not help the next prompt.

What to notice in this figure is that the learning target is the reflection writer itself. The actor is not updated. The rollout data becomes training data for the retrospective model, and the learned reward model scores which reflections are worth repeating.

The paper also notes that inference can use a best-of-n style sampler over candidate reflections. That fits the same logic: generate several possible retrospective prompts, score them, and keep the one most likely to help the next retry.

Why the method works when it works

Retroformer’s gains are largest when the failure pattern is repeatable and the environment score is meaningful. That is why HotPotQA and AlfWorld are the cleanest success stories.

On HotPotQA, the reinforced retrospective model improves faster over retries than the frozen-reflection baselines, and the final success rate after multiple retries is better than both ReAct and Reflexion. On AlfWorld, the method becomes even more striking: the strengthened reflection writer helps the agent reach perfect success after a few retries. The early retries matter most, which is exactly where better reflections should have the biggest effect.

WebShop is the cautionary case. The method still helps, but the improvement is smaller. That makes sense because web shopping asks for more open-ended exploration and more precise search behavior. Better reflection can steer the search, but it cannot remove the need to explore a large and noisy space.

So the result pattern is not “Retroformer solves every agent problem.” The result pattern is more specific: when the environment can score retries and the failure can be summarized into actionable advice, training the reflection writer pays off.

What is actually new here

The novelty is not that the paper uses reflection, memory, or PPO. Each of those ideas already existed in some form. The new move is to put them together in a way that makes the reflection module itself the learning target.

That changes the design question from “Can the actor discover the right behavior by itself?” to “Can we train the piece that writes the actor’s prompt so the actor gets better guidance on the next try?” That is a better question when the actor is closed, expensive, or simply not the component you want to tune.

It also changes the way to think about memory. Memory is not only a place to store summaries. It is part of the policy interface. If the reflection writer improves, the memory that enters the next prompt improves too.

What to remember

Retroformer is easiest to remember as this rule:

  • freeze the actor,
  • train the reflection writer,
  • score reflections by how much the next retry improves,
  • keep the useful lessons in memory,
  • and use PPO to make those lessons more likely to appear again.

That is why the paper is interesting. It makes self-reflection into something trainable, which gives autonomous agents a way to learn from retries even when the main LLM cannot be touched.

The main limitation is just as important: the method needs a reward signal that changes between retries, and it works best when failures can be compressed into clear, actionable reflections. When the environment is mostly an exploration problem, the gains are smaller. When the environment can tell you whether the next retry got better, Retroformer gives that signal a place to go.
