LLaMAC: Large Language Model-based Actor-Critic

LLaMAC is a coordination system for large teams of LLM-based agents. The paper starts from a practical problem: when the number of agents grows, asking every agent to reason deeply on every turn makes the interaction noisy, expensive, and brittle. LLaMAC’s answer is to stop making every agent do everything. A centralized critic proposes a plan, decentralized actors check that plan against what they actually see, and memory keeps the recent history visible so the next round of reasoning does not start from zero.

That sounds simple, but it is the whole trick. The paper is not trying to make one agent magically smarter in isolation. It is trying to make many agents share the work of planning without drowning in token cost or hallucinated coordination. The authors demonstrate the idea on up to 50 agents, which is where the coordination problem becomes real rather than hypothetical.

What to notice in Figure 1 is the loop shape. State is rendered as text, memory is retrieved, the critic suggests, actors answer back, actions execute, and the result is written back into memory. The system is not one big monologue. It is a repeated cycle of observation, proposal, checking, and action.

The method in one pass

You can read the architecture as a black box first:

The important design choice is that the expensive reasoning lives in the critic, while the actors stay locally grounded. That is how LLaMAC scales to many agents: the center does the heavy planning, but the edge agents still preserve local exploration and can reject a bad suggestion when they see one.

The execution module is the translation layer. It turns raw environment state into text the LLM can read, and it turns the chosen action back into the concrete move the environment needs. Depending on the task, that action can be a very small physical move or a higher-level instruction.

Why the memory module is there

The memory module is not just a log. It is what keeps the loop from forgetting what it already tried.

The paper uses two scopes of memory. Short-term memory keeps the latest state. Long-term memory keeps historical trajectory information and useful experience from earlier interactions. It also filters redundant details, so the critic does not have to reread the whole conversation every time.

That matters because the two testbeds in the paper are both history-sensitive, but in different ways. In the one-step allocation task, memory helps the critic infer the shape of the reward landscape from previous attempts. In grid transportation, memory helps the team remember which objects moved, which conflicts already happened, and which paths are still available.

A useful way to think about it is this: the memory module preserves the parts of the past that still change the next action.

TripletCritic: one planner, three voices

The central idea in LLaMAC is the TripletCritic. Instead of one critic producing one answer, the paper uses two critics with different preferences and a third component, the assessor, that reconciles them.

One critic leans toward exploration. It is the voice that says, “Do not settle too early. Keep looking around the promising region and gather enough evidence.” The other critic leans toward exploitation. It is the voice that says, “We already have a good-looking option. Use it and move the task forward.” The assessor sits between them and decides whether the plan is actually believable and well balanced.

What to notice in Figure 2 is that the critic does not just output a single plan and hope for the best. There is an internal debate first, then an external correction loop if an actor sees a problem. The internal side is about making the suggestion sane. The external side is about making it fit the local reality of the actor.

The assessor has two jobs.

First, it performs veracity scrutiny. In plain language, that means checking whether the joint plan is coherent, correctly formatted, and free of obvious conflicts.

Second, it performs belief correction. That is the balancing act between exploration and exploitation. The assessor does not let either bias dominate completely. If the exploratory plan is too speculative, it gets pulled back. If the exploitative plan gets stuck too early, it gets pushed to look a little wider.

That split is the reason the paper calls the design TripletCritic instead of just “multi-critic.” The third voice is not decorative. It is the piece that turns disagreement into a usable suggestion.

Symbols:

  • R(x) is the reward for a collective action sum x.
  • x is the sum of all agents’ actions, x = \sum_{i \in \mathcal{I}} a^i.
  • a^i is the action chosen by agent i.
  • \mathcal{I} is the set of agents.
  • \mu is the center of the reward peak.
  • \sigma controls how wide that peak is.

Read-aloud meaning: the team gets the best reward when its combined action lands near the target region, not when each agent simply pushes in the same direction without thinking.

Why it matters: this task makes the exploration-vs-exploitation tradeoff easy to see. A planner that explores too timidly can miss the peak. A planner that exploits too early can lock onto the wrong region. TripletCritic is built to keep both failure modes in check.

When actors push back

The second feedback loop is where LLaMAC stops being a pure central planner and becomes an actual multi-agent system.

After the TripletCritic sends a suggestion to each actor, the actor compares that suggestion with its local observation. If the suggestion looks fine, the actor executes it. If not, the actor sends external feedback back to the critic, which can then revise the suggestion.

This is not just an extra chat round. It is a gate. The paper uses plan confirmation so that the critic only gets called back when the local suggestion looks wrong or infeasible. That is where the token savings come from. The system does not ask every actor to fully replan from scratch every time. It asks them to confirm whether the central suggestion survives contact with the local view.

The appendix makes the check concrete in the grid tasks. Internal scrutiny looks for things like correct output grammar and whether joint actions conflict. External confirmation looks at things like action availability and whether the suggestion would reduce the Manhattan distance between an object and its target. Those are simple checks, but they are exactly the kind of checks that stop a fluent but wrong suggestion from going out into the environment.

The nice part of this design is that the actors still preserve local exploration. They are not passive executors. They can say, in effect, “this plan does not fit what I can actually do right now.” That keeps the system from becoming a brittle single-brain controller.

Why the design works

The main reason LLaMAC works is that it separates three jobs that are usually tangled together:

  • proposal generation
  • validity checking
  • correction

A single agent can try to do all three, but then every turn becomes a long, expensive reasoning pass. LLaMAC instead puts proposal generation in the TripletCritic, validity checking in the actor confirmation step, and correction in the feedback loop. That division makes the whole system easier to scale because not every agent needs a full fresh plan on every turn.

It also makes failure easier to localize. If the critic is over-exploring, the assessor can pull it back. If an actor sees an infeasible move, it can reject just that move instead of forcing the whole team to restart. If the plan is good enough, the system does not waste tokens on another round of negotiation.

So the real lesson is not “use more agents.” It is “spend reasoning where it changes the next action.”

What the tests are really checking

The paper uses two testbeds because they stress different parts of the same loop.

The first is system resource allocation, which is basically a one-step optimization game. It is useful because the reward surface has a visible peak, so you can see whether the critic is learning to explore around the right region instead of jumping to a local optimum too soon.

The second is grid transportation, which is a longer-horizon planning task. Here the agents must move objects to targets over multiple steps, and the hard version adds extra friction because objects move only along boundaries and multiple agents can compete for the same corner or create collisions. That is the kind of setting where a suggestion that looks fine in abstraction can fail immediately when an actor checks the local state.

The point of the evaluation is not just that LLaMAC scores well. It is that the architecture is aligned with the failure modes of the task. Single-step search stresses the exploration/exploitation balance. Long-horizon transport stresses local feasibility, memory, and correction under interaction.

What still depends on the prompts

LLaMAC is effective, but it is not magically general.

The method depends on carefully designed textual state descriptions and feedback prompts. That is a strength in the paper’s synthetic environments, because the system can spell out state, suggestions, and checks very clearly. It is also a limitation, because messy open-world tasks may not package their state so cleanly.

The paper is also honest about what it does not solve. TripletCritic reduces hallucination and local-optimum risk, but it does not remove them. And the evaluation is still on controlled resource allocation and grid-transport scenarios, so transfer to noisier domains remains open.

If you want the shortest possible memory hook, it is this: LLaMAC tries to make large-agent planning stable by separating who proposes, who validates, and who gets to correct the proposal.

What to remember

LLaMAC is easiest to remember as a proposal-and-check loop for many agents.

The critic proposes a shared direction. The assessor balances exploration and exploitation. Each actor checks that direction against its local view and only calls back when the suggestion is wrong. Memory keeps the recent story visible so the next round is not blind.

That is the paper’s main contribution. It is not a new way to let every agent talk more. It is a way to let a large team reason together without turning every step into a full debate.
