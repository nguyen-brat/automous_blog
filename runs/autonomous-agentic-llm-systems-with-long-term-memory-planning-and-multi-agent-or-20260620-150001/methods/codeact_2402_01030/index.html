Executable Code Actions Elicit Better LLM Agents

CodeAct is one of those ideas that sounds small until you follow the control flow all the way through. The paper is not saying that an LLM should merely generate code as an answer. It is saying that the code itself should be the action. That means the agent is not locked into a fixed JSON schema or a rigid text template. It can write Python, run it in an interpreter, inspect the result, and then decide what to do next.

That changes the whole shape of tool use. A fixed schema is like filling out a form. Executable Python is closer to writing a tiny program that can store intermediate results, branch on what happened, call libraries, and repair itself when the runtime complains.

The first figure makes that contrast very clear.

What to notice is the asymmetry between the two sides. With text or JSON, the model has to spell out each action in a narrow format and then wait for another round trip. With CodeAct, the model can bundle more of the reasoning into one executable action. That is why the paper keeps returning to control flow, data flow, and runtime feedback. Those are not side benefits. They are the point.

Why the format matters

The simplest way to understand the paper is to ask what an agent needs when it has to do more than one tool call.

It needs to:

  • keep intermediate values around,
  • reuse the output of one tool as the input to another,
  • branch when a condition changes,
  • and recover when a call fails.

JSON and free-form text can express some of that, but only awkwardly. They are best when the agent is choosing from a small menu of actions. They are much weaker when the task is really a workflow.

Code gives the agent four things for free.

First, it gives the model a syntax it already knows reasonably well. Large language models are trained on a lot of code, so Python is not an alien action grammar.

Second, it gives native control flow. A loop or an if statement lets the agent say “do this for every item” or “only do that if the result looks wrong” without inventing a new tool for each case.

Third, it gives native state. The agent can keep results in variables and reuse them later instead of re-sending the same information through the prompt.

Fourth, it gives runtime feedback. If the import fails or the function call is malformed, the interpreter says so immediately. That error message is not just a failure; it is a new observation the agent can use on the next turn.

A simple example makes this concrete. Suppose one tool finds a list of relevant links and another tool downloads the documents behind them. A fixed schema would usually make the agent spell out those steps separately. CodeAct lets the agent write one small script that stores the link list, loops over it, and passes each item along. The action is still interpretable, but now it behaves like a real program instead of a single command.

The loop is the real method

CodeAct is strongest when you see it as a loop rather than a format.

The agent receives an observation. That observation can come from the user as natural language or from the environment as code execution output. The agent then emits either a normal-language reply or a Python action. If the action goes to the environment, the interpreter runs it and returns the result, or the error, or both. That output becomes the next observation.

What to notice in the figure is the boundary between the two interaction types. The user side still matters, because the model needs to talk naturally and follow instructions. But the environment side is where CodeAct becomes interesting. The interpreter turns execution into feedback, and that feedback lets the agent revise later actions instead of committing to a one-shot guess.

That is the key difference from static code generation. In a static setting, the model writes a full program once, and if that first attempt is wrong, the whole attempt is a miss. In CodeAct, the code is part of a conversation with the environment. A bad import, a wrong variable, or a bad assumption can be corrected on the next turn because the runtime exposed the mistake.

This is also why the paper keeps stressing that code is a better action substrate than a fixed schema. The substrate is not only about how actions are formatted. It is about what the agent can do between turns.

From one tool call to a workflow

The paper tests that claim in two ways.

The first is an atomic tool-use setting on API-Bank. Here the task is deliberately simple: the model only needs to produce one correct tool call per step. That strips away the control-flow advantage of code and focuses more on whether the format itself feels natural to the model. CodeAct is competitive or better than text and JSON for most models, which is already useful. It says that Python is not just a convenient hack for complex tasks. Even when the action is tiny, the format is not a handicap.

The second test is the more important one: multi-step tool use on M^3ToolEval. This benchmark was created because the paper wanted tasks that really require multiple tools in multi-turn interactions, not just one isolated API call. The domains are varied, but the recurring pattern is the same: an agent has to coordinate several operations, not just pick a single command.

That is where CodeAct pulls ahead. The paper reports that the best closed model gets a large absolute improvement in success rate and also finishes in fewer turns on average. That pairing matters. Better tool use is not just about being correct more often. It is also about reaching the answer with less back-and-forth.

The reason is visible in the action format itself. When the agent can loop, branch, and carry state inside one action, it does not need to split every tiny substep into a separate tool call. The environment can still see the code. The human can still inspect it. But the model gets to think in program form instead of prompt fragments.

How CodeActInstruct teaches the pattern

The second half of the paper is not just “use code.” It is “teach the model to use code well in multi-turn interactions.”

That is what CodeActInstruct is for. The dataset contains 7,139 multi-turn trajectories, and they are not all drawn from the same kind of task. The paper repurposes four broad settings:

  • information seeking, where the agent searches for facts,
  • software package use, where the agent imports libraries and solves tasks with real Python tools,
  • external memory, where the agent works with tabular data through SQLite or pandas,
  • and robot planning, where the agent uses control APIs to complete embodied tasks.

That spread is important because it shows that CodeAct is not a one-trick paper about coding questions. It is trying to teach a general way of acting through Python.

The paper also repurposes some tasks into multi-turn episodes even when the original benchmark was single-turn. That matters because real agents rarely get everything right in one shot. The model needs to learn what to do after a partial answer, a failed attempt, or a new observation.

The most interesting part of the dataset design is the filtering. The authors do not just keep successful trajectories. They keep trajectories where the model first runs into trouble and then fixes the mistake later. That is the self-improvement signal the paper cares about. A trajectory where every action is wrong is not useful training material. A trajectory where the model notices an error, reflects, and repairs the plan is exactly the kind of example CodeAct is supposed to learn from.

What the training examples look like in practice

The paper shows a concrete interaction where the agent uses Python packages to get work done across several turns.

What to notice here is not just that the model can import packages. It is that the runtime opens up a whole practical workflow. The agent can use pandas to inspect tabular data, scikit-learn to split data and fit a model, and matplotlib to visualize the result. If something goes wrong, the interpreter tells it what failed, and the next turn can repair the script.

That is the self-debugging story in one picture. The model is not relying on a hand-built wrapper for every task. It is leaning on the Python ecosystem and using the error messages that already come with it.

What CodeActAgent learns

The paper then takes the idea one step further and finetunes an open-source agent on the CodeAct trajectories plus general conversation data.

The goal of the extra conversation data is simple: keep the model good at talking, not just acting. A tool-use agent still has to answer users in natural language, follow instructions, and stay useful outside the environment loop. Mixing in general conversation helps preserve that broader behavior.

The result is CodeActAgent, trained from LLaMA-2 and Mistral-7B backbones. The stronger backbone matters. The Mistral version improves more reliably on agent tasks and gets close to much larger models on the paper’s harder tool-use benchmark. The LLaMA-2 version is less consistent, which is a useful reminder that the action format is not the whole story. Backbone quality still matters.

The broader takeaway is that CodeAct is not just a prompting trick. The training data actually teaches the agent a habit: write code, observe the runtime, revise the plan, and keep going.

Where the idea is strong, and where it is not

The method has clear assumptions.

It assumes a working Python runtime. It assumes the needed libraries or APIs are available. It assumes the model can write usable code often enough to benefit from the interpreter.

And it inherits the familiar failure modes of language models. The paper explicitly notes hallucination-style mistakes, including state tracking errors where the model talks as if it has inspected a variable when it has not. CodeAct helps because the interpreter can expose mistakes quickly, but it does not magically remove them.

There is also a safety tradeoff. Giving an agent arbitrary code execution is powerful, which is exactly why it is useful for tool-heavy work and also why it needs sandboxing and careful control.

So the right way to read the paper is not “code solves agency.” The right way is: when the task requires composition, state, and correction across turns, a programming language is a much better action space than a fixed schema.

What to remember

The cleanest mental model is this:

CodeAct turns that loop into the action policy itself. That is why it works on tool-heavy tasks. The model can compose tools, keep intermediate values, and use the runtime as a source of feedback instead of treating failure as the end of the story.

If you remember only one sentence, keep this one: CodeAct is not just a different output format, it is a better substrate for agent action.
