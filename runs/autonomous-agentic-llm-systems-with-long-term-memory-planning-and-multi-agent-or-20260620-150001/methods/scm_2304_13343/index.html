Self-Controlled Memory (SCM)

SCM is not a bigger language model. It is a memory policy wrapped around a language model. The whole point is to stop treating long history as one giant blob of context. Instead, SCM asks three questions in order: do we need memory at all, which memories matter, and is the full memory too bulky to paste as-is?

That is the right framing for long dialogues and long documents. The failure is usually not “the model has never seen the fact”. The failure is that the fact is buried under enough irrelevant history that the model never gets to the useful part. SCM tries to remove that noise before the prompt is built.

What to notice here is simple: the model is not empty-headed. It is distracted. The clue is still somewhere in the transcript, but too much surrounding history makes it hard to surface at the right moment.

Think of SCM like a librarian for an ongoing conversation. If the question is generic, the librarian stays at the desk. If the question reaches back into older history, the librarian pulls the right cards from the archive. If those cards are too thick, the librarian hands back a summary card instead of the full folder.

The system first receives the new observation, then decides whether memory should even be activated, then retrieves the best candidates, then checks whether any of them should be summarized, then fuses the chosen memory into the prompt, and only then generates the response.

The memory stream is an archive, not a transcript dump

SCM keeps a memory stream that stores all historical interaction items. The important part is that each item is not just raw text. It carries enough structure to be searched and compressed later.

Each stored memory item includes:

  • the interaction index,
  • the observation,
  • the system response,
  • a memory summary,
  • an embedding for the interaction semantics.

That design is doing two jobs at once. The summary gives SCM a cheap fallback when the raw text is too long. The embedding gives SCM a way to retrieve by meaning instead of only by string overlap.

One subtle detail matters here: the paper builds the semantic vector from the observation summary and the response summary together. That balances the two sides of the interaction when one side is much longer than the other. Without that, a long user message or a long response could dominate the representation and blur the actual relationship between the two.

The result is a memory stream that behaves more like an indexed archive than a passive log.

The controller decides whether memory should enter the prompt at all

The most important design choice in SCM is the controller. It is also a language model, but it is not being used to answer the user directly at first. It is being used as a gatekeeper.

The controller asks itself two questions:

  1. Is memory needed for this new observation?
  2. If memory is needed, can a summary stand in for the full memory?

This is a much better shape than “always retrieve everything”. Some prompts obviously do not need archival context. If the user says “Tell me a joke”, pulling old memory is just noise. If the user says “Do you remember the conclusion we made last week on the fitness diets”, the archive matters immediately.

That first decision is important because retrieval itself is not free. Even a good retrieval step can add clutter if the current question does not actually need the past.

Formula Card: Rank memories by recency and relevance

Purpose: score candidate memories so SCM can keep the ones that are both useful and timely.

Intuition: a memory can be semantically relevant but stale, or recent but irrelevant. SCM wants both freshness and fit.

Formula:

rank_score=recency_score+relevance_scorerank\_score = recency\_score + relevance\_scorerank_score=recency_score+relevance_score

Symbols:

  • rank_score: the final score used to sort candidate memories.
  • recency_score: how recently the memory was accessed, which favors newer interactions.
  • relevance_score: how semantically close the memory is to the current observation, measured with cosine similarity between the query embedding and the memory embedding.

Read-aloud meaning: add a memory’s freshness and its semantic match, then keep the highest-scoring memories.

Why it matters: this is the core retrieval policy. SCM is not just doing vector search, and it is not just doing recency filtering. It combines both so the system can retrieve the memory that is both about the right thing and recent enough to trust.

Depending on the budget, SCM keeps the top few memories, with k typically in the 3 to 10 range. That is a practical choice, not a magic constant. The controller is trying to stay under the model’s input limit while still preserving the bits of history that matter.

Why summary fallback exists

After retrieval, SCM asks a second question: does the full memory need to be used, or is the summary enough?

This is where the memory controller becomes a compression gate instead of just a retrieval gate. The paper only performs this check when the retrieved activation bundle is already getting large, and it checks individual memories that are longer than 800 tokens. If the total activation memory crosses 2000 tokens, the controller evaluates whether the summary can answer the current question.

The logic is plain: if the summary already contains the answer-bearing facts, do not spend prompt space on the full text. That keeps the prompt tighter and reduces the chance that the important detail gets diluted by extra wording.

This is also where SCM differs from naive retrieval. Naive retrieval says, “Here is everything I found.” SCM says, “Here is what you actually need, and if that is too long, here is the compressed version.”

The same idea also works for books and meetings

The paper is not only about dialogue. The same memory policy is applied to book summarization and meeting summarization by turning the input into segments, processing each segment as an observation, and carrying forward relevant memory across blocks.

That makes SCM feel closer to iterative reading than to ordinary summarization. A long book or meeting transcript gets broken into chunks, each chunk is summarized locally, and then the summaries are summarized again hierarchically. The memory controller helps because the model can keep cross-block context alive without replaying the entire document every time.

This is a useful generalization. The method is still doing the same thing at every stage: decide whether the next chunk needs history, retrieve only the relevant history, compress it if it is too large, then fuse it into the current prompt.

What the evidence actually says

The strongest result in the paper is not a single headline number. It is the pattern in the ablations.

  • SCM davinci003 reached 77.1 answer accuracy and 94.0 memory retrieval recall on long-term dialogue.
  • Removing activation memory dropped answer accuracy to 10.5 and retrieval recall to 0.0.
  • Removing the memory controller dropped answer accuracy to 59.3 and multi-turn accuracy to 49.4.
  • Human annotators favored SCM on book and meeting summarization over RecursiveSum, especially on coverage.

The ablations tell the real story. Activation memory is what lets SCM recover distant facts. The controller is what prevents the system from flooding the prompt with everything it found. The summary fallback is what keeps the system usable when the retrieved memory is too large.

So the gains are not coming from “more tokens”. They come from better memory policy.

Limits and assumptions

SCM is practical, but it is not magical.

  • The paper evaluates up to 200 dialogue turns and 34k dialogue tokens, not unlimited live deployment.
  • It depends on a strong instruction-following backbone such as text-davinci-003 or gpt-3.5-turbo.
  • The controller and summarization logic are prompt-driven, not learned policies.

That means SCM is best understood as a strong wrapper around a capable LLM, not as a fully trained memory architecture. It improves long-context behavior without fine-tuning, but it also inherits the strengths and weaknesses of the underlying model.

What to remember

If you remember one sentence, make it this: when history gets long, memory quality matters more than memory quantity.

SCM’s recipe is: decide whether memory is needed, retrieve by recency plus similarity, compress bulky memories when possible, and only then fuse the result into the prompt. That is why the method works. It manages context before the model has to guess what matters.
