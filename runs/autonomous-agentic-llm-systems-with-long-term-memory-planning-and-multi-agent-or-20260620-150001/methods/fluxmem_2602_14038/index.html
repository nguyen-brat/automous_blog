FluxMem: Adaptive Multi-Structure Memory for LLM Agents

FluxMem starts from a simple complaint: long-horizon agents do not just need more memory, they need memory with the right shape. A conversation about how preferences changed over time wants chronology. A question about who is linked to whom wants relations. A discussion that keeps branching into subtopics wants a hierarchy. One fixed memory layout cannot serve all three well.

The paper’s answer is to treat memory organization as a control decision. FluxMem keeps recent turns in a short-term buffer, stores older conversations as episodic units in a middle layer, and distills durable facts into semantic memory. The middle layer is the interesting part: it can behave like a list, a graph, or a tree depending on the current conversation.

Think of it like a library with three floors. The front desk keeps today’s requests within arm’s reach. The middle floor arranges the recent past in different map styles depending on what the reader is looking for. The archive keeps stable facts that should survive long after the original conversation has moved on.

What to notice here is the mismatch the paper is trying to solve. Some queries are temporal, some are relational, and some are topical or hierarchical. FluxMem is not trying to make one structure do all the work. It is trying to choose the right structure before retrieval starts.

The memory stack

The main architecture figure makes the whole pipeline easier to keep in your head.

What to notice is the division of labor. Recent pages stay hot in short-term interaction memory, episodic history lives in a structure-aware middle layer, and durable knowledge is pushed into long-term semantic memory. FluxMem is not a single store with one retrieval rule. It is a staged memory system that changes behavior as information ages.

Short-term memory stays tiny on purpose

The short-term interaction memory, or STIM, is deliberately small. It holds only the most recent pages, and when the buffer overflows, the least recently used page is flushed into the middle layer. The paper even fixes the buffer at four pages, which is a nice reminder that short-term memory is supposed to be a working surface, not a dump truck.

That design matters for two reasons. First, recent turns are often the best clue for the next response, so they should be available immediately. Second, if the short-term buffer grows without limit, it turns into noise and starts interfering with the more structured layers below it.

So STIM is not where the system keeps the whole conversation. It is where the system keeps the conversation that is still alive.

Mid-term memory is where structure starts to matter

The mid-term episodic memory, or MTEM, is the bridge between raw recency and durable abstraction. Here, pages are grouped into episodic units instead of being appended one by one forever. Each session can carry the page sequence, timestamps, representations, continuity links, a summary, and an index structure tied to the chosen memory shape.

This is the point where FluxMem stops treating every page the same way. A session is not just a stack of turns. It is a structured unit whose internal layout should match the kind of reasoning the later query will need.

The paper also uses a lightweight utility view of these sessions. Episodes that get accessed often, matter in the interaction, and are still recent are more likely to be consolidated upward. That keeps MTEM useful without letting it grow into an unfiltered log.

Long-term memory is for stable knowledge

Long-term semantic memory, or LTSM, is where the system stores things that should outlive the episode that first revealed them. That includes user profiles, durable facts, general knowledge, and reusable interaction strategies.

The important idea is that FluxMem does not keep replaying old episodes forever. It compresses what is stable and discards what is not. LTSM is the archive for things that have earned their place by staying useful across interactions.

If STIM is the desk and MTEM is the filing room, LTSM is the shelf that only keeps the books worth rereading.

Why one episodic shape is not enough

FluxMem supports three episodic structures: linear, graph, and hierarchical. The reason is not elegance for its own sake. It is that long conversations naturally contain several kinds of order at once.

  • Linear memory is best when time matters and the answer depends on the sequence of events.
  • Graph memory is best when the question is really about entities, links, or multi-hop relations.
  • Hierarchical memory is best when the conversation branches into topics and subtopics, and the answer lives at the right level of abstraction.

This is a useful mental model. A query like “what happened first” wants a timeline. A query like “how are these people related” wants a network. A query that wanders across subtopics wants a tree.

The old mistake is to choose one structure at design time and force every query through it. FluxMem pushes that choice later, when the conversation has already given you clues about the right geometry.

How FluxMem chooses the shape

The selector does not read raw dialogue with a heavyweight reasoning pass. It looks at compact, interpretable conversation features that hint at the underlying structure of the exchange.

The paper groups those cues into a few useful buckets:

  • scale cues, such as page count and average page length
  • relational cues, such as entity density, relation indicators, and entity-centricity
  • topic cues, such as topic diversity, topic transitions, and semantic complexity
  • temporal cues, such as time span, temporal density, and question-answer style flow
  • branching cues, such as decision-tree-like dialogue patterns

This is a lightweight design choice, and that is the point. The selector should decide the memory shape without becoming a second language model inside the memory system.

The predictor itself is a shallow classifier, but the interesting part is how it gets supervised. The paper does not rely on direct ground-truth labels for the “best” structure, because those labels do not exist. Instead, it builds them offline.

For each interaction, FluxMem runs the agent under each candidate structure, scores the result with a reward that mixes response quality and memory utilization, and uses the highest-reward structure as the training label. That turns structure choice into ordinary supervised learning instead of unstable online reinforcement learning.

The practical result is easy to miss but important: structure choice is no longer a hidden engineering assumption. It becomes a learned control decision tied to downstream behavior.

The gate that decides whether a candidate belongs

Choosing the structure is only half of the memory problem. FluxMem also has to decide whether a newly retrieved candidate should be merged into an existing session or left out as noise.

This is where the Beta-mixture gate comes in. The paper first computes a matching score for each candidate session, then normalizes those scores to the unit interval so they can be modeled as a probability distribution. Instead of using a hand-tuned similarity threshold, FluxMem fits a two-component Beta mixture and uses the posterior of the high-compatibility component as the keep score.

Formula Card: A distribution-aware gate for memory fusion

Purpose: replace a brittle similarity cutoff with a learned posterior that decides whether a candidate memory is strong enough to merge.

Intuition: candidate scores usually split into two regimes. Some are weak matches that should be ignored. Others are strong enough to belong. A Beta mixture gives those two regimes a compact probabilistic model.

Read-aloud meaning: model the score distribution with two Beta-shaped regimes, then use the posterior probability of the better-fitting regime as the gate.

Why it matters: this is the fuse-or-drop decision that keeps FluxMem from stuffing weakly related memories into the wrong episode. It is also what makes the integration step adaptive when score distributions shift across conversations.

FluxMem keeps a minimum-keep fallback so the gate does not become overconfident and throw away too much. In plain language, the system would rather retain one plausible merge target than accidentally shut the door on the whole retrieval pass.

Once candidates survive the gate, they are merged into the best matching episodic unit under the chosen structure. If nothing fits well enough, FluxMem starts a new unit instead of forcing a bad merge.

The retrieval path is hybrid, not single-source

Another practical detail is worth keeping in mind: FluxMem does not trust one retrieval signal to do everything. It combines dense matching and BM25 with reciprocal rank fusion.

That hybrid step is useful because dense retrieval is good at semantic similarity while BM25 is good at exact lexical overlap. When the conversation is phrased loosely, dense matching helps. When the clue is a specific name, term, or phrase, BM25 helps. Reciprocal rank fusion lets the system benefit from both without pretending they score on the same scale.

So the full path is not just “select a structure and retrieve”. It is “select a structure, search it with complementary retrieval signals, then let the Beta-mixture gate decide what actually deserves to merge.”

Why the pieces fit together

FluxMem works because each layer solves a different part of the same bottleneck.

STIM protects recency. MTEM preserves episodic shape. The selector matches shape to the current conversation. The Beta-mixture gate filters noisy merges. LTSM keeps durable facts compact and reusable.

That separation is what makes the system feel coherent. It is not one giant memory store trying to be clever everywhere. It is a small set of decisions, each made at the point where it matters most.

The case studies in the paper line up with that story. A time-dependent query benefits from linear ordering, a relational query benefits from graph links, and a topic-evolution query benefits from hierarchical abstraction. Those are not different problems hidden under one label. They are different retrieval geometries.

What the numbers say

The evaluation backs up the intuition. On PERSONAMEM, FluxMem reaches 72.43% average accuracy, which is a 9.18% improvement over the strongest baseline. On LoCoMo, it reaches 51.16 average F1 and shows particularly strong gains on multi-hop, open, and single-hop settings.

The ablations are even more revealing than the headline averages. Removing the linear structure hurts temporal questions. Removing the graph structure hurts relational reasoning. Removing the hierarchical structure hurts abstraction-oriented queries. Removing the BMM gate hurts memory fusion across the board.

That pattern is exactly what you would expect if the method is doing real work rather than just adding another retrieval layer. Each component is covering a different failure mode.

The tradeoff is also clear. FluxMem adds control complexity and depends on offline reward labels for structure supervision. The selector also uses handcrafted conversation features, and the paper only explores three episodic families: linear, graph, and hierarchical. So this is not a final answer to memory design. It is a strong argument that memory shape should be a decision, not a constant.

What to remember

The easiest way to remember FluxMem is this: choose the map before you search.

Recent turns stay in a short-term buffer, older episodes are arranged in the memory shape that best fits the conversation, and stable facts are distilled into semantic memory. Then a distribution-aware gate decides whether a candidate memory is strong enough to merge. The core idea is not bigger storage. It is better control over memory geometry.
