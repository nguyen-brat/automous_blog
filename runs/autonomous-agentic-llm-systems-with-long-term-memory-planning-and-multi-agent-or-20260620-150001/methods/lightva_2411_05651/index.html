LightVA: Lightweight Visual Analytics With LLM Agent-Based Task Planning and Execution

Visual analytics usually breaks down at the handoff between intention and implementation. You know what you want to learn from the data, but you still have to decide which tasks to run, which columns matter, which chart to build, how to connect multiple views, and whether the current result is good enough to stop or should be split into smaller tasks. LightVA is built to handle that orchestration work. It uses an LLM agent to recommend tasks, write and run the analysis code, generate visualizations and insights, and decide whether the current task should be decomposed further.

That is why the paper calls the framework lightweight. It is not trying to replace the analyst with a fully automatic black box. It is trying to remove the most time-consuming part of visual analytics: translating a vague goal into a workable analysis loop.

The method is a loop, not a one-shot answer

The easiest way to understand LightVA is to stop thinking of visual analytics as a single chart-generation request. The paper treats it as a recursive workflow:

The important part is the feedback. The system does not just answer the user once and quit. It keeps turning intermediate results into the next round of planning.

What to notice here is the shape of the workflow. The user brings in a goal and data, the agent turns them into tasks, the executor turns tasks into charts and insights, and the user can then refine what the agent produced. The paper is really about keeping that back-and-forth visible and structured instead of burying it inside one giant prompt.

Three roles keep the loop from getting messy

LightVA splits the agent into three roles: planner, executor, and controller.

The planner has two jobs. First, it recommends initial exploratory tasks from the user goal and the dataset. Later, it proposes new tasks or decompositions after the system has already learned something from earlier results.

The executor does the work that analysts normally do by hand. It writes the Python code for the analysis, uses Altair and Vega-Lite to build the visualization, and reports the insight in a structured form.

The controller is what keeps the system lightweight. It decides whether the current task looks finished or whether the result is weak enough that the task should be split again. That choice matters because recursive decomposition is useful only when it is needed. If the system decomposes everything immediately, it wastes time and makes the analysis feel heavy.

What to notice in the architecture is that the planner is not a single module. It has a recommendation path and a decomposition path. That is the recursive heart of the system. The executor sits in the middle and produces concrete analysis artifacts, and the controller sits above that to decide whether the loop should continue or switch back to broader exploration.

LightVA needs a few clear primitives

The paper does something smart before it talks about prompts or code generation: it defines the objects the system is allowed to manipulate.

A goal is the vague, high-level thing the user wants to understand. In the paper’s own examples, it might be something like analyzing what influences fuel efficiency or finding high-risk events in a city.

A task is the actionable unit that makes that goal executable. LightVA treats tasks as having a type, a set of data variables, a method, and a progress state. That matters because the agent is not just saying “analyze this.” It has to say what kind of analysis it is, which columns it touches, how it will be solved, and whether it is finished.

An insight is the knowledge the task is supposed to produce. It has its own structure too: the kind of discovery, the relevant parameters, the data variables involved, and the actual values or patterns that support the claim.

A visualization is not just a chart. It also includes the encoding, the interaction, and the coordination behavior with other views. That extra structure is what lets LightVA build linked views instead of isolated pictures.

This structured vocabulary is the reason the recursive loop works. The planner can recommend a task, the executor can produce a chart and an insight, and the controller can judge whether the result is complete because all three stages are speaking about the same underlying objects.

First the agent recommends tasks, then it gets more selective

Task recommendation is the entry point. In the beginning, the agent receives the goal and the data, then maps the goal to the columns and task types that can actually support the analysis. The paper suggests broad task families like trend, correlation, category, and distribution, because those are often the first useful lenses on a dataset.

That first round is intentionally broad. The agent is supposed to help the user explore from several angles, not lock into one narrow interpretation too early.

After that, the recommendation logic changes. Once some tasks have already been explored, the agent starts using history. It can propose new tasks that build on earlier discoveries, and it can also remind the user about previously proposed tasks that have not yet been explored but still look relevant.

That historical context is important because visual analytics is cumulative. A good next question often depends on what the previous chart already revealed.

Execution is where LightVA becomes a real analysis assistant

Once the user accepts or writes a task, the executor takes over. This is the part of the system that most directly resembles the work people normally do in notebooks and dashboard tools.

The paper uses a two-step execution path. First, the agent writes the analysis and visualization code. Second, it runs the code and turns the result into a structured insight.

There is a reason the paper splits those steps instead of asking the model to do everything in one shot. Python is good for actual data manipulation and more complex modeling, while Altair and Vega-Lite are good for declarative chart construction. LightVA uses both. That lets it go beyond simple chart selection and handle things like regression or clustering when the task needs it.

The paper also tries to keep the code generation consistent by giving the model a scaffold to fill in. That is a practical choice. It reduces the chance that the analysis code and the chart code drift apart.

There is another useful constraint here: one task normally maps to one visualization. The system only starts combining views when the task really calls for multi-variable association analysis. That keeps the output easy to inspect and avoids turning every task into a dashboard by default.

When several charts belong together

LightVA does not automatically glue every chart into a multi-view display. The paper is more careful than that. It lets the user choose which visualizations should be merged, because subtasks often share variables and chart types in ways that make automatic merging awkward.

When the system does build linked views, it applies a few guardrails:

  • the charts need common key columns with compatible data formats,
  • each chart gets a suitable interaction mechanism, such as brushing or clicking,
  • linked charts should stay visually consistent,
  • colors should not be reused across unrelated data dimensions,
  • the layout should stay compact, with no more than three charts in a row,
  • users can merge up to six charts to avoid overloading the display.

These constraints are not decorative. They are what make the multi-view space usable. Without them, the dashboard would quickly become a pile of unrelated graphics.

The controller decides whether to stop or recurse

After the executor produces a result, the decomposer checks whether the task is really done. This is where LightVA’s recursive behavior becomes explicit.

The task is scored for completion. If the score is high enough, the controller can move on and ask the recommender for the next exploratory task. If the score is too low, the system proposes a decomposition plan.

The key design choice is that decomposition is on demand. The paper does not want to decompose before seeing any result. It wants to try the task first, then recurse only if the output looks incomplete or too shallow.

That gives the user faster feedback. It also keeps the analysis tree from exploding too early.

The paper also distinguishes two kinds of logic for decomposition:

  • AND means the subtasks are independent and can run in parallel.
  • DOWN means the subtasks depend on one another and should run step by step.

That distinction is easy to miss, but it is useful. Some analyses branch into parallel questions, while others require one subquestion to prepare the next.

The task flow itself is stored as a directed graph, so the system can keep track of what was recommended, what was explored, what was decomposed, and what still needs review. Progress then propagates upward through that tree, which gives the user a visible sense of where the analysis stands.

The interface makes the recursion visible

The paper does not hide the agent behind a chat box. It exposes the whole analysis process through four coordinated views.

The Chat view is where the user gives goals, accepts suggested tasks, or asks the agent to keep going.

The Visualization view shows the generated charts and insights as cards. That matters because the user should be able to inspect, edit, or export the code and the text rather than treating the agent’s output as final.

The Task flow view shows the tree of goals, tasks, subtasks, and progress. This is one of the most useful parts of the system because it turns an invisible planning process into something you can inspect and steer.

The Data table view uses a table-lens style overview of data usage. That sounds small, but it helps the user see which parts of the dataset have been explored heavily and which regions still look untouched.

The overall effect is that planning, execution, and progress are all visible at once. The user is not just reading answers. The user is watching the agent build the analysis.

The linked-view example shows what the loop is for

What to notice here is not the exact event story. The point is the interaction pattern. A broad task such as sentiment analysis or keyword exploration is only the start. Once the relevant charts exist, the analyst can brush across time, inspect a histogram, and use the map to connect patterns that would be hard to see in one view alone.

That is the real promise of LightVA: the agent helps you get to a usable analysis workspace much faster, and then the workspace itself helps you keep asking better questions.

In the paper’s scenario, that loop reduced a workflow that previously took roughly 108 hours of manual preprocessing, interface construction, and analysis to about 1 hour of guided work on the prepared data. The exact number is specific to that case, but the broader lesson is general: once the system can plan tasks, generate code, and compose linked views, a lot of the setup burden disappears.

When the code fails, the system tries to recover instead of collapsing

The authors do not pretend that LLM-generated analysis code is always stable. They explicitly discuss error handling because generated code can fail in boring but disruptive ways: the dataset may be unfamiliar, a variable name may not bind correctly, a serialization step may break, a transformation may go wrong, or the code may simply have syntax errors.

To reduce those failures, LightVA uses a few layers of defense.

Before generation, it uses prompting tricks such as few-shot examples and step-by-step reasoning guidance.

After generation, it lets the model inspect its own work and try to repair the code. The system also catches common low-level issues with rule-based fixes, especially simple syntax problems.

If the failure still matters, the user can edit the code manually and the system keeps the analysis history so the user can roll back to the previous step instead of losing the whole session.

That recovery path is important because it makes the system feel like an assistant rather than a fragile demo. The goal is not to eliminate every error. The goal is to keep the exploration moving.

Where the paper is careful

LightVA is useful, but it is not a complete automation story.

The paper says complex domains still need more guidance for task customization, visualization choice, and model selection. In other words, the agent can help structure the work, but it does not magically know every domain-specific rule.

It also does not include a built-in long-term memory module. The paper mentions memory as a possible future extension, which makes sense because recurring exploratory sessions would benefit from carrying prior analysis forward.

The expert study is also relatively small, so the results show promise rather than broad proof.

Those limits do not weaken the main idea. They just define the current scope: LightVA is a recursive orchestration framework for visual analytics, not a full replacement for domain expertise.

What to remember

The easiest way to remember LightVA is to think of it as a planner-executor-controller loop for analysis work.

The planner turns vague goals into candidate tasks and later into subtasks. The executor turns those tasks into code, charts, and insights. The controller decides whether the current task is good enough or needs deeper decomposition.

The user stays in the loop through a chat view, a task tree, a visualization panel, and a data coverage lens. That is what makes the system practical: the agent does not hide the analysis process. It helps organize it.

If you want the shortest possible summary, it is this: LightVA makes visual analytics recursive, visible, and less manual by letting an LLM agent plan the work, execute the analysis, and decide when to split the task again.
