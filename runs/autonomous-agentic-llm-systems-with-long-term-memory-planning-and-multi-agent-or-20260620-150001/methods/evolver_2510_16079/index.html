EvolveR: Self-Evolving LLM Agents through an Experience-Driven Lifecycle

EvolveR starts from a complaint that most agentic LLMs still fail in a very ordinary way: they can solve a task, but they do not turn that task into durable know-how. A successful run usually disappears the moment the answer is returned. The next question arrives, and the model starts again from scratch. EvolveR tries to break that cycle by turning each trajectory into a reusable principle, storing those principles in an experience base, and using that experience to guide the next round of reasoning.

You can think of the whole method as a loop that keeps converting action into guidance:

That is the paper’s core move. It is not just “add memory.” It is “teach the agent to write down what it learned, keep only the good parts, and then train it to use those lessons the next time.”

What to notice in the figure is the closed loop. The online phase produces trajectories, the offline phase compresses them into principles, and the resulting experience base feeds later online decisions. The system only becomes self-improving because the same agent both writes the lesson and reads it back.

The memory here is not a logbook

EvolveR does not try to store raw trajectories forever. That would quickly become noisy and repetitive. Instead, it stores short strategic principles. Each principle has two parts: a natural-language rule or warning, plus a small set of structured triples. Those triples are basically subject-predicate-object triplets, which matter because they make the lesson easier to match, deduplicate, and merge later.

That design is doing two jobs at once. The prose version is easy for the model to read during retrieval. The structured version makes the memory base more machine-like, so the system can tell whether two principles are really the same idea or just similar phrasing.

The action space is intentionally small:

  • <think> for internal reasoning,
  • <search_experience> to retrieve distilled principles from the experience base,
  • <search_knowledge> to fetch external facts,
  • <answer> to end the episode.

The important split is between experience and knowledge. External search fills factual gaps. Experience search supplies strategic guidance. EvolveR works because it keeps those roles separate instead of pretending that a retrieved web page and a retrieved lesson from past failures are interchangeable.

How a trajectory becomes a principle

The offline phase is where the experience base grows. After the agent finishes a task, EvolveR asks the same model to look back at the trajectory and summarize the lesson.

If the trajectory succeeded, the model writes a guiding principle. If it failed, it writes a cautionary principle. In both cases, the goal is not to restate the whole conversation. The goal is to extract the reusable insight that explains why the trajectory succeeded or failed.

That is why the paper emphasizes concise natural-language statements paired with triples. A useful principle is closer to “when the task is a comparison, gather evidence for both sides before concluding” than to a transcript of every search query the agent made. The distilled rule should be broad enough to reuse, but specific enough to change the next rollout.

The paper also does something important with duplicates. Because GRPO samples multiple rollouts for the same prompt, the same idea often appears several times with slightly different wording. EvolveR first checks for semantic equivalence among the newly distilled principles from the same problem, keeping only one representative per cluster. Then it compares the remaining candidate against the existing experience base with embedding similarity. If the candidate is genuinely new, it is added. If it is just another phrasing of an existing idea, the new trajectory is merged into that principle instead of creating another copy.

That gives the memory base a very practical shape: not a pile of transcripts, but a curated library of lessons with linked evidence behind each one.

Why the score matters

Once the base starts growing, the system needs a way to tell useful principles from noisy ones. EvolveR does that with a simple empirical score.

Formula Card: Principle quality score

Purpose: rank each principle by how reliably it helps when the agent retrieves it.

Intuition: a good rule is one that keeps paying off. The score should go up when the principle leads to success and down when it is frequently used but not very helpful.

Formula:

s(p)=csucc(p)+1cuse(p)+2s(p)=\frac{c_{\text{succ}}(p)+1}{c_{\text{use}}(p)+2}s(p)=cuse​(p)+2csucc​(p)+1​

Symbols:

  • s(p) is the quality score of principle p.
  • c_{\text{succ}}(p) is the number of successful uses associated with principle p.
  • c_{\text{use}}(p) is the number of times principle p has been used or retrieved.
  • p is one distilled principle in the experience base.

Read-aloud meaning: add one to the success count, add two to the usage count, and use that ratio to rank the principle.

Why it matters: this keeps the experience base compact and biased toward principles that actually help. The +1 and +2 are a small smoothing trick, so a brand-new rule does not look absurdly perfect or terrible after only a couple of observations.

The paper then periodically prunes principles whose score falls below a threshold. That matters more than it sounds. Without pruning, the memory base would accumulate vague advice, stale advice, and duplicated advice. The score-plus-prune loop keeps the library from turning into a junk drawer.

A useful way to remember the failure mode is that low-score principles are often not wildly wrong. They are just too generic to steer the agent. The paper’s manual checks show that the high-score tier is packed with specific, actionable rules, while the low-score tier is much more likely to contain vague advice that sounds reasonable but does not actually change behavior.

What changes when the agent starts solving a new task

The online phase is where the stored experience becomes useful. The agent runs its normal think-act-observe style loop, but now it can call <search_experience> at any point.

That search does not retrieve facts in the usual RAG sense. It retrieves strategic principles. A retrieved principle might say something like “for comparison questions, gather data on both items before concluding.” That does not answer the question directly. Instead, it changes the next move the agent makes: which thing to search first, whether to gather evidence for both sides, and how to frame the final answer.

That is the subtle but important difference between memory and wisdom. Experience search is not there to replace reasoning. It is there to bias reasoning toward patterns that already worked.

A toy example makes the flow easier to see. Imagine the agent is asked which of two candidates is more plausible. Without experience, it may jump into the first clue it finds and stop too early. With a retrieved principle about comparison tasks, the agent is nudged to collect evidence on both candidates first, compare the evidence side by side, and only then answer. The principle is short, but it changes the shape of the search.

This is also why the online phase generates better training data. A trajectory guided by a good principle is not a random walk. It is a structured example of how the agent used its own experience to solve a task. That makes it more valuable input for the next offline distillation pass.

How GRPO closes the loop

EvolveR finishes the cycle by training on those experience-guided trajectories. The reward has two parts. The outcome reward is sparse and strict: did the final answer exactly match the ground truth? The format reward is dense shaping: did the trajectory contain the right mix of thinking, experience search, knowledge search, and answering, and was it structurally complete?

The paper uses the format term carefully. It does not replace correctness. It just nudges the model toward trajectories that are both answerable and reusable. A trajectory that skips the reasoning/search structure entirely is not what EvolveR wants to reinforce.

What to notice in the figure is that the policy update sits on top of the experience loop rather than beside it. The model samples several rollouts for the same prompt, compares them through Group Relative Policy Optimization (GRPO), and updates the policy with a KL penalty to keep the change stable. There is no separate value network here; the method relies on group-relative comparisons, which is a good fit when rewards are sparse but multiple sampled rollouts can be compared against one another.

The most important part is what the RL stage is actually teaching. It is not just teaching the model to answer more often. It is teaching the model how to use its own distilled principles effectively while it is answering. That is the full loop: experience creates principles, principles guide rollouts, rollouts create training signal, and training makes the next round of experience better.

Why the paper thinks this is better than simpler alternatives

The ablations line up with the story. On the 3B model, self-distilled EvolveR reaches 0.382 average EM, while the teacher-distilled variant reaches 0.370. Removing experience retrieval drops the model to 0.340. That is a strong sign that the gains are not just coming from extra data or a bigger teacher. They come from the agent internalizing lessons that are aligned with its own policy and then actually consulting those lessons at inference time.

The scale story is equally important. Smaller models still benefit from an external teacher, which is a reminder that self-distillation is bounded by the base model’s reasoning ability. But once the base model is strong enough, the self-distilled version becomes the better fit because the lessons match the student’s own failure modes and action patterns more closely.

The memory system also scales better than you might expect. The paper reports an experience base growing to nearly 50,000 principles while top-3 retrieval stays around 0.20 seconds. That is not free, but it is small enough to keep the loop practical. The main risk is not raw latency. It is retrieval noise and principle drift, which is exactly why deduplication, scoring, and pruning are part of the method rather than afterthoughts.

What to remember

EvolveR is easiest to remember as a four-step habit:

  • solve a task,
  • distill the useful lesson,
  • store only the principles that keep proving themselves,
  • retrieve those principles on the next task and train the policy to use them well.

The paper’s real contribution is not just a new memory format. It is a closed-loop way to turn agent experience into durable, curated, and actionable knowledge. If a standard agent is a one-shot problem solver, EvolveR is trying to make the agent behave more like a learner: work, reflect, keep the good rule, and apply it next time.

Limitations remain. A weak base model cannot distill great principles from mediocre trajectories. Noisy retrieval can still mislead the agent. And the paper validates the loop mainly on complex QA settings, so broader task coverage is still open. But as a design pattern, the lesson is crisp: do not just store history. Compress history into principles, then train the model to consult them.
