Adaptive Memory via Multi-Agent Collaboration (AMA)

Long-term memory for LLM agents fails in a surprisingly ordinary way: a single retrieval style cannot serve every question. Some prompts need the exact wording from a past turn. Others need one clean fact. Others need a higher-level episode, where several turns are compressed into a coherent story. AMA’s core idea is to stop pretending that one chunk size can do all three jobs. Instead, it treats memory as a small workflow with separate roles for storing, searching, checking, and repairing.

The first figure captures the central problem nicely.

What to notice here is not the literal box drawing, but the choice it forces. Static memory systems must pick one granularity up front. If the chunks are too large, retrieval drags in irrelevant text and confuses the model. If the chunks are too small, the system loses the links that make a memory useful in the first place. AMA responds by choosing the memory lens based on the question.

The Constructor writes three different kinds of memory

The Constructor is the part that turns a conversation stream into something searchable later. It does not flatten everything into one summary. Instead, it builds three layers.

  • Raw text memory preserves the original utterance, plus references to the turns it came from and its timestamp/speaker metadata.
  • Fact knowledge memory breaks the turn into atomic factual statements.
  • Episode memory compresses a stretch of dialogue into a higher-level summary.

That split matters because different questions need different levels of detail. If a user later asks for the exact phrasing of a previous message, the raw record is the safest source. If they ask for a particular attribute or state, a fact is enough. If they ask for the gist of a longer exchange, the episode memory is the better abstraction.

The episode path is especially important because it is not generated for every turn. The paper only triggers it when the conversation actually calls for a higher-level abstraction, such as a topic shift, an explicit request, or a crowded context window. In other words, AMA does not pay the cost of summarizing everything all the time. It creates episodes when the dialogue starts to behave like an episode.

That is a subtle but useful design choice: the system keeps the raw trail for traceability, but it also builds compact forms that are easier to search when the reasoning task is more abstract.

The Retriever chooses the right lens before searching

Once memory exists, the Retriever decides how to ask for it. This is more than a nearest-neighbor lookup. AMA first rewrites the incoming query into a self-contained version so that pronouns, omissions, and half-finished references do not poison retrieval. That is a practical fix for dialogue, where the latest user message is often incomplete on its own.

Then the Retriever infers what kind of memory the query actually wants. The paper describes four intent signals:

  • fine-grained detail,
  • abstract summary,
  • cross-temporal event,
  • atomic fact.

The logic is straightforward. If the user wants exact detail, route to raw text. If they want an abstract or event-level answer, route to episode memory. Otherwise, fall back to fact memory. This is the part that makes AMA feel adaptive rather than merely organized.

A toy example helps. If the user asks, “What did I say about the migration plan last week?” the system is not looking for a single isolated fact. It wants a compressed episode that spans several turns. If the user asks, “What city did I mention as my office location?” a fact memory is enough. If the user asks, “What exact wording did I use in that message?” raw text is the only safe choice.

The Retriever also does not lock itself into a fixed retrieval depth. It predicts how many candidates it needs, but the system enforces a minimum so the query does not come back starved of evidence. That matters because a too-small retrieval set can be just as bad as a noisy one: the model ends up answering from an underpowered slice of history.

The Judge makes relevance and consistency separate questions

A lot of memory systems stop after retrieval, which is exactly where drift starts. AMA inserts the Judge between retrieval and use. That is a good idea because similarity is not the same thing as usefulness.

The Judge asks two questions in order. First: did we retrieve something relevant enough to trust? Second: do the retrieved memories still agree with the current situation?

The relevance check matters because vector search can easily return items that are semantically adjacent but not actually helpful. If the retrieved set looks too thin, the Judge does not force the model to guess. It sends the Retriever back for another pass and lets it broaden the search, but only up to a bounded number of rounds. That keeps the system from looping forever.

Only after relevance passes does the conflict check start. This is where AMA looks for contradictions between the current input and the retrieved memory. The paper’s example is the sort of thing long-term agents often get wrong: an old fact may still be stored, even though the user has updated it later. AMA treats that as a maintenance problem, not just a retrieval problem.

This ordering is important. A memory can be internally consistent and still be the wrong memory. The Judge prevents the system from confusing those two cases.

The Refresher is the maintenance loop that keeps memory honest

The Refresher is the piece that turns AMA from a smart retriever into a real memory manager. When the Judge finds a conflict, the Refresher decides whether to update or delete the stale entry.

Delete is reserved for narrow cases: the user explicitly asks to forget something, or the memory has lived past its retention limit. That is a conservative policy, and it makes sense. Deleting too eagerly would erase continuity. Most of the time, the system should preserve the memory and bring it up to date instead.

So the default action is update. The Refresher rewrites the stale memory so it reflects the latest state implied by the current turn. That preserves the useful part of the record while removing the contradiction. In plain language, AMA prefers to repair memory rather than throw it away.

This is where the method’s long-term consistency comes from. Without a repair step, old facts quietly stack up and future answers start inheriting the wrong state. With the Refresher, memory is not just accumulated. It is actively maintained.

Why the full loop matters more than any single piece

The paper’s ablations make the design feel less abstract. Table 3 is the clearest proof that the pieces are complementary rather than redundant.

The trend is simple:

  • using only one memory granularity works, but not as well as combining raw, fact, and episode memory;
  • the full three-layer design is strongest overall;
  • removing the Refresher causes knowledge-update performance to fall sharply.

That last point is the most revealing. It shows that retrieval quality alone is not enough for long-term agents. If stale entries are never corrected, the memory store slowly becomes an active source of error. AMA’s maintenance loop is what keeps the store usable over time.

The efficiency numbers tell the other half of the story. With the default retrieval-round cap of 2, AMA uses 3,613 input tokens versus 18,625 for FullContext on LoCoMo, yet reaches a 0.774 LLM Score. On LongMemEvals it averages 0.698 accuracy and reaches 0.897 on knowledge-update questions. Those are exactly the cases where stale memory would hurt most.

So the method is not just about being clever with retrieval. It is about spending compute where it matters: choose the right abstraction level, verify the evidence, and repair the store when the evidence says the history has gone stale.

What to remember

If you want one mental model, keep this one:

AMA treats memory like a tiered cache with a quality-control loop.

  • The Constructor writes the past at three levels: raw, fact, and episode.
  • The Retriever chooses the right level for the current question.
  • The Judge checks whether the retrieved memory is both relevant and consistent.
  • The Refresher updates or deletes stale records so the next turn starts from a cleaner state.

That is why AMA stands out. It does not assume that one representation of history is enough. It separates storage, search, verification, and repair, then lets each step do one job well.

The result is a memory system that is easier to reason about than a single monolithic controller and more robust than a static chunking strategy. For long-lived agents, that separation is the whole point.
