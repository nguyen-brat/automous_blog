AutoAgents: A Framework for Automatic Agent Generation

AutoAgents starts from a simple but important idea: a fixed agent roster is a bad default when the task itself changes the kind of expertise you need. A story wants a different mix of skills than a software project. Open-ended question answering wants a different mix again. Instead of forcing every job through the same hand-built team, AutoAgents first staffs the task, then writes the plan, and only then begins execution.

Think of it like a project manager that does not assume the same people should work on every assignment. It looks at the request, hires the right specialists for this job, sets up the work order, and keeps the team synchronized while the work is happening.

What to notice in the figure is the hard separation between staffing and doing the work. AutoAgents does not start by improvising a single answer. It first decides who should work, then decides how they should work together, and only after that does it let the generated team act.

The drafting stage is really a staffing problem

The drafting stage is the part of the system that asks, “What kind of team does this task need?” That sounds simple, but it is where a lot of agent systems quietly fail. If the roster is wrong, the plan is wrong before execution even starts.

AutoAgents handles this with three predefined agents:

  • The Planner proposes and refines the task-specific team and the execution plan.
  • The Agent Observer checks whether the proposed roles are sensible and complete.
  • The Plan Observer checks whether the proposed steps actually fit the task and the team.

This is not a one-shot generation step. The Planner and the two observers talk back and forth for several rounds. That matters because the first draft of a team is often the weakest one. A role that looks plausible in isolation may be redundant, or the plan may leave out a crucial specialist. The observers exist to catch exactly those failures.

The paper also makes a useful move here: it defines what a generated agent should contain. Each agent is not just a title. It is a compact role package with four parts:

  • a prompt, which defines the profile, goal, and constraints of the role,
  • a description, which gives the role a more concrete identity,
  • a toolset, which is chosen from the available tools instead of giving everything to everyone,
  • suggestions, which tell the agent how to execute the current task.

That structure matters because it stops role generation from becoming vague role-playing. A generated agent should know what it is, what it should try to accomplish, what it is allowed to use, and what kind of output is expected.

The limit on tools is also deliberate. The paper does not hand every tool to every agent, because too many tools create confusion instead of capability. AutoAgents is trying to reduce decision overload, not increase it.

The Agent Observer then checks whether each role is actually usable. It looks for missing prompt elements, checks whether the role matches the task, and removes redundant or missing roles. The Plan Observer does the same thing for the workflow itself. It makes sure each step has an assigned agent, that the steps are coherent and concise, and that the plan is complete enough to solve the task.

The result is a drafting panel that behaves less like a prompt generator and more like an interview loop. The Planner suggests, the observers critique, and the team gets revised until the roster and the plan look rational enough to trust.

That idea becomes very concrete in the paper’s software case study. If the task is to build a Tetris game, the system does not stop at “programmer.” It can staff a game design expert, a UI design expert, a programmer, and a debugging expert. The point is not that every task needs exactly those four roles. The point is that the team should be assembled for the job in front of it, not copied from a template.

The plan is a work order, not just a list of steps

The execution plan that comes out of drafting is also more structured than a normal checklist. Each step is written with a responsible agent, the input information that step needs, and the output it should produce.

That matters because a multi-agent system does not fail only when it picks the wrong roles. It also fails when the handoffs are fuzzy. If one agent does not know what the next agent expects, the team can drift even if every specialist is individually competent.

So the plan is really a chain of work orders:

  • who acts,
  • what that agent receives,
  • what that agent should return,
  • and how the next step can use it.

That is a small detail with a big effect. It makes the execution stage easier to supervise, because the system can judge not only whether a step was completed, but whether it produced the right kind of intermediate artifact for the next step.

Execution is where the team learns to act like a team

Once the team and plan are drafted, AutoAgents switches to execution. This is where the paper moves from “who should work?” to “how should they stay aligned while working?”

The answer is a vertical communication pattern centered on an Action Observer. The Action Observer acts like a team manager. It assigns subtasks, checks the outputs of each generated agent, and adapts the execution plan when the situation changes.

That vertical structure is important. It gives the team a clear coordination point instead of letting everyone talk to everyone in a free-for-all. The system is especially useful for tasks that need iterative decision-making, because the Action Observer can keep the team focused on the current objective instead of letting the conversation wander.

What to notice here is that AutoAgents uses two different improvement loops during execution. One loop stays inside a single agent. The other loop happens across several agents.

Self-refinement is the local loop. A single agent thinks through the task, writes a plan for itself, executes that plan, looks at the result, and then revises its own approach. This is useful when the task is narrow enough that one specialist can improve by checking its own work.

Collaborative refinement is the shared loop. Several agents take turns, each contributing from a different angle, until the group reaches a better answer or a consensus. This is the right move when the task needs more than one kind of expertise at once.

The difference is easy to miss, but it is central to the paper. Self-refinement improves one specialist’s competence. Collaborative refinement improves the team’s combined judgment. AutoAgents uses both because complex work usually needs both kinds of repair.

You can think of it this way: self-refinement helps an agent notice its own bad assumptions, while collaborative refinement helps the team notice blind spots that no single agent would catch alone.

Memory keeps the team coherent without stuffing the prompt

Long-horizon agent systems quickly run into a token budget problem. If every action, every discussion, and every intermediate result stays in the prompt forever, the context becomes too large and too noisy to use well. AutoAgents handles that by splitting memory into three layers.

The three layers have different jobs:

  • Short-term memory keeps the trace of one action’s self-refinement or collaborative refinement. It is the local scratchpad for the current step.
  • Long-term memory keeps the history of many actions across the whole task. It is the broader record of what the team has already done.
  • Dynamic memory is a focused slice pulled from long-term memory when the next action needs special attention.

That last layer is the most interesting one. The system does not simply dump the whole past back into the prompt. The Action Observer pulls out the part of the history that is actually relevant to the next action and turns it into a compact working memory. That is how the method stays useful even when the history gets long.

The logic here is practical rather than decorative. Short-term memory helps one step stay internally consistent. Long-term memory preserves the story of the whole task. Dynamic memory makes sure the right past information shows up at the right moment instead of burying the current decision under irrelevant history.

Why the paper thinks this works

The experiments are really checking whether the extra structure pays off. On open-ended question answering, AutoAgents outperforms Vicuna-13B, ChatGPT, and GPT-4 under both FairEval and HumanEval. On Trivia Creative Writing, it improves the metric score by about 10% over Standard Prompting.

Those results fit the design story. Open-ended questions benefit from a team that can bring different perspectives to the answer. Trivia Creative Writing benefits from a system that can pull in domain knowledge, keep the narrative coherent, and check itself while it writes. AutoAgents is doing exactly that by combining generated roles, observer-guided planning, and layered memory.

The ablations make the same point even more clearly:

  • removing observers from the drafting stage lowers performance by about 3%,
  • removing self-refinement drops performance by about 3%,
  • removing collaborative refinement drops performance by about 2%,
  • removing dynamic memory drops performance by about 1%.

That pattern is useful because it tells you the observers are not just bureaucracy, the refinement loops are not redundant, and the memory design is not just a token-saving trick. Each part contributes to the final result.

The strongest takeaway is that AutoAgents improves the output not by making one agent better at everything, but by making the team itself more appropriate for the task.

Limits and assumptions

The paper is also honest about what it does not solve.

The first limitation is that the generated roles are still mostly differentiated by prompts and tool choices. That means the specialization is useful, but still shallow. The system can create better role boundaries, but it does not magically create deep expert training.

The second limitation is model dependence. The framework leans heavily on GPT-4-level reasoning and text generation, so it may transfer poorly to weaker models unless the prompting and coordination logic are reworked.

The third limitation is that role generation and planning can still go wrong. AutoAgents improves the odds of assembling a sensible team, but it does not guarantee that the first team draft is rational. That is why the observers matter so much, and also why the paper treats team assembly as a problem worth revisiting instead of assuming the first answer is correct.

The open questions that fall out of that are straightforward:

  • How can agent generation be made more reliable without handcrafted observers?
  • How should long-term memory be compressed and retrieved for larger teams?
  • Can the same framework adapt cleanly to smaller or different model families?

Those are the right questions to ask after reading the paper. They are not side notes. They are the next layer of the method.

What to remember

AutoAgents is easiest to remember as a staffing system with supervision.

First, it asks what kind of team the task needs. Then it drafts the team and the plan with observer feedback. Then it executes the plan through a manager-like Action Observer. Finally, it keeps the team coherent with short-term, long-term, and dynamic memory.

The deepest lesson is that complex agent work is often a coordination problem before it is a reasoning problem. If the task needs different expertise, different handoffs, and different memory scopes, the system should adapt its team structure instead of pretending one fixed roster can do everything.
