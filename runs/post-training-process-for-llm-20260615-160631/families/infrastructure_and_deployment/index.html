Serving LLMs Without Wasting KV Cache Memory

Large language model serving often looks compute-heavy from the outside, but many real deployments hit a more mundane limit first: memory waste in the KV cache. Every active request accumulates key and value states token by token during generation. If the server reserves those states as large contiguous chunks, it quickly runs into fragmentation, stranded capacity, and poor batching efficiency. The result is not just higher memory use. It is lower throughput, more preemption, and less predictable latency.

This is why infrastructure work around KV cache layout matters. The common theme is not a single algorithm. It is a shared interface problem between generation, scheduling, and memory management: how do you keep causal attention correct while storing a dynamic, growing state efficiently enough for production serving?

The Serving Pipeline Where Memory Pressure Builds

The recurring setup in this subtopic is a standard online serving loop:

During prefill, the model processes the prompt and initializes the KV state. During decode, that state grows one token at a time for each live sequence. The scheduler decides which requests or beams get GPU time next. Memory management has to keep the growing cache available across those scheduling decisions, even when requests have different lengths, share prefixes, or get interrupted.

This pipeline matters because the KV cache is not an optional side structure. It is the working state that makes autoregressive decoding efficient. Any deployment strategy must preserve token order, per-token KV states, causal attention semantics, and the identity of shared prefixes. What can change is the storage layout and how the runtime allocates, shares, evicts, or rebuilds that state.

Where Contiguous Allocation Fails

The hard part is that request growth is uneven and unpredictable. One user stops after a few tokens, another generates a long answer, and a third fans out into multiple candidates. If the system stores each request’s KV cache as one contiguous tensor, it tends to reserve too much space up front or repeatedly copy data into larger buffers. Both choices waste memory.

That waste shows up as fragmentation. Some free memory exists, but not in the right shape for the next growing sequence. In that regime, the server can become memory-bound before the GPU’s compute is fully utilized. The compact evidence in this input is blunt about the scale of the issue: existing systems were reported to use only 20.4%-38.2% of KV cache memory for actual token states. The rest is overhead from the storage strategy, not useful model state.

The Design Axes That Matter in Practice

The embedded input supports a small set of design axes that are more useful than a paper-by-paper tour.

Block Size

One choice is whether to allocate cache in small or medium fixed-size blocks instead of as one contiguous region. Smaller blocks reduce wasted slack when sequences grow unevenly, but too many tiny blocks add more indirection overhead when attention kernels fetch them.

Eviction Strategy

When memory pressure rises, a server needs a recovery path. The input highlights two: swap blocks out or recompute them later. Swapping preserves prior work but depends on moving data between memory tiers. Recomputation avoids keeping everything resident but spends extra compute when a request resumes.

Layout

The key structural shift is from contiguous storage to a noncontiguous physical layout. This relaxes the allocator’s job. The server no longer needs one large free region per sequence; it needs enough free blocks somewhere in memory.

Scheduler Simplicity

The family synthesis points to FCFS scheduling in the representative setup. That is a useful reminder that memory efficiency does not require an exotic scheduler. A simpler scheduler can still benefit if the cache representation makes preemption and resumption less destructive.

Sharing Policy

Copy-on-write matters when multiple sequences reuse a shared prefix, such as beam search or shared-prompt workloads. Instead of duplicating the same KV state immediately, the system can share blocks until one branch writes new tokens and diverges.

Treating KV Cache Like Virtual Memory

The clearest example in this family is PagedAttention + vLLM. Its contribution is not “a better attention algorithm” in the abstract. It teaches a systems idea: treat the KV cache more like virtual memory.

The core move is to split the KV cache into fixed-size blocks, map each sequence’s logical block order onto noncontiguous physical blocks, and consult block tables when attention needs to read prior states. That preserves the model’s semantics while discarding the assumption that each request must own one contiguous memory segment.

This framing helps explain several deployment behaviors at once:

  • Allocation becomes demand-driven rather than reserve-everything-up-front.
  • Shared prefixes can reuse the same underlying blocks.
  • Diverging continuations only copy data when a write actually happens.
  • Preempted requests can recover through swap or recomputation rather than forcing a full contiguous re-layout.

The intuition is simple: if dynamic state grows token by token, contiguous preallocation wastes memory. Fixed blocks let the server allocate only what is needed, when it is needed, while keeping the model’s attention logic intact.

The Main Tradeoff: Less Waste, More Indirection

Paged layouts are not free. The main tradeoff in the input is explicit: block-table indirection adds kernel overhead. That overhead is the price of escaping fragmentation.

This creates a practical tuning problem. If blocks are too large, the old waste pattern returns because each sequence still holds too much unused reserved space. If blocks are too small, the runtime spends more effort managing mappings and gathering noncontiguous blocks during attention. The design question is not whether paging helps at all, but where the operating point should be for a given workload and memory regime.

That is also why this topic belongs under infrastructure and deployment rather than model architecture. The important variables are traffic shape, prompt sharing, beam behavior, available GPU memory, and the relative cost of memory movement versus recomputation.

What the Sparse Evidence Supports

The input provides enough evidence to anchor the idea without turning this page into a benchmark summary.

First, the motivation is strong: the reported 20.4%-38.2% effective utilization figure says contiguous KV storage wastes most of the reserved cache in some serving setups. Second, the representative method reports that vLLM improves throughput by 2-4x over FasterTransformer and Orca at similar latency. Third, the same mechanism shows clear memory savings for beam search, with 37.6%-55.2% savings on Alpaca and 44.3%-66.3% on ShareGPT. A shared-prefix translation workload also reportedly reaches 1.67x to 3.58x higher throughput than Orca (Oracle).

Those results do not prove that one block layout wins everywhere. They do support the narrower systems lesson: when serving is bottlenecked by KV cache fragmentation, changing the cache abstraction can unlock both higher utilization and better batching capacity.

What to Remember

KV cache management is a deployment problem about preserving semantics while changing storage assumptions. The durable idea is to stop treating per-request cache as a single contiguous object. Once the cache is broken into fixed blocks, the server can allocate on demand, share common prefixes, and recover under pressure without rewriting the model itself.

That is why this subtopic matters. In production LLM serving, memory layout decisions shape how much real work the GPU can sustain. Better infrastructure here does not make the model smarter. It makes the deployment less wasteful, more batchable, and more resilient under mixed workloads.

Open questions remain even in this compact view: how should block size adapt online to a changing trace, when should a system prefer swapping over recomputation, and how much more kernel overhead can be removed from paged access patterns? Those are deployment questions, but they are now central to how modern LLM systems scale.
