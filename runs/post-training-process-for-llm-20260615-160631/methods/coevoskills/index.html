CoEvoSkills

CoEvoSkills starts from a problem that shows up any time you try to turn an LLM into a real agent: the useful thing is usually not a single tool call, but a reusable procedure. A tool can be a neat, self-contained function. A skill is bigger than that. It is a bundle of instructions, scripts, and references that has to stay internally consistent across files and across revisions.

That difference sounds small until you try to generate the whole package in one shot. Then the cracks show up quickly. The agent may write something that looks plausible, but the files do not quite agree with each other, the procedure misses a step, or the skill works only for the easy cases the model happened to imagine. CoEvoSkills is built around the idea that this is not a prompting problem alone. It is a verification problem.

The paper’s main move is to make skill writing look more like iterative software development: one component writes the skill, a separate component writes tests, and a hidden oracle decides whether the whole package really works.

What to notice in the figure is the jump in complexity. A tool is one call. A skill is a workflow. Once the artifact becomes a workflow, the hard part is no longer just “generate text”; it is “keep a multi-file procedure aligned with the task it is supposed to solve.”

The basic shape of the loop

If you strip the method down to its essentials, the flow is:

  1. write a structured skill bundle,
  2. run the skill to produce outputs,
  3. have a separate verifier inspect those outputs and create deterministic checks,
  4. use the failures to revise the skill,
  5. if the surrogate checks pass, ask the hidden oracle to re-run the skill in a fresh environment,
  6. repeat when the oracle reveals a remaining gap.

That loop matters because the hidden task criteria are not visible to the agent. The agent never gets the ground-truth tests, only a pass/fail signal at the end. So the method has to build its own local feedback system that is informative enough to guide repair, but still isolated enough not to simply mirror the generator’s own assumptions.

The design looks a lot like having a code author and a test author in different rooms. The code author keeps rewriting the skill package. The test author keeps synthesizing more specific checks from the current outputs. The hidden oracle is the real acceptance test that nobody can inspect.

Why one-shot skill generation breaks

The paper’s starting complaint is not that agents cannot write anything useful. It is that one-pass generation is fragile when the output is a structured package rather than a single prompt. A reusable skill has to preserve task intent, file interfaces, tool workflow, and output format all at once. If any of those drift, the package becomes hard to use.

Human-written skills are not a perfect fix either. They are expensive to author, and they often encode a human’s idea of the workflow rather than the workflow the agent actually needs at execution time. That mismatch shows up in practice as inconsistent gains: some skills help a lot, others help little, and some can even hurt.

CoEvoSkills treats that as evidence that skill quality should be discovered through feedback, not guessed in advance.

Two sessions that keep each other honest

The two core pieces are the Skill Generator and the Surrogate Verifier.

The Skill Generator is the part that rewrites the skill bundle. It does not start from scratch on every round. Instead, it keeps a persistent context of the task instruction and the accumulated failure notes from earlier rounds. That persistence is important: if the same bug keeps coming back, the generator needs to remember what was already tried and why it failed.

The Surrogate Verifier is deliberately isolated from the generator. It runs in a separate LLM session and only sees the task instruction and the current outputs. It does not see the generator’s internal reasoning or its skill code. That isolation is the main safeguard against confirmation bias. If the verifier can read the generator’s plan, it is too easy for the two to agree for the wrong reasons.

The verifier’s job is not to give a vague review. It synthesizes deterministic assertions from the outputs it sees. Those assertions become a test suite that can be executed mechanically. If the suite fails, the verifier also produces structured failure diagnostics: which checks failed, what the likely root cause is, and what should be changed next.

That is the first important trick in the paper. The feedback is not free-form praise or criticism. It is a set of concrete checks and repair hints that the generator can actually act on.

The proxy score is just test coverage made explicit

The paper turns the verifier into a surrogate reward by measuring how many of its assertions pass. In compact form:

R~(x,V)=1∣V∣∑k=1∣V∣1[ek(x)]\tilde{\mathcal{R}}(x,\mathcal{V}) = \frac{1}{|\mathcal{V}|}\sum_{k=1}^{|\mathcal{V}|}\mathbf{1}[e_k(x)]R~(x,V)=∣V∣1​k=1∑∣V∣​1[ek​(x)]

Here, x is the output produced by running the skill, and \mathcal{V} is the verifier’s assertion set. If 17 of 20 checks pass, the surrogate reward is 0.85. The details are simple on purpose: the score needs to be deterministic, easy to compare across revisions, and cheap enough to compute repeatedly.

This is a very different kind of signal from a human-style review. A free-form critique might say “this looks incomplete.” A deterministic assertion says “this file is missing the required field” or “this output does not meet the format constraint.” That specificity is what makes iterative repair possible.

The catch is that the proxy is only a proxy. It can be wrong in two ways. It can miss a real hidden requirement, and it can also be too strict about something the hidden oracle does not actually care about. The paper leans into that limitation instead of pretending it does not exist.

What happens when the proxy and the oracle disagree

The hidden oracle is the truth gate. It re-executes the skill in a fresh environment and returns only pass or fail. No test content, no failing assertion, no explanation.

That binary response matters because it prevents the generator from overfitting to the hidden tests. But it also means the oracle cannot directly teach the agent how to fix the remaining issue. Instead, a mismatch between the surrogate and the oracle triggers another round of co-evolution. The verifier must escalate its own test suite, and the generator must revise the skill using the new diagnostics.

In practice, escalation means the verifier gets stricter or broader. It may add more cases, probe more edge conditions, or check a subtle requirement the previous suite missed. The point is not to repeat the same tests with a different tone. The point is to discover what the hidden oracle was still checking.

The exoplanet case study in the paper shows why this matters. A surrogate can be satisfied by a period estimate that is “close enough” under its own tolerance, while the hidden oracle requires exact five-decimal precision. The surrogate can also end up judging its own estimate against the generator’s estimate and flagging a tiny difference that is actually acceptable. That is exactly the kind of failure that justifies keeping the oracle separate.

Why the co-evolutionary loop works

The method works because it gives the generator dense local feedback without pretending that the local feedback is the final answer.

If the surrogate fails, the generator learns from a concrete diagnosis and repairs the skill. If the surrogate passes but the oracle fails, the verifier learns that its current tests were too weak and has to tighten them. So both sides improve for different reasons:

  • the generator gets better at producing a coherent, reusable skill package,
  • the verifier gets better at approximating hidden success conditions,
  • the oracle keeps both sides honest.

That alternating pressure is the paper’s real contribution. It is not just self-generation, and it is not just verification. It is the co-evolution of the thing being written and the thing that checks it.

The persistent context on the generator side is also important. Without it, the system would keep rediscovering the same mistakes. With it, the skill bundle becomes a living document that absorbs what earlier failures taught it.

What the experiments say, in plain language

The paper’s headline number is strong: CoEvoSkills reaches 71.1% pass rate on SkillsBench, which is 40.5 points above the no-skill baseline and 17.6 points above human-curated skills. That is the best evidence that the iterative loop is doing real work rather than just producing nicer prose.

The ablations make the mechanism even clearer. If you remove the surrogate verifier, performance drops sharply to 41.1%. That is the most direct proof that the dense diagnostic feedback matters. If you keep only background context without the full evolution loop, the score improves somewhat, but it stays far below the full system.

The transfer results are just as interesting. Skills evolved on one model still give large gains on six other models from different providers, with improvements in the roughly 36 to 44 point range. That suggests the final skill package is not just a model-specific artifact. It is capturing reusable task structure.

There is still a small caveat: the originating model can do a bit better with its own evolved skills than with transferred ones. So the method generalizes well, but it is not perfectly model-agnostic.

Where the method is still brittle

CoEvoSkills is not free. It needs repeated verification cycles, and those cycles consume oracle budget. The paper shows that the system converges in a few rounds for most tasks, but convergence still depends on how many times you can afford to re-run the loop.

There is also a structural blind spot in surrogate verification. A deterministic test suite can miss hidden precision constraints, aliasing rules, or other subtle criteria that are hard to infer from outputs alone. When that happens, the surrogate can say “good enough” even though the oracle still disagrees.

So the method is best understood as a disciplined way to reduce sparse feedback, not as a replacement for truth. It gives the generator something concrete to improve against, while reserving the final judgment for the hidden test.

What to remember

The simplest way to remember CoEvoSkills is this: it treats skill writing like iterative coding with a separate test writer.

The generator writes a structured skill package. The isolated verifier turns current outputs into deterministic checks and failure notes. The hidden oracle only says pass or fail, forcing the verifier to keep escalating when its approximation is too weak. Over a few rounds, that loop produces skills that are more reusable, more faithful to the task, and better aligned with how agents actually solve problems.

That is the paper’s core lesson: for agent skills, the important advance is not a clever one-shot prompt. It is a feedback loop that keeps improving both the skill and the test that evaluates it.
