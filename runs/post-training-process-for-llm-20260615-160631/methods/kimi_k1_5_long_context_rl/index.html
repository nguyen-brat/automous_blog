Kimi k1.5: Scaling Reinforcement Learning with LLMs

The easiest way to understand Kimi k1.5 is to think of it as a very direct answer to a familiar problem: pretraining alone eventually runs into data limits, but reasoning tasks still keep demanding more search, more revision, and more deliberate thinking. This paper asks a practical question instead of a glamorous one. If a task can be verified, can we teach a model to spend more tokens thinking during training, then compress that reasoning into something cheaper at inference time?

That is the whole shape of the method. The model is first warmed up on long chain-of-thought examples, then optimized with reinforcement learning over long contexts, partial rollouts, and verifiable rewards. After that, the long reasoning policy is used as a teacher for shorter policies through transfer methods such as merging, rejection sampling, DPO, or a second long2short RL phase.

The important idea is not just “use RL.” It is that context length becomes a compute budget for exploration. If the model can stay in a reasoning loop longer, it can try more paths, correct itself, and recover from mistakes. The paper’s claim is that this can work without explicit tree search, value networks, or process reward models, as long as the prompts are carefully chosen and the rewards are trustworthy.

Start with problems the model can actually check

Before any optimization begins, the paper spends effort curating prompts. That matters more than it sounds. If the model is trained on vague or easy-to-game questions, RL can drift toward reward hacking or superficial patterns. So the prompt set is built around three properties: it should be diverse, it should cover a range of difficulty, and it should be objectively evaluable.

That gives the rest of the pipeline something solid to stand on. Mathematics, coding, and other verifiable reasoning tasks are especially useful here because the reward can come from exact checks: test cases for code, rule-based verification when possible, or a reward model when the answer has to be judged against free-form ground truth.

The paper also uses the model itself to estimate difficulty. It samples a prompt multiple times from a supervised warm-start model and uses the pass rate as a proxy for how hard the prompt is. That is a neat move because difficulty is no longer a static label from an external taxonomy. It becomes a property of the current model. Easy questions can be filtered out early, and harder ones can be emphasized once the policy is ready for them.

There is another subtle filter here: remove questions that are too easy to hack. If a model can guess the answer without any real reasoning, the reward can look correct even when the thinking is shallow. The paper avoids that by rejecting prompt types that are especially prone to false positives and by testing whether a model can guess the answer in a small number of attempts. The goal is to keep the reward aligned with actual reasoning, not lucky surface matching.

Warm up the reasoning format before RL

The next stage is a small but important bridge: long-CoT supervised fine-tuning. This is not the main optimization stage, but it gives the policy the right habits before RL starts pushing it around.

The paper describes this warmup as a kind of carefully curated long reasoning dataset. The examples are chosen to show planning, evaluation, reflection, and exploration. That is useful because the model is not just learning to produce an answer. It is learning a style of internal work: outline a path, inspect it, revise it, and keep going when the first idea fails.

You can think of this stage as teaching the model the grammar of long reasoning. RL then has something to refine instead of starting from scratch. Without this warmup, the model would be more likely to wander, overfit to reward quirks, or learn a long format that looks busy but does not actually help.

The RL loop is about outcomes, not a value head

Once the prompt set and long-CoT warmup are in place, the paper switches to reinforcement learning. The setup is intentionally simpler than many earlier reasoning systems. The policy samples a chain of thought and then produces a final answer. The reward only checks whether the final answer is correct, or whether the answer passes the verifier.

That sounds ordinary, but the important design choice is what the paper does not add. There is no value network, no process reward model, and no explicit search tree. Instead, the method uses an off-policy variant of online mirror descent. In plain terms, the current policy is treated as the reference, sampled rollouts are reused, and the update tries to push the new policy toward higher-reward trajectories without making it jump too far in one step.

This is where the paper’s philosophy becomes clear. Rather than building a complicated planner outside the model, it tries to train the model to behave like a planner inside a long context window. The reasoning trace itself becomes the search space. More tokens mean more room for trial, error, and recovery.

What to notice in this figure is that training and inference are not treated as separate worlds. The system is built so that the same infrastructure can move weights between training and rollout phases, which matters because long-context RL is expensive if every phase has to be handled by a different stack. The paper’s design is really about keeping that loop moving efficiently enough that long reasoning is practical.

Why partial rollouts matter so much

Long reasoning is useful, but it can also stall the whole system if one trajectory monopolizes compute. That is why partial rollouts are such a central trick in this paper.

Instead of forcing every long trajectory to finish before the next update, the system can stop at a token budget boundary, save the unfinished continuation, and resume it in the next iteration. Previous segments are reused from the replay buffer, so the system does not pay the full cost of regenerating the same prefix again and again.

This is more than a performance optimization. It is what makes 128k-context RL workable in practice. Long traces are no longer all-or-nothing jobs. They become segmented work items that can be continued later. That lets the system keep many workers busy even when some trajectories are unusually long.

The same mechanism also helps with repeated text. If the model starts looping or repeating itself, the rollout can detect that pattern early and terminate it. That matters because repetition is one of the easiest ways for long reasoning to waste tokens without buying any new information.

Length is a signal, but also a cost

Once a model is allowed to explore longer traces, it tends to overthink. That is almost inevitable. If longer reasoning is rewarded indirectly through better correctness, the model will often discover that extra tokens help. But long answers are expensive, and humans do not always want more thinking when a shorter path would do.

The paper responds with a length penalty. The idea is simple: among correct answers, shorter outputs are preferred; among incorrect answers, long outputs are penalized more aggressively. The exact formula is less important than the intuition. The training signal should not only ask “Was the final answer right?” It should also gently ask “Did you get there without unnecessary detours?”

This is a careful balance. If the penalty is too strong too early, training slows down and the policy loses room to explore. So the paper warms the penalty in gradually: first train without it, then turn it on once the policy has learned enough long-form reasoning to make the pressure useful.

That is a good example of the paper’s overall approach. It does not try to stop long reasoning. It tries to make long reasoning disciplined.

Difficulty-aware sampling keeps RL focused

RL efficiency also depends on where you spend updates. The paper uses two simple sampling ideas to keep training focused.

Curriculum sampling starts with easier tasks and then moves toward harder ones. That is important because early in training the model may not produce many useful rollouts on very hard prompts, so compute is wasted on problems it cannot yet solve. Starting easier gives the policy a floor to stand on.

Prioritized sampling goes one step further. Problems where the model underperforms are sampled more often, so the training loop spends more time on weak spots instead of repeatedly polishing already-solved cases. Together, these strategies make the same RL budget go further.

Why the paper does not stop at a long-CoT policy

The long-CoT model is strong, but it is not the end product. It is a teacher. The paper’s second half is about compressing the reasoning priors into shorter models that cost less at inference time.

What to notice here is the basic tradeoff: the long-CoT policy is the source of better reasoning, but short-CoT models are the deployment target. The interesting part is that the paper does not assume compression has to be a crude loss of quality. Instead, it tries several ways of preserving the good parts of the long policy while trimming the token budget.

Model merging is the simplest version. Shortest rejection sampling keeps the shortest correct response from multiple samples and uses that as supervision. DPO turns those samples into preference pairs, treating longer responses as negatives when they are unnecessarily verbose. Long2short RL goes one step further and retrains a short model with a stricter length budget and the same length-aware pressure used in the main RL phase.

The paper reports that long2short RL is more token-efficient than DPO, shortest rejection sampling, and model merging. That matters because it suggests the compression step is not just an afterthought. It is part of the learning story: the short model is not merely copied from the long one, it is trained to inherit the useful reasoning prior under a tighter budget.

Why this works without explicit tree search

The most interesting conceptual claim in the paper is that long context can play the role of search. In classic planning, a model explores a tree of partial solutions and scores the branches with a value model or critic. Here, the trace itself is the search space. The model gets more room to try, fail, reflect, and continue inside one long context window.

That is a simpler system to run, but only if the reward is reliable enough to support exploration. If the verifier is weak, the model can loop, overfit, or exploit reward holes. So the paper’s training recipe is really a three-part bargain:

  • make the task verifiable,
  • let the policy explore in long context,
  • compress the result once the reasoning prior has been learned.

This is why the method feels practical rather than theoretical. It is not trying to solve every reasoning problem in one sweep. It is trying to build a usable post-training axis for problems where correctness can be checked and long-form exploration is actually worth paying for.

What to remember

If you want one sentence to keep in mind, make it this:

Kimi k1.5 treats long context as a reasoning budget, uses verifiable rewards to train exploration, and then compresses that reasoning into shorter policies when token cost matters.

The paper’s strongest contribution is not a single trick. It is the combination. Curate prompts carefully. Warm up the long reasoning format. Use off-policy RL with partial rollouts so long traces stay tractable. Add length pressure so exploration does not become rambling. Then transfer the resulting reasoning priors into a cheaper model.

That is the main lesson here. If the task is checkable, RL can teach a model to think longer and better. The remaining problem is not whether the model can reason. It is how to keep the reasoning useful once you have to ship it.
