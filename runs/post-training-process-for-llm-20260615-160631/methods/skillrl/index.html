SkillRL: Automatic Skill Discovery and Reinforcement Learning for General-Purpose LLM Agents

SkillRL starts from a simple but important complaint: raw agent trajectories are too noisy to be a good memory format. A rollout may eventually succeed, but it usually gets there through wandering, retries, backtracking, and accidental discoveries. A failed rollout can be even more valuable, because it shows exactly where the current strategy breaks, but it is also harder to reuse verbatim. If you store those episodes as-is, you get a transcript archive, not a reusable skill set.

The paper’s answer is to compress experience into a hierarchical SkillBank and then train the policy to actually use that bank. Instead of replaying long traces, the agent reads compact skills, starts with a cold-start demonstration stage so it learns how to use them, and then keeps expanding the library whenever validation failures expose a missing trick. That is the central move: experience becomes a small instruction manual, and the manual keeps evolving.

The whole loop at a glance

The easiest way to understand SkillRL is to follow the data flow from left to right:

  1. collect both successful and failed trajectories from the environment,
  2. distill them into reusable skills,
  3. organize those skills into a two-level library,
  4. teach the base model how to read and apply the library,
  5. fine-tune with reinforcement learning,
  6. add or revise skills when validation failures reveal gaps.

What matters in the figure is the loop, not just the boxes. SkillRL is not a one-time compression pass over old experience. It is a cycle that turns interaction traces into skills, skills into policy improvements, and new failures back into new skills. That recursive structure is what makes the method feel more like skill learning than memory storage.

Why raw trajectories are the wrong thing to remember

The paper makes a strong case that trajectory memory is the wrong granularity for agent improvement. A raw trace contains useful information, but it also contains a lot of clutter:

  • exploratory detours,
  • repeated actions,
  • dead ends,
  • irrelevant details that only matter in that one episode.

If the model sees the full trace at inference time, it can spend its context budget rereading noise instead of extracting a reusable rule. SkillRL treats that as the core bottleneck. The goal is not to remember more text. The goal is to preserve the decision pattern that actually mattered.

That is why the system keeps both success and failure. Successes show what worked. Failures show the boundary conditions and the mistaken assumptions that successes hide. A good post-training system needs both, because a skill is not only “how to win” but also “how not to repeat the same mistake.”

Turning episodes into skills

Skill distillation is the first real move in the pipeline. A teacher model looks at a trajectory together with the task description and rewrites it into a compact skill. For a successful rollout, it extracts the strategy that led to completion. For a failed rollout, it synthesizes a short failure lesson instead of dumping the whole episode into context.

That difference is crucial. A failure lesson is not just a shorter summary. It is a structured counterexample: where the agent went wrong, what reasoning or action was flawed, what should have happened instead, and what rule would prevent the same failure next time. In other words, the paper is preserving the lesson, not the transcript.

This is also why the method depends on a strong teacher. The teacher has to identify the useful pattern, compress it cleanly, and write it in a form that a later policy can actually consume. If the distillation is weak, the whole library becomes a pile of vague advice.

Why the SkillBank is hierarchical

Once the episodes are distilled, SkillRL does not throw them into one undifferentiated bucket. It splits the library into two layers.

General skills are the universal habits that help across the whole environment: systematic exploration, pre-action checks, state tracking, and goal confirmation. These are the kinds of strategies that keep an agent from looping or acting on bad assumptions.

Task-specific skills are narrower. They capture the procedures, constraints, and failure modes that matter for a particular task category. In ALFWorld, that can mean rules about object manipulation or state changes. In WebShop, it can mean search query formulation, variant checking, or purchase timing. The point is not to memorize every task instance. The point is to preserve the part of the experience that actually transfers within a family of tasks.

This two-level split is doing a lot of work. The general layer gives the policy a stable foundation, while the task-specific layer injects local expertise only when it is relevant. That keeps the context smaller and the guidance more useful.

Each skill is stored in a compact, human-readable format: a short title, a principle, and a when_to_apply condition. That format matters because retrieval is only half the problem. The agent also needs to know how to use the retrieved item once it is in context. A skill card is better than a raw trace because it already encodes the rule and the trigger.

How retrieval works at inference time

At inference, the agent receives the task description and retrieves support from the SkillBank. General skills are always included. Task-specific skills are fetched by semantic similarity, then filtered with a threshold and narrowed to the top matches.

That design choice is practical. Always-on general skills act like the agent’s basic operating manual. Retrieved task skills act like situation-specific reminders. The model does not need to scan the entire library every time, and it does not need to re-read long trajectories to get the same effect.

The important part is that the policy conditions on the retrieved skills before it chooses the next action. The input is no longer just the task and the current observation. It is the task, the observation, and a small set of reusable instructions that were distilled from past experience. That is the real input-output contract of the method.

SkillRL also reports a large compression benefit here. Because the library is distilled rather than replayed, the prompt is much shorter than a raw-memory baseline, yet it carries more actionable information. The paper reports roughly 10 to 20 times token compression relative to full trajectories, and a leaner prompt footprint than raw memory retrieval.

Why the model needs a cold start before RL

One subtle problem appears immediately: even if the library is good, the base model does not automatically know how to use it.

If you simply hand an unchanged model a skill library, it may ignore the skills, misuse them, or treat them like random extra text. SkillRL solves that with a cold-start supervised fine-tuning stage. A teacher generates skill-augmented demonstrations that show how to retrieve, interpret, and apply the skills during decision-making. The student then learns that pattern before reinforcement learning begins.

This stage is easy to underestimate, but it is one of the most important parts of the method. It teaches the model the “read the manual first” habit. Without that habit, the later RL stage would be trying to optimize a policy that still does not know how to consult its own memory.

After cold-start SFT, the policy is fine-tuned with GRPO. The RL objective rewards successful trajectories while keeping the policy close to the skill-aware reference model through a KL penalty. Conceptually, that means RL is allowed to improve behavior, but not to drift away from the skill-using behavior the SFT stage already established.

Why the skill library keeps changing

SkillRL would be much weaker if the SkillBank stayed static. A fixed library can only cover the scenarios the agent already knows about. As the policy improves, it explores new states and hits new failure modes. Those failures are exactly where the library needs to grow.

The recursive skill evolution loop handles that. After each validation epoch, the method checks success by task category. If a category falls below a threshold, it collects failed validation trajectories, samples them in a diversity-aware way, and asks the teacher model to do three things:

  • identify failure patterns the current library does not cover,
  • propose new skills for those gaps,
  • refine existing skills that turned out to be too weak.

The library is then updated with the new or revised skills, and the agent continues training with a better playbook. This is the core recursive idea in the paper. The policy learns from skills, but the skills themselves also learn from the policy’s failures. That co-evolution is what keeps the system from freezing into a static memory bank.

You can think of it as a very compact form of curriculum design. The current policy discovers what it still cannot do, and those failures become the next version of the instruction manual.

Why this works better than replaying memory

The method’s strength is not just that it stores less text. It stores the right abstraction level.

Raw memory can tell the model what happened. A skill tells the model what to do in similar situations. That sounds like a small distinction, but it is the difference between rereading a story and reading a playbook.

The experiments reflect that difference. On ALFWorld, SkillRL reaches 89.9% success, and on WebShop it reaches 72.7%. Removing the skill library drops performance sharply, which is a strong sign that the abstraction itself matters. The paper also reports a 47.1% average on search-augmented QA tasks, showing that the same idea can transfer beyond embodied and shopping-style environments.

The ablations make the story even clearer:

  • removing the hierarchy hurts, because general skills provide a useful base layer,
  • replacing the skill library with raw trajectories hurts the most, because noise overwhelms reuse,
  • removing cold-start SFT hurts, because the model never learns how to consume the skills,
  • disabling dynamic evolution hurts, because the library stops adapting to new failures.

Taken together, those results say the same thing in four different ways: abstraction, not replay, is what makes the experience useful.

What the method assumes

SkillRL is strong, but it is not free.

It depends on a capable teacher model to distill good skills and generate the cold-start demonstrations. It also depends on retrieval quality, because a missed skill is as bad as no skill. Semantic similarity and thresholding are useful, but they are still imperfect filters. And the method adds extra stages before RL can begin, so the training loop is more involved than plain fine-tuning.

Those tradeoffs are the price of making memory actionable. SkillRL is not trying to be the lightest possible recipe. It is trying to turn experience into a compact, reusable policy aid that can keep improving as the agent encounters new failures.

What to remember

SkillRL is a pipeline for converting noisy agent experience into a living skill library.

The short version is:

  • preserve both successes and failures,
  • distill them into general and task-specific skills,
  • teach the model to use those skills before RL,
  • and expand the library whenever validation exposes missing coverage.

If you remember one image, remember this: raw trajectories are the rough material, but skills are the finished tool. SkillRL is interesting because it does not just store the tool library. It keeps repairing and extending it while the agent learns to use it.
