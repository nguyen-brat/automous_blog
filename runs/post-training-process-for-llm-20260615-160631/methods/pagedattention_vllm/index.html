vLLM: Easy, Fast, and Cheap LLM Serving with PagedAttention

The easiest way to understand vLLM is to stop thinking about LLM serving as a pure compute problem. The paper argues that serving is often a memory-management problem in disguise.

Every request brings along a growing KV cache. That cache is needed for autoregressive decoding, but it grows token by token, varies from request to request, and is hard to pack neatly into GPU memory. If a serving system stores each request in one contiguous chunk, it ends up reserving too much space, wasting space inside that chunk, and fragmenting the rest of memory around it. That makes it harder to batch requests, which is exactly what you want to do to raise throughput.

vLLM’s answer is to treat the KV cache the way an operating system treats process memory: page it, map it, share it, and copy it only when two users diverge. PagedAttention is the attention kernel that makes this work. The KV cache manager and scheduler are the system pieces around it that keep the GPU workers fed.

What to notice in the system diagram is that vLLM is not just a new attention kernel dropped into an existing stack. The scheduler decides which requests run in each iteration. The block manager decides where KV blocks live physically. The GPU workers then execute attention using the block table instead of assuming the cache is one neat contiguous tensor.

Why contiguous KV cache becomes a trap

During prompt processing, a Transformer can parallelize well. During decoding, it cannot. New tokens are generated one at a time, and each step needs access to all previous key and value vectors. That means the KV cache is the long-lived state that determines whether a batch fits in memory.

The problem is that the cache does not behave like a fixed tensor. Its final length is unknown in advance, and different requests can have wildly different prompt and output lengths. If the serving system pre-allocates space for the maximum sequence length, most of that reservation often sits unused for a long time. If it tries to fit many requests with variable-sized contiguous chunks, fragmentation gets worse. Either way, the system runs out of room sooner than it should.

That is the basic bottleneck behind the paper’s headline numbers. vLLM raises throughput not by changing the model’s answers, but by fitting more useful KV state into the same GPU memory budget.

The core move: break the KV cache into blocks

PagedAttention starts with a small but powerful rethinking of layout. Instead of storing the KV cache for a sequence as one contiguous tensor, it splits the cache into fixed-size KV blocks.

Each block holds the keys and values for a handful of tokens. The blocks themselves do not need to sit next to each other in physical memory. A sequence is represented by a table that says which logical block maps to which physical block.

The important thing in this figure is that attention is still logically sequential, but the storage is no longer forced to be contiguous. The kernel walks through the block table, fetches one KV block at a time, and computes attention block by block. So the model still sees the correct token order and causal semantics, but the allocator gets much more freedom.

That freedom buys two things at once:

  1. It removes external fragmentation, because all blocks have the same size.
  2. It shrinks internal waste, because the last block of a sequence is the only one that can be partially empty.

This is why the paper keeps coming back to the same intuition: when state grows dynamically, fixed-size blocks are a better unit than giant pre-reserved chunks.

Logical blocks and physical blocks are not the same thing

The block table is the bridge between the abstraction and the hardware.

The logical blocks belong to a sequence. The physical blocks are the actual chunks of GPU or CPU memory. A sequence can have logical blocks that point to physical blocks anywhere in memory. Two sequences can even point to the same physical block if they share prefix state.

That separation matters because it lets vLLM grow memory on demand instead of reserving the full future length up front. It also lets the scheduler and block manager reshuffle memory without changing the model’s output semantics. The attention kernel just follows the table.

This is the same design pattern virtual memory uses for programs: the program thinks in logical addresses, while the OS maps them to physical pages behind the scenes. vLLM borrows that idea almost directly.

What happens during decoding

The decoding flow is easier to follow if you think about one request at a time first.

At prefill time, vLLM computes the prompt and stores the resulting KV cache into the first few logical blocks. It does not reserve the entire future output length. It only allocates what is needed so far, plus room for the current block to keep filling.

At each decoding step, the model reads the previous KV blocks through the block table, generates the next token, and writes the new KV state back into the current last block. When that block fills up, vLLM asks the block manager for a fresh physical block and extends the table.

That is the main loop:

  • read past KV blocks through the block table,
  • compute the next token,
  • append new KV state to the current logical block,
  • allocate a new physical block only when the current one is full.

Because the blocks are small and fixed-size, the system keeps memory pressure local. A sequence does not waste an entire giant reservation just because it might continue later. The only unavoidable waste is the slack space in the final block.

The block size is the tuning knob. If blocks are too large, the old fragmentation problem starts creeping back. If blocks are too small, indirection overhead and kernel work go up. The paper’s ablation shows that the best choice depends on workload, but a moderate default block size works well in practice.

Why sharing is such a big deal

PagedAttention becomes much more interesting once you have multiple sequences that can reuse part of the same prefix.

Parallel sampling is the cleanest example. One prompt fans out into multiple candidate continuations. All of those candidates share the same prompt KV cache. In a contiguous layout, that usually means duplicating prompt state across sequences. In vLLM, the prompt blocks can simply point to the same physical blocks, and the system uses reference counts to track how many logical blocks are sharing each physical block.

The same mechanism is even more valuable for beam search. Beam candidates often share not just the prompt, but several decoding steps before they diverge. vLLM can keep those shared blocks in place and only split them when a candidate actually writes new content into a shared block. That split uses copy-on-write at block granularity.

Copy-on-write is the key idea here. A shared block is cheap until one sequence needs to modify it. Then vLLM allocates a new block, copies the shared contents once, and lets the sequences diverge from there. That means the system copies only the part that truly needs to be different, not the whole history.

This is one of the paper’s most practical ideas. The memory savings are largest when decoding methods have shared prefixes or shared intermediate states, which is exactly what many real LLM workloads do.

A shared prefix is just cached memory with a smarter address map

Shared-prompt workloads are conceptually similar. If many user prompts start with the same instruction or example prefix, vLLM can keep that prefix’s KV blocks around and map new requests onto them immediately.

That is a good mental model for remembering the system: a shared prefix is not recomputed each time, and it is not duplicated each time. It is a reusable block sequence with a shared physical backing, just like a shared library in operating systems.

The last block still needs copy-on-write treatment, because the new request may extend from the shared prefix in a different direction. But the bulk of the prefix stays shared and reused.

How vLLM stays fair when memory runs out

The paper does not pretend memory pressure disappears. Requests can still arrive faster than the GPU can serve them, and some requests must be paused.

vLLM uses first-come-first-serve scheduling to keep the policy simple and fair. When it needs to preempt work, it treats each request or sequence group as an all-or-nothing unit. That is a good fit for KV cache management because the blocks for a sequence are meant to be accessed together.

For recovery, vLLM has two options:

  • swap the evicted blocks to CPU memory,
  • or recompute the KV cache later when the request is resumed.

Swapping is the more literal virtual-memory approach. It preserves the state, but it moves data across the PCIe boundary, which can be costly when blocks are small or when eviction happens often. Recomputation avoids that transfer, but it spends extra GPU compute to rebuild the cache later. The paper’s ablation shows that neither option is always best. The right choice depends on block size and the memory-compute balance of the system.

The important system point is that vLLM can make this decision at the block level without changing the model itself. The serving stack sees a request as a sequence of logical blocks, and the recovery policy only has to preserve those blocks.

What happens on multiple GPUs

The same paging idea also works when the model is split across GPUs. In that case, the scheduler still owns the mapping from logical blocks to physical blocks, and the workers still read through the same block tables.

Each GPU shard only stores the slice of KV cache it needs for its attention heads, but the control-plane view stays unified. That is useful because all model shards process the same tokens in the same order. The memory management policy does not need to be reinvented per worker.

So even in distributed execution, the core idea stays the same: one logical sequence, a shared mapping layer, and physical blocks that may live anywhere.

Why this works in practice

The method works because it attacks the real waste in serving systems.

Contiguous preallocation wastes space before a request has generated anything. Shared prefixes waste space when several sequences store the same history separately. Fragmentation wastes space in between requests, where the allocator cannot pack the leftovers efficiently. PagedAttention removes all three sources by changing the unit of allocation.

That does add overhead. The attention kernel now has to consult a block table, handle indirect access, and manage variable-length blocks. The paper’s microbenchmark shows that this makes the attention kernel slower than a highly optimized contiguous implementation. But the extra overhead is localized to attention, while the memory savings raise batch size across the whole serving system. In end-to-end terms, the throughput gains win.

That tradeoff is the heart of the paper: spend a little kernel complexity to recover a lot of usable memory.

The limits to keep in mind

PagedAttention is not a free lunch.

If blocks are too large, the system slides back toward fragmentation and loses sharing opportunities. If blocks are too small, the indirection overhead and block management work start to dominate. Swapping and recomputation also behave differently depending on how much GPU memory pressure you are under and how expensive PCIe transfers are relative to compute.

So the method is best thought of as a serving-system technique for dynamic, memory-bound workloads. It is not a universal accelerator trick. The paper says that explicitly in the discussion: the value comes from the combination of unpredictable sequence lengths, long-lived token state, and a throughput bottleneck caused by memory capacity.

What to remember

If you want one mental model for vLLM, make it this:

LLM KV cache behaves more like operating-system memory than like a fixed tensor.

Once you see that, the rest falls into place. Fixed-size blocks reduce fragmentation. Logical-to-physical mapping makes allocation flexible. Copy-on-write makes sharing cheap until sequences diverge. Swapping or recomputation gives the scheduler a recovery path when memory is tight.

That is why PagedAttention matters. It does not change what the model computes. It changes how the system makes room for that computation.
