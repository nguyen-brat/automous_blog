WebWatcher: A Multi-Modal Agent for Deep Research

WebWatcher is trying to do something that ordinary text-first research agents still struggle with: answer hard questions when the useful clue might live in an image, a webpage screenshot, a caption, a linked page, or a small piece of code. The paper’s answer is not to bolt vision onto retrieval as an afterthought. It is to train a multimodal investigator that can move between search, browsing, OCR, and computation in one continuous loop.

That is the real shift here. WebWatcher is not a better retriever with a camera attached. It is a research agent that has to decide which evidence source matters next.

What to notice in Figure 2 is the failure mode. A vision-only agent can inspect the image but miss the surrounding web context. A search-only agent can gather text but miss the visual clue. WebWatcher works because it can switch modes instead of forcing one tool to do every job.

In the simplest form, the method looks like this:

Why vision and search have to live in the same loop

The paper starts from a very practical observation: many real questions are not answerable from text alone. A chart may need OCR, a webpage may need clicking through several layers, and a question may require the model to connect a visual entity to a broader web trail before it can answer safely.

That is why WebWatcher is built around tool orchestration rather than a single model call. The agent can use:

  • web image search to recover visual evidence around an entity,
  • web text search to find contextual facts and related pages,
  • webpage visits to follow links and read deeper content,
  • OCR to pull text out of the image itself,
  • code to handle calculation or structured reasoning when the evidence demands it.

Each tool covers a different kind of missing information. The important part is not that the tools are powerful in isolation. It is that the agent can choose among them as the question unfolds.

How BrowseComp-VL turns that idea into data

The paper does not rely on ordinary VQA pairs, because ordinary VQA is usually too shallow. Instead, it introduces BrowseComp-VL, a benchmark that is deliberately hard for multimodal deep research. The questions are long, multi-hop, and often entity-obfuscated, so the model cannot just match a visible label to a memorized answer.

Figure 4 shows the basic data-generation idea.

What to notice in Figure 4 is the direction of the flow. The paper begins with hard textual QA, grounds it in real images, and only then turns it into multimodal VQA. That order matters. If you start from an image and ask for a question directly, you usually get something shallow. If you start from a hard reasoning chain and then force it to meet an image, you get a much richer task.

The benchmark has two difficulty levels:

  • Level 1 keeps the target entity explicit but still requires multi-hop reasoning.
  • Level 2 hides or fuzzes the entity and its attributes, so the model has to infer the answer from context instead of direct lookup.

That second level is the more interesting one. It is not testing whether the model can recognize a named object. It is testing whether the model can recover the identity of the object from partial clues and then use that identity to search, inspect, and synthesize evidence.

From text QA to image-grounded VQA

The construction pipeline is the main trick behind the benchmark.

First, the paper collects difficult QA pairs from real web structure. For the easier level, it starts from authoritative and knowledge-rich sources and follows hyperlinks to build deeper reasoning chains. For the harder level, it expands a small entity graph from a root page, samples subpaths through that graph, and then writes questions whose answer depends on connecting several hops.

The point of that graph-building step is simple: the question should depend on a relation chain, not a single isolated fact. If the answer can be found by a direct lookup, the agent does not need to reason.

Then the paper converts each textual QA pair into a VQA example.

  1. It keeps only target entities that can actually be grounded visually.
  2. It retrieves real web images for the retained entity.
  3. It rewrites the question so the entity name is masked by a visual reference like “this entity” or “the object in the image”.
  4. It pairs each image with the rewritten question and the original answer.

That last step is what scales the dataset. One textual QA pair can produce several multimodal examples, one for each retrieved image. The result is a training set that keeps the original reasoning structure but forces the answer to be tied back to visual evidence.

The paper is careful about image choice here. It uses authentic web images, not synthetic composites. That matters because the agent has to deal with the messy reality of online visuals instead of an artificially clean benchmark image.

Filtering is not cleanup, it is part of the method

The raw conversion step is not trusted on its own. The paper adds a quality-control stage because multimodal data can fail in very different ways.

The selector checks for obvious rewrite failures first. If the transformed question is basically unchanged, or if the entity name still appears directly, the masking step did not really do its job. The selector also checks whether the image and question actually fit together, so the model is not trained on visually irrelevant pairs.

The examiner then asks a stronger question: can the image set and its captions support the synthesized image query on their own? If the answer is no, the image-query pair is too weak to teach the agent reliable reasoning, so it gets removed.

This filtering matters because bad multimodal data are worse than no data. If the agent sees weak alignments during training, it learns to take shortcuts or trust accidental cues. The filter is trying to keep only examples where the visual clue genuinely contributes to the answer.

Teaching the agent to move between tools

Once the benchmark data exist, the paper turns to the agent itself. WebWatcher is trained on trajectories, not just answers. Each example is turned into a think-act-observe loop, so the model learns the rhythm of investigation rather than only the final sentence.

The trajectory format is straightforward:

  • think about what is missing,
  • act by calling a tool,
  • observe the returned evidence,
  • update the plan,
  • repeat until the answer is ready.

That is a ReAct-style loop, but the important detail is where the traces come from. The paper uses prompting to synthesize trajectories from actual action-observation behavior instead of hand-writing a generic rationale. That makes the supervision procedural rather than decorative. The model is learning what a useful tool sequence looks like in practice.

The paper also filters the trajectories before training on them:

  • The final answer has to match the ground truth.
  • The intermediate steps have to be logically consistent.
  • The trajectory has to use enough tools to count as real multi-step behavior.

That last filter is especially important. A one-click trace that happens to hit the right answer does not teach deep research behavior. The goal is to train a model that can explore, not a model that can occasionally guess well.

Why cold start comes before reinforcement

WebWatcher uses supervised fine-tuning as a cold start before reinforcement learning. That ordering is not cosmetic. It solves a stability problem.

If you jump directly into reinforcement learning from plain instruction-following data, the policy has not yet learned the tool protocol well enough. Then malformed tool calls, short answers, and format errors dominate the learning signal. The paper’s analysis shows that this leads to a brittle start and weak early reward.

The cold-start stage fixes that by teaching the model to predict the next correct tool action from the image, the question, and the prior tool history. In plain language, SFT teaches the agent the grammar of tool use before RL asks it to become strategic.

After that, the paper switches to Group-Relative Policy Optimization. The idea is easy to understand even without the formula: for one question, the model samples a group of complete trajectories, scores each one, and then updates toward the trajectories that performed better than their peers.

That relative comparison is useful because the reward only appears at the end of the episode. The method does not know which individual step caused success, so it compares whole trajectories instead of pretending it has perfect step-level supervision.

The reward itself mixes two concerns. Correct tool formatting matters, but answer quality matters more. In other words, the paper wants the agent to stay inside the tool protocol without turning format compliance into the real objective. A clipped update plus a penalty for drifting too far from the previous policy keeps training stable while still allowing improvement.

What the policy learns to do differently

The most convincing part of the analysis is not a leaderboard number. It is the behavior of the trained policy.

The tool mix changes with the task. On retrieval-heavy benchmarks, text search takes a bigger share of the work. On visual tasks, image search and OCR become more important. Code appears when the question really needs computation, not as a default reflex.

That is exactly what you want from a deep research agent. It should not have a favorite tool. It should have a reason for every tool call.

Where the method bends

WebWatcher is powerful, but it is not lightweight.

Training and inference both require substantial tool orchestration. That means more moving parts, more latency, and more places where the pipeline can fail. The method also depends on synthetic trajectory generation and strong filtering, so performance is partly a function of how good that data-engineering layer is.

There is also a benchmark-design caveat. BrowseComp-VL is intentionally synthetic and entity-obfuscated, which is good for stress-testing multimodal reasoning, but it also means the benchmark reflects the authors’ design choices. The results are strongest evidence that the system matches this style of hard visual-information seeking, not a guarantee that every visual domain will behave the same way.

The open questions are practical ones:

  • Does the same approach transfer cleanly to other visual domains without retraining?
  • Can the trajectory synthesis pipeline be simplified without losing quality?
  • How sensitive is the system to tool availability and latency?

Those are the right questions, because the method’s strength is also its cost.

What to remember

WebWatcher is easiest to remember as a careful multimodal investigator.

Start with a hard image-question pair. Ground it in real web images. Mask the obvious entity so the model has to reason instead of copy. Filter out weak alignments. Turn the surviving examples into real tool-use trajectories. Teach the agent to imitate those trajectories first, then refine it with reinforcement learning that compares whole rollouts against each other.

The paper’s main lesson is simple: if a question can require both visual evidence and web evidence, then the training data has to teach the model how to move between those evidence sources. WebWatcher is built around that idea from end to end.
