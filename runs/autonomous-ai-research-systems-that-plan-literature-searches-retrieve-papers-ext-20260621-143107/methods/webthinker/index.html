WebThinker: A Deep Research Agent for Autonomous Web Exploration and Report Generation

WebThinker is built around a simple but important claim: deep research works better when reasoning, search, and writing are not separated into rigid stages. The model should notice what it does not know, search the web, click deeper when the first page is not enough, keep the useful evidence alive, and start drafting the report while the investigation is still moving.

That changes the role of retrieval. In a normal RAG setup, retrieval happens before the answer. In WebThinker, retrieval is part of thinking.

That loop is the heart of the method. Everything else in the paper is there to make the loop deeper, more reliable, and easier to train.

What to notice in Figure 2 is where control lives. Standard RAG does one retrieval pass and then hands the context to the model. Iterative RAG repeats retrieval, but search is still mostly a side process. WebThinker pushes search, navigation, and drafting inside the reasoning loop itself. The model is not just receiving evidence. It is deciding when to go looking for more.

Two modes, one research brain

WebThinker uses the same core idea in two different settings.

In Problem-Solving Mode, the model treats the web like an evidence source for answering a hard question. It searches, explores, and returns a direct answer.

In Report Generation Mode, the same reasoning backbone also has to produce a long-form report. That means it must not only gather evidence, but also decide when a section is ready to draft, when the current structure needs checking, and when a revision is worth making.

Figure 3 makes the architecture easier to remember. The main model is the conductor. The Deep Web Explorer is the field scout. The writing tools are the scribe. WebThinker works because these parts do different jobs, but they all stay inside one continuous reasoning process.

The Deep Web Explorer is where shallow search stops

The most important technical addition in the paper is the Deep Web Explorer. This is not just a search box. It is a mini-agent inside the larger agent.

The explorer starts from a search intent or a gap in the current reasoning. It can search the web, inspect the returned pages, click links or buttons, and then decide whether it has enough evidence or needs to go deeper. That recursive behavior matters because many research questions do not end on the first page. The useful detail is often one click away, or one follow-up search away, or buried in a linked document that the first query would never surface.

Think about a deadline question where the first result is a conference page, but the exact date lives in a linked PDF. A shallow retriever might stop at the page summary. The explorer is designed to keep going until the relevant page tree gives up the answer.

The explorer also does not dump everything back into the main model. It returns a concise summary of the evidence that matters for the current gap. That keeps the reasoning thread clean. The main model can then continue thinking with a compact answer to “what did we just learn?” instead of a pile of raw snippets.

That design choice is the reason recursive navigation helps. Search broadens the candidate pool. Clicking deepens the evidence. Summarizing turns the result into something the reasoner can actually use.

Writing becomes part of the investigation

The second major idea is the Autonomous Think-Search-and-Draft strategy. This is the report-writing version of the same loop.

Instead of waiting until the end to write everything at once, WebThinker lets the model draft sections as soon as enough evidence has been gathered for them. The report is treated as a living artifact. Search can change it, and the draft can change what the model searches for next.

That is why the paper adds three writing tools:

  • Drafting writes a new section or fills in missing content.
  • Checking shows the current outline so the model can see the structure without pulling the whole report into live context.
  • Editing revises the existing report when later evidence or better organization makes a change worthwhile.

These tools are handled by an assistant writer model, while the main reasoning model stays focused on orchestration. That split is practical. The main model should spend its attention on what to search next and when a section is ready. The assistant writer can spend its attention on sentence-level editing and section-level composition.

The paper also keeps all explored pages in a document memory. That matters because report writing needs a stable evidence pool. If the model has already visited ten relevant pages, it should not have to rediscover them every time it edits a paragraph. The memory acts like a running notebook of the investigation.

So the real writing loop looks more like this:

  1. Search until a section has enough evidence.
  2. Draft that section.
  3. Check the report structure.
  4. Edit the parts that no longer fit.
  5. Keep searching if the next section still has gaps.

That is a much more realistic picture of research writing than “retrieve first, write once later.”

How the model learns to use the tools

WebThinker does not stop at prompting. It also trains the model to make better tool-using decisions.

The training idea is straightforward: sample several reasoning trajectories for the same task, compare them, and turn the comparisons into preference data. But the ranking rule matters more than it looks.

The paper uses a clear priority order:

  • First, prefer the trajectory that is actually correct, or that produces the better report.
  • If both are correct, prefer the one that uses fewer tool calls.
  • If they are still tied, prefer the shorter reasoning trace when the difference is meaningful.

That ordering is important because it keeps the model from learning the wrong lesson. A fast but wrong trajectory should never beat a correct one. Efficiency matters, but only after correctness. Conciseness matters, but only after efficiency.

Once those preference pairs are built, the model is trained with an online DPO loop. In plain language, the current policy is nudged to make better trajectories more likely and worse trajectories less likely, relative to a reference snapshot. Then the updated policy samples new trajectories, new preference pairs are built from those fresh runs, and the process repeats.

That iterative, on-policy part is what makes the training feel grounded in the real behavior of the agent. The model is not just copying a static dataset of ideal tool traces. It is learning from its own current mistakes and improvements.

Why this is more than “RAG with extra steps”

The paper’s main contribution is not just deeper search. It is the combination of deep search, live drafting, and preference training.

Compared with standard RAG, WebThinker does not treat retrieval as a front-loaded preprocessing step. Compared with iterative RAG, it gives the model recursive navigation and lets writing happen while the evidence is still being gathered. Compared with prompt-only agentic search, it learns which trajectories are worth repeating.

The ablation story matches that design. Removing the Deep Web Explorer, or even just link clicking, hurts complex problem solving. Removing autonomous drafting hurts report quality the most. Those are exactly the failure modes the architecture is trying to fix:

  • Without deep exploration, the model misses evidence that sits behind a link or a second search.
  • Without drafting, the model can gather facts but never turn them into a coherent report.
  • Without online preference learning, the model has less pressure to become efficient and reliable with its tool use.

So the components are not decorative. Each one fixes a different break in the loop.

What the paper leaves open

WebThinker is powerful, but it is not a general solution to every research task.

It is text-first, so it does not handle images or video. Its tool set is still limited, which means the method depends on a fairly specific research environment. GUI-based browsing is not supported yet, so tasks that require richer on-screen interaction are still out of reach.

There is also a systems tradeoff. The more autonomy you give the model, the more orchestration, latency, and tool dependence you have to manage. That is the price of letting search and writing happen inside the reasoning loop.

The broader open question is whether the same pattern transfers cleanly to other domains and tool stacks. The paper is convincing that the pattern works here, but it is not claiming the problem is solved in general.

What to remember

The simplest way to remember WebThinker is as a research loop:

reason, search, click deeper, store the useful evidence, draft a section, check the outline, edit the report, and repeat.

The key move is not any single tool. It is the decision to keep evidence gathering and synthesis interleaved, then train the model to prefer trajectories that are correct first, efficient second, and concise third.

That is why WebThinker feels like a real deep research agent rather than a search wrapper. It does not just look for information. It keeps turning that information into a better answer while the investigation is still in progress.
