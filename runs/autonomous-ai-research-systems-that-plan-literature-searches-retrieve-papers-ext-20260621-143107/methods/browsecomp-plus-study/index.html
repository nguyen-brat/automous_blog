Text Ranking in Deep Research: Retrievers, Re-rankers, and Query Reformulation

This paper is not trying to invent a brand-new retriever. It is doing something more useful for anyone building a deep-research agent: it opens the search stack and asks which classic IR components still matter when search happens inside an iterative reasoning loop.

That question matters because deep-research systems usually hide search behind a web API. The agent writes a query, gets back ranked results, reads a little, reasons a little, and then decides whether to search again. If the search quality is weak, the loop wastes time. If the retrieval unit is too large, the agent runs out of context. If the query format does not match what the retriever expects, even a strong model can look surprisingly bad.

The paper tests three practical levers:

  • whether the agent should retrieve whole documents or shorter passages,
  • whether reranking helps once search is already iterative,
  • whether agent-style web queries should be rewritten into natural-language questions before neural ranking.

The core idea is simple: the paper is about search quality inside the agent, not search quality in the abstract.

Why this benchmark matters

The study uses BrowseComp-Plus, which is valuable because it fixes the corpus and provides human-verified relevance judgments. That turns a normally black-box deep-research setup into a reproducible retrieval problem. The dataset ends up with 830 queries, and it distinguishes gold documents from broader evidence documents, so the paper can measure retrieval behavior directly instead of inferring everything from final answers.

The authors also compare two open-source agents, gpt-oss-20b and GLM-4.7-Flash, so the conclusions are not tied to one model family. That is important because the retrieval story changes with the agent’s context budget and search habits.

The loop the paper studies

The method is easiest to understand as a loop:

Once the system is iterative, the retriever is no longer serving a human who can browse 10 result pages at leisure. It is serving an agent with a limited context window, a search budget, and a need to keep moving.

That is why the paper looks at four evaluation signals instead of only one:

  • search calls, which show how much searching the agent needed,
  • recall, which shows how much evidence it managed to recover,
  • accuracy, which checks the final answer,
  • completion rate, which tells you how often the agent finished before running out of context or output budget.

Those metrics let the paper see both effectiveness and efficiency. A pipeline that improves ranking but causes the agent to wander longer is not obviously better for deep research.

When the paper evaluates passages, it has to bridge a label mismatch: BrowseComp-Plus provides document-level judgments, not passage-level ones. The paper handles this with Max-P, mapping retrieved passages back to their source documents by taking the best passage score for each document. That keeps passage retrieval comparable to document retrieval instead of punishing it for a ground-truth format mismatch.

Passages are the first big win

The clearest result in the paper is also the easiest to remember: in deep research, shorter evidence units usually work better than full documents.

The reason is not mysterious. When the agent sees a whole web page, most of that page is usually irrelevant to the current question. Some of it is useful, but it is buried inside a lot of noise. Passages cut away the noise and let the agent spend its context on the part that actually matters.

To make that comparison fair, the paper builds a passage corpus from the fixed BrowseComp-Plus document collection. Each document is split into passages of at most 250 words, and the title is prepended when available so the passage still carries a little source context. That sounds like a small preprocessing step, but it changes the search problem completely: the retriever no longer has to fight document length as much, and the agent no longer has to read around irrelevant bulk.

What matters here is the contrast behind the image. Full documents make retrieval work harder because long pages are harder to rank cleanly and harder to fit into an agent’s working memory. Passages move the system closer to “find the exact evidence slice” instead of “find the right page and hope the useful paragraph appears soon.”

That is also why the shorter-context agent benefits more from passages. When the context budget is tighter, every turn matters more. Compact evidence buys the agent more search rounds before it gets squeezed out by its own history.

One nice nuance: if the agent is already retrieving passages, adding a full-document reader can become redundant. On truncated documents, the reader helps because it recovers hidden evidence. On passages, the system already has a focused slice, so the reader adds less and can slightly hurt by adding complexity without adding much new signal.

BM25 is still a serious baseline

The paper also has a result that should make any search engineer pause: BM25 is very hard to beat when the agent’s queries look like web search.

That makes sense once you look at the query style. The agent does not usually type polished natural-language questions. It issues short keyword fragments, quoted exact matches, dates, numbers, and other web-search-style hints. That is exactly the kind of input lexical matching likes.

So on the passage corpus, BM25 becomes a surprisingly strong baseline. In fact, the best pre-reranking answer accuracy in the study comes from gpt-oss-20b with BM25 on passages, at 0.572.

The broader retriever comparison is also instructive. The paper covers the main IR families: lexical BM25, learned sparse SPLADE-v3, single-vector dense retrievers such as RepLLaMA and Qwen3-Embed-8B, and multi-vector ColBERTv2. On this benchmark, the smaller learned sparse and multi-vector methods often generalize better than the larger single-vector dense models, which is a reminder that bigger embeddings do not automatically solve query mismatch.

The deeper lesson is not “BM25 wins everywhere.” It is that retrieval quality depends on the match between query style and retriever style. The agent is not asking a human-style question, so a question-style encoder is not automatically the right tool.

The paper also shows why BM25 looked bad on full documents in earlier settings. The issue was not BM25 itself. The issue was length normalization. Long pages were being scored with defaults that were not well matched to this corpus. Once the authors either index only the first 512 tokens or retune BM25 for document retrieval, performance improves sharply.

That is an important reminder. Classic retrievers are not set-and-forget. Their defaults are corpus assumptions in disguise. When the corpus changes, the scoring rules may need to change too.

Reranking is where the big lift comes from

Once the first-stage retriever produces a candidate list, the paper adds reranking and asks a simple question: does a better shortlist actually help the agent?

The answer is yes, consistently.

The rerankers in the paper represent three levels of cost and style:

  • monoT5-3B, a relatively cheap pointwise reranker,
  • RankLLaMA-7B, an LLM-based reranker,
  • Rank1-7B, a reasoning-based reranker that writes a short reasoning trace before making its decision.

All three usually improve recall and final accuracy, and they also tend to reduce search calls. That last point matters. Better evidence ordering helps the agent converge faster because it spends less time wandering through bad search paths.

The strongest pipeline in the paper is BM25 followed by monoT5 at reranking depth 50. On the passage corpus, that setup reaches 0.716 recall and 0.689 accuracy, with fewer search calls than the unreranked baseline. That is the study’s best overall retrieval setup.

Deeper reranking usually helps because the reranker can rescue relevant items that the first-stage retriever placed lower in the list. If the candidate pool is too shallow, the reranker cannot help much. If the candidate pool is deep enough, reranking becomes a real lever.

The practical pattern is easy to remember: a strong first-stage retriever gives the reranker good candidates, and a deeper rerank lets the agent see them sooner. The result is not just a better ranking metric. It is a more efficient research loop.

Why reasoning is not automatically better

The reasoning-based reranker, Rank1, is the most interesting cautionary case.

You might expect a model that reasons before ranking to outperform a simpler scorer. But the paper does not find a clear advantage. The reason is that agent queries are often messy, web-style strings. Rank1 is trained on natural-language question data, so it can over-interpret fragments in the query and decide that unrelated terms belong together.

That failure mode is subtle but important. If the query contains a section number, a topic word, and a year, a reasoning model may try to build a coherent story out of those pieces even when the right interpretation is simply “these are search hints.” In that case, more reasoning does not help. It introduces a new way to be confidently wrong.

This is the paper’s best reminder that smarter is not the same as better matched. A reranker that thinks like a human question-answerer may still misread a web-search query.

Translating the query helps neural rankers

The paper’s answer to that mismatch is query-to-question reformulation, or Q2Q.

The idea is straightforward: take the agent’s raw search query and rewrite it as a natural-language question before sending it to the neural retriever or reranker. There are two versions. One uses only the query itself. The other also uses the agent’s recent reasoning trace, which gives the reformulator more context about what the agent is actually trying to find.

That second version matters a lot. A raw query like a few keywords may be ambiguous on its own. If the reformulator can see the current reasoning, it can preserve the search intent instead of guessing from the surface text alone.

This is where the paper draws a clear line between retriever families. Q2Q helps neural methods such as SPLADE-v3 and Qwen3-Embed-8B, because it bridges the gap between web-style queries and question-style training data. It does not help BM25 in the same way. For BM25, the raw query is often already the better fit.

So Q2Q is not a universal upgrade. It is a bridge for models that expect a different kind of input.

The paper shows this clearly on SPLADE-v3 and Qwen3-Embed-8B: using the query plus reasoning trace improves both recall and accuracy, while the query-only version helps little. In reranking too, giving Rank1 the reformulated query makes it behave more sensibly.

What the paper is really saying

The headline lesson is not that one retriever is permanently best. It is that deep research changes the retrieval contract.

The agent is not asking one clean user question. It is issuing many rough subqueries. Those queries often look like web search. The evidence has to fit into a limited context window. And the agent may need to search again immediately after reading the results.

Once you look at the system that way, the practical defaults become clearer:

  • use passages when context is tight,
  • do not assume neural retrievers will beat BM25 on keyword-heavy queries,
  • rerank deeper when the candidate pool is noisy,
  • translate queries into questions when the retriever expects question-like inputs,
  • and do not count on reasoning alone to fix query mismatch.

Limits and takeaways

The paper is a reproduction study, so its value is in pattern extraction rather than a new algorithm. It shows that several old IR truths still survive inside deep research:

  • passage-level retrieval is often better than whole-document retrieval,
  • BM25 is still very competitive when the query is web-like,
  • reranking helps a lot,
  • and query reformulation can rescue neural rankers from a bad input distribution.

It also shows where the old truths need correction. Document length normalization matters more than people often admit, reasoning-based reranking is not automatically superior, and a full-document reader is only helpful when the agent actually needs it.

There are also clear limits to the evidence. The study uses one deep-research benchmark, only two agents, and tuning without a validation set. So the exact numbers should not be treated as universal constants. The design lesson is sturdier than the scores: match the retrieval unit, query style, and reranking depth to the agent’s context budget.

What to remember

Passages keep evidence compact. BM25 is strong when queries are web-like. Reranking compounds the gain. Q2Q helps neural rankers by translating the query into the style they were trained on.

If you remember one line, make it this: deep research retrieval is about matching the retrieval unit, the query style, and the reranking depth to the way the agent actually searches.
