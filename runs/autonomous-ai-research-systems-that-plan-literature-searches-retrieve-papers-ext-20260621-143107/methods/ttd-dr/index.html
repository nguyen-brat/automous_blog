Test-Time Diffusion Deep Researcher (TTD-DR)

TTD-DR is easiest to understand as a research notebook that keeps editing itself. The system does not wait until the end to write the report. It starts with a rough draft, uses that draft to decide what to search next, brings back evidence, and then rewrites the draft again. That loop is the whole trick: the report becomes both the output and the steering wheel.

What to notice in Figure 1 is the rhythm. Human research is rarely line-by-line composition. People sketch a plan, write a messy first version, notice what is missing, search for more evidence, and then revise. TTD-DR is trying to give a research agent the same habit.

The paper calls this a test-time diffusion process, but the plain-English version is simpler: keep cleaning the draft, and let the cleaned draft decide where the next search should go.

Why the draft comes first

Traditional deep research agents often treat search and writing as separate phases. First they collect information, then they write. That sounds reasonable, but long research tasks usually fail because the search path drifts away from the eventual report structure. The agent may gather a lot of facts and still miss the thread that holds the answer together.

TTD-DR flips that around. The draft is created early, even before the evidence is complete. That rough draft is not meant to be correct yet. It is a scaffold. It preserves the shape of the report so the agent can see where it is heading, which sections are weak, and which claims still need support.

That is why the paper compares the process to diffusion. A diffusion model starts from noise and repeatedly denoises toward a usable sample. TTD-DR starts from a noisy draft and repeatedly revises it toward a trustworthy report. The important difference is that the “denoising” step is not just internal self-editing. It is guided by retrieval, so external evidence can correct the draft at every round.

What matters here is the direction of control. The draft is not just a place to store text. It actively shapes the next search query. Then the new evidence flows back into the draft. That back-and-forth is what keeps the research coherent.

The backbone research agent

The paper breaks the system into three stages so the loop stays understandable.

Stage 1 is the planning stage. The agent takes the user query and turns it into a research plan. Think of this as the outline of the final report. It does not need to contain all the facts yet, but it should know what sections the report needs and what kinds of evidence those sections will require.

Stage 2 is the search loop, and it has two parts. In Stage 2a, the agent generates a new search question from the research plan, the original user question, and the context from earlier search turns. In Stage 2b, it searches the available sources and synthesizes a concise answer from the retrieved documents. This is important: the system does not keep raw search junk around as its main memory. It compresses the retrieved evidence into a structured answer that can actually be reused.

Stage 3 is the final writing stage. Once the search loop ends, the agent writes the complete report from the plan and the sequence of question-answer pairs collected along the way. At that point the report has a stronger backbone than a one-shot answer, because the structure was preserved all the way through.

That structure matters for two reasons. First, it keeps the report aligned with the original topic instead of wandering through loosely related facts. Second, it limits information loss. If every search round only produced raw snippets, the model would have to rediscover the thread each time it revised the report. By storing plans and synthesized answers, TTD-DR keeps the useful context in a form the writer can reuse.

Self-evolution cleans up each step

The other major idea in TTD-DR is that each component in the workflow can improve itself before its output is passed downstream.

The picture to hold in your head is simple: do not trust the first version of a plan, question, answer, or report if you can cheaply produce several and let them critique each other.

The self-evolution loop has four parts.

  • First, the system samples multiple initial variants of a component. For example, it may generate several candidate answers to the same search question using different sampling settings.
  • Next, each variant is judged by an LLM-as-a-judge that assigns a score and writes feedback. The paper uses helpfulness and comprehensiveness as the main criteria for this kind of evaluation.
  • Then the variant is revised using that feedback. The goal is not to rewrite everything from scratch, but to move the draft toward the judged target.
  • Finally, the revised variants are merged into one output, so the main workflow receives the strongest pieces from the whole evolutionary path.

This is a useful design choice because it improves the quality of the intermediate artifacts, not just the final report. A better search question leads to better retrieval. A better answer summary preserves more useful evidence. A better plan gives the final report a clearer shape. The paper’s point is that deep research systems fail at multiple small steps, so the fix should also operate at multiple small steps.

Why retrieval has to stay in the loop

Self-evolution alone would still leave the system with a stale report if the draft never gets updated with fresh evidence. That is why TTD-DR adds report-level denoising with retrieval.

The logic is straightforward. The current draft is fed back into Stage 2a, so the next search question is shaped by what the report already says and what it still lacks. Stage 2b returns a synthesized answer from the retrieved documents. Then the draft is revised with that answer. The revised draft becomes the guide for the next search question, and the loop continues.

In practice, the draft can be improved in two ways. It can gain new details that were missing before, or it can be used to verify and tighten claims that were already there. That second part is easy to miss, but it is important: retrieval is not only for adding new facts. It is also for checking whether the current draft is actually supported.

This is the paper’s most interesting control idea. Search does not happen in a vacuum. It is always looking at the evolving report. That reduces drift because the agent can compare new evidence against the structure it is trying to build instead of chasing every promising query that happens to look relevant.

Put differently, the draft acts like a living outline. It remembers the global shape of the report, while retrieval fills in the missing pieces. That is why the final output is usually more coherent than a pipeline that searches first and writes later.

What the method is really buying you

TTD-DR is not new because it can search. Search is already standard. It is new because it turns the report into a persistent control object.

That gives it three practical advantages.

First, it preserves context across long trajectories. The agent does not have to reconstruct the report structure from scratch after every search round.

Second, it makes the search more targeted. Because the current draft exposes gaps, the next query is more likely to answer what the report actually needs rather than what the top-ranked search result happens to discuss.

Third, it improves the quality of the inputs that each stage receives. Self-evolution upgrades the plan, question, answer, and report artifacts before they are handed downstream, so the main loop is not constantly repairing low-quality intermediate text.

The tradeoff is latency. More revision rounds mean more search and more judge passes. The paper’s own analysis treats that as the central efficiency question: the method buys better coherence and evidence use, but it does so by spending more test-time computation.

What the paper does not try to solve

The scope is intentionally narrow. TTD-DR focuses on search tools, not broader browsing or coding tools. It is a test-time scaling recipe, not an agent-tuning or training framework. That makes the method easier to generalize across systems that already have accessible search, but it also means the approach is not a full answer to every kind of autonomous research task.

There is also an implicit dependency on the quality of the draft and the retrieval access. If the first draft is badly misaligned, the loop can waste time refining the wrong shape. If retrieval is weak, the denoising step has less corrective power. The method helps most when the system can keep adding evidence into a draft that is already pointing in the right direction.

What to remember

TTD-DR is easiest to remember as a loop with a living draft at the center.

Plan the report. Draft early. Use the draft to ask the next question. Retrieve evidence. Revise the draft. Improve each component with self-evolution. Repeat until the report is coherent.

That is the whole idea. Instead of treating research as search first and writing second, TTD-DR treats the report itself as the control signal that keeps the search on track.
