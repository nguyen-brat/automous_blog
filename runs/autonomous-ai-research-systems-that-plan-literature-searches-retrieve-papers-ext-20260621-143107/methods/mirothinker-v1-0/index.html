MiroThinker v1.0: Open-Source Research Agents with Interactive Scaling

MiroThinker starts from a practical complaint: open-source research agents are often asked to do research-grade work, but they are still boxed in by short tool chains, weak memory management, and training that does not really reward long interaction. Bigger models help. Longer context helps. But the paper argues that those two axes are not enough on their own if the agent never learns how to keep searching, checking, revising, and continuing until the evidence is actually good enough.

That is the paper’s main move. It treats interaction depth as a third scaling axis, alongside model size and context length. The goal is not just to make the model smarter in a vacuum. The goal is to make the model better at living inside a research loop.

What to notice in Figure 2 is the overall shape. MiroThinker is not a normal chat model with search bolted on at the end. It is a loop: think, choose a tool, read the observation, keep the useful parts, and think again. The paper is trying to make that loop deep enough to support real research, not just a quick lookup.

A loop that can keep going

The runtime agent is built under a ReAct-style setup, but the important part is not the label. The important part is that reasoning, tool use, and observation stay interleaved. The model does not do all the thinking first and all the searching later. It alternates between the two, because research tasks usually change as new evidence arrives.

To support that loop, the paper gives the agent three kinds of tools.

  • An isolated execution environment for commands and code, so the model can compute, inspect files, or manipulate local state safely.
  • File transfer tools, so evidence can move in and out of the sandbox without forcing everything through text.
  • Information retrieval tools, so the agent can search broadly and then extract only the detail it actually needs.

That last part matters more than it looks. The paper does not treat web retrieval as “return the page and hope the model reads it carefully.” It also uses a targeted scraper that condenses a page down to the task-relevant evidence at call time. In other words, the tool itself helps with compression. That is a good fit for long research runs, because the agent often needs the answer to “what did we learn?” rather than a giant raw page dump.

The practical result is that the agent can move across different kinds of evidence without getting trapped in one modality. It can compute in a sandbox, move files around, search the web, and extract focused snippets. Those are not fancy features by themselves, but they cover the main ways a research task can branch.

Context is not a diary

The other important design decision is context management. If the agent keeps every tool response forever, the prompt fills up with stale evidence, repeated snippets, and dead ends. If it throws everything away, the search trail collapses and the model loses the thread.

MiroThinker takes a middle path. It preserves the full sequence of thoughts and actions, but it keeps only the most recent tool outputs in active context. Earlier observations can be dropped once they are no longer needed. That is a very pragmatic idea: the agent usually does not need the full raw history of every page it ever saw. It needs the trail of decisions plus the freshest evidence.

Think of it like a research desk. The current notes stay on the desk. The older receipts go into the filing cabinet. You do not throw them away, but you also do not let them crowd out the next round of work.

This simple retention policy is important because the model is expected to handle very long interaction chains. The paper talks about up to 600 tool calls inside a 256K context window. That only makes sense if the agent can prune the dead weight while keeping the useful shape of the investigation.

The paper also truncates overly long tool outputs and marks them as shortened. That sounds minor, but it is exactly the kind of guardrail long research agents need. A command that dumps thousands of lines is usually not evidence; it is context pollution. Truncation keeps the loop moving.

The key idea here is that memory should support reasoning, not compete with it. MiroThinker is not trying to remember everything. It is trying to remember the right parts long enough to finish the task.

The training data is synthetic on purpose

What to notice in Figure 3 is that the training data is not just scraped and dumped into the model. The paper builds two separate kinds of synthetic data: multi-document question answering and agentic trajectories. One teaches factual grounding. The other teaches behavior.

Turning linked documents into multi-hop questions

For the multi-document QA side, the paper starts with a corpus of interlinked documents, then follows those links to build a connected document graph. The point is not to collect a random pile of pages. The point is to create a small network of documents that naturally require cross-document reasoning.

Once the document graph is built, the documents are consolidated into a cleaner markdown form, links outside the selected subgraph are pruned, and key facts are extracted from each document. Those facts are then turned into indirect constraints instead of literal lookups. A date might be generalized into a broader time period. A place might be recast as a type of location. Other entities may be described through related properties rather than direct naming.

That obfuscation step is doing real work. It stops the model from solving every question by string matching a single surface form. It forces the agent to combine clues across documents, which is much closer to how real research questions behave.

The final step is question generation. A language model is asked to combine multiple obfuscated clues into questions that require multi-hop reasoning. That is the bridge from document graph to training example. The end product is a question that cannot be answered from one isolated page, but also is not so vague that it becomes unlearnable.

Turning trajectories into research behavior

The second data stream is agentic trajectory synthesis. This is where the paper teaches the model how to act, not just what to know.

The authors generate traces using both single-agent ReAct-style behavior and multi-agent orchestration. They also use two styles of tool invocation: a structured function-calling interface and a more flexible context-based protocol. The point of mixing these styles is to expose the model to different interaction rhythms. A model that only sees one way of calling tools can overfit to one ritual. A model that sees several interaction patterns has a better chance of generalizing.

The paper also uses multiple strong language models to synthesize trajectories. That helps diversify the style of the traces and reduces the chance that the training set becomes too dependent on one model’s habits.

On top of that, the authors add open-source QA and post-training corpora so the agent does not only learn from synthetic traces. That broadens the coverage and keeps the model from becoming too specialized in one narrow research style.

The overall lesson is simple: MiroThinker is not trained on a single “ideal” agent trace. It is trained on a deliberately varied diet of facts, trajectories, and interaction patterns.

Three stages turn imitation into exploration

The training recipe is also easy to remember if you think of it as a ladder.

Each rung teaches a different part of the behavior.

Supervised fine-tuning teaches the basic loop

The first stage is supervised fine-tuning. Here the model learns to imitate expert trajectories that already contain thoughts, tool calls, and observations. This is where the agent learns the basic rhythm of the task: think, act, observe, continue.

The paper is careful about data quality here because raw synthesized trajectories are noisy. They can contain repeated text, duplicated responses, invalid tool calls, or malformed arguments. If those errors are left in place, the model learns bad habits very quickly. So the training data is filtered and repaired before it is used.

That filtering step is less glamorous than the rest of the paper, but it matters a lot. Supervised imitation only works if the examples are worth imitating.

Preference optimization teaches the model which traces are worth copying

The second stage is preference optimization with DPO. Instead of asking the model to imitate every trace equally, the paper compares a preferred trajectory with a rejected one.

The interesting design choice is how those preferences are decided. The paper does not try to enforce a rigid planning length or a fixed agent script. It prefers the trajectory that actually reaches the correct final answer, and it avoids hardcoding a single step count as the definition of quality. That matters because some research tasks are short and surgical, while others really do need a longer chain of checks.

The preferred and rejected traces also have to be clean enough to be meaningful. The chosen trace should have a coherent reasoning path and a clear correct answer. The rejected trace should still be a valid attempt, not just a broken artifact. That keeps the preference signal focused on actual decision quality rather than on superficial formatting noise.

The paper also keeps a supervised loss on the preferred trajectories during this stage. The point is to keep the model stable while the preference signal sharpens its behavior. In plain language: DPO says “choose better trajectories,” while the auxiliary supervised signal says “do not drift too far from the good behavior you already learned.”

Reinforcement learning teaches the model to explore deeper

The final stage is reinforcement learning with a GRPO-style update. This is where the agent stops copying traces and starts discovering better ones through interaction.

The environment is built to support many concurrent rollouts and several kinds of work: web search, scraping and summarization, code execution, and local system interaction. That matters because research tasks are long-tailed. Some finish quickly, but many do not. The system therefore streams tasks through workers, and unfinished tasks go back into the queue for another round. That keeps the rollout machinery busy instead of waiting for the slowest episode to finish.

The reward is intentionally simple: correctness is the main signal, and format failures are penalized. That keeps the agent from learning that a nicely shaped but wrong answer is good enough. The paper also filters out pathological trajectories, such as correct runs that only succeeded because of repeated tool failures or degenerate retry loops, as well as incorrect runs that failed for trivial format reasons or got stuck in action repetition.

That curation is important. In reinforcement learning, garbage in does not just mean low quality. It can teach the model to chase the wrong kind of effort.

What GRPO adds is a relative signal. The model samples several trajectories for the same prompt, then learns which ones beat the group average. That is a good fit for research agents, because the interesting question is often not “was this one sample perfect?” but “which behavior is consistently better than the rest of my current policy?”

What interactive scaling means in practice

This is where the paper’s main claim becomes visible. Once the model is trained with these stages, it does not just answer more often. It tends to stay in the loop longer, check more evidence, and use more tool calls before it commits to a final answer.

The paper reports that the RL-tuned model develops deeper interaction trajectories than the supervised model, and that this shift comes with roughly 8 to 10 points of average improvement across the evaluated tasks. The 72B version also reaches 81.9% on GAIA, 47.1% on BrowseComp, and 55.6% on BrowseComp-ZH, which is enough to show that the interaction story is not just a theory about better prompting. It is a measurable scaling effect.

That is the real meaning of interactive scaling. More interaction is not automatically better, but when the model is trained to use the interaction well, depth becomes a capability in its own right. The agent does not only need more parameters or more context. It needs more practice operating across a longer search-and-check-and-revise loop.

The release also includes 8B and 30B variants, which is useful because it shows the idea is not locked to one giant model. The same training recipe carries the scaling story across sizes.

The rough edges are still visible

The paper is fairly honest about what still hurts.

  • Some tool calls are redundant or only marginally useful.
  • Reinforcement learning can make the reasoning chain too long and repetitive.
  • Non-English inputs can trigger language mixing in the reasoning or output.
  • Sandbox use is still brittle, especially when the model should choose between code execution, file handling, and direct web retrieval.

That list is worth reading carefully because it shows the tradeoff behind interactive scaling. Deeper interaction improves capability, but it also exposes inefficiency. The agent can search more deeply, but it can also waste turns. It can reason longer, but it can also become verbose. It can do more in the sandbox, but it can also misuse the sandbox.

So the paper is not claiming that long interaction solves everything. It is claiming that long interaction is a real scaling dimension, and that the model can be trained to use it well enough to matter.

What to remember

If you want one memory hook, keep this:

  • MiroThinker is a research loop, not just a chat model with tools.
  • The loop is kept alive by selective memory, not by retaining every raw observation.
  • The training data teaches both the facts and the behavior of research.
  • SFT gives the agent the basic habit, DPO teaches it which trajectories are worth preferring, and RL pushes it to explore deeper.

That is why the paper matters. It argues that research capability does not only scale with model size and context length. It also scales with how well the agent can stay inside a useful interaction loop with the environment.
