Tongyi DeepResearch: An Open-Source Deep Research Agent

Tongyi DeepResearch is easier to understand if you think of it as a training system for a research agent, not just a chat model with tools attached. The paper’s main move is simple to say but hard to build: teach the model to behave like a careful researcher, give it stable environments where tool use does not collapse into noise, and then tighten that behavior with synthetic data, supervised cold start, and strict on-policy reinforcement learning.

This is the whole story in miniature. The model starts from Qwen3-30B-A3B-Base, learns research priors from long-context behavioral data, gets a supervised nudge from curated demonstrations, and then improves through real rollout-and-reward loops. The paper reports 32.9 on Humanity’s Last Exam, 43.4 on BrowseComp, and 90.6 on FRAMES while activating only 3.3 billion parameters per token out of 30.5 billion total.

What to notice in the pipeline figure is that the authors do not treat training as one blob. They split it into a stage that builds agentic instincts and a stage that sharpens those instincts with synthetic data and reward. That split is the paper’s core systems choice.

The agent does not keep the whole history

Tongyi DeepResearch uses the familiar ReAct pattern, which means the model alternates between thinking and acting. A thought leads to a tool call, the tool returns an observation, and that new observation shapes the next thought.

The important twist is context management. Instead of feeding the model the entire raw history forever, the paper keeps a compact running report, the current question, and the latest tool interaction. That makes the agent behave more like a researcher who keeps a working notebook than a chatbot that rereads every old message.

Why is that useful? Because long research tasks usually fail in one of two ways. Either the context overflows, or the model drifts away from the original question. A compressed running report solves both problems at once. It keeps the useful evidence alive, forces the model to restate what matters, and discards the noise that would otherwise bury the next decision.

This also changes the meaning of memory. The agent does not need every raw trace. It needs a faithful summary of what it has learned, what it still does not know, and what action makes sense next.

The environment is part of the method

The paper is very explicit that training a research agent is not only about the policy. The environment matters just as much. In fact, a lot of the paper reads like an argument that tool instability, not just bad reasoning, is what makes deep research hard to scale.

The authors separate the training world into three layers:

  • Prior world environment. This gives the agent task elements, tools, and state definitions without real feedback. It is perfectly stable and free, but it cannot tell the model whether its guess was actually right.
  • Simulated environment. This is a local replica of the web, built from a Wikipedia snapshot and local retrieval tools. It is cheap, fast, and reproducible, which makes it ideal for iteration and ablation.
  • Real-world environment. This is the actual web and tool stack. It is the most faithful, but also the noisiest and most expensive.

That split is more than a convenience. It is how the training curriculum is staged. Mid-training mostly leans on the stable worlds so the model can learn agentic behavior at scale. Post-training validates the ideas in simulation first, then moves the stronger policy into the real environment.

The real tool stack is also wrapped in a sandbox so failures do not get mixed up with policy mistakes. Search, Visit, Python, Scholar, and File Parser all sit behind a unified layer with retries, caching, rate limits, and fallback paths. That matters because if a search API times out, you do not want the model to learn the wrong lesson from a broken environment.

Mid-training builds research priors before RL ever starts

The first training stage is agentic continual pre-training. In plain English, the model sees lots of long, research-like trajectories before it is asked to optimize for final answers.

That data is not just random tool traces. The synthesis pipeline tries to cover the whole research cycle:

  • Question synthesis teaches the model what a hard research problem looks like.
  • Planning actions teach it how to decide the first move.
  • Reasoning actions teach it how to connect evidence across tool outputs.
  • Decision-making actions teach it how to pick between competing paths.

The paper also generates questions from an entity-anchored, open-world memory built from web-crawled data and past trajectories. That lets it produce different styles of hard problems, including multi-hop reasoning and numerical tasks, instead of overfitting to one narrow prompt shape.

The point of this stage is not to become perfect. The point is to develop agentic priors, meaning the model starts post-training already expecting that it should plan, search, summarize, and keep going across many turns. The context window grows from 32K to 128K during this phase so the model can practice longer traces without losing the thread.

This stage also scales general function calling by automatically constructing many simulated environments. That broadens the training distribution so the model learns the shape of tool use, not just one website or one schema.

Post-training turns those priors into a usable research policy

Once the model has those priors, the post-training stack tightens the behavior.

The synthetic data is doing a lot of the heavy lifting

The paper leans hard on fully automated data synthesis. It starts by building a connected knowledge graph from web search and real-world tables, samples subgraphs and subtables, and then deliberately increases uncertainty so the question becomes harder. In other words, it does not just collect examples. It manufactures difficult examples on purpose.

That is important because research agents need training data that looks more like real research than like ordinary instruction tuning. A shallow question is easy to annotate, but it does not teach long-horizon evidence gathering. A hard synthetic question can be checked more carefully and can force the model to reason across sources.

The authors are aiming for a flywheel. Good synthetic trajectories train a better model. A better model can then synthesize even better trajectories for the next round.

Cold-start SFT teaches the model the house style

Before reinforcement learning, the model gets supervised fine-tuning from curated demonstrations. This is the cold-start step.

The idea is straightforward. RL works better if the model already knows the rough shape of a good answer. So the paper first teaches it to imitate high-quality trajectories, including both ordinary ReAct mode and the compressed context-management mode. That gives the policy a stable starting point before it is thrown into reward-driven exploration.

The mixed training setup matters here. ReAct samples teach direct thought-and-action traces. Context-management samples teach the model how to summarize, carry state forward, and keep the investigation focused over longer stretches. The second mode is especially important because it trains the model to turn noisy tool output into a compact working memory.

RL then pushes the policy to improve by trying full rollouts

What to notice in the RL figure is the simple loop: try a full research attempt, score the final answer, and update the policy from that result. The paper is not trying to invent a flashy new RL algorithm. It is trying to make agentic RL stable enough to matter.

A few design choices make that work:

  • The reward is binary correctness, not a fancy format score. The SFT stage already teaches the model how to talk.
  • Training stays strictly on-policy so the reward is always tied to the current model, not a stale snapshot.
  • The curriculum is filtered so the model trains on tasks that are neither trivial nor impossible. Tasks that are always solved or always failed are removed because they do not teach much.
  • The active training set is refreshed in the background as the model gets stronger, so the policy always sees problems that are still in its learning zone.

That last point is a quiet but important one. The paper is basically saying that RL quality depends more on the difficulty of the data and the stability of the environment than on the exact optimizer trick. If the tasks are too easy, the agent stops learning. If they are too hard, it collapses. The data curation loop keeps the agent in the sweet spot.

Model merging is the final balancing act

The last stage is model merging. Instead of choosing one checkpoint and discarding the others, the authors combine variants that came from the same base model but developed different strengths.

That is a pragmatic move. One variant may be better at planning, another at synthesis, and another at long-context tool use. If the variants are compatible, averaging or interpolating them can produce a final model that keeps the useful behavior from each one without another expensive training run.

In other words, the paper does not treat the last checkpoint as the only candidate for deployment. It treats the model family as a set of complementary skills that can be blended back together.

Heavy mode is test-time scaling by parallel research

The test-time trick in the paper is called Heavy Mode, and it is easy to understand if you think in terms of multiple researchers working the same problem in parallel.

Each agent starts from the same question, but explores a different path. One might search one angle, another might inspect a different source cluster, and a third might follow a different reasoning route. Each agent produces a compressed final report, not a full raw trace. Then a synthesis model reads those reports and writes the final answer.

The compressed report is the clever part. Without it, you could not fit several full research trajectories into the context window. With it, the system can compare multiple lines of attack without drowning in raw history.

This is also a different scaling axis from ordinary language modeling. The paper is not just asking for more output tokens. It is asking for more environment interactions. That is a better fit for deep research, because the bottleneck is often evidence gathering, not word count.

What the paper gets right, and what it still cannot solve

The main strength of Tongyi DeepResearch is that it treats the hard parts of agentic research as one coupled system: memory, data, environment, and RL. That is a more realistic view than pretending the model can be fixed with a better prompt.

The tradeoff is that the stack is complex. It only works well when the environment is controlled, the synthetic data is good, and the tool layer is stable. That makes the method powerful, but also more brittle to port to a new tool ecosystem.

The paper is also candid about the limits:

  • 128K context is still not enough for the hardest long-horizon tasks.
  • A larger-scale model is still pending.
  • Broader tool-use robustness is still future work.
  • Heavy Mode is more expensive because it runs several agents and then synthesizes them. That cost is the price of parallel exploration.

What to remember

The easiest way to remember Tongyi DeepResearch is as a researcher with a notebook that never stops being edited.

First it learns what research behavior looks like. Then it learns to compress its own progress so it does not lose the thread. Then it gets rewarded for finishing full tasks correctly. Finally, it can run several research attempts in parallel and merge the best parts of each one.

That is why the paper matters. It is not just an open-source deep research model. It is a full training recipe for making long-horizon tool use more stable, more scalable, and more reproducible.

If you want the shortest possible summary, it is this: build agentic priors first, then reinforce them in stable environments, and only then scale test-time search with compressed multi-trajectory synthesis.
